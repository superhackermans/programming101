第19章 計算量分析（Complexity Analysis）— 重要なものを測る

重要な概念

コードを実行する前に、そのパフォーマンスを予測できる必要があります。AIが1000万件のレコードに対してネストしたループを生成したとき、それが何時間もかかると分かりますか？同僚がリストをループの中でスキャンする関数を書いたとき、スケールが非常に悪いと気づけますか？Big-O記法（Big-O notation）はそのための語彙を与えてくれます。エンジニアがパフォーマンスについて考える際に使う言語であり、すべてのデータ構造とアルゴリズムの議論の基礎となるものです。

Big-O記法とは何か

Big-O記法は、入力サイズnが大きくなるにつれて、関数が実行する操作数がどのように増加するかを表します。答える問いはひとつ：「入力を2倍にしたら、処理時間はどれだけ増えるか？」

私たちは秒数ではなく操作数を数えます。なぜなら秒数はCPU、OS、他に実行中のプロセスなどに依存するからです。操作数は決定論的です。入力サイズがnのとき関数がn回比較を行うなら、入力サイズが2nのときは常に2n回の比較を行います——ハードウェアに関係なく。

Big-Oには2つの簡略化ルールがあります：

  • 定数を落とす。 O(3n)はO(n)になります。O(100)はO(1)になります。定数は成長の形状を変えません。
  • 低次の項を落とす。 O(n² + n)はO(n²)になります。nが大きいとき、n²の項がn項を完全に支配するため、n項は無関係です。

具体的な比較でイメージを掴みましょう。タスクのリストがあり、特定のタイトルが存在するか確認したいとします：

# O(n) — scan the list one element at a time
def find_in_list(tasks, title):
    for task in tasks:
        if task["title"] == title:
            return True
    return False

# O(1) — look up in a dict (hash table) directly
task_index = {"Buy groceries": 1, "Write tests": 2, "Deploy API": 3}
def find_in_dict(task_index, title):
    return title in task_index

タスクが10件なら両方とも瞬時に終わります。タスクが1000万件になると、リストのスキャンは数秒かかる可能性がありますが、辞書の検索は依然としてマイクロ秒で終わります。Big-Oはベンチマークを一度も実行する前に、このことを教えてくれました。

主な計算量クラス

常に出会う計算量クラスが6つあります。それぞれ独自の成長形状と、すでに知っている現実世界の例があります。

主なBig-O計算量クラスとその成長特性
クラス名称例入力を2倍にすると → 時間への影響
O(1)定数（Constant）辞書の検索、配列のインデックスアクセス変化なし
O(log n)対数（Logarithmic）ソート済みリストの二分探索+1ステップ
O(n)線形（Linear）リストのスキャン、要素の合計2倍の時間
O(n log n)線形対数（Linearithmic）効率的なソート（マージソート、Timsort）約2倍の時間

# O(1) — Constant: dictionary lookup
def get_task(task_dict, task_id):
    return task_dict.get(task_id)  # one operation regardless of dict size

# O(log n) — Logarithmic: binary search
def binary_search(sorted_list, target):
    lo, hi = 0, len(sorted_list) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if sorted_list[mid] == target:
            return mid
        elif sorted_list[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1  # halves the search space each step

# O(n) — Linear: scan every element
def find_max(numbers):
    best = numbers[0]
    for num in numbers:
        if num > best:
            best = num
    return best  # touches each element exactly once

# O(n log n) — Linearithmic: Python's built-in sort
def sort_tasks(tasks):
    return sorted(tasks, key=lambda t: t["priority"])
    # Timsort: divides and conquers, n log n comparisons

# O(n^2) — Quadratic: check all pairs for duplicates
def has_duplicate_naive(items):
    for i in range(len(items)):
        for j in range(i + 1, len(items)):
            if items[i] == items[j]:
                return True
    return False  # compares every pair: n*(n-1)/2 operations

# O(2^n) — Exponential: generate all subsets
def all_subsets(items):
    if not items:
        return [[]]
    first = items[0]
    rest_subsets = all_subsets(items[1:])
    return rest_subsets + [[first] + s for s in rest_subsets]
    # each element doubles the number of subsets

自分のコードを分析する

ルールを暗記する必要はありません。2つのパターンを認識できれば十分です：

ルール1：順次ブロック → 加算（最大を取る）。 関数がまずリストをスキャンしO(n)、次にソートしO(n log n)なら、合計はO(n) + O(n log n) = O(n log n)です。支配的な項を残します。

def process_tasks(tasks):
    # Step 1: filter — O(n)
    active = [t for t in tasks if t["status"] == "active"]
    # Step 2: sort — O(n log n)
    active.sort(key=lambda t: t["priority"])
    # Step 3: return first 10 — O(1)
    return active[:10]
    # Total: O(n) + O(n log n) + O(1) = O(n log n)

ルール2：ネストしたループ → 掛け算。 ループの中にループがあると、内側のループが外側のループの各反復に対して1回実行されます。両方がn個の要素を反復すると、O(n) × O(n) = O(n²)になります。

def find_common_tasks(list_a, list_b):
    common = []
    for a in list_a:          # O(n)
        for b in list_b:      #   O(m) for each a
            if a["title"] == b["title"]:
                common.append(a)
    return common
    # Total: O(n * m). If both lists are size n, this is O(n^2)

より微妙な例を見てみましょう：

def group_by_status(tasks):
    groups = {}
    for task in tasks:           # O(n)
        status = task["status"]
        if status not in groups:
            groups[status] = []  # O(1) dict operations
        groups[status].append(task)  # O(1) amortized
    return groups
    # Total: O(n) — one pass, all dict operations are O(1)

そして、計算量が隠れているトリッキーな例：

def remove_duplicates_naive(items):
    unique = []
    for item in items:           # O(n)
        if item not in unique:   # O(k) where k = len(unique)
            unique.append(item)
    return unique
    # Worst case: all unique, so k grows to n
    # Total: O(n^2) — the "in" check on a list is O(n)!

def remove_duplicates_fast(items):
    seen = set()
    unique = []
    for item in items:           # O(n)
        if item not in seen:     # O(1) set lookup
            seen.add(item)       # O(1)
            unique.append(item)
    return unique
    # Total: O(n) — set lookup is O(1)

この2つの関数の違いは、O(n²)とO(n)の違いです。10,000個のアイテムでは、1億回対1万回の操作になります。2番目の関数は「少し速い」だけでなく、1万倍速いのです。

空間計算量（Space Complexity）

時間計算量は操作数を測定します。空間計算量（Space Complexity）は、アルゴリズムが入力そのもの以外に使用する追加メモリの量を測定します。

# O(1) space — modifies in place
def reverse_in_place(items):
    left, right = 0, len(items) - 1
    while left < right:
        items[left], items[right] = items[right], items[left]
        left += 1
        right -= 1
    # Uses only two integer variables — constant extra space

# O(n) space — creates a new list
def reverse_copy(items):
    return items[::-1]
    # Creates an entirely new list of the same size

空間計算量はデータが大きい場合に重要です。10要素のリストを逆にする？空間は関係ありません。1000万要素のリストを逆にする？コピーを作るとメモリ使用量が2倍になります。インプレイスアルゴリズムは可読性をメモリ効率と引き換えにします。

均しコスト分析（Amortized Analysis）

Pythonのlist.append()はO(1)と言われていますが、時々、内部配列がスペース不足になり、リサイズが必要になります——全要素を新しい大きな配列にコピーします。その単一のリサイズはO(n)です。では、appendはどうしてO(1)なのでしょう？

均しコスト分析（Amortized Analysis）は、一連の操作にわたってコストを平均します。動的配列が容量を2倍にする（たとえば8スロットから16スロットへ）と、次の8回のappendは「無料」——リサイズ不要です。高コストのO(n)コピーは、その後に続く安価なO(1)のappendに分散されます。平均すると、各appendはO(1)のコストです。

家賃の支払いのようなものです。月に一度大きな金額を支払いますが、30日間で均すと固定の日当たりコストになります。1日が高くても、平均費用は高くなりません。

# Demonstrating dynamic array growth
import sys

items = []
prev_size = 0
for i in range(65):
    items.append(i)
    curr_size = sys.getsizeof(items)
    if curr_size != prev_size:
        print(f"Length {len(items):3d} -> Memory {curr_size:5d} bytes")
        prev_size = curr_size
# You'll see memory jumps at irregular intervals as the array resizes

Big-Oが「嘘をつく」とき

Big-Oは成長率を教えますが、実際の速度は教えません。小さな入力では、定数因子が支配します。覚えておくべき重要な例外がいくつかあります：

  • 小さなn： 約50要素未満では、O(n²)の挿入ソートがO(n log n)のマージソートに勝つことがよくあります。なぜなら挿入ソートのオーバーヘッドが遥かに小さいからです（関数呼び出しが少なく、キャッシュの使用効率が良い）。PythonのTimsortは実際、マージソートの中で小さな部分列に挿入ソートを使用しています。
  • キャッシュ局所性（Cache locality）： 配列は要素をメモリに連続して格納します。CPUが1要素をロードするとき、近くの要素も無料でキャッシュにロードします。連結リストはノードをメモリ全体に散らばらせ、キャッシュミス（cache miss）を引き起こします——各アクセスがメインメモリを待つ可能性があります。キャッシュ効果により、O(n)の配列スキャンはO(n)の連結リストスキャンより10〜100倍速いことがあります。
  • 定数因子： 定数因子が1000のO(n)アルゴリズムは、定数因子が1のO(n log n)アルゴリズムより、n ≈ 10,000未満では遅くなります。Big-Oはこれらの定数を意図的に隠しています。

教訓：Big-Oは思考ツールであり、最終的な答えではありません。大規模な意思決定（数百万件のレコードに対してO(n)対O(n²)）では理論を信頼してください。マイクロ最適化（2つのO(n)アルゴリズムの選択）では、プロファイラで測定してください。

補足

現実の世界では、timeitやcProfileによるプロファイリングが理論的な分析と矛盾することがよくあります。辞書内包表記とforループは同じBig-Oですが、内包表記はバイトコード命令が少ないため20%速いことがあります。理論を信頼すべき時（大きなデータに対してO(n)かO(n²)かを選ぶとき）と、測定すべき時（異なる定数を持つ2つのO(n)アプローチを選ぶとき）を知ってください。シニアエンジニアは両方を行います：設計判断にはBig-O、最適化にはプロファイリング。

ヒント

この関数は線形に見えますが、実際はO(n²)です。なぜか分かりますか？

def build_string(n):
    result = ""

第20章 線形構造（Linear Structures）— 配列、連結リスト、スタックとキュー

重要な概念

すべてのプログラムはデータのシーケンスを格納します。それらがどのように格納されるか——連続（contiguous）対散在（scattered）、インデックス参照（indexed）対連結（linked）——を理解することで、コードがミリ秒で実行されるか数分かかるかが決まります。この章では4つの基本的な線形構造を取り上げ、それぞれをいつ使うべきかを学びます。

メモリ上の配列（Array）

配列（Array）は要素を連続した（隣り合った）メモリ位置に格納します。要素0がメモリアドレス1000から始まり、各要素が8バイト取るなら、要素5はアドレス1000 + (5 × 8) = 1040に存在します。このアドレス演算が配列のインデックスアクセスがO(1)である理由です——CPUはスキャンなしに任意の要素に直接ジャンプできます。

Pythonのlistは動的配列（Dynamic Array）です。内部では、Pythonオブジェクトへのポインタ（参照）の配列を格納しています。配列自体は連続していますが、指しているオブジェクトはメモリ上に散在している場合があります。

連続した格納は配列に大きな利点をもたらします：キャッシュ局所性（Cache locality）。現代のCPUは1バイトずつフェッチするのではなく、メモリからキャッシュライン（Cache line）（通常64バイト）全体をロードします。配列の要素0にアクセスすると、CPUは要素0〜7（またはそれ以上）を無料でキャッシュにロードします。次の要素がほぼ常にすでにキャッシュにあるため、配列の反復は高速です。

動的配列（Dynamic Array）

静的配列は作成時にサイズが固定されています。動的配列（Pythonのlistなど）は自動的に成長します。戦略は等比成長（Geometric Growth）です：配列がいっぱいになると、約2倍のサイズの新しい配列を割り当て、すべてをコピーし、古い配列を解放します。

これによりappend()は均しコストでO(1)になります（第19章で分析した通り）。しかし、配列の先頭での操作は高コストです：

Pythonのリスト操作の時間計算量
操作時間理由
list[i]（インデックスアクセス）O(1)アドレス演算
list.append(x)O(1) 均しコスト末尾に追加；時々リサイズ
list.insert(0, x)O(n)すべての要素が1つ右にシフト
list.pop()（最後の要素）O(1)末尾から削除
list.pop(0)（最初の要素）O(n)すべての要素が1つ左にシフト
x in listO(n)リスト全体をスキャンする必要がある

重要な洞察：Pythonリストの先頭への挿入または削除はO(n)であり、O(1)ではありません。挿入点以降のすべての要素がシフトする必要があります。これはPythonコードで最もよくあるパフォーマンスの罠のひとつです。

連結リスト（Linked List）

連結リスト（Linked List）は要素を個別のノード（Node）オブジェクトとして格納します。各ノードは値と次のノードへの参照（ポインタ）を含みます。ノードはメモリのどこにでも散在できます——物理的な位置ではなくポインタによって接続されています。

class Node:
    def __init__(self, value, next_node=None):
        self.value = value
        self.next = next_node

class LinkedList:
    def __init__(self):
        self.head = None
        self.size = 0

    def prepend(self, value):
        """Add to the front — O(1)."""
        self.head = Node(value, self.head)
        self.size += 1

    def get(self, index):
        """Access by index — O(n). Must walk the chain."""
        current = self.head
        for _ in range(index):
            if current is None:
                raise IndexError("Index out of range")
            current = current.next
        return current.value

    def __len__(self):
        return self.size

    def __iter__(self):
        current = self.head
        while current:
            yield current.value
            current = current.next

単方向連結リスト（Singly Linked List）では、各ノードは次のノードを指します。双方向連結リスト（Doubly Linked List）では、各ノードが前のノードも指し、両方向への走査とO(1)での両端からの削除（ノードへの参照があれば）が可能です。

連結リストが勝つとき（そして勝たないとき）

教科書は連結リストの先頭でのO(1)の挿入・削除を好みます。現実はより微妙です。Pythonのlist（動的配列）はほぼすべての実用的な場面で連結リストに勝ります。なぜなら：

  • キャッシュミスがパフォーマンスを破壊する。 配列の要素は連続しており、CPUがプリフェッチします。連結リストのノードは散在しており、各ノードアクセスは潜在的なキャッシュミスです。
  • メモリオーバーヘッド。 各連結リストのノードはヘッダ、値、ポインタを持つPythonオブジェクトです。100万の整数のリストは配列で約8MBですが、連結リストでは56MBです。
  • Pythonのオーバーヘッド。 数百万のNodeオブジェクトを作成すると、PythonのガベージコレクタGCに圧力がかかります。

連結リストが輝く特定のシナリオが2つあります：(1) インデックスアクセスなしに両端でO(1)の挿入・削除が必要な場合（固定サイズブロックの双方向連結リストであるcollections.dequeを使用）、(2) データをコピーせずにサブリストを結合したりノードを並べ替えたりする必要がある場合（Pythonでは稀ですが、システムプログラミングでは一般的）。

スタック（Stack）：LIFO

スタック（Stack）は後入れ先出し（Last-In, First-Out：LIFO）の構造です。アイテムを上にプッシュ（push）し、上からポップ（pop）します。皿の重ね積みを想像してください：上に追加し、上から取ります。

# Python list is a perfect stack
stack = []
stack.append("open file")    # push
stack.append("edit line 5")  # push
stack.append("delete word")  # push

last_action = stack.pop()    # pop -> "delete word"
print(last_action)           # most recent action

スタックのユースケース：

  • アンドゥシステム（Undo）—各アクションをプッシュし、アンドゥするためにポップします。これはまさにTaskForgeで構築するものです。
  • コールスタック（Call Stack）—Pythonが関数を呼び出すとき、コールスタックにフレームをプッシュします。関数が戻るとき、フレームをポップします。これが無限再帰が「スタックオーバーフロー（stack overflow）」を引き起こす理由です。
  • 括弧の対応チェック—開き括弧をプッシュし、閉じ括弧が来たらポップします。ポップ時にスタックが空、または最後にスタックが空でなければ、括弧は対応していません。
  • 式の評価—電卓は3 + 4 * 2を正しい優先順位で評価するためにスタックを使います。
def is_balanced(expression):
    """Check if parentheses are balanced — classic stack problem."""
    stack = []
    pairs = {"(": ")", "[": "]", "{": "}"}
    for char in expression:
        if char in pairs:
            stack.append(char)
        elif char in pairs.values():
            if not stack:
                return False
            if pairs[stack.pop()] != char:
                return False
    return len(stack) == 0

print(is_balanced("({[]})"))   # True
print(is_balanced("({[})"))    # False
print(is_balanced("((()"))     # False

キュー（Queue）とデック（Deque）：FIFO

キュー（Queue）は先入れ先出し（First-In, First-Out：FIFO）の構造です。アイテムは後ろから入り、前から出ます。切符売り場の列を想像してください。

PythonのリストをキューとしてPythonのlistを使いたくなるかもしれませんが、重大なパフォーマンス問題があります：

# BAD: Using list as a queue — pop(0) is O(n)!
queue = [1, 2, 3, 4, 5]
queue.append(6)       # enqueue at back — O(1), fine
front = queue.pop(0)  # dequeue from front — O(n), every element shifts!

# GOOD: Use collections.deque — popleft() is O(1)
from collections import deque
queue = deque([1, 2, 3, 4, 5])
queue.append(6)          # enqueue at back — O(1)
front = queue.popleft()  # dequeue from front — O(1)!

collections.deque（両端キュー：Double-Ended Queue）は両端でO(1)の操作をサポートします。固定サイズブロックの双方向連結リストとして実装されており、配列のキャッシュフレンドリーな動作と両端での連結リストの柔軟性を兼ね備えています。

キューのユースケース：

  • タスクスケジューリング—タスクが到着した順に処理します。プリントキュー、メッセージキュー、ジョブキュー。
  • 幅優先探索（Breadth-first search：BFS）—隣接ノードを層ごとに探索します（第23章のグラフで使用）。
  • レート制限—スライディングウィンドウで最近のリクエストのタイムスタンプを追跡します。
配列は要素をメモリ上に隣り合わせで格納するため、単一のCPUキャッシュラインが複数の要素を一度にロードします。連結リストのノードは散在しており、ほぼすべてのアクセスでキャッシュミスが発生します。これが、連結リストが特定の操作でより良い理論的計算量を持つ場合でも、実際には配列が速い理由です。
補足

キューが必要なとき、Pythonのcollections.dequeが正しい選択です。list.pop(0)を使うとO(n)——すべての要素がシフトします。deque.popleft()はO(1)です。これはAIが生成するコードがよく陥るパフォーマンスの罠です。AIの出力にqueue = []の後にqueue.pop(0)が続くのを見たら、dequeに置き換えるべきだと分かります。データ構造を1つ変えるだけ、アルゴリズムの変更なしで、一連のデキュー操作のパフォーマンスがO(n²)からO(n)になります。

ヒント

第21章 ハッシュテーブル（Hash Table）、集合（Set）とマップ（Map）

重要な概念

dictはPythonで最も強力なデータ構造です。内部がどう動くかを理解することで、いつO(1)で、いつ劣化し、なぜある値が辞書キーになれないかを知ることができます。if x in my_dictと書いて即座に結果が返るたびに、ハッシュテーブルが仕事をしています。TypeError: unhashable type: 'list'が出るたびに、それはハッシュテーブルがイミュータブルなキーを必要とするからです。この章でその理由を説明します。

ハッシュの仕組み

ハッシュ関数（Hash Function）はキー（任意のイミュータブルな値）をハッシュコード（Hash Code）と呼ばれる整数に変換します。その整数をテーブルサイズで割った余りが、キーと値のペアが入るバケット（Bucket）を決定します。

# Python's built-in hash function
print(hash("hello"))     # e.g., 2534023372954268
print(hash("hello") % 8) # e.g., 4 — bucket 4 in an 8-bucket table
print(hash(42))          # 42 — integers hash to themselves
print(hash((1, 2, 3)))   # tuples are hashable because they're immutable

my_dict["hello"]を検索するプロセス：

  1. hash("hello")を計算 → 何らかの大きな整数
  2. hash_code % table_sizeを計算 → バケットインデックス（例：4）
  3. バケット4に直接行き、値を見つける

スキャンも他のキーとの比較もありません。これが辞書の検索がO(1)である理由です。

衝突の解決（Collision Resolution）

2つの異なるキーが同じバケットにハッシュされることがあります。これを衝突（Collision）と呼びます。hash("hello") % 8 == 4かつhash("world") % 8 == 4なら、両キーがバケット4にマッピングされます。ハッシュテーブルはこれを処理しなければなりません。

主な2つの戦略があります：

  • チェイニング（Chaining）： 各バケットはエントリの連結リスト（または小さな配列）を保持します。衝突が発生すると、新しいエントリがチェーンに追加されます。検索はチェーンをスキャン——O(k)（kはチェーンの長さ）。ハッシュ関数が均等に分散すれば、チェーンは短いまま保たれます。
  • オープンアドレス法（Open Addressing）： 衝突が発生すると、決定論的なシーケンス（線形プロービング、二次プロービング、または二重ハッシュ）を使って次の空きスロットを探します。CPythonは擬似ランダムなプローブシーケンスを使ったオープンアドレス法の変形を使用しています。

両戦略にはトレードオフがあります：チェイニングはノードポインタでメモリを無駄にし、オープンアドレス法はクラスタリング（Clustering）（占有スロットが固まり、プローブシーケンスが長くなる）に悩まされます。実際には、テーブルがあまりいっぱいでない限り、どちらもうまく機能します。

負荷率（Load Factor）と再ハッシュ（Rehashing）

負荷率（Load Factor）はエントリ数とバケット数の比率：n / table_sizeです。負荷率が上がると衝突が増え、パフォーマンスはO(n)に向かって劣化します。

CPythonは負荷率を約2/3（0.67）以下に保ちます。新しいエントリの挿入がこのしきい値を超えると、テーブルは再ハッシュ（Rehash）します：新しい大きなテーブル（通常は2倍のサイズ）を割り当て、既存のすべてのエントリを再挿入します。これはO(n)の操作ですが、稀にしか起きません——挿入ごとの均しコストはO(1)、動的配列のリサイズと同じパターンです。

# You can observe rehashing indirectly via memory usage
import sys

d = {}
prev = sys.getsizeof(d)
for i in range(100):
    d[i] = i
    curr = sys.getsizeof(d)
    if curr != prev:
        print(f"At {i} entries: dict size jumped from {prev} to {curr} bytes")
        prev = curr
# You'll see jumps at approximately 5, 10, 21, 42, 85 entries

Python辞書の内部

CPython 3.6以降、辞書は2つの別々の配列を持つコンパクトレイアウトを使用します：

  • 密な配列へのインデックスを格納する疎なハッシュテーブル（大部分が空）
  • 挿入順にキーと値を格納する密なエントリ配列

これがPythonの辞書が3.7以降で挿入順を保持する理由です——密な配列が追加された順にエントリを記録します。また、辞書が予想より多くのメモリを使う理由も説明します：疎なハッシュテーブルは負荷率を低く保つために空きスロットを持っています。

Pythonの辞書についての重要な事実：

  • 平均的な検索、挿入、削除：O(1)
  • 最悪ケース（すべてのキーが衝突）：O(n)——しかしPythonのハッシュ関数ではこれは事実上起きない
  • 反復はO(n)で挿入順に従う
  • メモリ使用量はエントリごとに約50〜70バイト（リストよりずっと多い）

集合（Set）

集合（Set）は値なしのハッシュテーブルです——キーのみを格納します。これによりO(1)のメンバーシップテストが可能になります。これがリストよりも集合を使う最もよくある理由です。

# O(n) — membership test on a list
def has_overdue_list(task_ids, overdue_ids_list):
    for tid in task_ids:
        if tid in overdue_ids_list:  # O(n) each time
            return True
    return False
# Total: O(n * m) where n = task_ids, m = overdue list

# O(n) — membership test on a set
def has_overdue_set(task_ids, overdue_ids_set):
    for tid in task_ids:
        if tid in overdue_ids_set:  # O(1) each time
            return True
    return False
# Total: O(n) where n = task_ids

集合は効率的な数学的操作もサポートします：

team_a = {"Alice", "Bob", "Carol"}
team_b = {"Bob", "Dave", "Carol"}

print(team_a & team_b)   # Intersection: {"Bob", "Carol"}
print(team_a | team_b)   # Union: {"Alice", "Bob", "Carol", "Dave"}
print(team_a - team_b)   # Difference: {"Alice"}
print(team_a ^ team_b)   # Symmetric diff: {"Alice", "Dave"}

集合対リストの使い分け： コレクションに「Xが含まれているか？」を頻繁に確認する必要があるなら、集合に変換してください。100要素なら差は無視できます。100万要素ならx in my_listはミリ秒かかりますが、x in my_setはマイクロ秒——1000倍の差です。

ハッシュが機能しないとき

ハッシュテーブルには知っておくべき失敗モードがあります：

  • ミュータブルなキーは禁止。 リストを辞書キーとして使った後に変更できると、そのハッシュが変わり辞書が見つけられなくなります。Pythonはこれを防ぎます：TypeError: unhashable type: 'list'。先にタプルに変換してください。
  • ハッシュ衝突攻撃（Hash Collision Attack）。 ハッシュ関数を知っている攻撃者は、すべてが同じバケットにマップされる入力を作成でき、O(1)をO(n)に劣化させます。Pythonはハッシュランダム化（プロセスごとに異なるシード）でこれを軽減しています。
  • 病的な分布。 良いハッシュ関数があっても多くのキーが同じバケットにハッシュされると、パフォーマンスが劣化します。Pythonのハッシュでは極めて稀ですが、デバッグのために知っておく価値があります。
# This fails: lists are mutable, so unhashable
try:
    d = {[1, 2, 3]: "value"}
except TypeError as e:
    print(e)  # unhashable type: 'list'

# This works: tuples are immutable, so hashable
d = {(1, 2, 3): "value"}
print(d[(1, 2, 3)])  # "value"
8バケットに5つのキーと値のペアを保持するハッシュテーブル。バケット3に衝突があります：2つのキーが同じインデックスにハッシュされました。チェイニングがそれらをチェーンで繋ぎます。検索はバケットをチェックし、必要に応じてチェーンを辿ります。低い負荷率では、チェーンは短いまま保たれ、検索はO(1)のままです。
補足

検索にdict、set、listのどれを選ぶかは日々のエンジニアリングの判断です。if x in my_listはO(n)です。if x in my_setはO(1)です。100要素では問題になりません。100万要素では1秒と1マイクロ秒の差です。AIが生成したコードがユニークなIDをリストに格納してループ内でメンバーシップをチェックしているのを見たら、集合に置き換えるべきだと分かります。これがスケールする最適化です。

第22章 木構造（Tree）

重要な概念

木構造は気づかないところに至る所にあります。あなたのファイルシステムは木です。データベースのインデックスは木です。HTMLのDOMは木です。JSONは木です。os.walk()でディレクトリを走査するとき、あなたは木を走査しています。木を理解することは、これらすべてのシステムがどのようにパフォーマンスを発揮するか、そしてなぜ数百万行のテーブルでもデータベースクエリが速いかを理解することです。

二分木（Binary Tree）の基礎

二分木（Binary Tree）は、各ノードが最大2つの子を持つ階層構造で、慣例的に左（left）と右（right）と呼ばれます。最上位のノードをルート（root）といいます。子を持たないノードをリーフ（leaf）（葉）といいます。ノードの深さ（depth）はルートからの距離です。木の高さ（height）は最も深いリーフの深さです。

class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

# Build a small tree:
#        8
#       / \
#      3   10
#     / \    \
#    1   6    14
root = TreeNode(8,
    TreeNode(3, TreeNode(1), TreeNode(6)),
    TreeNode(10, None, TreeNode(14))
)

木の走査（Tree Traversal）

木のすべてのノードを訪問する標準的な方法が4つあります。各走査順序は異なる目的に役立ちます：

def inorder(node):
    """Left -> Root -> Right. For BSTs, visits nodes in sorted order."""
    if node is None:
        return []
    return inorder(node.left) + [node.value] + inorder(node.right)

def preorder(node):
    """Root -> Left -> Right. Copies the tree structure."""
    if node is None:
        return []
    return [node.value] + preorder(node.left) + preorder(node.right)

def postorder(node):
    """Left -> Right -> Root. Safe for deletion (children before parent)."""
    if node is None:
        return []
    return postorder(node.left) + postorder(node.right) + [node.value]

def levelorder(node):
    """Layer by layer (BFS). Uses a queue."""
    from collections import deque
    if node is None:
        return []
    result, queue = [], deque([node])
    while queue:
        current = queue.popleft()
        result.append(current.value)
        if current.left:
            queue.append(current.left)
        if current.right:
            queue.append(current.right)
    return result

# Using our tree from above:
print("Inorder:   ", inorder(root))    # [1, 3, 6, 8, 10, 14] — sorted!
print("Preorder:  ", preorder(root))   # [8, 3, 1, 6, 10, 14]
print("Postorder: ", postorder(root))  # [1, 6, 3, 14, 10, 8]
print("Levelorder:", levelorder(root)) # [8, 3, 10, 1, 6, 14]

二分探索木（Binary Search Tree：BST）

二分探索木（Binary Search Tree：BST）は順序規則を強制します：すべてのノードについて、左部分木の値はすべて小さく、右部分木の値はすべて大きい。この規則により、検索・挿入・削除はすべてO(h)（hは木の高さ）になります。

def bst_search(node, target):
    """Search a BST — O(h) where h = height."""
    if node is None:
        return False
    if target == node.value:
        return True
    elif target < node.value:
        return bst_search(node.left, target)   # go left
    else:
        return bst_search(node.right, target)  # go right

def bst_insert(node, value):
    """Insert into a BST — O(h)."""
    if node is None:
        return TreeNode(value)
    if value < node.value:
        node.left = bst_insert(node.left, value)
    elif value > node.value:
        node.right = bst_insert(node.right, value)
    # if value == node.value, duplicate — skip or handle as you wish
    return node

# Build a BST from scratch
tree = None
for val in [8, 3, 10, 1, 6, 14]:
    tree = bst_insert(tree, val)

print(bst_search(tree, 6))   # True
print(bst_search(tree, 7))   # False
print(inorder(tree))          # [1, 3, 6, 8, 10, 14] — always sorted

バランスが重要な理由

BSTのパフォーマンスは完全にその高さに依存します。nノードのバランスの取れたBSTは高さlog nとなり、O(log n)の操作を実現します。しかしBSTにソート済みデータを挿入すると、すべてのノードが右に行き、事実上高さnの連結リストになる退化した木（Degenerate Tree）が作られます。

# Balanced insertion: [8, 3, 10, 1, 6, 14]
# Creates a nice tree with height 3:
#        8
#       / \
#      3   10
#     / \    \
#    1   6    14
# Search: O(log n) = O(log 6) ≈ 3 comparisons

# Sorted insertion: [1, 3, 6, 8, 10, 14]
# Creates a degenerate tree with height 6:
#    1
#     \
#      3
#       \
#        6
#         \
#          8
#           \
#            10
#              \
#               14
# Search: O(n) = 6 comparisons — it's a linked list!

自己平衡木（Self-Balancing Tree）（AVL木、赤黒木）は挿入・削除後に自動的に構造を変えてO(log n)の高さを保証します。自分で実装することはほぼありません——それらはライブラリとデータベースの内部に存在します。しかし、なぜ存在するかを知る必要があります：O(n)の退化したケースを防ぐためです。Pythonのsorted containersライブラリとC++のstd::mapは赤黒木を使っています。データベースのインデックスはB木（後述）を使っています。

ヒープ（Heap）と優先度付きキュー（Priority Queue）

最小ヒープ（Min-Heap）は、すべての親がその子以下である完全二分木です。最小値は常にルートにあります。ヒープはインデックス演算を使って配列として格納されます：

  • ノードiの子：インデックス2i+1と2i+2
  • ノードiの親：インデックス(i-1)//2

Pythonのheapqモジュールは通常のリストによって支援される最小ヒープを提供します：

import heapq

# Build a heap from a list
tasks = [(3, "Low priority task"),
         (1, "URGENT task"),
         (2, "Medium task")]
heapq.heapify(tasks)  # O(n) — rearranges in place

# Pop the smallest (highest priority)
print(heapq.heappop(tasks))  # (1, "URGENT task")
print(heapq.heappop(tasks))  # (2, "Medium task")
print(heapq.heappop(tasks))  # (3, "Low priority task")

# Push a new item
heapq.heappush(tasks, (0, "Critical fix"))

# Get top-K without sorting the whole list
data = [42, 17, 93, 5, 68, 31, 84, 2]
print(heapq.nsmallest(3, data))  # [2, 5, 17] — O(n log k)
print(heapq.nlargest(3, data))   # [93, 84, 68]

トライ木（Trie）

トライ木（Trie）（「トライ」と発音）は各ノードが文字を表す木です。ルートからノードへのパスがプレフィックスを綴ります。トライ木はオートコンプリート機能を支えます：「pro」で始まるすべての単語を見つけるには、p-r-oのパスを辿ってその下のすべてを返します。

重要な特性：検索時間はO(m)（mはキーの長さ）で、格納されているキーの数に関係ありません。100万単語のトライ木は「programming」をちょうど11ステップで見つけます。

class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_end = False

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for char in word:
            if char not in node.children:
                node.children[char] = TrieNode()
            node = node.children[char]
        node.is_end = True

    def search(self, word):
        node = self.root
        for char in word:
            if char not in node.children:
                return False
            node = node.children[char]
        return node.is_end

    def starts_with(self, prefix):
        node = self.root
        for char in prefix:
            if char not in node.children:
                return False
            node = node.children[char]
        return True

t = Trie()
for word in ["task", "tag", "test", "team"]:
    t.insert(word)
print(t.search("task"))       # True
print(t.search("tas"))        # False (prefix, not complete word)
print(t.starts_with("ta"))    # True
print(t.starts_with("te"))    # True
print(t.starts_with("tx"))    # False

B木（B-Tree）：概念的理解

データベースはBSTを使いません。B木（B-Tree）——ディスクベースのストレージ向けに設計された、幅広く浅い木——を使います。

ディスク上のBSTの問題：各ノードは1回の比較で、ノードをディスクから読み取るのに約10ミリ秒（機械式ディスク）または約0.1ms（SSD）かかります。100万ノードのバランスBSTは高さ20で、20回のディスク読み取りが必要です。1回0.1msで、1クエリあたり2ms——許容範囲ですが理想的ではありません。

B木はノードを幅広くすることでこれを解決します：各ノードは数百のキーを保持し、数百の子を持ちます。分岐係数100で3レベルのB木は100³ = 100万キーを検索あたり3回のディスク読み取りのみで保持できます。これがSELECT * FROM tasks WHERE id = 42が数百万行のテーブルでも即座に返る理由です。

B木を実装することはありません。データベース（SQLite、PostgreSQL）を通じて使います。しかし存在を知ることで、データベースインデックスが速い理由と、CREATE INDEXがクエリを高速化する理由が分かります——インデックスを付ける列にB木を構築するからです。

同じ7つの数値が挿入順によって非常に異なる木を生み出します。ランダム順はO(log n)の検索を持つバランスの取れた木を生みます。ソート済み順は事実上の連結リストとなる退化した木を生み、O(n)の検索になります。自己平衡木は退化したケースを防ぎます。
補足

木を自分で実装することはほぼありません。しかし理解は不可欠です：データベースインデックスはB木で、ファイルシステムは木構造で、heapqは「最も重要なタスクトップ10を取得する」の適切なツールです。AIが最大値を見つけるためだけにリストをソートするコードを生成したとき、単一のmax()呼び出しがO(n)であることを知っており、さらに言えば、アイテムが変化するにつれて繰り返し最大値が必要なら、ヒープは更新ごとにO(log n)で、O(n log n)での再ソートより優れています。構造を知ることは適切なツールを知ることです。

ヒント

第23章 グラフ（Graph）

重要な概念

すべての関係が階層的なわけではありません。ソーシャルネットワーク、フライトルート、タスクの依存関係、Webページのリンク——これらはすべてグラフ（Graph）です。「AはBに依存する」「アリスはボブを知っている」「XからYへの道路がある」をモデル化する必要があるとき、木ではうまくいきません。木はサイクル、複数のパス、またはピアツーピアの関係を表現できないからです。グラフは接続をモデル化するための最も汎用的なデータ構造です。

グラフの用語

グラフ（Graph）はエッジ（Edge）（辺）で接続されたノード（Node）（頂点：Vertex）の集合です。これが定義のすべてです。それ以外はすべて特殊化です：

  • 有向対無向（Directed vs Undirected）： 有向グラフでは、エッジA→Bは一方向（AはBに依存するが、BはAに依存しない）。無向グラフでは、エッジは双方向（アリスがボブを知ることは、ボブがアリスを知ることを意味する）。
  • 重み付き対重みなし（Weighted vs Unweighted）： 重み付きエッジは値（距離、コスト、時間）を持ちます。重みなしエッジは接続のみを表します。
  • パス（Path）： 2つのノードを接続するエッジのシーケンス。「AからDに行けるか？」はパスが存在するかどうかの問いです。
  • サイクル（Cycle）： 同じノードで始まり終わるパス。サイクルのないグラフは非巡回（Acyclic）です。
  • 連結成分（Connected Component）： すべてのノードが他のすべてのノードに到達できるノードのグループ。グラフは複数の非連結成分を持つことがあります。
  • 次数（Degree）： ノードに接続されているエッジの数。有向グラフでは、入次数（In-Degree）は入ってくるエッジを、出次数（Out-Degree）は出ていくエッジを数えます。

グラフの表現方法

コードでグラフを格納する標準的な方法が2つあります。どちらも同じ情報を表しますが、パフォーマンス特性が異なります。

隣接リスト（Adjacency List）

各ノードをその隣接ノードのリスト（または集合）にマッピングする辞書です。これが最も一般的な表現方法で、デフォルトとすべきものです。

# Directed graph: A->B, A->C, B->C, C->D
adj_list = {
    "A": ["B", "C"],
    "B": ["C"],
    "C": ["D"],
    "D": []
}

# Checking neighbors of A: O(1) dict lookup
print(adj_list["A"])  # ["B", "C"]

# Checking if edge A->B exists: O(degree of A)
print("B" in adj_list["A"])  # True

隣接行列（Adjacency Matrix）

ノードiからノードjへのエッジがある場合にmatrix[i][j] = 1（または重み）、そうでない場合は0の2Dグリッドです。

# Same graph: A=0, B=1, C=2, D=3
#       A  B  C  D
adj_matrix = [
    [0, 1, 1, 0],  # A -> B, A -> C
    [0, 0, 1, 0],  # B -> C
    [0, 0, 0, 1],  # C -> D
    [0, 0, 0, 0],  # D -> (none)
]

# Checking if edge A->B exists: O(1)
print(adj_matrix[0][1])  # 1 — edge exists

グラフクラスの構築

class Graph:
    def __init__(self, directed=False):
        self.adj = {}
        self.directed = directed

    def add_node(self, node):
        if node not in self.adj:
            self.adj[node] = set()

    def add_edge(self, u, v):
        self.add_node(u)
        self.add_node(v)
        self.adj[u].add(v)
        if not self.directed:
            self.adj[v].add(u)

    def neighbors(self, node):
        return self.adj.get(node, set())

    def has_edge(self, u, v):
        return v in self.adj.get(u, set())

    def nodes(self):
        return list(self.adj.keys())

    def __repr__(self):
        lines = []
        for node, neighbors in self.adj.items():
            for n in neighbors:
                lines.append(f"  {node} -> {n}")
        return "Graph:\n" + "\n".join(lines)

social.add_edge("Alice", "Carol")

有向非巡回グラフ（Directed Acyclic Graph：DAG）

DAGはサイクルのない有向グラフです。DAGは依存関係のモデリングに不可欠です：タスクAがBに依存し、BがCに依存するのは問題ありません。しかしCもAに依存する場合、サイクルがあり——有効な実行順序が存在しません。

# Valid DAG (no cycles):
#   Write code -> Run tests -> Deploy
#   Write code -> Code review -> Deploy

# Invalid (cycle):
#   A -> B -> C -> A
# No task can go first because each depends on another

サイクルの検出は重要なバリデーションステップです：依存関係に基づいてタスクをスケジューリングする前に、依存グラフがDAGであることを確認してください。サイクルが含まれていれば、ユーザーは不可能な要件セットを持っています。

深さ優先探索（DFS）によるサイクル検出

再帰スタック（Recursion Stack）を使った深さ優先探索（Depth-First Search：DFS）で有向グラフのサイクルを検出できます。アイデア：DFSでより深く探索するとき、現在の探索パスにあるすべてのノードを追跡します。すでに現在のパスにあるノードに到達したら、サイクルを見つけたことになります。

def has_cycle(graph):
    """Detect cycles in a directed graph using DFS.

    Uses three states per node:
    - unvisited: not yet explored
    - in_stack: currently on the DFS path (ancestors of current node)
    - done: fully explored, no cycles through this node
    """
    visited = set()
    in_stack = set()

    def dfs(node):
        visited.add(node)
        in_stack.add(node)

        for neighbor in graph.neighbors(node):
            if neighbor not in visited:
                if dfs(neighbor):
                    return True
            elif neighbor in in_stack:
                return True  # found a cycle!

        in_stack.remove(node)
        return False

    for node in graph.nodes():
        if node not in visited:
            if dfs(node):
                return True
    return False

グラフはプロフェッショナルなエンジニアリングの至る所に現れます。pipはDAGとトポロジカルソートを使ってパッケージ依存関係を解決します。Google Mapsは重み付きグラフと最短経路アルゴリズム（ダイクストラ法）を使ってルートを見つけます。LinkedInの「知り合いかも」はソーシャルグラフのBFSを使います。DockerのレイヤーシステムはダウンDAGを形成します。GitのコミットヒストリーもDAGで、各コミットはその親を指しています。これらのアルゴリズムのいくつかはフェーズ6で実装します。

ヒント

タスクの依存関係をDAGとしてモデル化してください。タスクBがタスクAに依存する場合、有向エッジA → Bを追加します（AはBの前に完了する必要がある）。スケジューリング前にhas_cycle()でDAGを検証してください——サイクルは依存関係が不可能であることを意味します。フェーズ6では、すべてのタスクの有効な実行順序を見つけるトポロジカルソートを学びます。

やってみよう1：グラフを構築する

「タスクAは何にも依存しない。タスクBはAに依存する。タスクCはAに依存する。タスクDはBとCの両方に依存する。」

g = Graph(directed=True)

g.add_edge("A", "C")   # A must finish before C
g.add_edge("B", "D")   # B must finish before D
g.add_edge("C", "D")   # C must finish before D
print(has_cycle(g))     # False — valid DAG
print(g.neighbors("A")) # {"B", "C"}
# Valid execution orders: A, B, C, D  or  A, C, B, D

DからAに戻るエッジを追加するとどうなりますか？

g.add_edge("D", "A")    # D depends on A, but A depends on B/C which depend on D!
print(has_cycle(g))      # True — impossible dependency cycle

設計チャレンジ：「TaskForge依存関係グラフ」

タスクをノード、依存関係を有向エッジとして管理するTaskGraphクラスを構築してください。add_task(name)、add_dependency(task, depends_on)、依存グラフにサイクルがある場合にTrueを返すhas_cycle()を実装してください。

グラフをself.adj[node] = set()として格納します。集合にはそのノードがエッジを持つノードが含まれます。add_dependency("B", "A")では、エッジA -> Bを追加します（AはBの前に終わる必要がある）。

サイクル検出では、2つの集合でDFSを実装します：visited（見たすべてのノード）とin_stack（現在のDFSパスにあるノード）。in_stackにあるノードに到達したら、サイクルを見つけたことになります。

外側のループで、1つの開始ノードだけでなく、すべてのノードを反復することを忘れないでください。グラフには非連結成分が含まれる場合があります。
