第13章 APIとHTTP

重要な概念

現代のアプリケーションのほぼすべては、インターネットを通じて他のアプリケーションと通信しています。天気アプリは予報を取得し、決済システムは料金を処理し、AIツールはプロンプトを送って回答を受け取ります。それらが共通して話す言語がHTTPで、会話のパターンをAPIと呼びます。APIを理解することで、自分のマシン上で動くコードから、世界とつながるコードへと進化できます。

APIとは何か

API（Application Programming Interface／アプリケーション・プログラミング・インターフェース）は、2つのプログラム間の契約です。一方のプログラムが「この形式でリクエストを送ってくれれば、その形式でレスポンスを返す」と言います。APIはルールを定義します——何を要求できるか、どう要求するか、何が返ってくるか。

実はすでにAPIを使ったことがあります。Pythonで len([1, 2, 3]) を呼び出すとき、関数API（function API）を使っています——リストを渡すと、長さが返ってきます。Web APIも同じ仕組みで動いています。違いは、関数がリモートサーバー上にあり、呼び出しがインターネットを経由するだけです。

レストランで例えてみましょう。あなた（クライアント）はメニュー（APIドキュメント）を読み、ウェイターに注文します（リクエストを送る）。キッチン（サーバー）が料理を用意し、ウェイターが持ってきます（レスポンス）。あなたはキッチンには入りません。メニューが契約書です。

HTTP：Webの言語

HTTP（HyperText Transfer Protocol／ハイパーテキスト転送プロトコル）は、クライアントとサーバーが通信するためのプロトコルです。Webサイトを訪問するたびに、ブラウザはHTTPリクエストを送り、HTTPレスポンスを受け取っています。APIも同じプロトコルを使います。

HTTPのやり取りには2つの部分があります：

  • リクエスト（Request）—クライアントが送信。メソッド、URL、ヘッダー、オプションのボディを含む。
  • レスポンス（Response）—サーバーが送信。ステータスコード、ヘッダー、ボディを含む。
HTTPのリクエスト・レスポンスサイクル：クライアントがメソッドとURLを含むリクエストを送り、サーバーがそれを処理してステータスコードとデータを含むレスポンスを返します。

HTTPメソッド

メソッド（method）は、サーバーに何をしたいかを伝えます。知っておくべき4つのメソッドがあります：

HTTPメソッドとその用途
メソッド用途例
GETデータの読み取り・取得タスク一覧を取得する
POST新しいデータの作成新しいタスクを追加する
PUT既存データの更新タスクを完了済みにする
DELETEデータの削除タスクを削除する

この4つのメソッドはCRUD（Create・Read・Update・Delete）操作に対応しています。GETとDELETEは通常ボディを送りません。POSTとPUTはリクエストボディにデータを含めます（通常はJSON）。

ステータスコード

すべてのHTTPレスポンスにはステータスコード（status code）が含まれます——クライアントに何が起きたかを伝える3桁の数字です。すべてを暗記する必要はありません。次の5つのグループだけ知っておきましょう：

HTTPステータスコードの範囲とその意味
範囲意味主なコード
2xx成功200 OK、201 Created
3xxリダイレクト301 Moved Permanently
4xxクライアントエラー（こちらのミス）400 Bad Request、404 Not Found
5xxサーバーエラー（サーバー側の問題）500 Internal Server Error

何か問題が起きたら、最初にステータスコードを確認しましょう。404はURLが間違っていることを意味します。401は認証されていないことを意味します。500はサーバーが壊れていることを意味します——あなたのせいではありません。

JSON：データフォーマット

APIはデータを送受信するための共通フォーマットが必要です。標準はJSON（JavaScript Object Notation）です。JSONはすでに知っています——Pythonの辞書とリストとまったく同じように見えます：

{
    "id": 1,
    "title": "Buy groceries",
    "done": false
}

PythonのモジュールはJSONの文字列とPythonオブジェクトの間を変換します。ただし、requestsライブラリ（後述）を使う場合、この変換は自動的に行われます。

ヘッダー

ヘッダー（Headers）はリクエストとレスポンスに添付されるメタデータです。データそのものではなく、メッセージ自体についての情報を運びます。常によく見る2つのヘッダー：

  • Content-Type: application/json — 「JSONを送っています」とサーバーに伝える。
  • Authorization: Bearer <token> — 身元を証明する（パスワードのようなもの）。

ヘッダーを手動で設定することはほとんどありません——requestsライブラリのようなツールが大部分を自動的に処理してくれます。

注意

APIはプログラム同士が通信する手段です。データサイエンティストはAPIを使ってデータセットを取得します。DevOpsエンジニアはAPIを使ってクラウドインフラを管理します。AIエージェントはAPIを使って言語モデルを呼び出します。外部サービスとやり取りするコードを書くなら、APIを使っています。

注意

そうではありません。GETはデータを取得するもので、サーバー上の何かを変更してはいけません。POSTは新しいデータを作成します。間違ったメソッドを使うと、他の開発者を混乱させ、これらの規約に依存するツールを壊してしまいます。メソッドは意図を伝えます。

第14章 FlaskでWeb APIを構築する

重要な概念

第13章ではAPIを使う方法を学びました。今度は作る方です。これはユーザーからクリエイターへの橋渡しです——フェーズ4のゲートでも必要となります。

Flaskとは何か

Flaskは最小限のPython Webフレームワークです。数行のコードでPython関数をAPIエンドポイントに変えられます。大規模なフレームワーク（Django）とは異なり、Flaskは必要なものだけを提供して邪魔をしません。

補足

Flask（Python、無料）。APIとWebアプリを構築するための最小限のWebフレームワーク。インストール：pip install flask。

最初のFlaskアプリ

5行で完成するFlaskアプリ：

from flask import Flask

app = Flask(__name__)

@app.route("/")
def hello():
    return "Hello, World!"

if __name__ == "__main__":
    app.run(debug=True)

app.pyとして保存し、python3 app.pyで実行します。ブラウザでhttp://localhost:5000を開いてください。これでWebサーバーができあがりました。

@app.route("/")はデコレータ（decorator）です——どのURLがどの関数をトリガーするかをFlaskに伝えます。debug=Trueはコードが変更されたときに自動的にリロードし、便利なエラーページを表示します。

ルートとメソッド

各エンドポイント（endpoint）はURL + HTTPメソッドの組み合わせです。FlaskはデフォルトでGETのみです。他のメソッドを受け付けるには、明示的に指定します：

@app.route("/tasks", methods=["GET", "POST"])
def handle_tasks():
    if request.method == "GET":
        # すべてのタスクを返す
        pass
    elif request.method == "POST":
        # 新しいタスクを作成する
        pass

メソッドごとに別の関数を使うとよりすっきりします：

@app.route("/tasks", methods=["GET"])
def get_tasks():
    # すべてのタスクを返す
    pass

@app.route("/tasks", methods=["POST"])
def create_task():
    # 新しいタスクを作成する
    pass

JSONを返す

APIはHTMLではなくJSONを返します。Flaskはjsonify()を提供して、Pythonの辞書を正しいContent-Typeヘッダー付きのJSONレスポンスに変換します：

from flask import Flask, jsonify

@app.route("/status")
def status():
    return jsonify({"status": "running", "version": "1.0"}), 200

リクエストデータの読み取り

クライアントがAPIにデータを送ってきたとき、それを読み取る必要があります。Flaskには主に2つの方法があります：

from flask import Flask, request, jsonify

# JSONボディ（POST/PUTリクエスト用）
@app.route("/tasks", methods=["POST"])
def create_task():
    data = request.json          # JSONボディを解析する
    title = data.get("title")   # フィールドを安全に取得する
    return jsonify({"title": title}), 201

# クエリパラメータ（GETリクエスト用）
# GET /tasks?status=pending
@app.route("/tasks", methods=["GET"])
def get_tasks():
    status = request.args.get("status", "all")  # デフォルトは"all"
    return jsonify({"filter": status})

request.jsonはクライアントが送ったJSONボディを解析します。request.argsはURLのクエリパラメータ（?以降のすべて）を読み取ります。

TaskForge APIの構築

TaskForge用の完全なREST APIを段階的に構築しましょう。タスクの保存にはメモリ上のリストを使います——まだデータベースは不要です。

ステップ1：セットアップとデータストア

from flask import Flask, jsonify, request

app = Flask(__name__)

# メモリ上のタスクストレージ
tasks = []
next_id = 1

ステップ2：GET /tasks — すべてのタスクを一覧表示する

@app.route("/tasks", methods=["GET"])
def get_tasks():
    return jsonify(tasks), 200

タスクリスト全体をJSON配列として返します。シンプルで直接的です。

ステップ3：POST /tasks — タスクを追加する

@app.route("/tasks", methods=["POST"])
def create_task():
    global next_id
    data = request.json

    if not data or "title" not in data:
        return jsonify({"error": "title is required"}), 400

    task = {
        "id": next_id,
        "title": data["title"],
        "done": False
    }
    tasks.append(task)
    next_id += 1
    return jsonify(task), 201

バリデーションに注目してください：クライアントがtitleを送らなかった場合、明確なエラーメッセージとともに400 Bad Requestを返します。クライアントの入力を絶対に信用してはいけません。

ステップ4：PUT /tasks/<id>/done — タスクを完了済みにする

@app.route("/tasks/<int:task_id>/done", methods=["PUT"])
def complete_task(task_id):
    for task in tasks:
        if task["id"] == task_id:
            task["done"] = True
            return jsonify(task), 200

    return jsonify({"error": "task not found"}), 404

<int:task_id>はURL変数（URL variable）です——FlaskがURLから数値を抽出して関数パラメータとして渡します。タスクが見つからない場合は404を返します。

ステップ5：実行する

if __name__ == "__main__":
    app.run(debug=True)

第15章 データベースの基礎

重要な概念

現在のTaskForgeは閉じるとすべてのデータが消えます。データベースはこれを永続的に解決します。あらゆる実際のアプリケーションがデータベースを使っています。

データベースとは何か

データベース（database）はプログラムの実行を超えて持続する構造化されたストレージです。Pythonスクリプトを閉じると変数は消えます。データベースを使ったプログラムを閉じても、データは残ります。TaskForgeからGitHub、AnthropicのAPIまで、あらゆる実際のアプリケーションがデータをデータベースに保存しています。

SQLとNoSQL

データベースには2つの主要なファミリーがあります。このコースではSQLに焦点を当てます。

SQLデータベースとNoSQLデータベースの比較
特徴SQLNoSQL
構造行と列を持つテーブル柔軟なドキュメントまたはキーと値のペア
スキーマ厳格——データを挿入する前に列を定義する柔軟——各ドキュメントが異なってよい
クエリ言語SQL（Structured Query Language）データベースによって異なる

SQLite — 最初のデータベース

SQLiteから始めるのは、入門の障壁をすべて取り除いてくれるからです：

  • サーバー不要—サービスではなくライブラリです。インストールも設定もパスワードも不要。
  • Pythonに組み込み済み—import sqlite3がそのまま動きます。pip installは不要。
  • ファイルベース—データベース全体が1つのファイル（tasks.db）です。コピー、バックアップ、削除——普通のファイルと同じ。

SQLiteはおもちゃではありません。すべてのiPhone、すべてのAndroidスマートフォン、すべてのWebブラウザ、すべてのPythonで本番環境に使われています。学習用、およびTaskForgeのような単一ユーザーアプリケーションには最適な選択です。

基本的なSQL

SQLはデータベースと対話するための言語です。核心は5つのコマンドだけです：

CREATE TABLE — 構造を定義する

CREATE TABLE tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    done BOOLEAN DEFAULT 0
);

これで3つの列を持つテーブルが作られます。PRIMARY KEY AUTOINCREMENTは各タスクに一意の自動割り当てIDが付くことを意味します。NOT NULLはタイトルが必須であることを意味します。DEFAULT 0は新しいタスクが未完了で始まることを意味します。

INSERT — データを追加する

INSERT INTO tasks (title, done) VALUES ('Buy groceries', 0);
INSERT INTO tasks (title, done) VALUES ('Write tests', 0);
INSERT INTO tasks (title, done) VALUES ('Deploy API', 1);

SELECT — データを読み取る

-- すべてのタスクを取得
SELECT * FROM tasks;

-- 未完了のタスクのみ取得
SELECT * FROM tasks WHERE done = 0;

-- タイトルだけ取得
SELECT title FROM tasks WHERE done = 0;

UPDATE — データを変更する

UPDATE tasks SET done = 1 WHERE id = 1;

常にWHERE句を含めてください。なければすべての行を更新してしまいます。

DELETE — データを削除する

DELETE FROM tasks WHERE id = 1;

同じルール：常にWHEREを使ってください。WHEREなしのDELETE FROM tasksはすべてを削除します。

Pythonのsqlite3モジュール

Pythonは標準ライブラリにsqlite3を含んでいます。作業の流れ：

import sqlite3

# 1. データベースに接続する（ファイルがなければ作成される）
conn = sqlite3.connect("tasks.db")

# 2. カーソルを作成する（SQLを実行するオブジェクト）
cursor = conn.cursor()

# 3. SQLを実行する
cursor.execute("""
    CREATE TABLE IF NOT EXISTS tasks (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        title TEXT NOT NULL,
        done BOOLEAN DEFAULT 0
    )
""")

# 4. データを挿入する
cursor.execute("INSERT INTO tasks (title, done) VALUES (?, ?)", ("Buy groceries", 0))

# 5. 変更をコミットする（ディスクに保存）
conn.commit()

# 6. データを照会する
cursor.execute("SELECT * FROM tasks")
rows = cursor.fetchall()
for row in rows:
    print(row)  # (1, 'Buy groceries', 0)

# 7. 接続を閉じる
conn.close()

主な関数：connect()はデータベースを開き、cursor()は実行オブジェクトを作成し、execute()はSQLを実行し、fetchall()は結果を取得し、commit()は変更を保存し、close()はクリーンアップします。

パラメータ化クエリ

注意

常に?プレースホルダーを使ってください。f文字列、.format()、+連結でSQL文字列を組み立てることは絶対にしてはいけません。文字列フォーマットは攻撃者がデータベースに悪意のあるSQLを注入することを可能にします。これは理論上の話ではありません——SQLインジェクションは実際のアプリケーションで最も一般的なセキュリティ脆弱性の1つです。

# ユーザーが入力できる：'); DROP TABLE tasks; --
# するとテーブル全体が削除される。

?プレースホルダーはsqlite3に値を安全にエスケープするよう指示します。このルール1つでセキュリティ脆弱性のクラス全体を防げます。

ORM（Object-Relational Mapper／オブジェクト関係マッパー）はPythonオブジェクトとデータベースの行の間を橋渡しします。生のSQLを書く代わりに、テーブルにマップするPythonクラスを定義します：

脆弱なJSONファイルを本物のデータベースに置き換えました。TaskForgeのデータは再起動後も生き残り、クエリを効率的に処理し、インジェクション攻撃から守られています。これがすべての本番アプリケーションがデータを保存する方法です。

第16章 DockerとContainerについて

重要な概念

Dockerは「自分のマシンでは動く」問題を永久に解決します。プロフェッショナルチームが一貫した環境を保証する方法であり、フェーズ6でClaudeCodeエージェントを隔離されたコンテナで実行する方法でもあります。

Dockerとは何か

Dockerはアプリケーションとそれに必要なすべてのもの——Pythonのバージョン、ライブラリ、システムツール——を任意のマシンで同じように動く単一のコンテナ（container）にパッケージ化できます。コンテナ船で例えると：船が東京に向かうかロッテルダムに向かうかにかかわらず、箱は同じで中身は無事に届きます。Dockerコンテナはソフトウェアに対して同じように動作します。

コンテナは仮想マシン（VM）ではありません。VMは独自のカーネルを持つOS全体をエミュレートします——重くて起動が遅い。コンテナはホストOSのカーネルを共有し、アプリケーション層だけを隔離します。これによりコンテナは軽量（GBではなくMB）で起動が速く（分ではなく秒）なります。

-t taskforgeフラグはイメージに名前（タグ）を付けます。.はDockerに現在のディレクトリをビルドコンテキスト（DockerfileとコピーするファイルをどこにDOCKERFILEを見つけるか）として使うよう伝えます。

ボリューム：データの永続化

コンテナは一時的（ephemeral）です——コンテナを停止すると、内部で作成されたデータは失われます。これはTaskForgeのSQLiteデータベースにとって問題です。ボリューム（Volumes）はホストディレクトリをコンテナにマウントするので、データが再起動後も残ります：

Terminaldocker run -it -v $(pwd)/data:/app/data taskforge

これはホスト上の./dataをコンテナ内の/app/dataにマッピングします。SQLiteデータベースファイルはこの共有ディレクトリに置かれるので、コンテナが停止しても残ります。

第17章 CI/CDとGitHub Actions

今これを学ぶ理由

GitHubへのすべてのプッシュがテストを自動的に実行し、コードを確認し、アプリをデプロイさえできます。CI/CDは手動の品質チェックを自動化されたパイプラインに変えます。フェーズ5では、このインフラに依存するCIトリガーのエージェントを使います。

CI/CDとは何か

CI（Continuous Integration／継続的インテグレーション）は、コードがリポジトリにプッシュされるたびに自動的にテストを実行することを意味します。マージ前にpytestを実行することを覚えておく代わりに、システムが毎回欠かさず行います。

CD（Continuous Deployment/Delivery／継続的デプロイ/デリバリー）はさらに進みます：テストが通った後、コードが自動的に本番環境にデプロイされる（Deployment）か、ワンクリックリリースの準備が整ったパッケージになる（Delivery）。

核心のアイデアはシンプルです：バグが本番環境に届く前に捕捉する。テストが失敗すると、パイプラインが停止して通知します。壊れたコードがデプロイされることはありません。「テストを実行し忘れた」という災害はなくなります。

CI/CDの用語とパイプラインのステージ
用語意味実行されるタイミング
CIテストとチェックを自動的に実行するすべてのプッシュまたはプルリクエスト時
CD（デリバリー）コードをパッケージ化し、手動デプロイの準備をするCIがmainブランチで成功した後
CD（デプロイメント）自動的に本番環境にデプロイするCIがmainブランチで成功した後

チェックを追加する

テストだけでは十分ではありません。プロフェッショナルなパイプラインはコード品質もチェックします：

ruffでリンティング

リンティング（Linting）はコードを実行せずにスタイルの問題、未使用のインポート、潜在的なバグを検出します。ワークフローにリンティングステップを追加します：

Pythonコードで型ヒントを使っている場合、mypyのようなツールがCIでそれをチェックできます。これはより高度です——今のところ存在を知って、必要になったら別のステップとして追加できることを覚えておいてください。

マトリックス戦略：複数のPythonバージョン

コードがPython 3.12では動くのに3.11では壊れる場合があります。マトリックス戦略（matrix strategy）は複数のバージョンに対して並行してテストを実行します：

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.11', '3.12', '3.13']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install -r requirements.txt
      - run: python -m pytest

Pythonバージョンごとに1つずつ、合計3つの並行ジョブが作られます。いずれかが失敗した場合、どのバージョンに問題があるか正確にわかります。

シークレットの管理

一部のワークフローにはAPIキーやトークンが必要です（例：デプロイや外部サービスの呼び出し）。ワークフローファイルやソースコードに認証情報を絶対にハードコードしてはいけません。

GitHubはSecrets——リポジトリ設定に保存された暗号化変数——を提供しています。追加するには：GitHubのリポジトリに行き、Settings→Secrets and variables→Actionsに移動し、「New repository secret」をクリックしてください。

${{ secrets.NAME }}の構文でワークフロー内のシークレットを参照します：

      - run: python deploy.py
        env:
          API_KEY: ${{ secrets.API_KEY }}
          DATABASE_URL: ${{ secrets.DATABASE_URL }}

GitHubはログ内のシークレットを自動的にマスクしますが、ワークフローのステップでecho $API_KEYやprint(os.environ["API_KEY"])を使用しないでください。シークレットが漏洩した場合は、直ちにローテーション（再発行）してください。

よくある誤解

注意：「CIは大きなチームだけのもの」

個人プロジェクトでも自動テストは役に立ちます。プッシュ前にテストを実行し忘れることがあります。つながりに気づかなかった部分を壊してしまうこともあります。CIはこれらのミスを自動的に検出します——1人のチームにも50人のチームにも同じように有効なセーフティネットです。

注意：「ローカルでテストが通れば、CIでも通る」

CIは異なるオペレーティングシステム（通常Ubuntu）、異なるPythonバージョンの可能性があり、残存状態のないクリーンな環境で実行されます。ローカルで設定した環境変数はCIには存在しません。リポジトリ外のファイルも存在しません。まさにこれがCIのポイントです——CIはあなたのマシンだけでなく、クリーンな環境でコードが動作することを証明します。

Chapter 18 プロジェクトアーキテクチャ

補足

ここまでで、APIの構築、データの永続化、アプリケーションのコンテナ化、自動テストの実行ができるようになりました。しかしプロジェクトが大きくなると、コードの整理方法がコードそのものと同じくらい重要になります。アーキテクチャ（設計構造）が悪いと、動くプロジェクトが保守不可能なものに変わります。良いアーキテクチャがあれば、あなた自身もAIコーディングツールも、自信を持ってコードを読み、拡張し、リファクタリングできます。

アーキテクチャが重要な理由

TaskForgeが1つのファイルだったとき、プログラム全体を頭の中に収めることができました。今ではFlask API、データベース関数、テスト、Dockerfile、CIパイプラインがあります。これらすべてが1つのファイルにあったら、コーディングよりもスクロールに多くの時間を費やすことになるでしょう。

アーキテクチャとは、コードをファイル、フォルダ、モジュールに整理して、各部分が明確な目的を持つようにすることです。良いアーキテクチャは3つのことを提供します：

  • 見つけやすさ（Findability）——あらゆる機能をすばやく見つけられます。
  • 変更しやすさ（Changeability）——他の部分を壊さずに1つの部分を変更できます。
  • 読みやすさ（Readability）——新しい開発者（またはコードベースを読むAIエージェント）がすべての行を読まなくても構造を理解できます。

関心の分離

関心の分離（Separation of Concerns）とは、各ファイルやモジュールが1つの責務を担当することを意味します。APIレイヤーはHTTPリクエストとレスポンスを処理します。データベースレイヤーはデータの保存と取得を処理します。ビジネスロジックはアプリケーションのルールを処理します。これらの関心が分離されていれば、データの保存方法を変更してもAPIルートの変更は不要であり、その逆も同様です。

レストランに例えると分かりやすいでしょう。ウェイターが注文を受け（APIレイヤー）、キッチンが料理を作り（ビジネスロジック）、食料庫が食材を保管します（データレイヤー）。食料庫を整理しても、ウェイターの再教育は不要です。

一般的なPythonプロジェクトでは、ソースコードとテストを設定ファイルやインフラストラクチャファイルから分離します。各ファイルには単一の明確な責務があります。

設定管理

アプリケーションには設定が必要です：データベースパス、APIキー、デバッグフラグなど。これらの値をソースコードにハードコードすることは脆弱で安全ではありません。2つのアプローチがあります：

環境変数

プログラムが実行される環境で、コードの外側に設定を配置します：

import os

os.environ.get()は環境変数をフォールバックのデフォルト値付きで読み取ります。これにより、同じコードが開発環境（デフォルト値を使用）でも本番環境（DockerやCIで設定された環境変数を使用）でも動作します。

開発中は、プロジェクトルートの.envファイルに環境変数を保存できます：

.envを.gitignoreに追加して、決してコミットされないようにしてください。各開発者はローカルの.envを持ちます。本番環境ではデプロイシステムによって設定された実際の環境変数を使用します。

リファクタリングのタイミング

リファクタリング（Refactoring）とは、コードの動作を変えずに構造を再編成することです。以下のような場合にリファクタリングします：

  • ファイルが長すぎてナビゲーションしにくい（目安：200行以上）。
  • コードの2つの部分が異なる理由で変更される（例：APIルートとデータベースクエリ）。
  • 必要な部分にたどり着くために、大きなコードブロックをスクロールして通過している。
  • すべてが絡み合っているためテストが書きにくい。

以下のような場合にはリファクタリングしません：

  • 現在の構造が機能しており、理解しやすい。
  • 機能追加と再編成を同時に行おうとしている（一度に1つずつ行ってください）。
  • 50行のスクリプトに「ベストプラクティス」の構造をコピーしようとしている。シンプルなコードにはシンプルな構造がふさわしいです。
注意：「コードを書く前に完璧なアーキテクチャが必要」

そんなことはありません。動く最もシンプルな構造から始めてください。摩擦を感じたときに再編成します——それより前にではありません。100行のスクリプトにディレクトリ構造を過剰に設計するのは無駄な努力です。アーキテクチャは理論的な理想からではなく、実際のニーズから生まれるべきです。

よくある誤解

注意：「ファイルが多いほど整理されている」

30行のモジュールを5行ずつの6ファイルに分割すると、プロジェクトは理解しやすくなるのではなく、理解しにくくなります。各ファイルには意味のある、まとまりのある機能単位を含めるべきです。ファイルの役割を一文で説明できない場合、大きすぎるか（分割する）、曖昧すぎるか（関連するものと統合する）のどちらかです。
