第24章 探索とソート

重要な概念

データベースのクエリはすべて探索を使います。ソートされた表示はすべてソートを使います。AIは絶えずソートコードを生成します。そのアルゴリズムが適切かどうかを評価できなければなりません。誤った選択は、10ミリ秒の応答と10秒の応答の差を生みます。この章では、その判断を毎回正確に下すための語彙と理解を身につけます。

線形探索（Linear Search）

線形探索（linear search）は最もシンプルな探索アルゴリズムです。先頭から始めてすべての要素を確認し、対象が見つかるか、要素がなくなれば終了します。時間計算量はO(n)で、最悪の場合はすべての要素を調べます。

def linear_search(arr, target):
    """Return the index of target in arr, or -1 if not found."""
    for i in range(len(arr)):      # Walk through every index
        if arr[i] == target:       # Found it?
            return i               # Return the position
    return -1                      # Exhausted the list without finding target

# Usage
tasks = ["buy groceries", "write tests", "deploy app", "review PR"]
idx = linear_search(tasks, "deploy app")
print(idx)  # 2

線形探索が正しい選択となる状況は、思っているよりも多くあります：

  • 未ソートのデータ—ソートされていないコレクションには二分探索を使えません。データがランダムな順序で届き、1回だけ探索する場合、線形探索しか選択肢がありません。
  • 小さなコレクション—要素数が約50未満の場合、ソート済み構造の維持やハッシュテーブルの構築コストが、単純にスキャンするコストを上回ります。PythonのリストにおけるIn演算子は内部で線形探索を使っています。
  • 一度だけ探索するシナリオ—コレクションを1回だけ探索する場合、最初にソートして（O(n log n)）から二分探索（O(log n)）するより、単純な線形スキャン（O(n)）の方が速いです。

Pythonのリストにおけるin演算子は内部で線形探索を行います：

tasks = ["buy groceries", "write tests", "deploy app"]
print("write tests" in tasks)  # True — scans left to right, O(n)

二分探索（Binary Search）

二分探索（binary search）はソートされたデータが必要です。探索空間を繰り返し半分に切り分けます。対象と中央の要素を比較して、対象が含まれ得ない半分を捨てます。これによりO(log n)の時間計算量が得られます。100万件の要素を探索しても、最大20回の比較で済みます。

アルゴリズムのステップ：

  1. lo = 0、hi = len(arr) - 1 を設定します。
  2. lo <= hi の間：
    1. mid = (lo + hi) // 2 を計算します。
    2. arr[mid] == target なら mid を返します。
    3. arr[mid] < target なら対象は右半分にあります：lo = mid + 1 とします。
    4. arr[mid] > target なら対象は左半分にあります：hi = mid - 1 とします。
  3. ループが終了しても返っていなければ、対象は配列に存在しません。-1 を返します。
def binary_search(arr, target):
    """Return the index of target in sorted arr, or -1 if not found."""
    lo, hi = 0, len(arr) - 1           # 1. Initialize boundaries

    while lo <= hi:                     # 2. Loop while search space is non-empty
        mid = (lo + hi) // 2           #    Find the midpoint
        if arr[mid] == target:         #    2a. Exact match
            return mid
        elif arr[mid] < target:        #    2b. Target is in the right half
            lo = mid + 1               #         Discard left half including mid
        else:                          #    2c. Target is in the left half
            hi = mid - 1              #         Discard right half including mid

    return -1                          # 3. Not found

# Usage
sorted_ids = [2, 5, 8, 12, 16, 23, 38, 56, 72, 91]
print(binary_search(sorted_ids, 23))   # 5
print(binary_search(sorted_ids, 24))   # -1
注意

二分探索は誤りやすいことで有名です。よくあるバグ：

  • lo <= hi か lo < hi か—<= の代わりに < を使うと、lo == hi のケースをスキップしてしまい、最後の要素を確認できなくなります。
  • mid + 1 / mid - 1 か mid か—lo = mid + 1 の代わりに lo = mid と設定すると、lo == mid（hi - lo == 1 のとき）で無限ループが発生します。
  • 整数オーバーフロー（integer overflow）—C++ や Java などの言語では (lo + hi) がオーバーフローする場合があります。Python は任意精度の整数を扱うのでこの問題は起きませんが、他の言語で作業する際には注意が必要です。

探索空間上の二分探索

二分探索は配列内の要素を見つけるだけに限りません。「Xは十分大きいか？」という問いにYes/Noで答えられる問題なら、答えの空間に対して二分探索を適用できます。これは最も強力なアルゴリズムテクニックの一つで、実際のエンジニアリングでも頻繁に登場します。

問題：タスクの所要時間のリストとワーカー数が与えられています。各ワーカーはタスクを順番に処理します。どのワーカーの作業時間の最大値も最小になるようにタスクを割り当てるには？その最小の最大時間はいくらですか？

洞察：時間制限 T を推測すれば、T が実現可能かどうかを貪欲に確認できます。各ワーカーに順番にタスクを割り当て、現在のワーカーが T を超えそうになったら新しいワーカーに切り替えます。必要なワーカー数が利用可能な数を超えれば T は小さすぎます。以下なら実現可能です。答えは実現可能な最小の T です。

def min_max_time(durations, num_workers):
    """Find the minimum possible maximum time per worker."""

    def is_feasible(max_time):
        """Can we assign all tasks using <= num_workers if each handles <= max_time?"""
        workers_needed = 1
        current_load = 0
        for d in durations:
            if current_load + d > max_time:
                workers_needed += 1        # Start a new worker
                current_load = d           # This task goes to the new worker
            else:
                current_load += d          # Add task to current worker
        return workers_needed <= num_workers

    lo = max(durations)          # Lower bound: at least the longest single task
    hi = sum(durations)          # Upper bound: one worker does everything

    while lo < hi:
        mid = (lo + hi) // 2
        if is_feasible(mid):
            hi = mid             # mid works — try to find something smaller
        else:
            lo = mid + 1         # mid is too small — need more time

    return lo

# Example: 5 tasks, 3 workers
print(min_max_time([10, 20, 30, 40, 50], 3))  # 60

要素探索の二分探索との微妙な違いに注目してください：ここでは lo < hi（<= ではなく）を使い、lo と hi が同じ値に収束するまで続けます。これが「最大値を最小化する」二分探索の標準テンプレートです。

挿入ソート（Insertion Sort）

挿入ソート（insertion sort）は配列のソート済み部分を1要素ずつ構築します。新しい要素ごとに、より大きな要素を右にシフトしてスペースを作り、正しい位置に挿入します。

def insertion_sort(arr):
    """Sort arr in-place using insertion sort."""
    for i in range(1, len(arr)):       # Start from the second element
        key = arr[i]                   # The element to insert
        j = i - 1                      # Start comparing with the element to its left

        # Shift elements that are larger than key one position to the right
        while j >= 0 and arr[j] > key:
            arr[j + 1] = arr[j]        # Shift right
            j -= 1                     # Move comparison pointer left

        arr[j + 1] = key               # Insert key in the correct position

    return arr

print(insertion_sort([5, 2, 8, 1, 9, 3]))  # [1, 2, 3, 5, 8, 9]

計算量： 最悪ケース（逆順ソート済み入力）はO(n²)ですが、データがほぼソート済みの場合は最良ケースO(n)です。これにより、挿入ソートは小さな配列や「ほぼソート済み」の配列に優れています。これは実際には驚くほど多く見られます。PythonのTimsort自体も小さな連続区間（run）に挿入ソートを使っています。

マージソート（Merge Sort）

マージソート（merge sort）は分割統治法（divide-and-conquer）アルゴリズムです。各部分が1要素になるまで（それ自体でソート済み）再帰的に配列を半分に分割し、ソート済みの半分を結合します。入力の順序に関わらずO(n log n)の時間を保証し、安定ソート（stable sort）です。等しい要素は元の相対順序を保ちます。

def merge_sort(arr):
    """Return a new sorted list using merge sort."""
    if len(arr) <= 1:                  # Base case: 0 or 1 elements are already sorted
        return arr

    mid = len(arr) // 2                # Find the midpoint
    left = merge_sort(arr[:mid])       # Recursively sort the left half
    right = merge_sort(arr[mid:])      # Recursively sort the right half

    return merge(left, right)          # Merge the two sorted halves

def merge(left, right):
    """Merge two sorted lists into one sorted list."""
    result = []
    i = j = 0                         # Pointers into left and right

    while i < len(left) and j < len(right):
        if left[i] <= right[j]:        # <= preserves stability
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1

    # One of the halves may have remaining elements
    result.extend(left[i:])
    result.extend(right[j:])
    return result

print(merge_sort([38, 27, 43, 3, 9, 82, 10]))  # [3, 9, 10, 27, 38, 43, 82]

再帰木： 8要素の配列は2つの4要素配列に、それが4つの2要素配列に、さらに8つの1要素配列に分割されます。分割のレベル数はlog₂ nです。各レベルでマージステップがすべての要素に1回触れるため、各レベルのコストはO(n)です。合計：O(n log n)。

空間計算量（space complexity）： O(n)。マージ関数は各レベルで新しいリストを作成します。これがクイックソートのような「インプレース（in-place）」ソートとのトレードオフです。

マージソートは配列を再帰的に分割し（上半分）、ソート済みの部分配列を結合します（下半分）。各マージレベルですべての要素に1回触れるため、合計の作業量はO(n log n)です。

クイックソート（Quicksort）

クイックソート（quicksort）も分割統治法アルゴリズムですが、中点で分割する代わりに、ピボット（pivot）要素を基準に配列を分割（partition）します。ピボットより小さい要素はその左へ、大きい要素はその右へ移動します。そして左右の部分配列を再帰的にソートします。

def quicksort(arr):
    """Return a new sorted list using quicksort."""
    if len(arr) <= 1:
        return arr

    pivot = arr[len(arr) // 2]                # Choose middle element as pivot
    left = [x for x in arr if x < pivot]      # Elements smaller than pivot
    middle = [x for x in arr if x == pivot]    # Elements equal to pivot
    right = [x for x in arr if x > pivot]     # Elements larger than pivot

    return quicksort(left) + middle + quicksort(right)

print(quicksort([10, 7, 8, 9, 1, 5]))  # [1, 5, 7, 8, 9, 10]

これは「Lomutoスタイル」のリスト内包表記バージョンで、明快ですが余分なメモリを使います。インプレースのHoare分割スキームはメモリ効率が良いですが、読みにくいです：

def quicksort_inplace(arr, lo=0, hi=None):
    """Sort arr in-place using quicksort with Hoare partition."""
    if hi is None:
        hi = len(arr) - 1
    if lo >= hi:
        return

    # Partition
    pivot = arr[(lo + hi) // 2]
    i, j = lo, hi
    while i <= j:
        while arr[i] < pivot:
            i += 1
        while arr[j] > pivot:
            j -= 1
        if i <= j:
            arr[i], arr[j] = arr[j], arr[i]
            i += 1
            j -= 1

    quicksort_inplace(arr, lo, j)
    quicksort_inplace(arr, i, hi)

data = [10, 7, 8, 9, 1, 5]
quicksort_inplace(data)
print(data)  # [1, 5, 7, 8, 9, 10]

計算量： 平均ケースはO(n log n)ですが、最悪ケースはO(n²)です。ピボットが常に最小または最大の要素になる場合（例：最初の要素をピボットにするとソート済み入力がこれに該当）に発生します。これを避けるためのピボット選択戦略：

  • ランダムピボット—ランダムな要素を選びます。最悪ケースの発生確率が天文学的に低くなります。
  • 3の中央値（median-of-three）—最初、中間、最後の要素の中央値を取ります。ソート済み入力をうまく処理します。
  • 中央要素—上記で使用した方法です。シンプルで通常は効果的です。

ヒープソート（Heapsort）

ヒープソート（heapsort）は配列から最大ヒープ（max-heap）を構築し（第22章で解説）、最大要素を繰り返し取り出して末尾に配置します。O(n log n)の保証時間を達成し、インプレース（in-place）で動作します。余分な配列が不要です。

def heapsort(arr):
    """Sort arr in-place using heapsort."""
    n = len(arr)

    def sift_down(start, end):
        """Restore the max-heap property from start down to end."""
        root = start
        while 2 * root + 1 <= end:
            child = 2 * root + 1        # Left child
            if child + 1 <= end and arr[child] < arr[child + 1]:
                child += 1              # Right child is larger
            if arr[root] < arr[child]:
                arr[root], arr[child] = arr[child], arr[root]
                root = child
            else:
                return

    # Phase 1: Build a max-heap (heapify)
    for i in range(n // 2 - 1, -1, -1):
        sift_down(i, n - 1)

    # Phase 2: Extract max repeatedly
    for end in range(n - 1, 0, -1):
        arr[0], arr[end] = arr[end], arr[0]   # Move max to sorted position
        sift_down(0, end - 1)                  # Restore heap on reduced array

    return arr

data = [4, 10, 3, 5, 1]
print(heapsort(data))  # [1, 3, 4, 5, 10]

ヒープソートは余分なメモリなしでO(n log n)を保証したい場合に有用です。実際には、クイックソートはキャッシュの局所性（cache locality）が良いため速いことが多いですが、ヒープソートの最悪ケースは厳密にO(n log n)で、不運なピボットの心配がありません。

非比較ソート（Non-Comparison Sorts）

上記のソートはすべて要素のペアを比較します。比較の下限（comparison lower bound）と呼ばれる数学的な結果は、比較ベースのソートがO(n log n)より良くなれないことを証明しています。しかし、データが既知の範囲の整数で構成される場合、この壁を突破できます。

カウントソート（counting sort）は各値の出現回数を数えてから、カウントから配列を再構築します。時間計算量：O(n + k)（kは値の範囲）。

def counting_sort(arr, max_val):
    """Sort non-negative integers using counting sort. O(n + k)."""
    counts = [0] * (max_val + 1)        # One counter per possible value
    for x in arr:
        counts[x] += 1                  # Count occurrences

    result = []
    for val, count in enumerate(counts):
        result.extend([val] * count)    # Emit each value 'count' times
    return result

print(counting_sort([4, 2, 2, 8, 3, 3, 1], 8))  # [1, 2, 2, 3, 3, 4, 8]

基数ソート（radix sort）は整数を最下位桁から最上位桁へと桁ごとにソートし、各桁で安定ソート（カウントソートなど）を使います。時間計算量：O(d × (n + k))（dは桁数、kは基数（通常10））。固定サイズの整数では実質的にO(n)です。

安定性（Stability）

等しいキーを持つ要素が元の相対順序を維持するソートを安定ソート（stable sort）と言います。複数のキーでソートする際に重要です。

例：作成日でソートされたタスクがあります。今度は優先度でソートします。安定ソートを使えば、同じ優先度のタスクは作成日の順序を保ちます。不安定ソートでは、優先度レベル内での順序が予測できなくなります。

Timsort：Pythonが実際に使うアルゴリズム

Pythonで sorted() や list.sort() を呼び出すと、Timsortが使われます。これはTim Petersが2002年にPython用に発明したハイブリッドアルゴリズムです。Timsortはマージソートと挿入ソートを組み合わせ、実世界のデータに現れるパターンを活用するように設計されています：

  • 入力内の既存のソート済み「ラン」（run：昇順または降順の連続）を見つけます。
  • 短いランは挿入ソートを使って最小長まで延長されます。
  • ランはペアでマージされ、不均一なマージには「ギャロッピング」モードを使った修正マージソートが適用されます。

Timsortの最悪ケースはO(n log n)、ソート済みデータではO(n)で、安定ソートです。その効果は非常に高く、Java、Rust、Swiftも標準ライブラリに採用しています。

実践的な結論：Pythonの組み込み sorted() を使いましょう。C言語で実装されており、Timsortの実世界最適化を使うため、手書きより速いことはほぼ確実です。自分でソートを書くのは学習のため、または制約のある環境で作業する場合だけにしましょう。

エンジニアリングの視点

補足

カスタム key パラメータを使った sorted() で実際のソートニーズの99%に対応できます。タスクを優先度でソートしたいですか？ sorted(tasks, key=lambda t: t["priority"])。降順が必要ですか？ reverse=True を追加。第2ソートキーが必要ですか？タプルを返せばOK：key=lambda t: (t["priority"], t["due_date"])。

データベースの ORDER BY 句はBツリーインデックススキャンを使って実際にソートせずにソート済み結果を返します。データはすでにインデックスで整理されています。ソート済みデータを繰り返し探索する必要がある場合は、1回ソートして毎回二分探索を行いましょう：1回のO(n log n)ソートとk回のO(log n)探索は、k > log n の場合にk回のO(n)線形スキャンより速くなります。

やってみよう

TaskForge向けの複数キーソートを実装してみましょう：まず優先度でソートし（高優先度を先に）、同じ優先度内では期日（最も早いものを先に）でソートし、さらに作成日をタイブレーカーとして使います。これはタプルキーを使った sorted() の動作と全く同じです：

第25章 再帰と動的計画法

重要な概念

多くの問題には優雅な再帰的解法があります。しかし、単純な再帰は天文学的に遅くなる場合があります。Fibonacci(50)を再帰で計算すると100兆回以上の関数呼び出しが発生します。動的計画法（dynamic programming）はこれらの指数的アルゴリズムを多項式時間に変換します。AIは再帰コードをよく生成します。効率的なときと本番システムをクラッシュさせるトラップになるときを見極める必要があります。

再帰的思考（Recursive Thinking）

再帰関数（recursive function）は同じ問題の小さなバージョンで自身を呼び出します。すべての再帰には2つの部分が必要です：

  1. ベースケース（base case）—再帰を停止する条件です。これがないと関数は永遠に自身を呼び続けます。
  2. 再帰ケース（recursive case）—ベースケースに近づく小さな入力で自身を呼び出します。

最もシンプルな例は階乗（factorial）です：

def factorial(n):
    """Compute n! = n * (n-1) * ... * 1."""
    if n <= 1:             # Base case: 0! = 1! = 1
        return 1
    return n * factorial(n - 1)  # Recursive case: n! = n * (n-1)!

print(factorial(5))  # 120 = 5 * 4 * 3 * 2 * 1

実行の流れを追ってみましょう：

factorial(5)
  = 5 * factorial(4)
  = 5 * (4 * factorial(3))
  = 5 * (4 * (3 * factorial(2)))
  = 5 * (4 * (3 * (2 * factorial(1))))
  = 5 * (4 * (3 * (2 * 1)))           # Base case reached
  = 5 * (4 * (3 * 2))
  = 5 * (4 * 6)
  = 5 * 24
  = 120

次はフィボナッチ数列（Fibonacci）。各数は前の2つの和です：0, 1, 1, 2, 3, 5, 8, 13, 21, ...

def fib(n):
    """Return the nth Fibonacci number (naive recursive)."""
    if n <= 1:             # Base cases: fib(0) = 0, fib(1) = 1
        return n
    return fib(n - 1) + fib(n - 2)  # Recursive case

print(fib(10))  # 55

これは動作しますが、大きなnに対しては壊滅的に遅くなります。なぜなら、同じ部分問題を何度も繰り返し計算するからです。fib(5) は fib(3) を2回、fib(2) を3回、fib(1) を5回呼び出します。呼び出し総数は指数関数的に増加します：O(2ⁿ)。

コールスタック（Call Stack）

各再帰呼び出しはスタックフレーム（stack frame）を追加します。これは関数のローカル変数とリターンアドレスを格納するメモリブロックです。再帰が深くなるほどフレームが積み上がり、各呼び出しが返るたびにアンワインド（unwind）されます。

# Visualizing the call stack for factorial(4):
#
# Frame 4: factorial(1) -> returns 1         [TOP]
# Frame 3: factorial(2) -> waiting for frame 4
# Frame 2: factorial(3) -> waiting for frame 3
# Frame 1: factorial(4) -> waiting for frame 2  [BOTTOM]
#
# After factorial(1) returns, frames unwind from top to bottom.

Pythonはデフォルトの再帰制限（recursion limit）を1000に設定しています。確認・変更できます：

import sys
print(sys.getrecursionlimit())  # 1000
sys.setrecursionlimit(5000)     # Increase it
注意

再帰が制限に達した場合、本当の問題はアルゴリズムの深さが指数関数的になっていることであり、制限が低すぎることではありません。制限を増やすことはクラッシュを先延ばしにし、利用可能なメモリをすべて消費するリスクがあります。正しい対処法は、反復に変換するか、メモ化を追加することです。

スタックオーバーフロー（Stack Overflow）

再帰が深くなりすぎると、RecursionError（Pythonのスタックオーバーフロー）が発生します。これは次の場合に起こります：

  • ベースケースを忘れた
  • ベースケースが間違っていて一度も発動しない
  • 問題が再帰の深さに対して大きすぎる（例：100,000要素の連結リストを再帰的に処理する）

末尾再帰（tail recursion）は、再帰呼び出しが関数の最後の操作となるパターンです。一部の言語（Scheme、Scala）ではこれを自動的にループに最適化し、スタックの増加を排除します。Pythonは末尾呼び出し最適化（tail-call optimization）を行いません。 Pythonで深い再帰が必要な場合は、スタックデータ構造を使った明示的なループに変換しましょう。

# Recursive (limited by call stack):
def sum_list_recursive(lst, i=0):
    if i == len(lst):
        return 0
    return lst[i] + sum_list_recursive(lst, i + 1)

# Iterative (no stack limit):
def sum_list_iterative(lst):
    total = 0
    for x in lst:
        total += x
    return total

メモ化（Memoization）

メモ化（memoization）は高コストな関数呼び出しの結果をキャッシュし、同じ引数で再び呼ばれた場合に再計算せずにキャッシュ済みの結果を返します。これにより、O(2ⁿ)のフィボナッチがO(n)に変わります。

def fib_memo(n, cache={}):
    """Fibonacci with manual memoization."""
    if n in cache:             # Already computed? Return cached result.
        return cache[n]
    if n <= 1:
        return n
    cache[n] = fib_memo(n - 1, cache) + fib_memo(n - 2, cache)
    return cache[n]

print(fib_memo(50))  # 12586269025 — instant, not heat-death-of-the-universe

Pythonにはこれを自動的に行う組み込みデコレータがあります：

from functools import lru_cache

@lru_cache(maxsize=None)    # Cache all results (no eviction)
def fib_cached(n):
    if n <= 1:
        return n
    return fib_cached(n - 1) + fib_cached(n - 2)

print(fib_cached(100))  # 354224848179261915075

その違いは劇的です。計測してみましょう：

ボトムアップの表化（Bottom-Up Tabulation）

メモ化は「トップダウン」です。大きな問題から始めて再帰で降りていきます。表化（tabulation）は「ボトムアップ」です。ベースケースからテーブルを構築し、上向きに進みます。再帰もスタックのオーバーヘッドもありません。

def fib_table(n):
    """Fibonacci using bottom-up tabulation."""
    if n <= 1:
        return n

    table = [0] * (n + 1)     # table[i] will hold fib(i)
    table[0] = 0               # Base case
    table[1] = 1               # Base case

    for i in range(2, n + 1):
        table[i] = table[i - 1] + table[i - 2]   # Build from smaller subproblems

    return table[n]

print(fib_table(100))  # 354224848179261915075

前の2つの値だけが必要なので、空間計算量をO(1)に最適化することもできます：

def fib_optimal(n):
    """Fibonacci with O(1) space."""
    if n <= 1:
        return n
    prev2, prev1 = 0, 1
    for _ in range(2, n + 1):
        prev2, prev1 = prev1, prev2 + prev1
    return prev1

print(fib_optimal(100))  # 354224848179261915075

古典的なDPパターン

コイン問題：目標金額を作る最少枚数

問題：コインの額面（例：[1, 5, 10, 25]）と目標金額が与えられています。必要なコインの最少枚数を求めてください。

これはDPの典型的な問題です。重要な洞察：金額 a に必要な最少コイン枚数は、すべてのコイン c について金額 a - c の答えの最小値に1を足したものです。

def coin_change(coins, amount):
    """Return minimum coins to make amount, or -1 if impossible."""
    # dp[i] = minimum coins needed to make amount i
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0                          # Base case: 0 coins for amount 0

    for a in range(1, amount + 1):     # For each amount from 1 to target
        for c in coins:                # Try each coin denomination
            if c <= a:                 # Coin must not exceed the current amount
                dp[a] = min(dp[a], dp[a - c] + 1)

    return dp[amount] if dp[amount] != float('inf') else -1

print(coin_change([1, 5, 10, 25], 36))  # 3 (25 + 10 + 1)
print(coin_change([3, 7], 5))            # -1 (impossible)

計算量：O(amount × len(coins))。テーブルの各セルは1回だけ計算され、コインごとに一定の作業量を使います。

最長共通部分列（LCS：Longest Common Subsequence）

問題：2つの文字列が与えられます。最長共通部分列（LCS）の長さを求めてください。LCSとは両方の文字列に同じ順序で現れる最長の文字列（ただし連続でなくてもよい）です。

これはdiffアルゴリズム（git diff など）の基盤です。2つのファイルのLCSを見つけ、LCSに含まれないものすべてが変更です。

def lcs(s1, s2):
    """Return the length of the longest common subsequence."""
    m, n = len(s1), len(s2)
    # dp[i][j] = LCS length of s1[:i] and s2[:j]
    dp = [[0] * (n + 1) for _ in range(m + 1)]

    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s1[i - 1] == s2[j - 1]:          # Characters match
                dp[i][j] = dp[i - 1][j - 1] + 1  # Extend the LCS
            else:                                  # Characters differ
                dp[i][j] = max(dp[i - 1][j],      # Skip char from s1
                               dp[i][j - 1])      # Skip char from s2

    return dp[m][n]

print(lcs("ABCBDAB", "BDCAB"))  # 4 ("BCAB")

0/1ナップサック問題：重量制限内で価値を最大化

問題：重さと価値を持つアイテムがあります。重量制限のあるナップサックがあります。制限を超えずに合計価値を最大化してください。各アイテムは最大1回しか取れません。

def knapsack(values, weights, capacity):
    """Return maximum value achievable within capacity."""
    n = len(values)
    # dp[i][w] = max value using items 0..i-1 with capacity w
    dp = [[0] * (capacity + 1) for _ in range(n + 1)]

    for i in range(1, n + 1):
        for w in range(capacity + 1):
            dp[i][w] = dp[i - 1][w]               # Don't take item i
            if weights[i - 1] <= w:                 # Can we take item i?
                take = dp[i - 1][w - weights[i - 1]] + values[i - 1]
                dp[i][w] = max(dp[i][w], take)     # Take it if better

    return dp[n][capacity]

# Items: (value, weight)
values  = [60, 100, 120]
weights = [10,  20,  30]
capacity = 50
print(knapsack(values, weights, capacity))  # 220 (items 2 and 3)

計算量：O(n × capacity)。これは擬多項式時間（pseudo-polynomial）と呼ばれます。入力サイズだけでなく、capacityの数値にも依存するからです。

DP問題の見極め方

すべての問題がDPで恩恵を受けるわけではありません。2つの性質が成立する必要があります：

  1. 重複する部分問題（overlapping subproblems）—同じ部分問題が複数回解かれます。すべての部分問題が一意であれば（マージソートのように）、DPはオーバーヘッドを加えるだけで利益がありません。
  2. 最適部分構造（optimal substructure）—問題全体の最適解が部分問題の最適解から構築できます。サブ決定を変えても他のサブ決定に影響しない場合、その問題は最適部分構造を持ちます。

簡易テスト：再帰解を書けますか？同じ部分問題を再計算していますか？両方「はい」なら、メモ化を追加しましょう。部分問題テーブルの次元を特定できれば、ボトムアップバージョンを書きましょう。

左：素朴な再帰フィボナッチは部分問題を指数関数的に再計算します。右：メモ化バージョンは結果をキャッシュし（網掛けボックス）、O(n)のユニークな計算に削減します。

エンジニアリングの視点

補足

DPは思っているよりも多く本番システムに登場します。テキストdiffアルゴリズム（git diff、コードレビューツール）はLCSを使います。スペルチェッカーは編集距離（edit distance：1つの単語を別の単語に変換するための最少の挿入・削除・置換回数を数えるDPの変種）を使います。リソース配分システムはナップサック型の最適化を使います。ルートプランナーはDPを使って最短経路を求めます。

AIが再帰解を生成したら、最初のチェックは：部分問題を再計算していますか？ はいなら @lru_cache で包めば10,000倍の高速化が得られるかもしれません。再帰の深さが大きすぎる場合は、ボトムアップの表化に変換しましょう。

TaskForgeのタスクセットがあり、それぞれ時間コストと価値（優先度スコア）を持っています。スプリントの時間予算があります。時間予算内で合計価値を最大化するにはどのタスクを選ぶべきですか？これは0/1ナップサック問題です。

def select_tasks(tasks, time_budget):
    """Select tasks to maximize total value within time_budget.
    Each task is a dict with 'title', 'time', and 'value' keys.
    Returns list of selected task titles.
    """
    n = len(tasks)
    dp = [[0] * (time_budget + 1) for _ in range(n + 1)]

    for i in range(1, n + 1):
        t = tasks[i - 1]
        for w in range(time_budget + 1):
            dp[i][w] = dp[i - 1][w]              # Don't take this task
            if t["time"] <= w:
                take = dp[i - 1][w - t["time"]] + t["value"]
                dp[i][w] = max(dp[i][w], take)

    # Backtrack to find which tasks were selected
    selected = []
    w = time_budget
    for i in range(n, 0, -1):
        if dp[i][w] != dp[i - 1][w]:
            selected.append(tasks[i - 1]["title"])
            w -= tasks[i - 1]["time"]

    return selected

第26章 アルゴリズムパターン

重要な概念

ほとんどのアルゴリズム問題は少数のパターンに当てはまります。パターンを認識することが解決の80%です。これらは単なる面接対策のトリックではありません。実際のエンジニアリング解法の構成要素です。レートリミッターはスライディングウィンドウを使います。スケジューラーは貪欲アルゴリズムを使います。設定バリデーターはバックトラッキングを使います。パターンを見れば、どこにでも見えるようになります。

2ポインタ法（Two Pointers）

2ポインタ（two-pointer）法は配列を移動する2つのインデックスを使います。両端から互いに近づくか、同じ方向に異なる速度で移動します。多くのO(n²)のブルートフォース解をO(n)に変換できます。

パターン1：収束するポインタ（両端から）

問題：ソート済み配列と目標合計が与えられています。合計が目標と等しくなる要素のペアを見つけてください。

def two_sum_sorted(arr, target):
    """Find two numbers in sorted arr that sum to target. Return their indices."""
    left, right = 0, len(arr) - 1

    while left < right:
        current_sum = arr[left] + arr[right]
        if current_sum == target:
            return (left, right)
        elif current_sum < target:
            left += 1          # Need a bigger sum — move left pointer right
        else:
            right -= 1         # Need a smaller sum — move right pointer left

    return None                # No pair found

print(two_sum_sorted([1, 3, 5, 7, 11, 15], 16))  # (2, 4) — 5 + 11

なぜ機能するか：配列がソート済みのため、左ポインタを右に動かすと合計が増え、右ポインタを左に動かすと合計が減ります。すでに除外したペアを再チェックする必要がありません。

パターン2：回文チェック（Palindrome Check）

def is_palindrome(s):
    """Check if string is a palindrome, ignoring case and non-alphanumeric chars."""
    left, right = 0, len(s) - 1

    while left < right:
        while left < right and not s[left].isalnum():
            left += 1
        while left < right and not s[right].isalnum():
            right -= 1
        if s[left].lower() != s[right].lower():
            return False
        left += 1
        right -= 1

    return True

print(is_palindrome("A man, a plan, a canal: Panama"))  # True

パターン3：インプレースで重複を除去

def remove_duplicates(arr):
    """Remove duplicates from sorted arr in-place. Return new length."""
    if not arr:
        return 0

    write = 1                          # Position to write next unique element
    for read in range(1, len(arr)):    # Scan through the array
        if arr[read] != arr[read - 1]: # Found a new value
            arr[write] = arr[read]     # Write it
            write += 1

    return write

data = [1, 1, 2, 2, 2, 3, 4, 4, 5]
new_len = remove_duplicates(data)
print(data[:new_len])  # [1, 2, 3, 4, 5]

スライディングウィンドウ（Sliding Window）

スライディングウィンドウ（sliding window）法は「ウィンドウ」（連続する部分配列または部分文字列）を維持し、データ全体をスライドさせます。2種類あります：

固定サイズのウィンドウ：k個の連続要素の最大合計

def max_sum_k(arr, k):
    """Return the maximum sum of k consecutive elements."""
    if len(arr) < k:
        return None

    # Compute the sum of the first window
    window_sum = sum(arr[:k])
    max_sum = window_sum

    # Slide: add the next element, remove the leftmost element
    for i in range(k, len(arr)):
        window_sum += arr[i] - arr[i - k]   # Slide the window
        max_sum = max(max_sum, window_sum)

    return max_sum

print(max_sum_k([2, 1, 5, 1, 3, 2], 3))  # 9 (5 + 1 + 3)

各位置でk要素の合計を再計算する（O(n × k)）代わりに、1要素分の調整（O(1)）で済み、合計O(n)になります。

可変サイズのウィンドウ：重複文字のない最長部分文字列

def longest_unique_substring(s):
    """Return the length of the longest substring without repeating characters."""
    seen = {}                  # Map character -> its most recent index
    max_len = 0
    start = 0                  # Left edge of the window

    for end in range(len(s)):  # Right edge expands
        if s[end] in seen and seen[s[end]] >= start:
            start = seen[s[end]] + 1   # Shrink: jump past the duplicate
        seen[s[end]] = end
        max_len = max(max_len, end - start + 1)

    return max_len

print(longest_unique_substring("abcabcbb"))  # 3 ("abc")
print(longest_unique_substring("pwwkew"))    # 3 ("wke")

def min_subarray_sum(arr, target):
    """Return the length of the smallest subarray with sum >= target, or 0."""
    min_len = float('inf')
    window_sum = 0
    start = 0

    for end in range(len(arr)):
        window_sum += arr[end]             # Expand window

        while window_sum >= target:        # Shrink while condition holds
            min_len = min(min_len, end - start + 1)
            window_sum -= arr[start]
            start += 1

    return min_len if min_len != float('inf') else 0

貪欲法（Greedy）

貪欲法（greedy）アルゴリズムは各ステップで局所最適な選択をし、大局的な最適解を目指します。問題が貪欲選択特性（greedy choice property）を持つ場合に機能します。局所最適な選択が大局的最適解に繋がる性質です。

貪欲法が機能する場合：区間スケジューリング

問題：区間（start, end）のセットが与えられます。重複しない区間の最大数を求めてください。

戦略：常に最も早く終わる区間を選びます。これにより後の区間のための余地が最大になります。

def max_non_overlapping(intervals):
    """Return max number of non-overlapping intervals.
    Each interval is (start, end).
    """
    # Sort by end time
    intervals.sort(key=lambda x: x[1])

    count = 0
    last_end = float('-inf')

    for start, end in intervals:
        if start >= last_end:      # No overlap with previous selection
            count += 1
            last_end = end         # Update the end boundary

    return count

meetings = [(1, 3), (2, 5), (3, 6), (5, 7), (8, 10), (6, 9)]

分割統治法（Divide and Conquer）

分割統治法（divide and conquer）は問題を独立した部分問題に分割し、各々を再帰的に解き、結果を結合します。マージソート（第24章）はすでに知っています。このパターンはソートを超えて適用されます：

  • 最近点対問題（closest pair of points）—点をx座標で分割し、各半分で最近点対を見つけ、分割線をまたぐ点対を確認します。
  • 転倒数の計算（counting inversions）—順序が逆になったペアの数を数える修正マージソートです。
  • Karatsuba乗算—大きな数を半分に分割して乗算し、O(n²)からO(n¹·⁵⁸)に削減します。

DPとの重要な違い：分割統治法の部分問題は独立しています（重複なし）。DPの部分問題は重複します。

バックトラッキング（Backtracking）

バックトラッキング（backtracking）はすべての可能な選択肢を探索しますが、有効な解に繋がりえない枝を刈り込みます（枝刈り：pruning）。行き止まりに時間を無駄にせずすべてを試す体系的な方法です。

パターン：選択を行い、再帰し、選択を元に戻し（バックトラック）、次の選択肢を試す。

例：全部分集合の生成

def subsets(nums):
    """Generate all subsets of nums."""
    result = []

    def backtrack(start, current):
        result.append(current[:])          # Add a copy of the current subset

        for i in range(start, len(nums)):
            current.append(nums[i])        # Choose
            backtrack(i + 1, current)      # Explore
            current.pop()                  # Un-choose (backtrack)

    backtrack(0, [])
    return result

print(subsets([1, 2, 3]))
# [[], [1], [1,2], [1,2,3], [1,3], [2], [2,3], [3]]

def solve_n_queens(n):
    """Return the number of ways to place n queens on an n x n board
    so that no two queens attack each other."""
    solutions = 0
    cols = set()          # Columns with queens
    diag1 = set()         # "row - col" diagonals with queens
    diag2 = set()         # "row + col" diagonals with queens

    def backtrack(row):
        nonlocal solutions
        if row == n:               # All queens placed successfully
            solutions += 1
            return

        for col in range(n):
            if col in cols or (row - col) in diag1 or (row + col) in diag2:
                continue           # Prune: this position is attacked

            # Place queen
            cols.add(col)
            diag1.add(row - col)
            diag2.add(row + col)

            backtrack(row + 1)     # Try next row

            # Remove queen (backtrack)
            cols.remove(col)
            diag1.remove(row - col)
            diag2.remove(row + col)

    backtrack(0)
    return solutions

print(solve_n_queens(8))  # 92 solutions

重要な概念

グラフは関係をモデル化します：道路で繋がる都市、友人関係で繋がるユーザー、依存関係で繋がるタスク。グラフアルゴリズムは実際の問題を解決します：最短ルート、依存関係の順序付け、ネットワーク接続性。この章では中核のアルゴリズムと適切なものを選ぶためのフレームワークを提供します。

幅優先探索（BFS：Breadth-First Search）

幅優先探索（breadth-first search）はグラフをレベルごとに探索し、次のレベルに移る前に現在のノードの隣接ノードをすべて訪問します。第20章で学んだキュー（queue：FIFO）データ構造を使います。BFSは始点からの距離順にノードを発見するため、重みなしグラフの最短経路を見つけます。

from collections import deque

def bfs(graph, start):
    """Perform BFS from start node. Return dict of {node: distance}.
    graph: dict of {node: [neighbors]}
    """
    visited = {start: 0}              # node -> distance from start
    queue = deque([start])

    while queue:
        node = queue.popleft()         # Dequeue the front node
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited[neighbor] = visited[node] + 1
                queue.append(neighbor) # Enqueue unvisited neighbor

    return visited

# Example graph:
#   A -- B -- D
#   |         |
#   C -- E -- F
graph = {
    'A': ['B', 'C'],
    'B': ['A', 'D'],
    'C': ['A', 'E'],
    'D': ['B', 'F'],
    'E': ['C', 'F'],
    'F': ['D', 'E'],
}
distances = bfs(graph, 'A')
print(distances)
# {'A': 0, 'B': 1, 'C': 1, 'D': 2, 'E': 2, 'F': 3}

計算量：O(V + E)（Vは頂点数、Eは辺数）。各ノードは1回訪問され、各辺は1回調べられます。

BFS最短経路（経路の再構築）

from collections import deque

def bfs_path(graph, start, end):
    """Return the shortest path from start to end, or None."""
    if start == end:
        return [start]

    visited = {start}
    queue = deque([(start, [start])])  # (node, path_so_far)

    while queue:
        node, path = queue.popleft()
        for neighbor in graph[node]:
            if neighbor == end:
                return path + [neighbor]
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append((neighbor, path + [neighbor]))

    return None  # No path exists

print(bfs_path(graph, 'A', 'F'))  # ['A', 'B', 'D', 'F'] or ['A', 'C', 'E', 'F']

深さ優先探索（DFS：Depth-First Search）

深さ優先探索（depth-first search）はバックトラックする前に各枝を可能な限り深く探索します。スタック（stack：LIFO）を使います。コールスタック（再帰経由）または明示的なスタックデータ構造のどちらかです。

再帰DFS

def dfs_recursive(graph, node, visited=None):
    """Perform DFS from node. Return list of visited nodes in order."""
    if visited is None:
        visited = []
    visited.append(node)

    for neighbor in graph[node]:
        if neighbor not in visited:
            dfs_recursive(graph, neighbor, visited)

    return visited

print(dfs_recursive(graph, 'A'))  # ['A', 'B', 'D', 'F', 'E', 'C']

反復DFS

def dfs_iterative(graph, start):
    """Perform DFS from start using an explicit stack."""
    visited = []
    stack = [start]

    while stack:
        node = stack.pop()
        if node not in visited:
            visited.append(node)
            # Add neighbors in reverse to visit them in order
            for neighbor in reversed(graph[node]):
                if neighbor not in visited:
                    stack.append(neighbor)

    return visited

print(dfs_iterative(graph, 'A'))  # ['A', 'B', 'D', 'F', 'E', 'C']

DFSの応用：

  • 閉路検出（cycle detection）—有向グラフのDFS中に、現在の再帰スタックにあるノード（単に訪問済みではなく）に遭遇した場合、閉路があります。
  • トポロジカルソート（topological sort）—後述します。
  • 連結成分（connected components）—未訪問のノードから各DFSを実行します。各DFSが1つの成分を発見します。
  • 経路探索（path finding）—DFSはある経路を見つけますが、必ずしも最短ではありません。
BFS（左）はノードをレベルごとに訪問します。距離1のノードをすべて訪問してから距離2のノードへ進みます。DFS（右）はバックトラックする前に1つの枝を深く進みます。数字は訪問順序を示します。

ダイクストラ法（Dijkstra's Algorithm）

ダイクストラ法（Dijkstra's algorithm）は非負の重みを持つ重み付きグラフで最短経路を見つけます。第22章の最小ヒープを使った優先度キュー（priority queue）を利用して、常に最も近い未訪問ノードを最初に処理します。

アルゴリズムのステップ：

  1. 始点ノードへの距離を0、その他すべての距離を無限大に設定します。
  2. 始点ノードを距離0で優先度キューに追加します。
  3. 優先度キューが空でない間：
    1. 最小距離のノードを取り出します。
    2. 各隣接ノードについて、現在のノードを経由した距離を計算します。
    3. この距離が隣接ノードの現在の距離より短い場合、更新してキューに追加します。
import heapq

def dijkstra(graph, start):
    """Find shortest distances from start to all nodes.
    graph: dict of {node: [(neighbor, weight), ...]}
    Returns: dict of {node: shortest_distance}
    """
    distances = {node: float('inf') for node in graph}
    distances[start] = 0
    pq = [(0, start)]             # (distance, node)
    visited = set()

    while pq:
        dist, node = heapq.heappop(pq)

        if node in visited:        # Already processed with a shorter distance
            continue
        visited.add(node)

        for neighbor, weight in graph[node]:
            new_dist = dist + weight
            if new_dist < distances[neighbor]:
                distances[neighbor] = new_dist
                heapq.heappush(pq, (new_dist, neighbor))

    return distances

# Weighted graph:
#   A --4-- B --2-- D
#   |       |       |
#   1       3       1
#   |       |       |
#   C --5-- E --1-- F
weighted = {
    'A': [('B', 4), ('C', 1)],
    'B': [('A', 4), ('D', 2), ('E', 3)],
    'C': [('A', 1), ('E', 5)],
    'D': [('B', 2), ('F', 1)],
    'E': [('B', 3), ('C', 5), ('F', 1)],
    'F': [('D', 1), ('E', 1)],

import heapq

def dijkstra_path(graph, start, end):
    """Find the shortest path and its distance from start to end."""
    distances = {node: float('inf') for node in graph}
    distances[start] = 0
    previous = {node: None for node in graph}  # Track the path
    pq = [(0, start)]
    visited = set()

    while pq:
        dist, node = heapq.heappop(pq)
        if node in visited:
            continue
        visited.add(node)

        if node == end:
            break

        for neighbor, weight in graph[node]:
            new_dist = dist + weight
            if new_dist < distances[neighbor]:
                distances[neighbor] = new_dist
                previous[neighbor] = node
                heapq.heappush(pq, (new_dist, neighbor))

    # Reconstruct path
    path = []
    current = end
    while current is not None:
        path.append(current)
        current = previous[current]
    path.reverse()

    return path, distances[end]

path, dist = dijkstra_path(weighted, 'A', 'F')
print(f"Path: {path}, Distance: {dist}")
# Path: ['A', 'B', 'D', 'F'], Distance: 7

トポロジカルソート（Topological Sort）

トポロジカルソート（topological sort）は有向非閉路グラフ（DAG：Directed Acyclic Graph）のノードを線形順序で並べます。すべての辺 u → v について、uがvの前に現れます。依存関係を解決するアルゴリズムです：ビルドシステム、パッケージマネージャ、科目の前提条件、タスクスケジューリング。

カーンのアルゴリズム（BFSベース：Kahn's Algorithm）

カーンのアルゴリズムは入次数（in-degree）の追跡を使います。入来辺がないノード（入次数0）から始め、処理し、その隣接ノードの入次数を減らし、繰り返します。

from collections import deque

def topological_sort_kahn(graph):
    """Return a topological ordering of the DAG, or None if a cycle exists.
    graph: dict of {node: [dependents]} (edges point from dependency to dependent)
    """
    # Step 1: Compute in-degrees
    in_degree = {node: 0 for node in graph}
    for node in graph:
        for neighbor in graph[node]:
            in_degree[neighbor] = in_degree.get(neighbor, 0) + 1

    # Step 2: Start with all nodes that have in-degree 0
    queue = deque([node for node in graph if in_degree[node] == 0])
    result = []

    # Step 3: Process nodes
    while queue:
        node = queue.popleft()
        result.append(node)

        for neighbor in graph[node]:
            in_degree[neighbor] -= 1       # One dependency satisfied
            if in_degree[neighbor] == 0:   # All dependencies satisfied
                queue.append(neighbor)

    # Step 4: Check for cycles
    if len(result) != len(graph):
        return None  # Cycle detected! Not all nodes were processed.

    return result

#   "setup" -> "backend" -> "deploy"
#   "setup" -> "frontend" -> "deploy"
#   "backend" -> "tests"
deps = {
    "setup":    ["backend", "frontend"],
    "backend":  ["tests", "deploy"],
    "frontend": ["deploy"],
    "tests":    [],
    "deploy":   [],
}
print(topological_sort_kahn(deps))

def topological_sort_dfs(graph):
    """Topological sort using DFS (post-order reverse)."""
    visited = set()
    stack = []         # Will contain the reverse post-order
    in_progress = set()  # For cycle detection

    def dfs(node):
        if node in in_progress:
            return False           # Cycle detected
        if node in visited:
            return True

        in_progress.add(node)
        for neighbor in graph[node]:
            if not dfs(neighbor):
                return False
        in_progress.remove(node)
        visited.add(node)
        stack.append(node)         # Post-order: add after all descendants
        return True

    for node in graph:
        if node not in visited:
            if not dfs(node):
                return None        # Cycle

    stack.reverse()                # Reverse post-order = topological order
    return stack

print(topological_sort_dfs(deps))
# ['setup', 'frontend', 'backend', 'tests', 'deploy']

計算量：両アルゴリズムともO(V + E)で動作します。

Union-Find（素集合データ構造：Disjoint Set）

Union-Find（素集合データ構造）は連結成分を効率よく追跡します。2つの操作をサポートします：

  • find(x)—xの成分のルート代表を返します。
  • union(x, y)—xとyを含む成分を結合します。

経路圧縮（path compression：find中に木を平坦化）とランクによる結合（union by rank：小さい木を大きい木の下に付ける）により、両操作は実質的に定数時間で動作します：O(α(n))の償却時間（αはアッカーマン関数の逆関数）、実際上はすべての目的においてO(1)です。

class UnionFind:
    """Disjoint Set with path compression and union by rank."""

    def __init__(self, n):
        self.parent = list(range(n))   # Each node is its own parent initially
        self.rank = [0] * n            # Rank (approximate tree height)
        self.components = n            # Number of distinct components

    def find(self, x):
        """Return the root of x's component (with path compression)."""
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])  # Flatten the tree
        return self.parent[x]

    def union(self, x, y):
        """Merge the components of x and y. Return True if they were separate."""
        rx, ry = self.find(x), self.find(y)
        if rx == ry:
            return False               # Already in the same component

        # Attach smaller tree under larger tree
        if self.rank[rx] < self.rank[ry]:
            self.parent[rx] = ry
        elif self.rank[rx] > self.rank[ry]:
            self.parent[ry] = rx
        else:
            self.parent[ry] = rx
            self.rank[rx] += 1

        self.components -= 1
        return True

    def connected(self, x, y):
        """Check if x and y are in the same component."""
        return self.find(x) == self.find(y)

# Example: network of 6 servers
uf = UnionFind(6)
uf.union(0, 1)   # Connect server 0 and 1
uf.union(2, 3)   # Connect server 2 and 3
uf.union(1, 3)   # Connect the two clusters

print(uf.connected(0, 3))  # True — all four are connected
print(uf.connected(0, 4))  # False — server 4 is isolated
print(uf.components)        # 3 (cluster {0,1,2,3}, isolated 4, isolated 5)

グラフ問題に直面したとき、この体系的なアプローチを使いましょう：

    • 重みなしグラフの最短経路 → BFS

from collections import deque

def resolve_task_order(tasks):
    """Given a dict of {task: [dependencies]}, return a valid execution order.
    Raises ValueError if there is a circular dependency.
    """
    # Build the forward graph and compute in-degrees
    graph = {t: [] for t in tasks}
    in_degree = {t: 0 for t in tasks}

    for task, deps in tasks.items():
        for dep in deps:
            graph[dep].append(task)    # dep must come before task
            in_degree[task] += 1

    # Kahn's algorithm
    queue = deque([t for t in tasks if in_degree[t] == 0])
    order = []

    while queue:
        task = queue.popleft()
        order.append(task)
        for dependent in graph[task]:
            in_degree[dependent] -= 1
            if in_degree[dependent] == 0:
                queue.append(dependent)

    if len(order) != len(tasks):
        raise ValueError("Circular dependency detected! Cannot determine order.")

    return order

# TaskForge sprint tasks with dependencies
sprint = {
    "Set up database":    [],
    "Build API":          ["Set up database"],
    "Build frontend":     ["Build API"],
    "Write tests":        ["Build API"],
    "Deploy":             ["Build frontend", "Write tests"],
}

print(resolve_task_order(sprint))
# ['Set up database', 'Build API', 'Build frontend', 'Write tests', 'Deploy']

# Detect circular dependency
try:
    circular = {
        "A": ["B"],
        "B": ["C"],
        "C": ["A"],  # Cycle!
    }
    resolve_task_order(circular)
except ValueError as e:
    print(e)  # "Circular dependency detected! Cannot determine order."
