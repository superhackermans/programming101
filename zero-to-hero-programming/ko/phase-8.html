Chapter 34 시스템 설계 기초 — 확장을 위한 구축

지금 이것이 중요한 이유

지금까지 작성한 모든 프로그램은 단일 머신에서 단일 사용자를 대상으로 실행됩니다. 학습에는 충분하지만, 실제 소프트웨어는 수천 명 — 때로는 수백만 명 — 의 사용자를 동시에 서비스합니다. 시스템 설계(system design)는 수요가 증가함에 따라 애플리케이션이 빠르고 안정적이며 비용 효율적으로 유지되도록 서버, 데이터베이스, 캐시, 큐를 어떻게 배치할지 선택하는 분야입니다. 이 결정들은 엔지니어가 내리는 가장 중요한 결정 중 하나입니다. 잘못된 아키텍처 선택은 수개월의 재작업을 초래할 수 있는 반면, 좋은 선택은 제품을 수년간 지탱할 수 있습니다.

확장 사고: 수직 확장 vs. 수평 확장

애플리케이션이 부하 아래에서 느려지기 시작하면 더 빠르게 만들기 위한 두 가지 기본적인 선택지가 있습니다. 수직 확장(vertical scaling)(또는 "스케일 업"이라고도 함)은 기존 서버에 더 많은 자원 — 더 많은 CPU 코어, 더 많은 RAM, 더 빠른 디스크 — 을 부여하는 것입니다. 가장 단순한 전략으로, 코드를 전혀 변경하지 않고 더 큰 머신으로 옮기기만 하면 됩니다. 문제는 수직 확장에 물리적 한계가 있다는 것입니다. 임대할 수 있는 가장 큰 단일 서버는 시간당 수천 달러에 달하고, 그것조차 결국 감당할 수 없는 수준이 됩니다.

수평 확장(horizontal scaling)(또는 "스케일 아웃"이라고도 함)은 애플리케이션을 여러 머신에서 실행하고 트래픽을 분산시키는 것입니다. 이것이 모든 주요 웹 애플리케이션이 작동하는 방식입니다. 이론적 한계가 없어 머신을 계속 추가할 수 있습니다. 하지만 수평 확장은 수직 확장이 피할 수 있는 복잡성을 도입합니다. 트래픽을 어떻게 분산하나요? 머신들이 데이터를 어떻게 공유하나요? 한 머신이 충돌하면 어떻게 되나요?

수평 확장을 가능하게 하는 핵심 개념은 무상태성(statelessness)입니다. 무상태 서비스(stateless service)는 요청 사이에 사용자별 데이터를 저장하지 않습니다. 모든 요청은 처리에 필요한 모든 정보를 담아 전달됩니다 (주로 HTTP 헤더의 토큰을 통해). 단일 서버가 아무것도 "기억"하지 않기 때문에 어느 서버든 어느 요청이나 처리할 수 있고, 서버를 자유롭게 추가하거나 제거할 수 있습니다.

반면 상태 보존 서비스(stateful service)는 요청 사이에 메모리에 데이터를 유지합니다 — 예를 들어 플레이어 위치를 추적하는 게임 서버가 그렇습니다. 상태 보존 서비스는 각 사용자를 동일한 서버로 라우팅하거나("스티키 세션"이라고 함) 머신 간에 상태를 복제해야 하므로 확장이 더 어렵고 비용이 많이 들며 오류가 발생하기 쉽습니다.

“가능한 모든 곳에서 무상태로 만드십시오. 상태는 이를 관리하도록 설계된 데이터베이스와 캐시에 위임하십시오.”

로드 밸런싱(Load Balancing)

여러 서버를 보유하게 되면 들어오는 요청을 서버들에게 분배할 무언가가 필요합니다. 그것이 바로 로드 밸런서(load balancer)입니다 — 사용자와 서버 사이에 위치하는 특수 구성 요소입니다. 사용자는 로드 밸런서에 연결하고, 로드 밸런서는 각 요청을 애플리케이션 서버 중 하나로 전달합니다.

로드 밸런서가 각 요청을 어떤 서버에 보낼지 결정하는 데 사용할 수 있는 여러 전략이 있습니다:

최소 연결(Least Connections)현재 활성 연결이 가장 적은 서버로 각 요청을 전송요청 처리 시간이 다양한 경우

주의: “로드 밸런서는 모든 단일 장애 지점을 제거한다”

로드 밸런서 자체가 단일 장애 지점입니다. 프로덕션에서는 로드 밸런서를 쌍으로 운영합니다 (액티브/스탠바이). 액티브 로드 밸런서가 다운되면 스탠바이로 전환되는 플로팅 IP 주소를 사용합니다. 항상 자문하십시오: “이 구성 요소가 실패하면 어떻게 되는가?”

캐싱(Caching): 가장 빠른 요청은 처음부터 만들지 않는 것

캐싱(caching)은 비용이 많이 드는 작업의 결과를 저장하여 동일한 데이터에 대한 향후 요청을 즉시 처리할 수 있도록 합니다. 잘 배치된 캐시는 200ms의 데이터베이스 쿼리를 1ms의 메모리 조회로 바꿀 수 있습니다. 캐싱은 여러 레이어에 존재합니다:

  • 브라우저 캐시(browser cache) — 사용자의 브라우저가 정적 파일(CSS, JS, 이미지)을 로컬에 저장합니다. Cache-Control 같은 HTTP 헤더로 제어됩니다.
  • CDN 캐시(CDN cache) — 콘텐츠 전달 네트워크(Content Delivery Network)가 전 세계 엣지 위치에 정적 자산의 복사본을 캐시하여 먼 사용자의 지연 시간을 줄입니다.
  • 애플리케이션 캐시(application cache) — 서버가 자주 접근하는 데이터를 인메모리 저장소(Redis나 Memcached 등)에 유지하여 매번 데이터베이스를 조회하지 않습니다.
  • 데이터베이스 쿼리 캐시(database query cache) — 데이터베이스 자체가 최근 쿼리 결과를 캐시합니다.

캐싱의 어려운 문제는 캐시 무효화(cache invalidation)입니다 — 캐시된 데이터가 오래되어 갱신이 필요한 시점을 아는 것입니다. 세 가지 주요 전략이 있습니다:

캐시 무효화 전략과 트레이드오프
전략작동 방식트레이드오프
TTL (Time-To-Live, 생존 시간)캐시된 데이터가 고정된 시간(예: 60초) 후 만료됨단순하지만 TTL 기간 동안 데이터가 오래될 수 있음

특히 까다로운 문제는 캐시 스탬피드(cache stampede)(또는 "썬더링 허드(thundering herd)"라고도 함)입니다. 인기 있는 캐시 항목이 만료되고 10,000개의 요청이 동시에 해당 데이터를 요청한다고 상상해 보십시오. 10,000개 모두 데이터베이스를 한 번에 조회하여 데이터베이스를 압도할 수 있습니다. 해결책으로는 잠금(하나의 요청만 캐시를 재생성하고 다른 요청은 대기), 확률적 조기 만료(TTL이 실제로 만료되기 전에 무작위로 갱신), stale-while-revalidate(백그라운드에서 재생성하는 동안 오래된 데이터를 제공)가 있습니다.

콘텐츠 전달 네트워크(CDN)

CDN(Content Delivery Network, 콘텐츠 전달 네트워크)은 많은 지리적 위치에 분산된 서버 네트워크입니다("엣지 위치" 또는 "거점"). 도쿄의 사용자가 버지니아에 호스팅된 애플리케이션에서 이미지를 요청하면, CDN은 대신 도쿄의 서버에서 해당 이미지를 제공합니다. 결과: 정적 자산의 지연 시간이 극적으로 감소합니다.

CDN은 자주 변경되지 않는 콘텐츠 — 이미지, CSS, JavaScript 번들, 폰트, 동영상 — 에 가장 효과적입니다. 일부 CDN은 동적 콘텐츠의 지연 시간을 더욱 낮추기 위해 엣지 위치에서 애플리케이션 로직의 일부를 실행하는 "엣지 컴퓨트(edge compute)"도 지원합니다.

메시지 큐(Message Queue): 작업 분리

모든 작업을 사용자에게 응답하기 전에 완료해야 하는 것은 아닙니다. 사용자가 가입하면 즉시 계정을 생성해야 하지만 — 환영 이메일은 몇 초 후에 보내도 됩니다. 메시지 큐(message queue)를 사용하면 이러한 작업을 분리할 수 있습니다. 애플리케이션 서버는 큐에 메시지("user@example.com에게 환영 이메일 발송")를 넣고 즉시 사용자에게 응답합니다. 별도의 워커(worker) 프로세스가 메시지를 가져가 이메일을 비동기적으로 발송합니다.

이 패턴 — 프로듀서(producer)가 메시지를 쓰고 컨슈머(consumer)가 읽는 — 은 여러 이점을 제공합니다: API가 느린 작업을 기다리지 않으므로 빠른 상태를 유지하고; 프로듀서와 컨슈머가 독립적으로 확장할 수 있으며; 컨슈머가 충돌하면 다른 컨슈머가 가져갈 때까지 메시지가 큐에 남아 있습니다.

이해해야 할 두 가지 중요한 전달 보장:

  • 최소 한 번 전달(at-least-once delivery) — 큐는 모든 메시지가 전달됨을 보장하지만, 일부 메시지는 두 번 이상 전달될 수 있습니다 (컨슈머가 처리 후 확인하기 전에 충돌할 경우). 컨슈머는 멱등성(idempotent)을 가져야 합니다 — 동일한 메시지를 두 번 처리해도 같은 결과가 나와야 합니다.
  • 정확히 한 번 전달(exactly-once delivery) — 큐는 각 메시지가 정확히 한 번 전달되고 처리됨을 보장합니다. 이는 구현이 훨씬 어렵고 대부분의 시스템은 멱등성 있는 컨슈머와 함께 최소 한 번 전달로 타협합니다.

여러 번 재시도 후에도 메시지를 처리할 수 없을 때(잘못된 형식의 데이터 때문일 수 있음), 해당 메시지는 데드 레터 큐(dead letter queue, DLQ)로 이동합니다 — 엔지니어가 실패한 메시지를 검사하고 처리 방법을 결정할 수 있는 보관 영역입니다.

속도 제한(Rate Limiting): 시스템 보호

속도 제한(rate limiting)은 클라이언트가 시간 창 내에 만들 수 있는 요청 수를 제한합니다. API는 남용(우발적 또는 악의적)을 방지하고, 클라이언트 간 공정한 사용을 보장하며, 백엔드 시스템이 압도되지 않도록 속도 제한이 필요합니다.

토큰 버킷 알고리즘(token bucket algorithm)이 가장 일반적인 구현입니다. 고정된 수의 토큰(예: 100개)을 담는 버킷을 상상해 보십시오. 각 요청은 토큰 하나를 소모합니다. 일정한 속도(예: 초당 10개)로 버킷에 토큰이 추가됩니다. 버킷이 비어 있으면 요청이 거부됩니다 (일반적으로 HTTP 429 "Too Many Requests"로). 이는 장기적인 평균 속도를 유지하면서 단기적인 트래픽 급증을 허용합니다.

슬라이딩 윈도우(sliding window) 알고리즘은 고정 간격 대신 롤링 시간 창(예: 최근 60초)에서 요청을 카운트합니다. 윈도우 경계에서 의도한 속도의 최대 2배를 허용할 수 있는 고정 창 카운팅보다 더 정밀합니다.

모놀리스 vs. 마이크로서비스: 솔직한 비교

모놀리스(monolith)는 애플리케이션의 모든 기능을 포함하는 단일 배포 가능 단위입니다. 모든 코드가 하나의 저장소, 하나의 프로세스, 하나의 배포 파이프라인에 있습니다. 모놀리스는 나쁜 평판을 얻고 있지만 진정한 장점이 있습니다:

  • 개발이 더 단순 — 모듈 간 함수 호출이 빠르고 컴파일 시 타입 체크됨.
  • 배포가 더 단순 — 하나의 빌드, 하나의 아티팩트, 하나의 배포.
  • 디버깅이 더 단순 — 스택 트레이스가 전체 이야기를 알려줌; 추적할 네트워크 홉 없음.
  • 테스트가 더 단순 — 통합 테스트가 네트워크 호출을 모킹(mocking)하지 않고 단일 프로세스에서 실행됨.
  • 트랜잭션 일관성 — 단일 데이터베이스 트랜잭션이 여러 테이블을 원자적으로 업데이트할 수 있음.

마이크로서비스(microservices)는 애플리케이션을 각각 하나의 비즈니스 기능을 담당하는 여러 개의 작고 독립적으로 배포 가능한 서비스로 분리합니다. 각 서비스는 자체 데이터베이스, 자체 배포 파이프라인, 자체 팀을 가집니다. 장점은 다음과 같습니다:

  • 독립적인 확장 — 검색 서비스가 청구 서비스보다 100배 더 많은 트래픽을 받음; 독립적으로 확장 가능.
  • 독립적인 배포 — 결제 서비스에 위험을 주지 않고 알림 서비스 배포 가능.
  • 기술 유연성 — 각 서비스가 필요에 가장 적합한 언어와 데이터베이스 사용 가능.
  • 팀 자율성 — 소규모 팀이 소규모 서비스를 엔드투엔드로 소유.

하지만 마이크로서비스는 엄청난 복잡성을 도입합니다:

  • 네트워크는 신뢰할 수 없음 — 서비스 간 호출이 실패하거나, 타임아웃되거나, 잘못된 데이터를 반환할 수 있음. 이 모든 것을 처리해야 함.
  • 분산 데이터 일관성 — 서비스 간에 단일 데이터베이스 트랜잭션을 사용할 수 없음. 사가(saga), 최종 일관성, 또는 다른 패턴이 필요함 (아래에서 다룸).
  • 운영 오버헤드 — 20개의 서비스를 모니터링, 배포, 디버깅하는 것은 하나를 모니터링하는 것보다 근본적으로 어려움.
  • 통합 테스트 — 서비스 간 상호작용 테스트에는 비용이 많이 드는 엔드투엔드 테스트나 계약 테스트가 필요함.
“잘 구조화된 모놀리스를 구축할 수 없다면, 마이크로서비스가 답이라고 생각하는 이유는 무엇입니까?” — Simon Brown

언제 분리할까: 모놀리스로 시작하십시오. 명확한 이유가 있을 때만 서비스로 분리하십시오 — 모듈에 독립적인 확장이 필요하거나, 팀이 배포 자율성이 필요하거나, 기술 제약이 다른 런타임을 요구하는 경우. 기술 레이어가 아닌 자연스러운 비즈니스 경계(사용자, 주문, 알림)를 따라 분리하십시오.

CAP 정리(CAP Theorem)

CAP 정리(CAP theorem)는 네트워크 파티션이 발생했을 때 분산 시스템은 일관성(Consistency) 또는 가용성(Availability) 중 하나만 제공할 수 있으며, 둘 다는 불가능하다고 말합니다. 정확하게 정의하면:

  • 일관성(C, Consistency) — 모든 읽기는 가장 최근의 쓰기를 반환합니다. 모든 노드가 동시에 동일한 데이터를 봅니다.
  • 가용성(A, Availability) — 일부 노드가 다운되더라도 모든 요청이 응답(오류 없음)을 받습니다.
  • 파티션 허용성(P, Partition Tolerance) — 일부 노드 간의 네트워크 통신이 끊어져도 시스템이 계속 작동합니다.

모든 분산 시스템에서 네트워크 파티션은 반드시 발생합니다 (케이블이 끊어지고, 스위치가 실패하고, 데이터 센터의 전원이 나갑니다). 따라서 파티션 허용성은 선택사항이 아닙니다 — 항상 P가 필요합니다. 진정한 선택은 파티션 발생 시 C와 A 사이입니다:

선택파티션 중 동작예시
CP (일관성 + 파티션 허용성)오래된 데이터를 가질 수 있는 노드로의 요청 거부. 일부 요청은 실패하지만 오래된 읽기는 없음.뱅킹 시스템 — 잘못된 잔액을 절대 보여서는 안 됨

CAP는 파티션 중 동작에 관한 것임에 주의하십시오. 네트워크가 정상적일 때는 세 가지 모두를 가질 수 있습니다. 이 정리는 문제가 발생했을 때 사용자에게 가장 중요한 것이 무엇인지 생각하도록 강요합니다.

이벤트 기반 아키텍처(Event-Driven Architecture)

전통적인 아키텍처에서 서비스는 서로 직접 호출하여 통신합니다: 서비스 A가 서비스 B에게 요청을 보내고 응답을 기다립니다. 이벤트 기반 아키텍처(event-driven architecture)에서 서비스는 이벤트를 발행하고 구독하여 통신합니다. 중요한 일이 발생하면 서비스는 이벤트("태스크 생성됨", "결제 성공")를 발행하고, 관심 있는 서비스가 이에 반응할 수 있습니다.

이벤트(event)와 커맨드(command)의 구분이 중요합니다. 커맨드는 "이것을 하라"고 말합니다 (이메일 발송, 카드 청구) — 특정 서비스를 향합니다. 이벤트는 "이것이 발생했다"고 말합니다 (주문 접수, 사용자 가입) — 관심 있는 누구에게나 브로드캐스트됩니다. 발행자가 누가 듣고 있는지 알지도 신경 쓰지도 않기 때문에 이벤트는 더 느슨한 결합(loose coupling)을 만듭니다.

이벤트 소싱(event sourcing)은 이를 더 발전시킵니다: 엔티티의 현재 상태를 저장하는 대신, 해당 상태를 만들어낸 이벤트의 시퀀스를 저장합니다. 은행 계좌는 "잔액: $500"이 아닙니다 — 완전한 역사입니다: $1000으로 개설, $200 출금, $100 입금, $400 출금. 이벤트를 재생하여 항상 현재 상태를 재구성할 수 있고, 완전한 감사 추적이 무료로 제공됩니다.

CQRS(Command Query Responsibility Segregation, 커맨드 쿼리 책임 분리)는 읽기 모델을 쓰기 모델에서 분리합니다. 쓰기는 커맨드 측을 통해 이루어지는 반면(이벤트 소싱을 사용할 수 있음), 읽기는 별도의 최적화된 읽기 모델(비정규화된 데이터베이스 뷰일 수 있음)에서 제공됩니다. 이를 통해 읽기와 쓰기를 독립적으로 최적화할 수 있습니다.

대규모 API 설계

여러 서비스와 팀이 API를 공유할 때, 사소해 보이는 설계 결정이 중요해집니다. 주요 고려 사항은 다음과 같습니다:

버전 관리(versioning): API는 변경됩니다. 기존 클라이언트를 깨지 않고 발전시키는 전략이 필요합니다.

# URL path versioning (most common, most visible)
GET /api/v1/tasks
GET /api/v2/tasks

# Header versioning (cleaner URLs, harder to test in browser)
GET /api/tasks
Accept: application/vnd.taskforge.v2+json

# Query parameter versioning
GET /api/tasks?version=2

페이지네이션(pagination): 무제한 목록을 반환하지 마십시오. 두 가지 방법:

# Offset pagination (simple, but slow for large offsets)
GET /api/tasks?offset=100&limit=20

# Cursor pagination (efficient, stable under inserts/deletes)
GET /api/tasks?cursor=eyJpZCI6MTAwfQ&limit=20
# Response includes: "next_cursor": "eyJpZCI6MTIwfQ"

멱등성 키(idempotency key): 클라이언트가 실패한 요청을 재시도할 때(응답이 전송 중에 손실된 경우), 작업이 두 번 수행되지 않도록 해야 합니다. 클라이언트는 각 요청과 함께 고유한 Idempotency-Key 헤더를 전송합니다. 서버는 이 값을 키로 결과를 저장하고 재시도 시 저장된 결과를 반환합니다.

하위 호환성(backward compatibility): 응답에 필드를 추가하는 것은 안전합니다. 필드를 제거하거나 유형을 변경하면 클라이언트가 깨집니다. 항상 추가하고, 절대 제거하지 마십시오.

서비스 통신 패턴

서비스는 동기적(synchronous) (요청/응답)으로 또는 비동기적(asynchronous) (메시지 큐를 통해)으로 통신할 수 있습니다. 동기는 더 단순하지만 강한 결합을 만듭니다 — 서비스 B가 다운되면 서비스 A의 요청이 실패합니다. 비동기는 더 탄력적이지만 추론하기 더 어렵습니다.

서킷 브레이커 패턴(circuit breaker pattern)은 동기 통신에서 연쇄 장애를 방지합니다. 전기 회로 차단기처럼 세 가지 상태가 있습니다:

  1. 닫힘(Closed) (정상) — 요청이 통과됩니다. 실패가 임계값을 초과하면 차단기가 "트립"됩니다.
  2. 열림(Open) — 다운스트림 서비스를 호출하지 않고 모든 요청이 즉시 실패합니다. 이미 어려움을 겪고 있는 서비스를 압도하는 것을 방지합니다.
  3. 반열림(Half-Open) — 타임아웃 후 단일 테스트 요청이 통과됩니다. 성공하면 차단기가 닫힙니다. 실패하면 차단기가 다시 열립니다.

지수 백오프를 사용한 재시도(retry with exponential backoff)는 또 다른 필수 패턴입니다. 요청이 실패하면 즉시 재시도하지 마십시오 — 1초, 그 다음 2초, 4초, 8초를 기다리십시오. 많은 클라이언트가 동시에 재시도하는 것을 방지하기 위해 작은 무작위 "지터(jitter)"를 추가하십시오.

import random
import time

def retry_with_backoff(operation, max_retries=5):
    """Retry an operation with exponential backoff and jitter."""
    for attempt in range(max_retries):
        try:
            return operation()
        except Exception as e:
            if attempt == max_retries - 1:
                raise  # Final attempt failed; propagate the error
            wait = (2 ** attempt) + random.uniform(0, 1)
            print(f"Attempt {attempt + 1} failed: {e}. Retrying in {wait:.1f}s")
            time.sleep(wait)

분산 시스템의 데이터 일관성

각 마이크로서비스가 자체 데이터베이스를 소유할 때, 서비스 간 작업을 단일 트랜잭션으로 묶을 수 없습니다. 사가 패턴(saga pattern)이 표준 해결책입니다: 사가는 각각 다른 서비스에 있는 로컬 트랜잭션의 시퀀스로, 각 단계에는 나중 단계가 실패할 경우 이를 되돌리는 보상 작업(compensating action)이 있습니다.

예시: 주문 생성은 (1) 재고 예약, (2) 결제 청구, (3) 배송 라벨 생성을 포함할 수 있습니다. 3단계가 실패하면 사가는 보상 작업을 실행합니다: 결제 환불 (2단계 취소), 재고 해제 (1단계 취소).

최종 일관성(eventual consistency)은 쓰기 후 모든 읽기가 즉시 새 데이터를 보지 못할 수 있지만 — 새로운 쓰기 없이 충분한 시간이 지나면 모든 읽기가 결국 동일한 값을 보게 된다는 것을 의미합니다. 강한 일관성이 필요해 보이는 많은 시스템이 실제로는 최종 일관성으로도 잘 작동합니다. 사용자의 친구 목록이 새 친구를 반영하는 데 2초가 걸리는 것이 중요한가요? 보통은 그렇지 않습니다.

분산 트랜잭션(distributed transaction)(2단계 커밋)은 서비스 간에 강한 일관성을 제공하지만, 심각한 비용이 따릅니다: 느리고, 프로토콜 중에 자원을 잠그며, 코디네이터가 충돌하면 모든 참가자가 막힙니다. 절대적으로 필요한 경우가 아니면 마이크로서비스 아키텍처에서 분산 트랜잭션을 피하십시오.

주의: “마이크로서비스는 항상 모놀리스보다 낫다”

많은 성공적인 회사들이 엄청난 규모에서 모놀리스를 운영합니다. Shopify, Stack Overflow, Basecamp 모두 모놀리식 아키텍처를 사용합니다. 마이크로서비스는 기술 문제보다 조직 문제(여러 팀이 하나의 제품에서 작업)를 더 많이 해결합니다. 소규모 팀이라면 모놀리스가 거의 확실히 올바른 선택입니다. 최선의 아키텍처는 팀이 안정적으로 출시할 수 있게 해주는 것입니다.

TaskForge 연결: 서비스 분해

TaskForge가 성장하여 마이크로서비스 아키텍처가 필요해진다면, 합리적인 분해 방법은 다음과 같습니다:

  • 태스크 서비스(Task Service) — 태스크의 CRUD, 태스크 상태 전환, 할당 로직. 태스크 데이터베이스 소유.
  • 사용자 서비스(User Service) — 인증, 권한 부여, 사용자 프로필. 사용자 데이터베이스 소유.
  • 알림 서비스(Notification Service) — task.assigned, task.overdue 같은 이벤트를 구독하고 이메일과 푸시 알림을 발송.

태스크 서비스는 태스크 할당 시 사용자 ID 검증을 위해 (서킷 브레이커와 함께) 사용자 서비스를 호출합니다. 태스크가 생성되면 태스크 서비스는 메시지 버스에 task.created 이벤트를 발행합니다. 알림 서비스는 이 이벤트를 구독하고 알림을 발송합니다. 알림 서비스가 다운되면 메시지가 큐에 쌓이고 복구되면 처리됩니다 — 태스크가 손실되지 않습니다.

프로파일링(Profiling): 병목 찾기

프로파일링(profiling)은 프로그램이 시간(또는 메모리)을 정확히 어디에 사용하는지 측정하는 것입니다. 추측과 앎의 차이입니다. 흔한 실수는 중요하지 않은 코드를 최적화하는 것입니다 — 전체 런타임의 0.1%를 차지하는 함수를 10배 빠르게 만드는 데 몇 시간을 소비하는 것입니다. 프로파일링은 코드의 어떤 5%가 느린 속도의 95%를 담당하는지 알려줍니다.

CPU 프로파일링(CPU profiling)은 각 함수에서 소요되는 시간을 측정합니다. Python의 내장 cProfile 모듈이 표준 도구입니다:

import cProfile

def slow_function():
    total = 0
    for i in range(1_000_000):
        total += i ** 2
    return total

def fast_function():
    return sum(i ** 2 for i in range(1_000_000))

def main():
    slow_function()
    fast_function()

# Profile the main function
cProfile.run('main()')
# Output shows: function name, call count, time per call, cumulative time

프로파일 출력은 어떤 함수가 시간을 소비하는지 정확하게 보여줍니다. 먼저 "누적 시간(cumulative time)" 열을 보십시오 — 함수가 호출하는 모든 함수를 포함하여 해당 함수에서 소요된 총 시간을 알려줍니다. 그런 다음 "호출당 시간(time per call)"을 보고 개별적으로 느린 함수를 찾으십시오.

메모리 프로파일링(memory profiling)은 프로그램이 할당하는 메모리의 양을 추적합니다. 메모리 누수가 몇 시간 또는 며칠에 걸쳐 충돌을 일으키는 장시간 실행 서버 프로세스에 중요합니다. memory_profiler 같은 도구는 줄별로 메모리 사용량을 보여줄 수 있습니다.

플레임 그래프(flame graph)는 프로파일링 데이터의 시각적 표현입니다. x축은 각 함수에서 소요된 시간의 비율을 나타내고, y축은 호출 스택을 보여줍니다. 그래프 상단의 넓은 막대가 병목입니다. 복잡한 애플리케이션에서 시간이 어디에 소비되는지 빠르게 이해하는 가장 강력한 도구 중 하나입니다.

실제 캐싱

34장에서 캐싱 아키텍처를 논의했습니다. 이제 코드 수준의 구현을 살펴보겠습니다. Python의 functools.lru_cache 데코레이터는 메모이제이션(memoization)을 제공합니다 — 인수를 기반으로 함수의 반환 값을 캐싱합니다:

from functools import lru_cache

@lru_cache(maxsize=128)
def fibonacci(n):
    """Without caching, this is O(2^n). With caching, it is O(n)."""
    if n < 2:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

# First call: computes and caches
print(fibonacci(100))  # Instant, thanks to caching

# Check cache statistics
print(fibonacci.cache_info())
# CacheInfo(hits=98, misses=101, maxsize=128, currsize=101)

캐시 어사이드 패턴(cache-aside pattern)(또는 "지연 로딩(lazy loading)"이라고도 함)은 가장 일반적인 애플리케이션 수준 캐싱 전략입니다. 코드가 먼저 캐시를 확인하고, 미스(miss)가 발생하면 데이터베이스에서 읽어 캐시에 저장하고 반환합니다:

def get_task(task_id, cache, database):
    """Cache-aside pattern: check cache first, fall back to database."""
    # 1. Check cache
    cached = cache.get(f"task:{task_id}")
    if cached is not None:
        return cached  # Cache hit!

    # 2. Cache miss - read from database
    task = database.query("SELECT * FROM tasks WHERE id = %s", (task_id,))

    # 3. Store in cache for next time (with 60-second TTL)
    cache.set(f"task:{task_id}", task, ttl=60)

    return task

Redis는 가장 널리 사용되는 애플리케이션 캐시입니다. 문자열, 리스트, 집합, 해시, 정렬된 집합을 지원하는 인메모리 키-값 저장소입니다. 일반적인 사용 사례로는 세션 저장, 속도 제한 카운터, 리더보드(정렬된 집합 사용), 범용 캐싱이 있습니다.

관측 가능성의 세 가지 기둥

관측 가능성(observability)은 시스템의 출력을 검사하여 내부에서 무슨 일이 일어나고 있는지 이해하는 능력입니다. 세 가지 기둥은 로그(logs), 메트릭(metrics), 트레이스(traces)입니다. 각각은 다른 질문에 답하며, 완전한 그림을 위해 세 가지 모두가 필요합니다.

구조화된 로깅(Structured Logging)

전통적인 텍스트 로그는 다음과 같습니다:

2026-03-18 14:23:05 ERROR Failed to create task for user 42: database timeout

사람이 읽기는 쉽지만 기계에는 적대적입니다. "사용자 42의 모든 오류"를 검색하려면 불안정한 정규식 파싱이 필요합니다. 구조화된 로깅(structured logging)은 일관된 필드를 가진 JSON 객체로 로그를 출력합니다:

import json
import time
import uuid

class StructuredLogger:
    """Logger that outputs JSON-formatted log entries."""

    def __init__(self, service_name):
        self.service_name = service_name

    def log(self, level, message, **context):
        entry = {
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
            "level": level,
            "service": self.service_name,
            "message": message,
            **context
        }
        print(json.dumps(entry))

logger = StructuredLogger("taskforge-api")

# Usage with correlation ID for request tracing
correlation_id = str(uuid.uuid4())
logger.log("INFO", "Task created",
    correlation_id=correlation_id,
    user_id=42,
    task_id=1001,
    duration_ms=45
)
# Output:
# {"timestamp":"2026-03-18T14:23:05Z","level":"INFO","service":"taskforge-api",

상관 관계 ID(correlation ID)(때로는 "요청 ID" 또는 "추적 ID"라고도 함)는 요청이 시스템에 들어올 때 생성되는 고유 식별자입니다. 모든 서비스가 로그에 이 ID를 포함하므로, 요청이 접촉한 모든 서비스에서 단일 요청의 모든 로그 항목을 필터링할 수 있습니다.

표준 로그 레벨(log level)은 심각도를 전달합니다:

표준 로그 레벨과 각각의 사용 시기
레벨사용 시기예시
DEBUG개발자를 위한 상세한 진단 정보변수 값, 쿼리 파라미터
INFO기록할 가치 있는 정상 작업사용자 로그인, 태스크 생성
WARNING예상치 못하지만 깨지지는 않은 것캐시 미스율이 임계값 초과
ERROR이 요청에 대해 무언가 실패함데이터베이스 쿼리 타임아웃
CRITICAL시스템이 다운될 위험이 있음디스크 꽉 참, 메모리 부족

모니터링과 알림: 황금 신호(Golden Signals)

Google의 Site Reliability Engineering 책은 모든 서비스가 모니터링해야 할 네 가지 황금 신호(golden signal)를 정의합니다:

  1. 지연 시간(latency) — 요청이 걸리는 시간. 평균만이 아닌 백분위수(p50, p95, p99)를 추적하십시오. 100ms의 평균은 요청의 1%가 5초 걸린다는 사실을 숨길 수 있습니다.
  2. 트래픽(traffic) — 시스템에 가해지는 수요. 초당 요청, 동시 사용자 수, 또는 데이터 처리량.
  3. 오류(errors) — 실패한 요청의 비율. 명시적 실패 (HTTP 500)와 암묵적 실패 (HTTP 200이지만 잘못된 콘텐츠) 모두 포함.
  4. 포화도(saturation) — 시스템이 얼마나 "가득 찬" 상태인지. CPU 사용률, 메모리 사용량, 큐 깊이, 디스크 I/O. 포화도는 문제가 발생하기 전에 미래 문제를 예측합니다.

이 신호들은 SLI, SLO, SLA와 연결됩니다:

  • SLI (Service Level Indicator, 서비스 수준 지표) — 정량적 측정. 예시: "요청의 99.2%가 200ms 미만에 완료됨."
  • SLO (Service Level Objective, 서비스 수준 목표) — 내부적으로 설정한 목표. 예시: "요청의 99.5%가 200ms 미만을 목표로 함."
  • SLA (Service Level Agreement, 서비스 수준 계약) — 위반 시 결과를 포함한 고객에 대한 계약상 약속. 예시: "99.9% 가동 시간을 보장하며 그렇지 않으면 크레딧을 제공함."

좋은 알림은 원인 (CPU 80% 초과)이 아닌 증상 (오류율 5% 초과)을 기반으로 합니다. 높은 CPU는 사용자에게 보이는 영향을 초래할 때만 문제입니다. 사용자가 경험하는 것에 대해 알림을 설정하고, 대시보드와 로그를 사용하여 원인을 찾으십시오.

분산 추적(Distributed Tracing)

단일 사용자 요청이 다섯 개의 서비스를 통과하고 총 응답 시간이 2초일 때, 어떤 서비스가 병목입니까? 분산 추적(distributed tracing)은 서비스 경계 전반에 걸쳐 요청을 추적함으로써 이에 답합니다.

각 서비스는 스팬(span)을 생성합니다 — 수행한 작업, 소요 시간, 그리고 부모 스팬의 기록입니다. 스팬들이 함께 요청의 완전한 라이프사이클을 보여주는 트레이스(trace) 트리를 형성합니다. 모든 스팬을 연결하기 위해 추적 ID가 서비스 간에 전달됩니다 (보통 HTTP 헤더로).

# Conceptual trace structure (simplified)
trace = {
    "trace_id": "abc-123",
    "spans": [
        {"service": "api-gateway",    "duration_ms": 1950, "parent": None},
        {"service": "auth-service",   "duration_ms": 150,  "parent": "api-gateway"},
        {"service": "task-service",   "duration_ms": 1700, "parent": "api-gateway"},
        {"service": "database",       "duration_ms": 1500, "parent": "task-service"},
    ]
}
# Diagnosis: 1500ms in the database query - that is the bottleneck!

프로덕션 디버깅: 조사 마인드셋

프로덕션 디버깅은 개발 중 디버깅과 근본적으로 다릅니다. 디버거를 연결할 수 없고, 출력 문을 추가할 수 없으며, 문제가 간헐적일 수 있습니다. 접근 방식은 체계적이어야 합니다:

  1. 관찰(Observe) — 증상은 무엇입니까? 대시보드, 오류율, 최근 로그를 확인하십시오. 언제 시작되었습니까? 최근에 무엇이 변경되었습니까 (배포, 설정 변경, 트래픽 급증)?
  2. 가설 수립(Hypothesize) — 원인에 대한 이론을 형성하십시오. "새 배포가 느린 쿼리를 도입했다." "트래픽 급증으로 데이터베이스가 과부하되었다."
  3. 테스트(Test) — 가설을 검증하십시오. 쿼리 성능 메트릭을 확인하십시오. 데이터베이스 연결 수를 살펴보십시오. 배포 전후의 오류율을 비교하십시오.
  4. 조치(Act) — 즉각적인 문제를 수정하고 (배포 롤백, 데이터베이스 스케일 업), 그런 다음 근본 원인을 해결하십시오.

인시던트 해결 후 포스트모텀(postmortem)을 수행하십시오. 좋은 포스트모텀은 비난 없음(blameless)입니다 — 개인 책임 ("Bob이 나쁜 쿼리를 작성했다")이 아닌 시스템적 원인 ("코드 리뷰 프로세스가 이것을 잡지 못했다")에 집중합니다. 구체적인 실행 항목을 도출합니다: "태스크 목록 엔드포인트에 부하 테스트 추가", "데이터베이스 지연 시간이 500ms를 초과할 때 알림 추가."

“모든 인시던트는 선물입니다 — 시스템에 있었는지 몰랐던 약점에 대해 배울 기회입니다.”
주의: 관측 가능성은 선택 사항이 아닙니다

많은 개발자들이 로깅과 모니터링을 나중에 추가할 것으로 취급합니다. 실제로 적절한 관측 가능성 없이 발생하는 첫 번째 프로덕션 인시던트는 처음부터 관측 가능성을 설정하는 것보다 더 많은 엔지니어링 시간을 소비하게 됩니다. 첫날부터 구축하십시오. 구조화된 로깅, 기본 메트릭 (네 가지 황금 신호), 상관 관계 ID는 구축하는 모든 서비스에서 표준이 되어야 합니다.

TaskForge 연결: 프로덕션을 위한 계측

TaskForge의 프로덕션 준비된 관측 가능성의 모습:

  • 구조화된 로깅: 모든 API 요청이 메서드, 경로, 상태 코드, 지속 시간, 사용자 ID, 상관 관계 ID를 기록합니다.

  • 알림: 오류율이 5분 동안 5%를 초과하거나, p99 지연 시간이 2초를 초과하거나, 데이터베이스 연결 풀이 80% 사용률을 초과할 때 트리거됩니다.
  • 프로파일: 태스크 목록 엔드포인트에서 cProfile을 실행하여 tasks.user_id에 누락된 인덱스가 전체 테이블 스캔을 유발한다는 것을 발견합니다. 인덱스를 추가하면 지연 시간이 800ms에서 15ms로 떨어집니다.

제곱합의 수학 공식: n * (n + 1) * (2 * n + 1) // 6. 결과를 정수로 유지하려면 정수 나눗셈(//)을 사용하십시오.

함수를 측정하려면 호출 전후에 time.time()을 기록하십시오. 차이가 초 단위의 경과 시간입니다.

지금 이것이 중요한 이유

지금까지의 모든 튜토리얼, 연습, 프로젝트는 처음부터 시작했습니다. 하지만 현실 세계에서는 거의 그렇지 않습니다. 경력의 대부분은 이미 존재하는 코드베이스—다른 사람이 작성한, 때로는 몇 년 전에 만든, 종종 불완전한 문서만 있는 코드베이스—에서 작업하게 됩니다. 낯선 코드를 읽고, 이해하고, 안전하게 발전시키는 능력은 전문 엔지니어가 개발할 수 있는 가장 중요한 실전 스킬입니다. 이 챕터는 그것을 위한 체계적인 접근법을 가르칩니다.

낯선 코드 읽기: 체계적인 접근 방식

100,000줄의 코드베이스를 처음 열면 압도감을 느낍니다. 핵심은 교과서를 외우려는 학생이 아닌 미지의 영역을 지도로 만드는 탐험가처럼 접근하는 것입니다. 모든 것을 이해할 필요는 없습니다 — 중요한 부분의 정신적 지도를 구축해야 합니다.

전략 1: 진입점 찾기. 모든 프로그램에는 시작점이 있습니다. 웹 서버의 경우 HTTP 요청 수신을 시작하는 파일입니다. CLI 도구의 경우 if __name__ == "__main__" 블록입니다. 라이브러리의 경우 __init__.py의 공개 API입니다. 이 앵커를 찾으면 추적을 시작할 위치가 생깁니다.

전략 2: 해피 패스(happy path) 추적. 가장 일반적인 사용자 작업 (예: "태스크 생성")을 선택하고 HTTP 라우트 핸들러에서 서비스 레이어를 거쳐 데이터베이스로 들어갔다가 돌아오는 코드를 따라가십시오. 오류 처리, 엣지 케이스, 유틸리티 모듈에 의해 산만해지지 마십시오. 정상 경로를 먼저 이해하십시오.

전략 3: 의존성 그래프 그리기. 어떤 모듈이 어떤 다른 모듈을 가져오는지 스케치하십시오. 이것은 아키텍처를 드러냅니다: 어떤 모듈이 중심적인지 (많은 곳에서 가져옴), 어떤 것이 주변부인지 (적은 곳에서 가져옴), 어떤 것이 고립되었는지 (아무데서도 가져오지 않음 — 데드 코드일 가능성).

전략 4: 하향식 vs. 상향식. 하향식(top-down)은 main()에서 시작하여 함수 호출로 파고듭니다. 이는 큰 그림을 빠르게 제공합니다. 상향식(bottom-up)은 테스트와 소규모 유틸리티 함수에서 시작하여 구체적인 예에서 이해를 구축합니다. 둘 다 사용하십시오: 아키텍처는 하향식으로 시작하고, 특정 모듈을 이해해야 할 때 상향식으로 전환하십시오.

전략 5: 솔기(seam) 찾기. 솔기(seam)는 모듈 사이의 경계 — 한 모듈이 정의된 인터페이스를 통해 다른 모듈과 상호작용하는 장소입니다. 솔기를 이해하면 어디서 안전하게 변경을 할 수 있는지 (모듈 내부, 솔기 뒤에서)와 변경이 코드베이스 전반에 파급될 곳 (솔기에서, 모든 소비자에게 영향)을 알 수 있습니다.

레거시 시스템 이해: 고고학적 접근 방식

레거시 코드(legacy code)는 비하적인 표현이 아닙니다 — 여전히 실행될 만큼 가치 있는 코드입니다. 도전 과제는 왜 그렇게 작성되었는지 이해하는 것입니다. 답은 보통 역사에 있습니다.

git blame은 파일의 각 줄을 마지막으로 누가 언제 수정했는지 보여줍니다. 더 중요한 것은 왜 그 변경이 이루어졌는지 설명하는 커밋 메시지를 검사할 수 있는 커밋 해시를 보여준다는 것입니다. "왜 이 코드가 이상한가?"라는 많은 질문들이 이를 도입한 커밋을 찾음으로써 답해집니다.

안전한 변경: 깨지지 않고 수정하는 방법

코드베이스에서 가장 위험한 순간은 모든 영향을 이해하지 못한 채 무언가를 변경할 때입니다. 안전하게 변경하는 검증된 기술들이 있습니다.

특성화 테스트(characterization test)(또는 "골든 마스터(golden master)" 테스트라고도 함)는 그 동작이 올바른지 여부에 관계없이 코드의 현재 동작을 캡처합니다. 리팩터링 전에 코드가 실제로 하는 것을 단언하는 테스트를 작성합니다 — 코드가 해야 한다고 생각하는 것이 아니라. 그런 다음 리팩터링하고, 테스트가 여전히 통과하면 동작이 보존되었음을 알 수 있습니다.

스트랭글러 피그 패턴(strangler fig pattern)은 레거시 시스템을 점진적으로 교체하는 전략입니다. 기존 나무를 감싸며 자라다가 결국 대체하는 무화과나무에서 이름을 따왔습니다. 패턴의 작동 방식:

  1. 오래된 시스템 앞에 파사드(facade)(프록시)를 배치합니다.
  2. 동일한 파사드 뒤의 새 시스템에 새 기능을 구현합니다.
  3. 오래된 시스템에서 새 시스템으로 더 많은 트래픽을 점진적으로 라우팅합니다.
  4. 모든 트래픽이 새 시스템에 있으면 오래된 것을 폐기합니다.

모든 단계에서 두 시스템이 실행 중이며 트래픽을 오래된 시스템으로 되돌려 즉시 롤백할 수 있습니다.

기능 플래그(feature flag)를 사용하면 사용자에게 배포하지 않고 코드를 배포할 수 있습니다. 새 동작을 조건부 검사로 감싸고 (if feature_enabled("new_priority_algorithm")) 배포합니다. 플래그가 처음에는 꺼져 있어 사용자가 변경 사항을 보지 못합니다. 그런 다음 1%의 사용자에게 활성화하고, 문제를 모니터링하고, 100%까지 점진적으로 롤아웃할 수 있습니다. 문제가 발생하면 즉시 플래그를 끄면 됩니다 — 배포가 필요 없습니다.

기술 부채: 실용적인 프레임워크

기술 부채(technical debt)는 Ward Cunningham이 만든 은유입니다: 재정적 부채가 지금 무언가를 사고 나중에 (이자와 함께) 지불하게 하듯이, 기술적 지름길은 지금 더 빠르게 출시하게 해주지만 미래의 변경을 더 느리게 만듭니다 (이자). 모든 기술 부채가 나쁜 것은 아닙니다 — 때로는 빠르게 출시하는 것이 올바른 비즈니스 결정입니다. 문제는 부채가 너무 많이 쌓이고 상환을 멈출 때 발생합니다.

기술 부채를 측정하는 실용적인 방법들:

  • 새 개발자가 생산적이 되는 데 얼마나 걸립니까? 3일 대신 3주가 걸린다면 온보딩 부채가 있습니다.
  • 한 영역의 변경이 다른 영역을 얼마나 자주 깨뜨립니까? 자주 그렇다면 결합 부채가 있습니다.
  • 테스트 스위트는 얼마나 걸립니까? 45분이 걸린다면 개발자들이 건너뛰고 버그가 출시됩니다. 이것이 테스트 인프라 부채입니다.
  • 코드의 얼마나 많은 부분이 한 사람만 이해합니까? 이것이 지식 부채입니다 — 버스 팩터(bus factor)가 1인 것입니다.

부채를 포괄적으로가 아닌 전략적으로 상환하십시오. 아무도 건드리지 않는 코드를 리팩터링하는 것은 시간 낭비입니다. 변경 빈도가 높고 결함률이 높은 모듈에 집중하십시오 — 이것이 부채가 실제로 비용을 초래하는 영역입니다.

시니어 기술로서의 코드 리뷰

코드 리뷰는 버그를 찾는 것만이 아닙니다. 좋은 코드 리뷰는 코드를 개선하고 동시에 작성자를 성장시킵니다. 경험 많은 리뷰어가 집중하는 것들:

코드 리뷰 우선순위와 피드백 예시
우선순위확인 사항피드백 예시
1. 정확성주장하는 것을 하는가? 엣지 케이스? 레이스 컨디션?“여기서 user_id가 None이면 어떻게 됩니까?”
2. 가독성코드에 익숙하지 않은 사람이 이해할 수 있는가?“이것을 이름 있는 함수로 추출할 수 있습니까? 의도가 명확하지 않습니다.”
3. 테스트 가능성코드가 테스트 가능한가? 테스트가 있는가? 올바른 것을 테스트하는가?“이 테스트는 동작이 아닌 구현 세부 사항을 검증합니다.”
4. 설계기존 아키텍처에 맞는가? 올바른 추상화 수준인가?“이 서비스 함수가 HTTP 요청에 직접 접근합니다. 필요한 데이터만 전달할 수 있습니까?”
5. 성능명백한 비효율? N+1 쿼리? 불필요한 할당?“이 루프가 반복마다 데이터베이스를 조회합니다. 이것을 배치로 처리할 수 있습니까?”

자전거 창고 토론(bikeshedding)을 피하십시오 — 구조적 문제를 무시하면서 사소한 문제(변수 이름, 형식)에 불균형적인 시간을 소비하는 것. 팀에 린터와 포맷터가 있다면 스타일은 도구가 처리하게 두십시오. 리뷰 시간은 로직, 설계, 엣지 케이스에 사용하십시오.

대규모 팀과 함께 일하기

코드베이스가 성장함에 따라 컨벤션(convention)이 필수적이 됩니다. 스타일 가이드와 린터는 일관성을 강제하여 서로 다른 사람이 작성한 코드가 마치 같은 사람이 작성한 것처럼 보이게 합니다. 이는 코드를 읽을 때 인지 부하를 크게 줄입니다.

아키텍처 결정 기록(ADR, Architecture Decision Record)은 중요한 기술 결정과 그 이유를 문서화합니다. ADR은 일반적으로 다음을 포함합니다:

# ADR-007: Use PostgreSQL for Task Storage

## Status: Accepted

## Context
TaskForge needs a relational database. We considered PostgreSQL, MySQL,
and SQLite. We need concurrent write support, full-text search, and
JSONB columns for flexible task metadata.

## Decision
We will use PostgreSQL.

## Consequences
- Positive: Native JSONB support, excellent concurrent performance,
  strong community, good tooling.
- Negative: More complex to set up than SQLite; team needs PostgreSQL
  expertise. We accept this trade-off because SQLite cannot handle
  our concurrency requirements.

## Alternatives Considered
- MySQL: Lacks native JSONB; full-text search requires plugins.
- SQLite: Cannot handle concurrent writes from multiple workers.

ADR은 무엇이 결정되었는지뿐만 아니라 왜와 어떤 대안이 고려되었는지도 기록하기 때문에 매우 가치 있습니다. 6개월 후에 누군가 "왜 SQLite를 사용하지 않나요?"라고 물을 때, ADR은 원래 토론을 기억할 필요 없이 답을 제공합니다.

RFC 프로세스(Request for Comments, 의견 요청)는 더 큰 변경에 대해 유사한 목적을 제공합니다. 주요 기능이나 아키텍처 변경을 구현하기 전에 설계 문서를 작성하고, 피드백을 위해 순환시키고, 반복합니다. 이는 비용이 많이 드는 놀라움을 방지하고 팀 간 조율을 보장합니다.

주의: “좋은 코드는 문서가 필요 없다”

좋은 코드는 무엇을 하고 어떻게 하는지에 대해 자체 문서화됩니다. 하지만 코드는 결정이 왜 내려졌는지, 어떤 대안이 고려되었는지, 또는 당시 어떤 제약이 있었는지를 결코 알려줄 수 없습니다. 문서화가 이 간극을 채웁니다. 가장 유용한 문서는 의도, 트레이드오프, 맥락 — 코드가 표현할 수 없는 것들 — 을 설명합니다.

TaskForge 연결: 레거시 구조 작전

다음 문제가 있는 "레거시" 버전의 TaskForge를 상속받았다고 상상해 보십시오:

  • 갓 클래스(god class): 단일 TaskManager 클래스가 라우팅, 비즈니스 로직, 데이터베이스 쿼리, 이메일 발송을 모두 처리합니다.
  • 테스트 없음: 자동화 테스트가 전혀 없습니다. 변경에 대한 안전망이 없습니다.
  • 하드코딩된 설정: 데이터베이스 URL, API 키, 이메일 자격 증명이 코드 전체에 문자열 리터럴로 흩어져 있습니다.
  • SQL 인젝션: 사용자 입력이 SQL 쿼리에 직접 연결됩니다.

접근 방식: (1) 현재 동작을 캡처하는 특성화 테스트 작성. (2) 설정을 환경 변수로 추출. (3) SQL 인젝션 수정 (가장 위험한 항목). (4) 스트랭글러 피그 패턴을 사용하여 갓 클래스를 점진적으로 분해 — 데이터베이스 접근을 위한 TaskRepository, 알림을 위한 EmailService를 도입하고 TaskManager가 이들에 위임하도록 합니다. 각 단계는 작고, 테스트되고, 되돌릴 수 있습니다.

직접 해보기: ADR 작성하기

이 시나리오에 대한 간략한 아키텍처 결정 기록을 작성하십시오: 팀이 TaskForge에 Redis 캐시를 추가할지 결정하고 있습니다. 현재 태스크 목록 페이지는 매번 데이터베이스를 조회하기 때문에 800ms가 걸립니다. Redis를 사용하면 캐시된 결과에 5ms가 걸립니다. 트레이드오프는 운영 복잡성 (유지할 또 다른 서비스)과 오래된 데이터를 제공할 위험입니다. 팀은 30초 TTL을 사용하여 Redis를 진행하기로 했습니다.

연습문제: 특성화, 그 다음 리팩터링

아래는 너무 많은 일을 하는 지저분한 함수 process_task_data입니다: 유효성 검사, 변환, 우선순위 계산, 출력 형식화를 하나의 덩어리로 처리합니다. 먼저 동작을 연구하십시오. 그런 다음 원본과 동일한 결과를 생성하는 세 개의 깔끔한 함수 — validate_task, calculate_priority, format_task_summary — 를 구현하십시오. 특성화 테스트가 리팩터링된 코드가 원본 동작과 정확히 일치하는지 검증합니다.

validate_task부터 시작하십시오: 원본과 동일한 조건을 확인하십시오 (빈/비문자열 제목, 100자 초과 제목, 빈/비문자열 담당자). 유효하면 None을 반환하고, 그렇지 않으면 {"error": "..."}를 반환하십시오.

calculate_priority: 0에서 시작하고, 긴급하면 50 추가, days_until_due 범위에 따라 추가, title_length > 50이면 5 빼기, 0과 100 사이로 클램프하십시오.

format_task_summary: 우선순위에서 긴급 레이블을 결정하고 (>=80 CRITICAL, >=50 HIGH, >=30 MEDIUM, 그 외 LOW), 다섯 개의 키로 결과 딕셔너리를 구성하십시오.

process_task_refactored에서: validate_task를 먼저 호출하고 (None이 아니면 오류 딕셔너리 반환), 제목을 정리하고 (strip + upper), title_length에 len(title)을 사용하여 우선순위를 계산하고, 형식화하십시오.

  4. 레거시 코드 리팩터링 — TaskForge 모듈의 "지저분한 버전"을 가져오십시오. 먼저 특성화 테스트를 작성한 다음 깔끔하고 잘 구조화된 코드로 리팩터링하십시오. 모든 특성화 테스트는 리팩터링 전후 모두 통과해야 합니다.
