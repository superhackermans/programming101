from flask import Flask, jsonify, request

app = Flask(__name__)

tasks = []

@app.route("/api/tasks", methods=["GET"])
def get_tasks():
    """Return all tasks, optionally filtered by status."""
    status = request.args.get("status")
    if status:
        filtered = [t for t in tasks if t["status"] == status]
        return jsonify(filtered)
    return jsonify(tasks)

@app.route("/api/tasks", methods=["POST"])
def create_task():
    """Create a new task from JSON body."""
    data = request.get_json()
    if not data or "title" not in data:
        return jsonify({"error": "title is required"}), 400
    task = {
        "id": len(tasks) + 1,
        "title": data["title"],
        "status": data.get("status", "todo"),
    }
    tasks.append(task)
    return jsonify(task), 201

if __name__ == "__main__":
    app.run(debug=True)

부록 D 문제 해결

페이즈별로 정리된 흔한 문제들입니다. 각 항목에는 증상(현상), 원인, 해결 방법이 포함됩니다.

Phase 1 문제

Python을 찾을 수 없음
▶
증상
터미널에서 python3: command not found 또는 'python3' is not recognized.
원인
Python이 설치되지 않았거나 설치 디렉토리가 시스템 PATH에 없습니다.
해결
python.org에서 Python을 재설치하고 "Add to PATH"가 체크되어 있는지 확인하세요. macOS에서는 brew install python3를 실행하세요. which python3로 확인하세요.
"command not found" 오류
▶
증상
설치 후 임의의 도구(git, node, code)에서 "command not found"가 반환됩니다.
원인
실행 파일의 디렉토리가 셸의 PATH에 없거나, 셸 세션이 재시작되지 않았습니다.
해결
터미널을 닫고 다시 여세요. 그래도 안 되면 설치 경로를 ~/.zshrc 또는 ~/.bashrc에 추가하고 source ~/.zshrc를 실행하세요.
REPL이 시작되지 않음
▶
증상
python3을 입력하면 멈추거나 예상치 못한 버전이 표시됩니다.
원인
여러 Python 설치가 충돌하거나, 별칭이 잘못된 버전을 가리키고 있습니다.
해결
which python3를 실행하여 어떤 바이너리가 실행되는지 확인하세요. 전체 경로(예: /usr/local/bin/python3)를 사용하거나, 셸 설정에서 PATH 순서를 수정하세요.
VS Code에서 Python을 찾을 수 없음
▶
증상
VS Code에서 "No Python interpreter selected" 또는 린팅 실패가 표시됩니다.
원인
VS Code의 Python 확장이 Python 설치 위치를 모릅니다.
해결
명령 팔레트(Cmd+Shift+P)를 열고, "Python: Select Interpreter"를 입력한 후 올바른 Python 3 설치를 선택하세요.

Phase 2 문제

IndentationError(들여쓰기 오류)
▶
증상
IndentationError: unexpected indent 또는 unindent does not match.
원인
파일에서 들여쓰기에 탭과 공백이 혼용되거나, 들여쓰기 수준이 일관되지 않습니다.
해결
편집기를 4칸 공백(탭 아님)으로 설정하세요. VS Code: 설정 → "Editor: Tab Size" = 4, "Editor: Insert Spaces" = 체크. 해당 파일을 다시 들여쓰기하세요.
ModuleNotFoundError(모듈 없음 오류)
▶
증상
ModuleNotFoundError: No module named 'requests' (또는 다른 서드파티 패키지).
원인
현재 사용 중인 Python 환경에 패키지가 설치되지 않았거나, Python이 여러 개 설치되어 있습니다.
해결
python3 -m pip install requests를 실행하세요(python3 -m pip 사용으로 올바른 Python에 설치됩니다). 가상 환경을 사용하여 의존성을 격리하세요.
json.decoder.JSONDecodeError(JSON 파싱 오류)
▶
증상
JSON 파일을 읽을 때 json.decoder.JSONDecodeError: Expecting value 발생.
원인
파일이 비어 있거나, 잘못된 JSON을 포함하거나, 후행 쉼표 또는 큰따옴표 대신 작은따옴표가 있습니다.
해결
jsonlint.com에서 JSON을 검증하세요. 파일이 비어 있지 않은지 확인하세요. 직접 편집하는 대신 json.dumps()를 사용하여 JSON을 작성하세요.
pytest를 찾을 수 없음
▶
증상
python3 -m pytest 실행 시 No module named pytest.
원인
현재 Python 환경에 pytest가 설치되지 않았습니다.
해결
python3 -m pip install pytest를 실행하세요. 가상 환경을 사용 중이라면 먼저 활성화하세요 (source venv/bin/activate).

Phase 3 문제

Permission denied(파일/SSH 권한 거부)
▶
증상
파일 읽기 또는 SSH로 git 원격에 연결할 때 Permission denied.
원인
파일 권한이 너무 제한적이거나, SSH 키가 SSH 에이전트 또는 GitHub에 추가되지 않았습니다.
해결
파일의 경우: chmod 644 filename. SSH의 경우: ssh-add ~/.ssh/id_ed25519를 실행하고 GitHub 설정 → SSH Keys에서 키가 추가되어 있는지 확인하세요.
Git "detached HEAD" 상태
▶
증상
Git이 HEAD detached at abc1234라고 표시하고 커밋이 사라지는 것처럼 보입니다.
원인
브랜치 대신 특정 커밋이나 태그를 체크아웃했기 때문에 새 커밋을 추적할 브랜치가 없습니다.
해결
현재 위치에서 브랜치를 만드세요: git checkout -b my-branch. 커밋은 이름 있는 브랜치에 있으면 안전합니다.
병합 충돌(Merge conflict)
▶
증상
파일에 <<<<<<< 마커가 표시되고 git이 병합을 완료하지 못합니다.
원인
두 브랜치가 동일한 줄을 편집했습니다. Git이 어떤 버전을 유지할지 자동으로 결정할 수 없습니다.
해결
충돌된 각 파일을 열고 마커 사이에서 올바른 코드를 선택한 후, 모든 <<<<<<< / ======= / >>>>>>> 줄을 제거하고 git add와 git commit을 실행하세요.

▶

▶

네트워크가 api.anthropic.com으로의 아웃바운드 HTTPS 연결을 차단하거나, 기업 방화벽/VPN 뒤에 있습니다.
해결
curl https://api.anthropic.com/v1/messages -I로 연결을 테스트하세요. 차단된 경우 방화벽 설정을 확인하거나 다른 네트워크를 시도하세요. 기업 프록시 뒤에 있다면 HTTPS_PROXY 환경 변수를 설정하세요: export HTTPS_PROXY=http://your-proxy:port.
프록시 설정
▶
증상
Claude Code는 설치되었지만 API에 연결할 수 없습니다. npm install 자체도 네트워크 오류로 실패합니다.
원인
npm과 Claude Code는 프록시 설정을 별도로 필요로 합니다.
해결
npm의 경우: npm config set proxy http://your-proxy:port와 npm config set https-proxy http://your-proxy:port. Claude Code의 경우: 셸 설정 파일(~/.zshrc 또는 ~/.bashrc)에 HTTPS_PROXY를 설정한 후 source ~/.zshrc를 실행하세요.

Phase 9 문제

Dispatch를 찾을 수 없음
▶
증상
dispatch: command not found 또는 /dispatch 명령이 인식되지 않습니다.
원인
Dispatch 도구가 설치되지 않았거나, 현재 Claude Code 버전에서 사용할 수 없습니다.
해결
Claude Code를 최신 버전으로 업데이트하세요: npm update -g @anthropic-ai/claude-code. 플랜의 기능 목록에서 Dispatch 사용 가능 여부를 확인하세요.
백그라운드 에이전트가 조용히 실패함
▶
증상
디스패치된 작업이 "완료"로 표시되지만 출력이 없거나 잘못된 결과를 생성합니다.
원인
에이전트의 컨텍스트가 불충분하거나, 하네스(테스트/린터)가 실패를 감지하지 못했습니다.
해결
에이전트의 출력 로그에서 오류를 확인하세요. 하네스에 충분한 테스트 커버리지가 있는지 확인하세요. 실패가 명시적으로 드러나도록 디스패치 프롬프트에 제약 조건을 추가하세요.
병렬 에이전트의 병합 충돌
▶
증상
여러 에이전트 브랜치를 main에 병합할 때 충돌이 발생합니다.
원인
두 개 이상의 에이전트가 동일한 파일을 수정했습니다. 병렬 작업을 디스패치할 때 예상되는 상황입니다.
해결
각 에이전트를 격리하기 위해 워크트리를 사용하세요. 가능하면 에이전트에게 겹치지 않는 파일을 배정하세요. 한 번에 하나씩 브랜치를 병합하고 충돌을 점진적으로 해결하세요.
Docker 권한 거부
▶
증상
permission denied while trying to connect to the Docker daemon socket.
원인
사용자가 docker 그룹에 없거나 Docker Desktop이 실행되지 않고 있습니다.
해결
macOS: Docker Desktop을 시작하세요. Linux: sudo usermod -aG docker $USER를 실행하고 로그아웃/로그인하세요. docker ps로 확인하세요.

Phase 10 문제

에이전트 팀의 파일 변경 충돌
▶
증상
두 에이전트가 동시에 같은 파일을 편집하여, 하나가 다른 하나의 변경 사항을 덮어씁니다.
원인
병렬 에이전트 간 파일 잠금 또는 조율이 없습니다.
해결
Ch 46의 허브 앤 스포크 패턴을 사용하세요. 각 에이전트에게 별도의 파일이나 섹션을 배정하세요. 에이전트가 파일을 공유해야 한다면 오케스트레이터를 통해 쓰기를 직렬화하세요.
백그라운드 에이전트 타임아웃 또는 중단
▶
증상
백그라운드 에이전트가 출력을 멈추고 완료되지 않습니다.
원인
에이전트가 무한 재시도 루프에 진입하거나, 속도 제한에 도달하거나, 받을 수 없는 사용자 입력을 기다리고 있습니다.
해결
에이전트 설정에서 max_iterations를 설정하세요. 에이전트 출력 로그를 모니터링하세요. 막혔다면 프로세스를 종료하고 더 좁은 작업으로 새 에이전트를 디스패치하세요.
