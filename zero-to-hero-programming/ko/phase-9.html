시작 전에: LLM이 실제로 무엇인지 이해하기

이제 AI 코딩 도구를 사용하기 시작할 것입니다. 그 전에, 이 도구들이 무엇인지—그리고 무엇이 아닌지—를 이해해야 합니다.

대규모 언어 모델(LLM, Large Language Model)은 방대한 텍스트에서 학습한 패턴을 바탕으로 가장 그럴듯한 다음 단어를 예측합니다. 인간 엔지니어처럼 여러분의 프로젝트를 "이해"하지 않습니다. 세션 간 기억도 없습니다. 자신의 출력을 검증할 수도 없습니다. 패턴에 근거해 올바르게 보이는 텍스트를 생성하며—종종 진정으로 유용하지만—자신 있고 그럴듯한 헛소리를 생성할 수도 있습니다.

수백만 개의 코드베이스를 읽었지만 여러분의 코드는 한 번도 본 적 없는 뛰어난 연구 보조자라고 생각하세요. 유창한 코드를 작성할 수 있지만, 아키텍처를 오해하거나, 존재하지 않는 API를 만들어내거나, 표면적으로는 통과하지만 엣지 케이스에서 실패하는 로직을 생성할 수 있습니다.

에이전트(Agent)는 마법 같은 존재가 아닙니다. 모델 호출을 중심으로 구축된 워크플로우입니다: 프롬프트가 들어가고, 응답이 나오며, 도구가 행동을 실행하고, 규칙이 행동을 제한합니다. 에이전트가 하는 모든 "지능적인" 것은 여러분이 제공한 컨텍스트, 구성한 도구, 설정한 제약으로 거슬러 올라갑니다. 설정이 좋을수록 에이전트 성능이 좋아집니다.

이것은 두 가지를 의미합니다: (1) 여러분의 판단이 품질 게이트이고, (2) 페이즈 1~4에서 배운 모든 것—코드 읽기, 데이터 흐름 이해, 테스트 작성—이 그 판단을 가능하게 합니다.

38장 AI 생성 코드 평가와 지시

지금 이것이 중요한 이유

AI 도구는 코드를 빠르게 생성합니다. 하지만 판단 없는 속도는 위험합니다. 이 장에서는 품질 게이트가 되는 법—AI 출력이 코드베이스에 들어가기 전에 올바른지, 완전한지, 안전한지 결정하는 사람—을 배웁니다.

이것은 페이즈 9에서 가장 중요한 장입니다. 이 기술 없이는 이후의 모든 장이 출력을 평가할 판단력 없이 AI 도구를 사용하는 독자를 만들어냅니다.

핵심 문제: 비결정론(Non-Determinism)

지금까지 배운 모든 프로그래밍 개념은 결정론적(deterministic)입니다. 2 + 2를 백만 번 실행해도 항상 4를 얻습니다. sorted([3,1,2])를 호출하면 결과는 항상 [1,2,3]입니다. 함수에는 입력과 출력이 있습니다. 테스트는 통과하거나 실패합니다. 이 예측 가능성이 소프트웨어 엔지니어링의 전체 기반입니다.

대규모 언어 모델(LLM)은 근본적으로 비결정론적입니다. 같은 질문을 두 번 하면 다른 답을 받을 수 있습니다. 함수 작성을 요청하면 변수 이름, 구조, 엣지 케이스 처리, 심지어 알고리즘 선택도 실행마다 달라질 수 있습니다. 이것은 버그가 아닙니다—기술이 작동하는 방식입니다. LLM은 토큰에 대한 확률 분포에서 샘플링하며, 작은 부동소수점 차이가 연쇄적으로 다른 출력을 만들어냅니다.

이것이 모든 것을 바꿉니다

계산기를 사용할 때는 출력을 신뢰합니다. LLM을 사용할 때는 출력을 검증합니다. 이것은 철학적 입장이 아닙니다—엔지니어링 요구사항입니다. AI 보조 개발이라는 전체 분야가 존재하는 이유는 LLM 출력을 기본적으로 신뢰할 수 없기 때문입니다.

이것은 역설을 만듭니다: 불안정한 생성기에서 신뢰할 수 있는 소프트웨어를 원합니다. 비결정론적 시스템이 결정론적인 것처럼 동작하게 만드는 유일한 방법은 극도로 견고한 가드레일로 둘러싸는 것입니다:

  • 테스트(Tests) — 코드가 어떻게 생성되었든 상관없이 통과하거나 통과하지 못합니다
  • 명세(Specifications) — 정밀한 지시가 가능한 출력의 공간을 줄입니다
  • 컨텍스트 문서 (CLAUDE.md, 규칙 파일) — 모델의 선택을 프로젝트 패턴에 맞게 제한합니다
  • 검토 파이프라인(Review pipelines) — 차이(diff), 린팅, 타입 검사가 모델이 놓친 것을 잡아냅니다
  • 결정론적 검증(Deterministic validation) — CI/CD, 자동화된 검사, 판단에 의존하지 않는 수락 기준

페이즈 5와 6의 모든 기술—컨텍스트 엔지니어링, 복합 규칙, 하네스 엔지니어링, Docker 격리—은 비결정론적 출력을 결정론적 경계로 제한하기 위해 존재합니다. 가드레일은 선택적 추가 사항이 아닙니다. 바로 핵심입니다. 그것 없이는 복권 티켓을 배포하는 셈입니다.

AI는 생성합니다. 여러분은 검증합니다. 가드레일이 적용합니다. 그것이 계약입니다. 이후의 모든 장은 더 나은 가드레일을 구축하는 법을 가르칩니다.

차이(Diff) 읽기

차이(diff)는 줄 단위 변경을 보여줍니다: + = 추가됨, - = 제거됨. Claude Code가 변경하면, 수락하기 전에 차이를 검토합니다. 확인 사항: 변경이 내가 요청한 것과 일치하는가? 관련 없는 변경이 있는가? 잘못 삭제된 것이 있는가?

실습 예제: 실제 차이 검토하기

숙련된 개발자가 차이를 검토하는 방법입니다. Claude에게 "load_tasks 함수에 에러 처리를 추가해줘"라고 요청했다고 가정합니다. 차이는 다음과 같습니다:

 def load_tasks(filepath):
-    with open(filepath) as f:
-        data = json.load(f)
-    return data
+    if not isinstance(filepath, str) or not filepath.strip():

+    try:

이제 줄 단위 주석입니다:

  • 1~2번 줄 (추가됨): 좋음—filepath에 대한 입력 검증이 추가되었습니다. None이나 빈 문자열이 open()에 도달해서 발생하는 불분명한 오류를 방지합니다.
  • 5번 줄 (except:): 문제—이것은 KeyboardInterrupt와 SystemExit을 포함한 모든 예외를 잡는 벌거벗은 except입니다. 예상된 실패만 잡으려면 except (FileNotFoundError, json.JSONDecodeError):이어야 합니다.
  • 6번 줄 (return []): 질문—빈 리스트를 조용히 반환하면 오류가 숨겨집니다. 의도적인 것인가요, 아니면 경고를 로그해야 할까요? 프로젝트의 오류 철학에 달려 있습니다.
  • 8~9번 줄 (data 타입 검사): 좋음—반환 전에 JSON 구조를 검증합니다. 손상된 파일을 조기에 잡아냅니다.
  • 10번 줄 (import os 제거됨): 질문—왜 이 임포트가 제거되었나요? 다른 곳에서 os를 사용하지 않는다면 괜찮은 정리입니다. 하지만 먼저 나머지 파일을 확인하세요.

판정: 수정 필요. 벌거벗은 except는 실제 버그입니다—Claude에게 특정 예외를 잡도록 요청하세요. 나머지는 탄탄합니다.

미니 연습

1: 환각 잡기

AI에게 질문하세요: "XQVZ 형식 파일을 파싱하는 데 가장 좋은 Python 패키지가 무엇인가요?" 자신 있게 패키지를 추천하면 PyPI에서 검색해보세요. 해당 형식은 만들어낸 것이므로 거의 확실히 존재하지 않습니다.

2: AI 출력 평가하기

AI에게 is_valid_email을 작성하게 합니다. 확인 사항: 빈 문자열을 처리하는가? 여러 개의 @는? 공백은? 빠진 것이 있다면 AI가 실패한 것이고—여러분이 잡아낸 것입니다.

직접 해보기

AI가 임의의 함수를 생성하게 합니다. 평가 파이프라인을 적용하세요: (1) 출력 읽기. (2) 요청한 것과 대조. (3) 옳은 것 하나 찾기. (4) 틀리거나 빠진 것 하나 찾기. (5) 판정: 수락, 수정, 또는 거부. 평가를 3~5문장으로 작성하세요.

검증: 평가에 구체적으로 올바른 요소 하나와 구체적인 문제 하나가 포함되어야 합니다.

작동하지 않으면: 문제를 찾을 수 없다면 더 어려운 것—엣지 케이스가 5개 이상인 함수—을 요청하세요.

방금 AI 생성 코드를 평가하고 실제 문제를 발견했습니다. 그것은 트집 잡기가 아닙니다—AI를 사용하는 사람과 AI를 감독하는 사람을 구분하는 기술입니다.

팰린드롬은 앞뒤로 읽어도 같습니다. Python은 s[::-1]로 문자열을 뒤집을 수 있습니다.

전체 함수는 다음과 같이 할 수 있습니다: 문자열을 정리하고, 소문자로 변환하고, 뒤집은 것과 비교합니다.

AI는 re, functools를 임포트하고, 메모이제이션이 있는 재귀를 사용하고, 타입 검사를 추가했습니다—단순한 팰린드롬 검사에는 필요하지 않은 것들입니다.

누락된 엣지 케이스 찾기

AI가 KEY=VALUE 설정 파일을 파싱하는 parse_config(text)를 작성했습니다. 유효한 입력에는 동작하지만 3가지 일반적인 엣지 케이스에서 충돌합니다. 3가지 다른 잘못된 입력으로 parse_config를 호출하고 실패를 잡아내는 test_edge_cases()를 작성하세요. 그런 다음 모든 엣지 케이스가 충돌하는 대신 합리적인 결과를 반환하도록 함수를 수정하세요.

parse_config("")을 시도해 보세요—""를 =로 분할하려 하면 ["key", "value"]가 아닌 [""]를 얻습니다.

=로 분할하기 전에 빈 줄과 #으로 시작하는 줄을 건너뜁니다.

=를 포함하는 값이 깨지지 않도록 첫 번째 =에서만 분할하는 line.split("=", 1)을 사용합니다.

40장 레벨 3 — 컨텍스트 엔지니어링

지금 이것이 중요한 이유

평범한 AI 출력과 탁월한 출력의 차이는 거의 모델에 있지 않습니다—컨텍스트에 있습니다. 모델이 어떤 파일을 보는가? 어떤 지시가 이끄는가? 어떤 제약이 나쁜 출력을 막는가? 컨텍스트 엔지니어링은 AI 보조 개발에서 가장 레버리지가 높은 기술입니다.

컨텍스트 엔지니어링(Context engineering)은 모델이 보는 것을 제어해 모든 토큰이 유용한 작업을 하도록 하는 것입니다. "모든 토큰은 프롬프트에서 자리를 차지할 이유가 있어야 합니다."

컨텍스트 창(Context Window)

컨텍스트 창을 모델의 작업 메모리 예산이라고 생각하세요. 모델이 한 번에 고려할 수 있는 모든 것—프롬프트, 시스템 지시, 파일 내용, 대화 기록—이 이 예산에 맞아야 합니다. 채워지면 오래된 정보가 밀려납니다. 이 예산을 관리하는 것이 컨텍스트 엔지니어링의 핵심 기술입니다.

네 가지 표면

1
CLAUDE.md

모든 세션 시작 시 읽히는 지속적 지시사항. 200줄 이하, 구체적이고 실행 가능해야 합니다.

2
도구 설명(Tool Descriptions)

모델이 어떤 도구를 호출할지 결정하기 위해 읽습니다. 명확한 설명 = 더 나은 도구 사용.

3
대화 관리(Conversation Management)

컨텍스트 ~50%에서 /compact. 탐색에는 서브에이전트(subagent). 작업 전환에는 /clear.

4
선택적 컨텍스트(Selective Context)

전체 코드베이스가 아닌 관련 파일만 포함합니다. 노이즈가 적을수록 = 더 나은 출력.

TaskForge를 위한 CLAUDE.md 예시

# TaskForge
Command-line task manager with JSON persistence and Flask API.

## Commands
- `python3 -m pytest` — run tests
- `python3 -m flask run` — start API server
- `python3 src/taskforge/main.py` — run CLI

## Architecture
- src/taskforge/ — core logic (main.py, models.py, api.py)
- tests/ — pytest test files
- data/ — JSON storage (gitignored)

## Code Style
- Type hints on all function signatures
- Docstrings on all public functions
- No bare try/except — always catch specific exceptions

## Gotchas
- Task IDs are auto-incrementing integers, NOT UUIDs
- JSON file is the single source of truth
- Flask API mirrors CLI functionality exactly

전/후: 컨텍스트 품질

같은 모델이라도 다른 컨텍스트로는 극적으로 다른 결과를 생성합니다:

모호한 프롬프트
"TaskForge에 검색 추가해줘"
제약된 프롬프트
"TaskForge에 작업 설명을 대소문자 구분 없이 매칭하는 search_tasks(keyword: str) -> list[dict]를 추가해줘. 매칭 없으면 빈 리스트 반환. 테스트 3개 추가: 매칭 있음, 매칭 없음, 빈 키워드."

왜 더 나은가: 타입이 명시됨. 동작이 정의됨. 엣지 케이스가 나열됨. 모델이 틀릴 여지가 줄어듭니다.
노이즈가 많은 컨텍스트
2,000줄짜리 전체 코드베이스를 프롬프트에 붙여넣고, 관련 없는 README 텍스트와 오래된 변경 로그도 추가.
선택적 컨텍스트
models.py(함수가 들어갈 곳)와 test_models.py(테스트가 들어갈 곳)만 포함합니다. 다른 파일은 모델이 필요 시 찾도록 합니다.

왜 더 나은가: 노이즈가 적으면 모델이 컨텍스트 창을 관련 없는 텍스트가 아닌 관련 코드에 사용합니다.
약한 CLAUDE.md
"이것은 Python 프로젝트입니다. 좋은 코드를 작성해주세요. 모범 사례를 따르세요."
실행 가능한 CLAUDE.md
"테스트는 pytest 사용. 모든 시그니처에 타입 힌트. bare except 금지. 작업 ID는 UUID가 아닌 정수. JSON 파일이 단일 정보 소스."

왜 더 나은가: 모든 줄이 모델이 따를 수 있는 구체적인 제약입니다. 추측이 필요 없습니다.
컨텍스트 창에는 예산이 있습니다. 중요한 정보를 시작과 끝에 배치하세요. 응답을 위해 20~30%를 남겨두세요.

지시 스캐폴드(Instruction Scaffold) 구축하기

소규모 프로젝트용 CLAUDE.md는 간단합니다. 하지만 목표가 더 크다면—Claude Code에게 처음부터 전체 환경을 설정하는 방법을 알려주는 것처럼요? 여기서 지시 스캐폴드를 구축합니다: 비결정론적 모델이 거의 항상 같은 올바른 출력을 생성할 만큼 정밀한 문서입니다.

38장의 핵심 문제를 기억하세요: LLM은 비결정론적입니다. Claude에게 "이 프로젝트에 Docker를 설정해줘"를 다섯 번 요청하면 다섯 가지 다른 Dockerfile, 다른 compose 설정, 다른 인증 방식을 받게 됩니다. 일부는 작동합니다. 일부는 작동하지 않습니다. 변동성이 적입니다.

해결책은 모든 결정 지점에서 모호성을 제거하는 스캐폴드입니다:

1
한 문장으로 목표 명시하기

"Docker 설정해줘"가 아니라 "컨테이너 하나. exec로 들어가면 저장소를 가리키는 Claude Code가 있어야 함. CMD는 sleep infinity. 그게 전부."

2
정확한 버전과 명령어 명시하기

"Claude Code 설치"가 아니라 curl -fsSL https://claude.ai/install.sh | bash. "Debian 사용"이 아니라 FROM debian:bookworm-slim. 명시되지 않은 모든 선택은 모델이 절반의 확률로 틀릴 동전 던지기입니다.

3
잘못된 접근 방식 명시적으로 금지하기

"npm install 절대 사용 금지—인증/PATH 손상 유발." "환경 블록에 CLAUDE_MODEL 추가 금지—실제 환경 변수가 아님." 음수 제약은 양수 제약만큼 중요합니다. 모델의 학습 데이터에도 모든 잘못된 접근 방식이 포함되어 있기 때문입니다.

4
정확한 파일 내용 제공하기

Dockerfile이 포함해야 할 내용을 설명하지 말고—Dockerfile을 직접 작성하세요. 모델이 즉흥적으로 만드는 것이 아니라 복사해야 합니다. 처음부터 생성하는 모든 줄은 실행마다 달라질 수 있는 줄입니다.

5
검증 체크리스트 추가하기

출력이 올바른지 검증하는 예/아니오 확인 목록. "기본 이미지가 debian:bookworm-slim인가?" "CMD가 sleep infinity인가?" ".env에 기존 줄이 모두 있는가?" 이것이 비결정론적 출력 주변의 결정론적 가드레일입니다.

6
기존 파일 보호하기

".env 절대 덮어쓰기 금지—추가만 허용." "프로젝트의 기존 Dockerfile 건드리지 말 것." 파괴는 되돌릴 수 없습니다. 스캐폴드는 올바른 일을 하기보다 망가뜨리기를 더 어렵게 만들어야 합니다.

이 수준의 정밀도가 중요한 이유

"Claude Code용 Docker 설정해줘"와 같은 모호한 지시는 모델에게 수천 가지 가능한 해석을 줍니다. 정확한 파일 내용, 금지된 접근 방식, 검증 체크리스트가 있는 정밀한 스캐폴드는 본질적으로 하나로 좁힙니다. 질문을 할 수 있는 인간 동료를 위한 지시를 작성하는 것이 아닙니다—해석의 여지를 남기면 자신 있게 잘못된 일을 할 시스템을 위한 제약을 작성하는 것입니다. 가드레일이 촘촘할수록 출력이 결정론적입니다.

24장 (하네스 엔지니어링)에서 이러한 원칙들을 모두 적용해 Claude Code용 Docker 환경을 설정하는 완전한 스캐폴드를 볼 수 있습니다. 300줄이 넘으며—모든 줄은 빠뜨렸을 때 실제로 실패가 발생했기 때문에 존재합니다.

41장 레벨 4 — 복합 엔지니어링(Compounding Engineering)

지금 이것이 중요한 이유

복합 없이는 매 세션마다 같은 문제를 해결합니다. Claude가 같은 실수를 하고, 같은 문제를 수정하며, 진전이 쌓이지 않습니다. 복합 엔지니어링은 모든 수정을 영구적인 개선으로 만듭니다—제자리에서 뛰는 것과 동력을 쌓는 것의 차이입니다.

이 장에서 다루는 내용

이 장은 두 가지 핵심 주제를 다룹니다: 복합 엔지니어링(compounding engineering) (교훈을 지속적 규칙과 컨텍스트로 성문화하여 시간이 지남에 따라 AI를 더 효과적으로 만들기)과 명세 기반 개발 워크플로우(spec-driven development workflow) (AI 보조 소프트웨어 개발의 전문적 접근 방식으로, 브레인스토밍, 명세 작성, 프롬프트 계획 생성, 단계별 실행).

복합 엔지니어링은 현재 세션 이후 모든 세션을 개선합니다. 루프: 계획(Plan) → 위임(Delegate) → 평가(Assess) → 성문화(Codify). 성문화 단계가 복합을 만드는 것입니다.

성문화 단계가 복합 엔지니어를 다른 사람들과 구분합니다. 그것 없이는 같은 문제를 반복해서 해결합니다.

명세 기반 워크플로우(Spec-Driven Workflow)

AI 보조 개발에서 가장 효과적인 단일 워크플로우. 코드로 바로 뛰어드는 대신: 브레인스토밍 → 명세 → 계획 → 실행으로 진행합니다.

명세 기반 워크플로우: 절대 코드로 시작하지 마세요. 이해로 시작하고, 적어두고, 단계로 나누고, 실행하세요.

1단계: 브레인스토밍

코드가 아닌 대화로 시작합니다. AI에게 여러분에게 질문하도록 요청합니다:

I want to build [feature]. Before we write any code, ask me clarifying questions
one at a time to understand: what problem this solves, who uses it, what the core
features are, what it should NOT do, and what edge cases matter.

AI가 명확화 질문을 합니다. 여러분이 답합니다. 출력: 문제에 대한 공유된 이해.

2단계: 명세 작성 (spec.md)

브레인스토밍을 서면 명세로 전환합니다:

  • 시스템이 하는 것 (요구사항)
  • 시스템이 하지 않는 것 (명시적 범위 경계)
  • 데이터 모델 (저장되는 것)
  • API 또는 인터페이스 설계 (입력, 출력, 오류)
  • 엣지 케이스와 오류 처리
  • 수락 기준 (완료를 어떻게 아는지)

3단계: 계획 생성 (prompt_plan.md)

명세를 순서대로 된 구현 단계로 나눕니다. 각 단계는 AI에게 줄 프롬프트이며, 5분 이내에 검토할 수 있을 만큼 작아야 합니다.

4단계: 단계별 실행

계획에서 각 프롬프트를 하나씩 줍니다. 출력을 검토합니다. 테스트를 실행합니다. 다음으로 넘어가기 전에 문제를 수정합니다. 검토를 건너뛰지 마세요. 여러 단계를 일괄 처리하지 마세요.

왜 효과적인가: 명세가 AI의 기억이 아닌 정보 소스입니다. 계획이 체크포인트를 만듭니다. 무언가 잘못되면 정확히 어디서인지 압니다. 그리고 복합됩니다 — 프로젝트마다 명세 품질이 향상됩니다.

추가 읽기: 명세 기반 접근법

이 워크플로우는 매일 AI로 프로덕션 소프트웨어를 구축하는 실무자들에게서 적용했습니다:

  • Harper Reed: My LLM Codegen Workflow — 원래의 명세 기반 워크플로우.
  • Harper Reed: Basic Claude Code — Claude Code 작업을 위한 실용적 패턴.

연습: 나쁜 명세 분석하기

잘못 작성된 명세가 주어지면 문제를 파악하세요. 누락된 섹션이 있는 딕셔너리를 반환합니다.

"requirements", "data model", "edge case", "acceptance"와 같은 키워드가 텍스트에 나타나는지 확인합니다(대소문자 구분 없음).

"should work well" 또는 "make it nice"와 같은 모호한 표현을 찾으세요 — 테스트할 수 없기 때문에 문제입니다.

각 섹션 키워드를 확인합니다. "명시적 요구사항 목록 없음", "데이터 모델 정의 없음", "모호한 요구사항 포함: 'should work well'"과 같은 문제를 추가합니다.
