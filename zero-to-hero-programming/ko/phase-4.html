13장 API와 HTTP

지금 이것이 중요한 이유

거의 모든 현대 애플리케이션은 인터넷을 통해 다른 애플리케이션과 통신합니다. 날씨 앱은 예보를 가져오고, 결제 시스템은 요금을 처리하고, AI 도구는 프롬프트를 보내고 완성된 결과를 받습니다. 이 모든 것이 공통으로 사용하는 언어가 HTTP이고, 따르는 대화 패턴을 API라고 합니다. API를 이해하면 자신의 컴퓨터에서만 실행되던 코드를 세상과 연결하는 코드로 발전시킬 수 있습니다.

API란 무엇인가?

API(Application Programming Interface, 애플리케이션 프로그래밍 인터페이스)는 두 프로그램 사이의 계약입니다. 한 프로그램이 "이 형식으로 요청을 보내면 저 형식으로 응답을 돌려주겠다"고 말하는 것입니다. API는 규칙을 정의합니다—무엇을 요청할 수 있는지, 어떻게 요청하는지, 무엇을 돌려받는지.

사실 이미 모르는 사이에 API를 사용해 왔습니다. Python에서 len([1, 2, 3])을 호출할 때 함수 API를 사용하는 것입니다—리스트를 전달하면 길이를 반환합니다. 웹 API도 동일하게 작동하지만, 함수가 원격 서버에 있고 호출이 인터넷을 통해 전달된다는 점이 다릅니다.

식당에 비유해 보겠습니다. 여러분(클라이언트(client))은 메뉴(API 문서(API documentation))를 읽고, 웨이터에게 주문을 합니다(요청(request) 전송). 주방(서버(server))이 음식을 준비하고, 웨이터가 가져다 줍니다(응답(response)). 여러분은 절대 주방에 들어가지 않습니다. 메뉴가 바로 계약입니다.

HTTP: 웹의 언어

HTTP(HyperText Transfer Protocol, 하이퍼텍스트 전송 프로토콜)는 클라이언트와 서버가 통신할 때 사용하는 프로토콜입니다. 웹사이트를 방문할 때마다 브라우저는 HTTP 요청을 보내고 HTTP 응답을 받습니다. API도 동일한 프로토콜을 사용합니다.

HTTP 교환은 두 부분으로 구성됩니다:

  • 요청(Request)—클라이언트가 보냅니다. 메서드, URL, 헤더, 그리고 선택적으로 본문(body)을 포함합니다.
  • 응답(Response)—서버가 보냅니다. 상태 코드(status code), 헤더, 그리고 본문을 포함합니다.
HTTP 요청-응답 주기: 클라이언트가 메서드와 URL을 포함한 요청을 보내면, 서버가 처리하고 상태 코드와 데이터가 담긴 응답을 반환합니다.

HTTP 메서드

메서드(method)는 서버에 무엇을 원하는지 알려줍니다. 알아야 할 네 가지가 있습니다:

HTTP 메서드와 그 목적
메서드목적예시
GET데이터 읽기/조회할 일 목록 가져오기
POST새 데이터 생성새 할 일 추가
PUT기존 데이터 수정할 일을 완료로 표시
DELETE데이터 삭제할 일 삭제

GET과 DELETE는 일반적으로 본문을 보내지 않습니다. POST와 PUT은 요청 본문에 데이터(보통 JSON)를 담아 보냅니다.

상태 코드

모든 HTTP 응답에는 상태 코드(status code)가 포함됩니다—클라이언트에게 무슨 일이 일어났는지 알려주는 세 자리 숫자입니다. 전부 외울 필요는 없고, 다음 다섯 그룹만 알면 됩니다:

HTTP 상태 코드 범위와 의미
범위의미주요 코드
2xx성공200 OK, 201 Created
3xx리다이렉트301 Moved Permanently
4xx클라이언트 오류 (요청 측 실수)400 Bad Request, 404 Not Found
5xx서버 오류 (서버 측 문제)500 Internal Server Error

문제가 생겼을 때 가장 먼저 확인해야 할 것이 상태 코드입니다. 404는 URL이 잘못됐다는 뜻입니다. 401은 인증되지 않았다는 뜻입니다. 500은 서버가 망가진 것이므로 여러분의 잘못이 아닙니다.

JSON: 데이터 형식

API는 데이터를 주고받기 위한 공통 형식이 필요합니다. 표준은 JSON(JavaScript Object Notation, 자바스크립트 객체 표기법)입니다. 이미 JSON을 알고 있습니다—Python 딕셔너리 및 리스트와 똑같이 생겼습니다:

{
    "id": 1,
    "title": "Buy groceries",
    "done": false
}

Python의 json 모듈은 JSON 문자열과 Python 객체 사이를 변환합니다. 아래에서 소개할 requests 라이브러리를 사용하면 이 변환이 자동으로 이루어집니다.

헤더

헤더(headers)는 요청과 응답에 첨부된 메타데이터입니다. 데이터 자체가 아니라 메시지에 관한 정보를 전달합니다. 자주 보게 될 두 가지 헤더:

  • Content-Type: application/json — 서버에 "JSON을 보내고 있습니다"라고 알립니다.
  • Authorization: Bearer <token> — 신원을 증명합니다(비밀번호와 같은 역할).

헤더를 직접 설정하는 경우는 드뭅니다—requests 라이브러리 같은 도구가 대부분을 처리해 줍니다.

GET 요청 만들기

import requests

response = requests.get("https://api.github.com/users/octocat")

print(response.status_code)  # 200
print(response.json())       # Python dict with user data

requests.get()은 URL에 GET 요청을 보냅니다. 응답 객체는 .status_code로 상태 코드를, .json()으로 파싱된 JSON 본문을 제공합니다.

POST 요청 만들기

import requests

data = {"title": "Buy groceries", "done": False}
response = requests.post("http://localhost:5000/tasks", json=data)

print(response.status_code)  # 201
print(response.json())       # {"id": 1, "title": "Buy groceries", "done": false}

json=data 매개변수는 Python 딕셔너리를 JSON으로 자동 변환하고 Content-Type 헤더도 설정해 줍니다.

네 가지 메서드 모두 사용하기

import requests

BASE = "http://localhost:5000"

# GET    - 읽기
r = requests.get(f"{BASE}/tasks")

# POST   - 생성
r = requests.post(f"{BASE}/tasks", json={"title": "New task"})

# PUT    - 수정
r = requests.put(f"{BASE}/tasks/1/done")

# DELETE - 삭제
r = requests.delete(f"{BASE}/tasks/1")

흔한 오해

"API는 웹 개발자만 사용한다"

API는 프로그램들이 서로 대화하는 방식입니다. 데이터 과학자는 API로 데이터셋을 가져오고, DevOps 엔지니어는 API로 클라우드 인프라를 관리하고, AI 에이전트는 API로 언어 모델을 호출합니다. 외부 서비스와 상호작용하는 코드를 작성한다면 API를 사용하는 것입니다.

"GET과 POST는 서로 바꿔 쓸 수 있다"

그렇지 않습니다. GET은 데이터를 조회하며 서버에서 아무것도 수정해서는 안 됩니다. POST는 새 데이터를 생성합니다. 잘못된 메서드를 사용하면 다른 개발자들이 혼란스러워하고, 이러한 관례에 의존하는 도구들이 오작동합니다. 메서드는 의도를 전달합니다.

Flask란 무엇인가?

Flask는 최소한의 Python 웹 프레임워크(web framework)입니다. 몇 줄의 코드만으로 Python 함수를 API 엔드포인트(endpoint)로 만들 수 있습니다. 더 큰 프레임워크(Django)와 달리, Flask는 필요한 것만 제공하고 나머지는 개발자의 방식을 따릅니다.

첫 번째 Flask 앱

5줄로 완성하는 Flask 앱:

from flask import Flask

app = Flask(__name__)

@app.route("/")
def hello():
    return "Hello, World!"

if __name__ == "__main__":
    app.run(debug=True)

app.py로 저장하고 python3 app.py로 실행하세요. 브라우저에서 http://localhost:5000을 열어보세요. 방금 웹 서버를 구축했습니다.

@app.route("/")는 데코레이터(decorator)입니다—어떤 URL이 어떤 함수를 실행할지 Flask에 알려줍니다. debug=True는 코드 변경 시 자동으로 다시 로드하고 유용한 오류 페이지를 표시합니다.

요청 데이터 읽기

클라이언트가 API에 데이터를 보내면 이를 읽어야 합니다. Flask는 두 가지 주요 방법을 제공합니다:

from flask import Flask, request, jsonify

# JSON 본문 (POST/PUT 요청용)
@app.route("/tasks", methods=["POST"])
def create_task():
    data = request.json          # JSON 본문 파싱
    title = data.get("title")   # 안전하게 필드 가져오기
    return jsonify({"title": title}), 201

# 쿼리 파라미터 (GET 요청용)
# GET /tasks?status=pending
@app.route("/tasks", methods=["GET"])
def get_tasks():
    status = request.args.get("status", "all")  # 기본값은 "all"
    return jsonify({"filter": status})

request.json은 클라이언트가 보낸 JSON 본문을 파싱합니다. request.args는 URL 쿼리 파라미터(? 뒤의 모든 것)를 읽습니다.

TaskForge API 구축하기

TaskForge를 위한 완전한 REST API를 단계별로 구축해 보겠습니다. 데이터베이스 없이 메모리 내 리스트를 사용하여 할 일을 저장합니다.

1단계: 설정 및 데이터 저장소

from flask import Flask, jsonify, request

app = Flask(__name__)

# 메모리 내 할 일 저장소
tasks = []
next_id = 1

2단계: GET /tasks — 모든 할 일 목록

@app.route("/tasks", methods=["GET"])
def get_tasks():
    return jsonify(tasks), 200

전체 할 일 목록을 JSON 배열로 반환합니다. 단순하고 직접적입니다.

3단계: POST /tasks — 할 일 추가

@app.route("/tasks", methods=["POST"])
def create_task():
    global next_id
    data = request.json

    if not data or "title" not in data:
        return jsonify({"error": "title is required"}), 400

    task = {
        "id": next_id,
        "title": data["title"],
        "done": False
    }
    tasks.append(task)
    next_id += 1
    return jsonify(task), 201

유효성 검사에 주목하세요: 클라이언트가 title을 보내지 않으면 명확한 오류 메시지와 함께 400 Bad Request를 반환합니다. 클라이언트 입력을 절대 신뢰하지 마세요.

4단계: PUT /tasks/<id>/done — 할 일 완료 처리

@app.route("/tasks/<int:task_id>/done", methods=["PUT"])
def complete_task(task_id):
    for task in tasks:
        if task["id"] == task_id:
            task["done"] = True
            return jsonify(task), 200

    return jsonify({"error": "task not found"}), 404

<int:task_id>는 URL 변수(URL variable)입니다—Flask가 URL에서 숫자를 추출하여 함수 매개변수로 전달합니다. 일치하는 할 일이 없으면 404를 반환합니다.

5단계: 실행하기

if __name__ == "__main__":
    app.run(debug=True)
요청이 도착하면 Flask가 URL을 라우트에 매칭하고, 핸들러 함수가 실행되어 데이터를 반환하며, 클라이언트는 JSON 응답을 받습니다.

API에서의 오류 처리

좋은 API는 클라이언트가 무엇이 잘못됐는지와 어떻게 수정해야 하는지 알 수 있도록 명확한 오류 응답을 반환합니다:

# 잘못된 요청: 필수 필드 누락
@app.route("/tasks", methods=["POST"])
def create_task():
    data = request.json
    if not data or "title" not in data:
        return jsonify({"error": "title is required"}), 400

    if not isinstance(data["title"], str) or len(data["title"].strip()) == 0:
        return jsonify({"error": "title must be a non-empty string"}), 400

    # ... 할 일 생성

# 찾을 수 없음
@app.route("/tasks/<int:task_id>/done", methods=["PUT"])
def complete_task(task_id):
    for task in tasks:

@app.errorhandler(500)

15장 데이터베이스 기초

지금 이것이 중요한 이유

TaskForge는 현재 닫을 때마다 모든 데이터를 잃어버립니다. 데이터베이스(database)가 이 문제를 영구적으로 해결합니다. 모든 실제 애플리케이션이 데이터베이스를 사용합니다.

데이터베이스란 무엇인가?

데이터베이스는 프로그램 실행이 종료된 이후에도 지속되는 구조화된 저장소입니다. Python 스크립트를 닫으면 변수가 사라집니다. 데이터베이스를 사용하는 프로그램을 닫아도 데이터는 남아 있습니다. TaskForge부터 GitHub, Anthropic API까지 모든 실제 애플리케이션은 데이터베이스에 데이터를 저장합니다.

SQL vs NoSQL

데이터베이스에는 두 가지 주요 계열이 있습니다. 이 과정에서는 SQL에 집중합니다.

SQLite — 첫 번째 데이터베이스

SQLite부터 시작하는 이유는 시작하는 모든 장벽을 없애주기 때문입니다:

  • 서버 불필요—서비스가 아닌 라이브러리입니다. 설치, 설정, 비밀번호 없이 사용 가능합니다.
  • Python 내장—import sqlite3가 바로 작동합니다. pip install 불필요합니다.
  • 파일 기반—전체 데이터베이스가 단일 파일입니다(tasks.db). 일반 파일처럼 복사, 백업, 삭제할 수 있습니다.

SQLite는 장난감이 아닙니다. 모든 iPhone, 모든 Android 폰, 모든 웹 브라우저, 그리고 모든 Python 설치에서 프로덕션으로 사용됩니다. 학습용이나 TaskForge 같은 단일 사용자 애플리케이션에 적합한 선택입니다.

기본 SQL

SQL(Structured Query Language)은 데이터베이스와 대화하는 언어입니다. 핵심은 단 5개의 명령어입니다:

CREATE TABLE — 구조 정의하기

CREATE TABLE tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    done BOOLEAN DEFAULT 0
);

세 개의 컬럼이 있는 테이블을 생성합니다. PRIMARY KEY AUTOINCREMENT는 각 할 일이 고유하게 자동 할당된 ID를 갖는다는 의미입니다. NOT NULL은 title이 필수임을 의미합니다. DEFAULT 0은 새 할 일이 미완료 상태로 시작한다는 의미입니다.

INSERT — 데이터 추가하기

INSERT INTO tasks (title, done) VALUES ('Buy groceries', 0);
INSERT INTO tasks (title, done) VALUES ('Write tests', 0);
INSERT INTO tasks (title, done) VALUES ('Deploy API', 1);

SELECT — 데이터 읽기

-- 모든 할 일 가져오기
SELECT * FROM tasks;

-- 미완료 할 일만 가져오기
SELECT * FROM tasks WHERE done = 0;

-- 제목만 가져오기
SELECT title FROM tasks WHERE done = 0;

UPDATE — 데이터 수정하기

UPDATE tasks SET done = 1 WHERE id = 1;

항상 WHERE 절을 포함하세요. 없으면 모든 행이 수정됩니다.

DELETE — 데이터 삭제하기

DELETE FROM tasks WHERE id = 1;

동일한 규칙: 항상 WHERE를 사용하세요. WHERE 없이 DELETE FROM tasks를 실행하면 모든 것이 삭제됩니다.

Python의 sqlite3 모듈

Python은 표준 라이브러리에 sqlite3를 포함합니다. 작업 흐름은 다음과 같습니다:

import sqlite3

# 1. 데이터베이스에 연결 (파일이 없으면 생성)
conn = sqlite3.connect("tasks.db")

# 2. 커서 생성 (SQL을 실행하는 객체)
cursor = conn.cursor()

# 3. SQL 실행
cursor.execute("""
    CREATE TABLE IF NOT EXISTS tasks (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        title TEXT NOT NULL,
        done BOOLEAN DEFAULT 0
    )
""")

# 4. 데이터 삽입
cursor.execute("INSERT INTO tasks (title, done) VALUES (?, ?)", ("Buy groceries", 0))

# 5. 변경 사항 커밋 (디스크에 저장)
conn.commit()

# 6. 데이터 조회
cursor.execute("SELECT * FROM tasks")
rows = cursor.fetchall()
for row in rows:
    print(row)  # (1, 'Buy groceries', 0)

# 7. 연결 종료
conn.close()

주요 함수: connect()는 데이터베이스를 열고, cursor()는 실행자를 만들고, execute()는 SQL을 실행하고, fetchall()은 결과를 가져오고, commit()은 변경 사항을 저장하고, close()는 정리합니다.

매개변수화된 쿼리

SQL 인젝션(SQL Injection) — 문자열 서식 절대 사용 금지

항상 ? 플레이스홀더를 사용하세요. f-문자열, .format(), 또는 + 연결로 SQL 문자열을 만들지 마세요. 문자열 서식을 사용하면 공격자가 악성 SQL을 데이터베이스에 주입할 수 있습니다. 이것은 이론적인 이야기가 아닙니다—SQL 인젝션은 실제 애플리케이션에서 가장 흔한 보안 취약점 중 하나입니다.

# 위험 — 절대 이렇게 하지 마세요
title = input("Task title: ")
cursor.execute(f"INSERT INTO tasks (title) VALUES ('{title}')")
# 사용자가 입력할 수 있음: '); DROP TABLE tasks; --
# 그러면 전체 테이블이 삭제됩니다.

# 안전 — 항상 이렇게 하세요
title = input("Task title: ")
cursor.execute("INSERT INTO tasks (title) VALUES (?)", (title,))
# ? 플레이스홀더가 이스케이프를 자동으로 처리합니다.

? 플레이스홀더는 sqlite3에 값을 안전하게 이스케이프하도록 알려줍니다. 이 한 가지 규칙으로 보안 취약점의 전체 범주를 예방할 수 있습니다.

import sqlite3

def get_connection():
    conn = sqlite3.connect("tasks.db")
    conn.row_factory = sqlite3.Row  # 컬럼 이름으로 접근 가능
    return conn

def init_db():
    conn = get_connection()
    conn.execute("""
        CREATE TABLE IF NOT EXISTS tasks (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            title TEXT NOT NULL,
            done BOOLEAN DEFAULT 0
        )
    """)
    conn.commit()
    conn.close()

def add_task(title):
    conn = get_connection()
    conn.execute("INSERT INTO tasks (title) VALUES (?)", (title,))
    conn.commit()
    conn.close()

def list_tasks():
    conn = get_connection()
    rows = conn.execute("SELECT * FROM tasks").fetchall()
    conn.close()
    return [dict(row) for row in rows]

def complete_task(task_id):
    conn = get_connection()
    conn.execute("UPDATE tasks SET done = 1 WHERE id = ?", (task_id,))
    conn.commit()
    conn.close()

가장 큰 차이점: 더 이상 전체 파일을 메모리에 로드하지 않고, 매번 변경 시 전체 파일을 다시 쓰지 않아도 됩니다. 데이터베이스는 수천 개의 할 일이 있어도 읽기와 쓰기를 효율적으로 처리합니다.

Python 코드가 sqlite3 모듈을 통해 SQL을 전송하고, sqlite3 모듈이 단일 데이터베이스 파일을 읽고 씁니다. 서버 불필요합니다.

ORM — 미리 보기

ORM(Object-Relational Mapper, 객체-관계형 매퍼)은 Python 객체와 데이터베이스 행 사이의 간격을 채웁니다. 원시 SQL을 작성하는 대신, 테이블에 매핑되는 Python 클래스를 정의합니다:

# 원시 SQL (위에서 배운 방법)
cursor.execute("SELECT * FROM tasks WHERE done = 0")
rows = cursor.fetchall()

# ORM 방식 (개념적 — SQLAlchemy)
pending_tasks = Task.query.filter_by(done=False).all()

Python에서 가장 인기 있는 ORM은 SQLAlchemy입니다. 여기서는 깊이 다루지 않습니다—원시 SQL은 ORM이 내부적으로 무엇을 하는지 더 잘 이해하게 해줍니다. 하지만 ORM이 존재하며 대부분의 프로덕션 애플리케이션이 사용한다는 것을 알아두세요.

흔한 오해

"JSON 파일도 저장소로 충분하다"

간단한 경우에는 작동하지만, 동시 접근(두 프로세스가 동시에 쓰면 파일이 손상됨)을 처리하지 못하고, 전체 데이터셋을 메모리에 로드해야 하며, 쿼리 기능이 없습니다—모든 것을 로드하지 않고는 "미완료 할 일 모두 주세요"라고 요청할 수 없습니다. 데이터베이스는 이 세 가지 문제를 모두 해결합니다.

"SQL은 어렵다"

기본은 5개의 명령어입니다: CREATE, INSERT, SELECT, UPDATE, DELETE. 방금 배웠습니다. 고급 기능(조인, 인덱스, 트랜잭션)도 있지만, 기본만으로도 실제 애플리케이션을 구축할 수 있습니다.

16장 Docker와 컨테이너

지금 이것이 중요한 이유

Docker는 "내 컴퓨터에서는 됩니다" 문제를 영구적으로 해결합니다. 전문 팀이 일관된 환경을 보장하는 방법이며, 6단계에서 격리된 컨테이너에서 Claude Code 에이전트를 실행하는 방법입니다.

Docker란 무엇인가?

Docker는 애플리케이션과 필요한 모든 것—Python 버전, 라이브러리, 시스템 도구—을 어떤 컴퓨터에서도 동일하게 실행되는 단일 컨테이너(container)로 패키징할 수 있게 해줍니다. 해운 컨테이너를 생각해 보세요: 배가 도쿄로 가든 로테르담으로 가든 상관없이, 박스는 동일하고 내용물은 그대로 도착합니다. Docker 컨테이너도 소프트웨어에서 동일하게 작동합니다.

컨테이너는 가상 머신(VM)이 아닙니다. VM은 자체 커널이 있는 전체 운영 체제를 에뮬레이션합니다—무겁고 시작하는 데 느립니다. 컨테이너는 호스트 OS 커널을 공유하고 애플리케이션 레이어만 격리합니다. 이로 인해 컨테이너는 가볍고(기가바이트 대신 메가바이트) 빠르게 시작됩니다(분 대신 초).

첫 번째 컨테이너

Docker의 내장 테스트 이미지를 실행하여 모든 것이 작동하는지 확인하세요:

터미널docker run hello-world

Docker는 Docker Hub(이미지의 공개 레지스트리)에서 hello-world 이미지를 가져와, 컨테이너를 만들고, 실행하고, 성공 메시지를 출력합니다. 전체 작업 흐름이 바로 이것입니다: 풀, 생성, 실행.

더 유용한 것을 시도해 보겠습니다—컨테이너 내부의 Python REPL:

터미널docker run -it python:3.12 python3

-it 플래그는 인터랙티브 터미널을 제공합니다. 이제 호스트 머신과 완전히 격리된 Python 3.12 환경 내부에 있습니다. exit()을 입력하여 나가세요.

Dockerfile 작성하기

Dockerfile은 Docker에게 이미지를 빌드하는 방법을 알려주는 텍스트 파일입니다. TaskForge용 예시입니다:

FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "taskforge.py"]

각 줄은 하나의 지침입니다:

Dockerfile 지침 한 줄씩 설명
줄하는 일
FROM python:3.12-slim공식 Python 3.12 이미지(slim 변형, 더 작은 크기)로 시작
WORKDIR /app컨테이너 내부 작업 디렉터리를 /app으로 설정
COPY requirements.txt .캐싱을 위해 requirements 파일을 먼저 컨테이너에 복사
RUN pip install -r requirements.txt컨테이너 내부에서 Python 의존성 설치
COPY . .모든 프로젝트 파일을 컨테이너에 복사
CMD ["python", "taskforge.py"]컨테이너 시작 시 기본 명령어

왜 requirements.txt를 다른 것보다 먼저 복사할까요? Docker는 각 레이어를 캐싱합니다. 코드는 변경되었지만 의존성은 변경되지 않았다면, Docker는 캐시된 의존성 레이어를 재사용합니다—리빌드를 훨씬 빠르게 만듭니다.

빌드하고 실행하기

이미지를 빌드하고 그로부터 컨테이너를 실행하세요:

터미널# 현재 디렉터리에서 "taskforge"라는 이름의 이미지를 빌드 docker build -t taskforge . # 이미지로부터 컨테이너 실행 docker run -it taskforge

-t taskforge 플래그는 이미지에 이름(태그)을 부여합니다. .은 Docker에게 현재 디렉터리를 빌드 컨텍스트(Dockerfile과 복사할 파일을 찾을 위치)로 사용하라고 알립니다.

볼륨: 데이터 지속하기

컨테이너는 일시적(ephemeral)입니다—컨테이너를 중지하면 내부에서 생성된 모든 데이터가 손실됩니다. 이것은 TaskForge의 SQLite 데이터베이스에 문제가 됩니다. 볼륨(volumes)은 호스트 디렉터리를 컨테이너에 마운트하여 재시작 후에도 데이터가 살아남게 합니다:

터미널docker run -it -v $(pwd)/data:/app/data taskforge

이는 호스트의 ./data를 컨테이너 내부의 /app/data에 매핑합니다. SQLite 데이터베이스 파일이 이 공유 디렉터리에 있으므로 컨테이너가 중지되어도 지속됩니다.

Docker Compose 기초

프로젝트에 여러 서비스(웹 API, 데이터베이스 등)가 필요할 때, Docker Compose는 단일 파일로 모두 관리합니다:

# docker-compose.yml
services:
  web:
    build: .
    ports:
      - "5000:5000"
    volumes:
      - ./data:/app/data
    command: ["python", "-m", "flask", "run", "--host=0.0.0.0"]

한 명령어로 모두 시작하세요:

터미널# 모든 서비스 시작 docker compose up # 모든 서비스 중지 docker compose down

ports 매핑 "5000:5000"은 다음을 의미합니다: 호스트의 포트 5000을 컨테이너의 포트 5000으로 전달합니다. 이를 통해 브라우저에서 Flask API에 접근합니다.

흔한 오해

주의: "Docker는 배포에만 쓴다"

Docker는 개발 중에도 똑같이 유용합니다. 팀의 모든 개발자가 동일한 환경을 갖도록 보장합니다—동일한 Python 버전, 동일한 라이브러리 버전, 동일한 시스템 도구. 더 이상 "제 컴퓨터에서는 되는데요"가 없습니다.

주의: "컨테이너는 가상 머신이다"

컨테이너는 호스트 OS 커널을 공유합니다. 하드웨어를 에뮬레이션하거나 전체 OS를 부팅하지 않습니다. 이 때문에 컨테이너는 초 단위로 시작하지만 VM은 분 단위가 걸리고, 컨테이너 이미지는 메가바이트이지만 VM 이미지는 기가바이트인 것입니다.

TaskForge 연계

TaskForge를 컨테이너화하세요. Dockerfile을 작성하고, 이미지를 빌드하고, 실행하세요. TaskForge가 호스트 머신에서와 동일하게 컨테이너 내부에서 작동하는지 확인하세요. 이것이 6단계에서 Docker로 Claude Code 에이전트를 실행하기 위한 기반입니다.

컨테이너는 격리되어 있지만 Docker 엔진과 호스트 OS를 공유합니다. 각 컨테이너는 고유한 파일시스템, 네트워크, 프로세스 공간을 갖지만 커널을 공유하기 때문에 가볍습니다.

직접 해보기

TaskForge의 Flask API를 포트 5000에서 실행하고 SQLite 데이터베이스를 위한 볼륨 마운트가 있는 docker-compose.yml을 만드세요:

docker-compose.ymlservices: web: build: . ports: - "5000:5000" volumes: - ./data:/app/data command: ["python", "-m", "flask", "run", "--host=0.0.0.0"]

docker compose up으로 시작하고, 다른 터미널에서 확인하세요:

터미널curl http://localhost:5000/tasks

확인: TaskForge가 Docker 내부에서 실행되고 호스트 머신에서 접근 가능합니다. docker compose down과 docker compose up 사이에도 데이터가 유지됩니다.

작동하지 않을 경우: (1) 포트 충돌 → "5000:5000"을 "5001:5000"으로 변경하고 curl http://localhost:5001/tasks를 사용하세요. (2) permission denied → Docker Desktop이 실행 중인지 확인하세요. (3) 빌드 실패 → requirements.txt가 프로젝트 루트에 있는지 확인하세요.

어느 곳에서나 동일하게 실행되도록 애플리케이션을 컨테이너화했습니다—내 노트북, 팀원의 컴퓨터, 또는 클라우드 서버. 이것이 격리된 AI 에이전트 컨테이너를 가능하게 하는 인프라입니다.

17장 CI/CD와 GitHub Actions

지금 이것이 중요한 이유

GitHub에 푸시할 때마다 자동으로 테스트를 실행하고, 코드를 확인하고, 심지어 앱을 배포할 수 있습니다. CI/CD는 수동 품질 검사를 자동화된 파이프라인으로 전환합니다. 5단계에서는 이 인프라에 의존하는 CI 트리거 에이전트를 사용합니다.

CI/CD란 무엇인가?

CI(Continuous Integration, 지속적 통합)는 코드가 저장소에 푸시될 때마다 자동으로 테스트를 실행하는 것을 의미합니다. 머지 전에 pytest를 실행하는 것을 기억해야 하는 대신, 시스템이 항상, 예외 없이 해줍니다.

CD(Continuous Deployment/Delivery, 지속적 배포/전달)는 한 단계 더 나아갑니다: 테스트가 통과되면 코드가 자동으로 프로덕션에 배포되거나(deployment), 패키징되어 원클릭 배포가 가능한 상태가 됩니다(delivery).

핵심 아이디어는 단순합니다: 버그가 프로덕션에 도달하기 전에 잡아내는 것입니다. 테스트가 실패하면 파이프라인이 중단되고 알림을 보냅니다. 깨진 코드가 배포되지 않습니다. "테스트 실행을 잊었어요" 사고가 없습니다.

GitHub Actions 기초

GitHub Actions는 GitHub의 내장 CI/CD 시스템입니다. 공개 저장소에는 무료이며 개인 저장소에는 넉넉한 무료 사용 시간을 제공합니다. 핵심 개념들:

GitHub Actions 핵심 개념
개념무엇인가비유
워크플로우(Workflow)자동화 파이프라인을 정의하는 YAML 파일순서가 있는 단계들의 레시피
트리거(Trigger)워크플로우를 시작하는 이벤트"시작" 신호
잡(Job)동일한 머신에서 실행되는 단계의 집합한 스테이션의 한 요리사
스텝(Step)잡 내의 단일 명령어 또는 액션레시피의 한 지침

워크플로우는 저장소 내의 .github/workflows/에 있습니다. 각 워크플로우는 YAML 파일(중괄호 대신 들여쓰기를 사용하는 JSON과 유사한 사람이 읽을 수 있는 데이터 형식)입니다.

일반적인 트리거:

  • push — 지정된 브랜치에 코드가 푸시될 때 실행
  • pull_request — PR이 열리거나 업데이트될 때 실행
  • schedule — cron 일정으로 실행(예: 야간 테스트)

첫 번째 워크플로우

모든 푸시와 풀 리퀘스트에서 테스트를 실행하는 완전하고 작동하는 GitHub Actions 워크플로우입니다:

name: Run Tests
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install -r requirements.txt
      - run: python -m pytest

각 줄을 살펴보겠습니다:

GitHub Actions 워크플로우 파일 한 줄씩 설명
줄하는 일
name: Run TestsGitHub Actions UI에 표시되는 사람이 읽을 수 있는 이름
on: push: branches: [main]main 브랜치에 코드가 푸시되면 이 워크플로우 실행
on: pull_request: branches: [main]PR이 main을 대상으로 할 때도 실행
jobs: test:test라는 잡 정의

uses: actions/checkout@v4저장소 코드를 VM에 체크아웃—없으면 VM이 비어 있음
uses: actions/setup-python@v5VM에 지정된 Python 버전 설치
with: python-version: '3.12'Python 3.12 사용
run: pip install -r requirements.txt프로젝트 의존성 설치
run: python -m pytest테스트 스위트 실행—테스트가 실패하면 워크플로우 실패

저장소에서 .github/workflows/test.yml로 저장하세요. GitHub에 푸시하면 워크플로우가 자동으로 실행됩니다.

더 많은 검사 추가하기

테스트만으로는 충분하지 않습니다. 전문적인 파이프라인은 코드 품질도 검사합니다:

ruff로 린팅하기

린팅(Linting)은 코드를 실행하지 않고도 스타일 문제, 사용하지 않는 임포트, 잠재적 버그를 잡습니다. 워크플로우에 린팅 단계를 추가하세요:

      - run: pip install ruff
      - run: ruff check .

ruff는 빠르며 Python의 흔한 실수를 많이 잡아냅니다. 수년간 표준이었던 flake8이 대안입니다.

타입 검사

Python 코드에서 타입 힌트를 사용한다면, mypy 같은 도구가 CI에서 검사할 수 있습니다. 이것은 더 고급이므로—지금은 존재하며 또 다른 단계로 추가할 수 있다는 것만 알아두세요.

매트릭스 전략: 여러 Python 버전

코드가 Python 3.12에서는 작동하지만 3.11에서는 깨질 수 있습니다. 매트릭스 전략(matrix strategy)은 여러 버전에 대해 병렬로 테스트합니다:

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.11', '3.12', '3.13']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install -r requirements.txt
      - run: python -m pytest

각 Python 버전당 하나씩 세 개의 병렬 잡을 만듭니다. 하나라도 실패하면 어떤 버전에 문제가 있는지 정확히 알 수 있습니다.

${{ secrets.NAME }} 구문으로 워크플로우에서 시크릿을 참조하세요:

      - run: python deploy.py
        env:
          API_KEY: ${{ secrets.API_KEY }}
          DATABASE_URL: ${{ secrets.DATABASE_URL }}

주의

오해: "로컬에서 테스트가 통과하면 CI에서도 통과할 것이다"—CI는 다른 운영체제(보통 Ubuntu), 다른 Python 버전, 그리고 남아 있는 상태가 없는 깨끗한 환경에서 실행됩니다. 로컬에서 설정한 환경 변수는 CI에 존재하지 않습니다. 저장소 밖의 파일도 존재하지 않습니다. 이것이 바로 핵심입니다—CI는 여러분의 컴퓨터가 아닌, 깨끗한 환경에서 코드가 작동하는지 증명합니다.

확인 방법: 첫 번째 푸시에서 Actions 탭에 초록색 체크마크가 표시됩니다. 테스트를 깨뜨리고 다시 푸시하면 빨간 X가 표시됩니다. 로그에서 테스트 출력을 확인할 수 있습니다.

작동하지 않을 때: (1) 워크플로우가 트리거되지 않음 → on: 섹션이 브랜치 이름과 일치하는지 확인하세요 (예: main vs master). (2) Python을 찾을 수 없음 → actions/setup-python 단계가 포함되어 있는지 확인하세요. (3) 의존성 누락 → 테스트 전에 pip install -r requirements.txt가 실행되는지 확인하세요. (4) Actions 탭이 없음 → 저장소가 로컬이 아닌 GitHub에 있는지 확인하세요.

방금 매번 푸시할 때마다 품질 검사가 자동으로 실행되도록 설정했습니다—전문적인 워크플로우에서 CI 트리거 에이전트가 코드를 리뷰하고, 테스트를 실행하고, 프로덕션을 보호하는 데 사용하는 것과 동일한 인프라입니다.

Chapter 18 프로젝트 아키텍처

핵심 개념

이제 API를 구축하고, 데이터를 영속화하고, 애플리케이션을 컨테이너화하고, 자동화된 테스트를 실행할 수 있습니다. 하지만 프로젝트가 커지면 코드를 조직하는 방법이 코드 자체만큼 중요해집니다. 나쁜 아키텍처(Architecture)는 작동하는 프로젝트를 유지보수 불가능한 프로젝트로 만듭니다. 좋은 아키텍처는 여러분과 AI 코딩 도구가 자신 있게 탐색하고, 확장하고, 리팩터링할 수 있게 해줍니다.

아키텍처가 중요한 이유

TaskForge가 단일 파일이었을 때는 프로그램 전체를 머릿속에 담을 수 있었습니다. 이제는 Flask API, 데이터베이스 함수, 테스트, Dockerfile, CI 파이프라인이 있습니다. 이 모든 것이 하나의 파일에 있다면, 코딩보다 스크롤하는 데 더 많은 시간을 보내게 될 것입니다.

아키텍처는 코드를 파일, 폴더, 모듈로 조직하여 각 부분이 명확한 목적을 갖도록 하는 방법입니다. 좋은 아키텍처는 세 가지를 제공합니다:

  • 찾기 쉬움(Findability)—어떤 기능이든 빠르게 찾을 수 있습니다.
  • 변경 용이성(Changeability)—다른 부분을 깨뜨리지 않고 한 부분을 수정할 수 있습니다.
  • 가독성(Readability)—새로운 개발자(또는 코드베이스를 읽는 AI 에이전트)가 모든 줄을 읽지 않고도 구조를 이해할 수 있습니다.

관심사의 분리

관심사의 분리(Separation of Concerns)란 각 파일이나 모듈이 하나의 책임만 처리한다는 의미입니다. API 계층은 HTTP 요청과 응답을 처리합니다. 데이터베이스 계층은 데이터 저장과 조회를 처리합니다. 비즈니스 로직은 애플리케이션의 규칙을 처리합니다. 이러한 관심사가 분리되면, 데이터 저장 방식을 변경해도 API 라우트를 변경할 필요가 없으며 그 반대도 마찬가지입니다.

레스토랑으로 비유해 보겠습니다. 웨이터는 주문을 받고(API 계층), 주방은 음식을 요리하고(비즈니스 로직), 식료품 저장실은 재료를 보관합니다(데이터 계층). 식료품 저장실을 재정리해도 웨이터는 재교육이 필요 없습니다.

프로젝트는 성장하면서 예측 가능한 단계를 거칩니다:

단계 1: 단일 파일

# taskforge.py — everything in one file

단계 3: 디렉토리 구조를 갖춘 패키지

관련 모듈을 디렉토리로 그룹화합니다. 최상위 수준에 설정, 테스트, 인프라 파일을 추가합니다.

디렉토리 구조 규칙

다음은 일반적인 Python 프로젝트 레이아웃입니다. 외울 필요는 없습니다—이 패턴이 존재한다는 것을 알아서 인식하고 따를 수 있으면 됩니다.

일반적인 Python 프로젝트는 소스 코드와 테스트를 설정 및 인프라 파일과 분리합니다. 각 파일은 하나의 명확한 책임을 갖습니다.

주요 규칙:

  • 소스 코드는 __init__.py 파일이 있는 패키지 디렉토리(프로젝트와 같은 이름)에 위치합니다.
  • 테스트는 소스 구조를 반영하는 별도의 tests/ 디렉토리에 위치합니다.
  • 설정 파일(Dockerfile, requirements.txt, .gitignore)은 프로젝트 루트에 위치합니다.
  • CI 워크플로우는 .github/workflows/에 위치합니다.

설정 관리

애플리케이션에는 설정이 필요합니다: 데이터베이스 경로, API 키, 디버그 플래그 등입니다. 이러한 값을 소스 코드에 하드코딩하면 깨지기 쉽고 안전하지 않습니다. 두 가지 접근 방법이 있습니다:

환경 변수

코드 바깥, 프로그램이 실행되는 환경에서 설정을 지정합니다:

import os

DB_PATH = os.environ.get("TASKFORGE_DB", "taskforge.db")
DEBUG = os.environ.get("FLASK_DEBUG", "0") == "1"

os.environ.get()는 기본값을 지정하여 환경 변수를 읽습니다. 이렇게 하면 동일한 코드가 개발 환경(기본값 사용)과 프로덕션 환경(Docker나 CI가 설정한 환경 변수 사용) 모두에서 작동합니다.

.env 파일

개발 중에는 프로젝트 루트의 .env 파일에 환경 변수를 저장할 수 있습니다:

# .env — local development configuration
TASKFORGE_DB=taskforge.db
FLASK_DEBUG=1

.env를 .gitignore에 추가하여 절대 커밋되지 않도록 하세요. 각 개발자는 자신만의 로컬 .env를 가집니다. 프로덕션에서는 배포 시스템이 설정한 실제 환경 변수를 사용합니다.

주의

API 키, 데이터베이스 비밀번호, 토큰은 절대로 소스 코드에 나타나거나 git에 커밋되어서는 안 됩니다. 환경 변수나 시크릿 관리(Chapter 17)를 사용하세요. 실수로 시크릿을 커밋했다면 즉시 교체하세요—파일을 삭제한 후에도 시크릿은 git 히스토리에 영원히 남아 있습니다.

TaskForge가 각 Phase를 거치며 어떻게 성장했는지, 그리고 각 구조 변경이 왜 필요했는지 살펴보겠습니다:

Phase 3.gitignore 추가, GitHub에 푸시버전 관리와 협업을 위해 추적하는 파일과 추적하지 않는 파일을 분리해야 합니다.

이 진행은 자연스러운 것입니다: 파일이 너무 길어질 때, 두 부분이 서로 다른 이유로 변경될 때, 또는 부분을 독립적으로 테스트해야 할 때 코드를 분리합니다. 단순히 분리를 위해 재구성하지 마세요—현재 구조가 마찰을 일으킬 때 재구성하세요.

리팩터링할 때

리팩터링(Refactoring)이란 코드가 하는 일을 바꾸지 않으면서 구조를 재편성하는 것을 의미합니다. 다음과 같을 때 리팩터링합니다:

  • 파일이 너무 길어서 탐색하기 어려울 때 (대략적인 기준: 200줄 이상).
  • 코드의 두 부분이 서로 다른 이유로 변경될 때 (예: API 라우트와 데이터베이스 쿼리).
  • 필요한 부분에 도달하기 위해 큰 코드 블록을 스크롤해야 할 때.
  • 모든 것이 얽혀 있어서 테스트 작성이 어려울 때.

다음과 같을 때는 리팩터링하지 않습니다:

  • 현재 구조가 작동하고 이해하기 쉬울 때.

관심사의 분리(Separation of Concerns)란 무엇입니까?

설계 도전: "모놀리스 리팩터링"

이 30줄짜리 스크립트는 입력 파싱, 유효성 검사, 출력 포맷팅을 섞고 있습니다. 3개의 함수로 리팩터링하세요: parse_input(raw), validate(data), format_output(data). 메인 로직은 이 3개의 함수를 호출하기만 해야 합니다.

parse_input은 분리와 공백 제거만 해야 합니다 — 유효성 검사는 하지 않습니다.

validate는 데이터 딕셔너리를 검사하고 오류 문자열 또는 None을 반환해야 합니다.

format_output은 검증된 딕셔너리를 받아서 포맷된 문자열을 반환합니다.
