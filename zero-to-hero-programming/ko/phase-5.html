챕터 19 복잡도 분석 (Complexity Analysis) — 중요한 것을 측정하기

핵심 개념

코드를 실행하기 전에 성능을 예측할 수 있어야 합니다. AI가 천만 개의 레코드에 대해 중첩 루프를 생성할 때, 그것이 몇 시간이 걸릴 것임을 알 수 있나요? 동료가 루프 안에서 리스트를 스캔하는 함수를 작성할 때, 그것이 끔찍하게 확장된다는 것을 발견할 수 있나요? Big-O 표기법이 그 어휘를 제공합니다. 이것은 엔지니어들이 성능에 대해 추론하는 데 사용하는 언어이며, 모든 자료 구조와 알고리즘 논의가 기반으로 하는 토대입니다.

Big-O 표기법이란?

Big-O 표기법은 함수가 수행하는 연산의 수가 입력 크기 n이 커짐에 따라 어떻게 증가하는지를 설명합니다. 하나의 질문에 답합니다: "입력을 두 배로 늘리면 얼마나 더 오래 걸리는가?"

초를 세는 대신 연산을 세는 이유는, 초는 CPU, 운영체제, 다른 실행 중인 프로그램 등에 따라 달라지기 때문입니다. 연산은 결정론적(deterministic)입니다. 함수가 크기 n의 입력에 대해 n번 비교를 수행한다면, 크기 2n의 입력에 대해서는 항상 2n번 비교를 수행합니다 — 하드웨어와 관계없이.

Big-O에는 두 가지 단순화 규칙이 있습니다:

  • 상수를 제거합니다. O(3n)은 O(n)이 됩니다. O(100)은 O(1)이 됩니다. 상수는 성장 모양을 바꾸지 않습니다.
  • 하위 항을 제거합니다. O(n² + n)은 O(n²)이 됩니다. n이 클 때, n² 항이 너무 완전히 지배하여 n 항은 무관합니다.

구체적인 비교로 이를 실감해봅시다. 작업 목록이 있고 특정 제목이 존재하는지 확인하려고 한다고 가정합니다:

# O(n) — scan the list one element at a time
def find_in_list(tasks, title):
    for task in tasks:
        if task["title"] == title:
            return True
    return False

# O(1) — look up in a dict (hash table) directly
task_index = {"Buy groceries": 1, "Write tests": 2, "Deploy API": 3}
def find_in_dict(task_index, title):
    return title in task_index

작업이 10개일 때는 둘 다 즉시 완료됩니다. 작업이 천만 개일 때는 리스트 스캔이 몇 초가 걸릴 수 있지만 딕셔너리 조회는 여전히 마이크로초 안에 완료됩니다. Big-O는 벤치마크를 한 번 실행하기 전에 이것이 일어날 것을 알려줍니다.

일반적인 복잡도 클래스

항상 마주치게 되는 여섯 가지 복잡도 클래스가 있습니다. 각각은 뚜렷한 성장 모양과 이미 알고 있는 실세계 예시를 갖고 있습니다.

일반적인 Big-O 복잡도 클래스와 성장 동작
클래스이름예시입력 두 배 → 시간 영향
O(1)상수 (Constant)딕셔너리 조회, 배열 인덱스 접근변화 없음
O(log n)로그 (Logarithmic)정렬된 리스트에서의 이진 탐색+1 단계

O(n log n)선형로그 (Linearithmic)효율적인 정렬 (병합 정렬, Timsort)~2배 시간
O(n²)이차 (Quadratic)모든 쌍을 비교하는 중첩 루프4배 시간

각 클래스에 대한 Python 예시로 코드에서 패턴을 확인해 보겠습니다:

# O(1) — Constant: dictionary lookup
def get_task(task_dict, task_id):
    return task_dict.get(task_id)  # one operation regardless of dict size

# O(log n) — Logarithmic: binary search
def binary_search(sorted_list, target):
    lo, hi = 0, len(sorted_list) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if sorted_list[mid] == target:
            return mid
        elif sorted_list[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1  # halves the search space each step

# O(n) — Linear: scan every element
def find_max(numbers):
    best = numbers[0]
    for num in numbers:
        if num > best:
            best = num
    return best  # touches each element exactly once

# O(n log n) — Linearithmic: Python's built-in sort
def sort_tasks(tasks):
    return sorted(tasks, key=lambda t: t["priority"])
    # Timsort: divides and conquers, n log n comparisons

# O(n^2) — Quadratic: check all pairs for duplicates
def has_duplicate_naive(items):
    for i in range(len(items)):
        for j in range(i + 1, len(items)):
            if items[i] == items[j]:
                return True
    return False  # compares every pair: n*(n-1)/2 operations

# O(2^n) — Exponential: generate all subsets
def all_subsets(items):
    if not items:
        return [[]]
    first = items[0]
    rest_subsets = all_subsets(items[1:])
    return rest_subsets + [[first] + s for s in rest_subsets]
    # each element doubles the number of subsets

자신의 코드 분석하기

규칙을 외울 필요는 없습니다. 두 가지 패턴을 인식하면 됩니다:

규칙 1: 순차적 블록 → 더하기 (최댓값 취하기). 함수가 먼저 리스트를 O(n)으로 스캔하고, 그 다음 O(n log n)으로 정렬하면, 총합은 O(n) + O(n log n) = O(n log n)입니다. 지배적인 항을 유지합니다.

def process_tasks(tasks):
    # Step 1: filter — O(n)
    active = [t for t in tasks if t["status"] == "active"]
    # Step 2: sort — O(n log n)
    active.sort(key=lambda t: t["priority"])
    # Step 3: return first 10 — O(1)
    return active[:10]
    # Total: O(n) + O(n log n) + O(1) = O(n log n)

규칙 2: 중첩 루프 → 곱하기. 루프 안의 루프는 내부 루프가 외부 루프의 각 반복마다 한 번씩 실행된다는 의미입니다. 둘 다 n개의 항목을 반복하면 O(n) × O(n) = O(n²)이 됩니다.

def find_common_tasks(list_a, list_b):
    common = []
    for a in list_a:          # O(n)
        for b in list_b:      #   O(m) for each a
            if a["title"] == b["title"]:
                common.append(a)
    return common
    # Total: O(n * m). If both lists are size n, this is O(n^2)

좀 더 미묘한 예시를 살펴보겠습니다:

def group_by_status(tasks):
    groups = {}
    for task in tasks:           # O(n)
        status = task["status"]
        if status not in groups:
            groups[status] = []  # O(1) dict operations
        groups[status].append(task)  # O(1) amortized
    return groups
    # Total: O(n) — one pass, all dict operations are O(1)

그리고 복잡도가 숨겨진 까다로운 예시입니다:

def remove_duplicates_naive(items):
    unique = []
    for item in items:           # O(n)
        if item not in unique:   # O(k) where k = len(unique)
            unique.append(item)
    return unique
    # Worst case: all unique, so k grows to n
    # Total: O(n^2) — the "in" check on a list is O(n)!

def remove_duplicates_fast(items):
    seen = set()
    unique = []
    for item in items:           # O(n)
        if item not in seen:     # O(1) set lookup
            seen.add(item)       # O(1)
            unique.append(item)
    return unique
    # Total: O(n) — set lookup is O(1)

이 두 함수의 차이는 O(n²)와 O(n)의 차이입니다. 항목이 10,000개일 때, 그것은 100,000,000 대 10,000번의 연산입니다. 두 번째 함수는 "약간 더 빠른" 것이 아닙니다 — 10,000배 더 빠릅니다.

공간 복잡도 (Space Complexity)

시간 복잡도는 연산을 측정합니다. 공간 복잡도는 알고리즘이 입력 자체를 넘어 사용하는 추가 메모리를 측정합니다.

# O(1) space — modifies in place
def reverse_in_place(items):
    left, right = 0, len(items) - 1
    while left < right:
        items[left], items[right] = items[right], items[left]
        left += 1
        right -= 1
    # Uses only two integer variables — constant extra space

# O(n) space — creates a new list
def reverse_copy(items):
    return items[::-1]
    # Creates an entirely new list of the same size

공간 복잡도는 데이터가 클 때 중요합니다. 10개 원소 리스트를 뒤집을 때는? 공간이 중요하지 않습니다. 천만 개 원소 리스트를 뒤집을 때는? 복사본을 만들면 메모리 사용량이 두 배가 됩니다. 제자리(in-place) 알고리즘은 가독성을 메모리 효율성으로 교환합니다.

분할 상환 분석 (Amortized Analysis)

Python의 list.append()는 O(1)로 알려져 있지만, 가끔 기반 배열이 공간 부족으로 크기를 조정해야 합니다 — 모든 원소를 새로운 더 큰 배열에 복사합니다. 그 단일 크기 조정은 O(n)입니다. append가 어떻게 O(1)일 수 있을까요?

분할 상환 분석은 일련의 연산에 걸쳐 비용을 평균냅니다. 동적 배열이 용량을 두 배로 늘릴 때(예: 8에서 16 슬롯으로), 다음 8번의 append는 "무료"입니다 — 크기 조정이 필요하지 않습니다. 비싼 O(n) 복사는 이후의 저렴한 O(1) append에 걸쳐 분산됩니다. 평균적으로 각 append는 O(1)의 비용이 듭니다.

월세 납부와 비슷하게 생각하면 됩니다. 한 달에 한 번 큰 금액을 내지만, 30일로 분할 상환하면 고정된 일일 비용이 됩니다. 하루가 비싸다고 해서 평균 지출이 높아지지는 않습니다.

# Demonstrating dynamic array growth
import sys

items = []
prev_size = 0
for i in range(65):
    items.append(i)
    curr_size = sys.getsizeof(items)
    if curr_size != prev_size:
        print(f"Length {len(items):3d} -> Memory {curr_size:5d} bytes")
        prev_size = curr_size
# You'll see memory jumps at irregular intervals as the array resizes

Big-O가 거짓말할 때

Big-O는 성장률에 대해 알려주지, 실제 속도에 대해 알려주지 않습니다. 작은 입력에서는 상수 인자가 지배합니다. 기억해야 할 몇 가지 중요한 예외가 있습니다:

  • 작은 n: ~50개 미만의 원소에서는 O(n²) 삽입 정렬이 O(n log n) 병합 정렬을 자주 이깁니다. 삽입 정렬의 오버헤드가 훨씬 낮기 때문입니다(함수 호출 적음, 캐시 사용 좋음). Python의 Timsort는 실제로 병합 정렬 내부의 작은 실행에 삽입 정렬을 사용합니다.
  • 캐시 지역성 (Cache locality): 배열은 원소를 메모리에 연속적으로 저장합니다. CPU가 하나의 원소를 로드할 때, 인접한 원소들을 무료로 캐시에 로드합니다. 연결 리스트는 노드를 메모리 전체에 분산시켜 캐시 미스(cache miss)를 유발합니다 — 각 접근이 메인 메모리를 기다릴 수 있습니다. O(n) 배열 스캔은 캐시 효과로 인해 O(n) 연결 리스트 스캔보다 10–100배 빠를 수 있습니다.
  • 상수 인자: 상수 인자가 1000인 O(n) 알고리즘은 n이 약 10,000 미만일 때 상수 인자가 1인 O(n log n) 알고리즘보다 느립니다. Big-O는 설계상 이 상수들을 숨깁니다.

교훈: Big-O는 생각 도구이지, 최종 답이 아닙니다. 대규모 결정(수백만 레코드에 대해 O(n) 대 O(n²))에는 이론을 믿으세요. 미시 최적화(두 O(n) 알고리즘 중 선택)에는 프로파일러로 측정하세요.

참고

실제 세계에서 timeit과 cProfile을 이용한 프로파일링은 종종 이론적 분석에 모순됩니다. 딕셔너리 컴프리헨션과 for 루프는 동일한 Big-O를 갖지만, 컴프리헨션은 바이트코드 명령어가 적어 20% 더 빠를 수 있습니다. 이론을 언제 믿어야 하는지(대용량 데이터에 대해 O(n)과 O(n²) 중 선택)와 언제 측정해야 하는지(상수가 다른 두 O(n) 접근 방식 중 선택)를 알아야 합니다. 시니어 엔지니어들은 둘 다 합니다: 설계 결정에는 Big-O, 최적화에는 프로파일링.

입력 크기가 커짐에 따라 다른 복잡도 클래스가 어떻게 성장하는지. O(1)과 O(log n)은 어떤 규모에서도 관리 가능합니다. O(n²)과 O(2ⁿ)은 빠르게 사용 불가능해집니다. 올바른 알고리즘 선택은 밀리초와 몇 시간의 차이를 의미할 수 있습니다.

챕터 20 선형 구조 (Linear Structures) — 배열, 연결 리스트, 스택 & 큐

핵심 개념

모든 프로그램은 데이터 시퀀스를 저장합니다. 어떻게 저장되는지 — 연속적 대 분산, 인덱스 대 연결 — 를 이해하는 것이 코드가 밀리초 단위로 실행될지 분 단위로 실행될지를 결정합니다. 이 챕터는 네 가지 기본 선형 구조를 다루고 각각을 언제 사용해야 하는지 가르칩니다.

메모리에서의 배열 (Array)

배열(array)은 원소를 연속적인(나란한) 메모리 위치에 저장합니다. 원소 0이 메모리 주소 1000에서 시작하고 각 원소가 8바이트를 차지한다면, 원소 5는 주소 1000 + (5 × 8) = 1040에 있습니다. 이 주소 산술이 배열 인덱스 접근이 O(1)인 이유입니다 — CPU가 스캔 없이 어떤 원소에도 직접 점프할 수 있습니다.

Python의 list는 동적 배열(dynamic array)입니다. 내부적으로 Python 객체에 대한 포인터(참조) 배열을 저장합니다. 배열 자체는 연속적이지만, 가리키는 객체들은 메모리에 분산되어 있을 수 있습니다.

연속적 저장은 배열에 큰 이점을 줍니다: 캐시 지역성(cache locality). 현대 CPU는 한 번에 한 바이트를 가져오지 않습니다 — 메모리에서 전체 캐시 라인(cache line)(일반적으로 64바이트)을 로드합니다. 배열의 원소 0에 접근할 때, CPU는 원소 0부터 7 이상을 무료로 캐시에 로드합니다. 배열을 반복하는 것은 다음 원소가 거의 항상 이미 캐시에 있기 때문에 빠릅니다.

동적 배열 (Dynamic Array)

정적 배열은 생성 시 고정된 크기를 갖습니다. 동적 배열(Python의 list와 같은)은 자동으로 성장합니다. 전략은 기하급수적 성장(geometric growth)입니다: 배열이 가득 차면 대략 두 배 크기의 새 배열을 할당하고, 모든 것을 복사하고, 이전 배열을 해제합니다.

이것은 append()를 분할 상환 O(1)으로 만듭니다(챕터 19에서 분석한 대로). 하지만 배열의 시작 부분에서의 연산은 비쌉니다:

연결 리스트 (Linked List)

연결 리스트(linked list)는 원소를 개별 노드(node) 객체로 저장하며, 각 노드는 값과 다음 노드에 대한 참조(포인터)를 포함합니다. 노드는 메모리 어디에든 분산될 수 있습니다 — 물리적 위치가 아닌 포인터로 연결됩니다.

class Node:
    def __init__(self, value, next_node=None):
        self.value = value
        self.next = next_node

class LinkedList:
    def __init__(self):
        self.head = None
        self.size = 0

    def prepend(self, value):
        """Add to the front — O(1)."""
        self.head = Node(value, self.head)
        self.size += 1

    def get(self, index):
        """Access by index — O(n). Must walk the chain."""
        current = self.head
        for _ in range(index):
            if current is None:
                raise IndexError("Index out of range")
            current = current.next
        return current.value

    def __len__(self):
        return self.size

    def __iter__(self):
        current = self.head
        while current:
            yield current.value
            current = current.next

단일 연결 리스트(singly linked list)에서 각 노드는 다음 노드를 가리킵니다. 이중 연결 리스트(doubly linked list)에서 각 노드는 이전 노드도 가리켜 양방향 순회와 양쪽 끝에서의 O(1) 제거가 가능합니다(노드에 대한 참조가 있을 때).

연결 리스트가 이길 때 (그리고 이기지 못할 때)

교과서는 헤드에서의 O(1) 삽입/삭제로 연결 리스트를 좋아합니다. 현실은 더 미묘합니다. Python의 list(동적 배열)는 거의 모든 실용적인 시나리오에서 연결 리스트를 이깁니다:

  • 캐시 미스가 성능을 파괴합니다. 배열 원소는 연속적이어서 CPU가 미리 가져옵니다. 연결 리스트 노드는 분산되어 있어 모든 노드 접근이 잠재적 캐시 미스입니다.
  • 메모리 오버헤드. 각 연결 리스트 노드는 헤더, 값, 포인터를 가진 Python 객체입니다. 백만 개 정수 리스트는 배열로 약 8 MB, 연결 리스트로는 56 MB를 사용합니다.
  • Python 오버헤드. 수백만 개의 Node 객체를 만들면 Python의 가비지 컬렉터에 압력을 줍니다.

연결 리스트가 빛나는 두 가지 특정 시나리오: (1) 인덱스 접근 없이 양쪽 끝에서 O(1) 삽입/삭제가 필요할 때(collections.deque 사용, 고정 크기 블록의 이중 연결 리스트), (2) 데이터를 복사하지 않고 서브리스트를 연결하거나 노드를 재정렬해야 할 때(Python에서는 드물지만 시스템 프로그래밍에서는 흔함).

스택 — LIFO (Last-In, First-Out)

스택(stack)은 후입선출(LIFO: Last-In, First-Out) 구조입니다. 항목을 위에 push하고 위에서 pop합니다. 접시 더미를 생각하세요: 위에 추가하고 위에서 가져옵니다.

# Python list is a perfect stack
stack = []
stack.append("open file")    # push
stack.append("edit line 5")  # push
stack.append("delete word")  # push

last_action = stack.pop()    # pop -> "delete word"
print(last_action)           # most recent action

스택의 사용 사례:

  • 실행 취소 시스템 — 각 동작을 push하고, 실행 취소하려면 pop합니다. 이것이 TaskForge에서 구축할 것입니다.
  • 호출 스택(call stack) — Python이 함수를 호출할 때 호출 스택에 프레임을 push합니다. 함수가 반환되면 프레임을 pop합니다. 이것이 무한 재귀가 "스택 오버플로우"를 일으키는 이유입니다.
  • 괄호 매칭 — 여는 괄호를 push하고, 닫는 괄호를 볼 때 pop합니다. pop할 때 스택이 비어 있거나 끝에서 비어 있지 않으면 괄호가 불균형합니다.
  • 식 평가 — 계산기는 스택을 사용하여 올바른 우선순위로 3 + 4 * 2를 평가합니다.
def is_balanced(expression):
    """Check if parentheses are balanced — classic stack problem."""
    stack = []
    pairs = {"(": ")", "[": "]", "{": "}"}
    for char in expression:
        if char in pairs:
            stack.append(char)
        elif char in pairs.values():
            if not stack:
                return False
            if pairs[stack.pop()] != char:
                return False
    return len(stack) == 0

print(is_balanced("({[]})"))   # True
print(is_balanced("({[})"))    # False
print(is_balanced("((()"))     # False

큐와 덱 — FIFO (First-In, First-Out)

큐(queue)는 선입선출(FIFO: First-In, First-Out) 구조입니다. 항목은 뒤에서 들어오고 앞에서 나갑니다. 매표소 줄처럼 생각하세요.

Python list를 큐로 사용하고 싶을 수 있지만, 중요한 성능 문제가 있습니다:

# BAD: Using list as a queue — pop(0) is O(n)!
queue = [1, 2, 3, 4, 5]
queue.append(6)       # enqueue at back — O(1), fine
front = queue.pop(0)  # dequeue from front — O(n), every element shifts!

# GOOD: Use collections.deque — popleft() is O(1)
from collections import deque
queue = deque([1, 2, 3, 4, 5])
queue.append(6)          # enqueue at back — O(1)
front = queue.popleft()  # dequeue from front — O(1)!

collections.deque(양방향 큐, double-ended queue)는 양쪽 끝에서 O(1) 연산을 지원합니다. 고정 크기 블록의 이중 연결 리스트로 구현되어 있어 배열의 캐시 친화적 동작과 끝에서의 연결 리스트 유연성을 얻습니다.

큐의 사용 사례:

  • 작업 스케줄링 — 도착한 순서대로 작업을 처리합니다. 인쇄 큐, 메시지 큐, 작업 큐.
  • 너비 우선 탐색(BFS) — 층별로 이웃을 탐색합니다(챕터 23에서 그래프와 함께 사용합니다).
  • 속도 제한 — 슬라이딩 윈도우에서 최근 요청의 타임스탬프를 추적합니다.
배열은 원소를 메모리에 나란히 저장하므로 단일 CPU 캐시 라인이 여러 원소를 한 번에 로드합니다. 연결 리스트 노드는 분산되어 있어 거의 모든 접근에서 캐시 미스가 발생합니다. 이것이 특정 연산에서 연결 리스트가 더 좋은 이론적 복잡도를 가지더라도 배열이 실제로 더 빠른 이유입니다.
참고

큐가 필요할 때 Python의 collections.deque가 올바른 선택입니다. list.pop(0)는 O(n)입니다 — 모든 원소가 이동합니다. deque.popleft()는 O(1)입니다. 이것은 AI가 생성한 코드가 자주 빠지는 성능 함정입니다. AI 출력에서 queue = [] 다음에 queue.pop(0)를 보면 deque로 교체해야 함을 알 수 있습니다. 자료 구조 하나 교체, 알고리즘 변경 없이, 일련의 디큐 연산에서 성능이 O(n²)에서 O(n)으로 향상됩니다.

import time
from collections import deque

n = 100000

# List: pop from front
data = list(range(n))
start = time.time()
while data:
    data.pop(0)
list_time = time.time() - start

# Deque: pop from front
data = deque(range(n))
start = time.time()
while data:
    data.popleft()
deque_time = time.time() - start

print(f"list.pop(0):    {list_time:.4f}s")
print(f"deque.popleft():{deque_time:.4f}s")
print(f"Deque is {list_time/deque_time:.0f}x faster")

핵심 개념

dict는 Python의 가장 강력한 자료 구조입니다. 내부적으로 어떻게 작동하는지 이해하면 O(1)인 시점, 저하되는 시점, 그리고 특정 것들이 딕셔너리 키가 될 수 없는 이유를 알 수 있습니다. if x in my_dict를 작성하고 즉시 반환될 때마다, 해시 테이블이 작업을 합니다. TypeError: unhashable type: 'list'를 받을 때마다, 해시 테이블이 불변 키를 요구하기 때문입니다. 이 챕터는 그 이유를 설명합니다.

해싱(Hashing)이 작동하는 방법

해시 함수(hash function)는 키(불변 값)를 해시 코드(hash code)라고 불리는 정수로 변환합니다. 그 정수를 테이블 크기로 나눈 나머지가 키-값 쌍이 있는 버킷(bucket)을 결정합니다.

# Python's built-in hash function
print(hash("hello"))     # e.g., 2534023372954268
print(hash("hello") % 8) # e.g., 4 — bucket 4 in an 8-bucket table
print(hash(42))          # 42 — integers hash to themselves
print(hash((1, 2, 3)))   # tuples are hashable because they're immutable

my_dict["hello"]를 조회하는 과정:

  1. hash("hello") 계산 → 어떤 큰 정수
  2. hash_code % table_size 계산 → 버킷 인덱스 (예: 4)
  3. 버킷 4로 직접 이동하여 값 찾기

스캔 없음, 다른 키와의 비교 없음. 이것이 딕셔너리 조회가 O(1)인 이유입니다.

충돌 해결 (Collision Resolution)

두 개의 다른 키가 같은 버킷으로 해시될 수 있습니다. 이것을 충돌(collision)이라고 합니다. hash("hello") % 8 == 4이고 hash("world") % 8 == 4라면, 두 키 모두 버킷 4로 매핑됩니다. 해시 테이블은 이를 처리해야 합니다.

두 가지 주요 전략이 있습니다:

  • 체이닝(Chaining): 각 버킷은 항목들의 연결 리스트(또는 작은 배열)를 보유합니다. 충돌이 발생하면 새 항목이 체인에 추가됩니다. 조회는 체인을 스캔합니다 — O(k) (k는 체인 길이). 해시 함수가 균등하게 분배하면 체인이 짧게 유지됩니다.
  • 개방 주소법(Open addressing): 충돌이 발생하면 결정론적 시퀀스(선형 탐사, 이차 탐사, 이중 해싱)를 사용하여 다음 빈 슬롯을 탐사합니다. CPython은 의사 랜덤 탐사 시퀀스를 사용하는 개방 주소법의 변형을 사용합니다.

두 전략 모두 트레이드오프가 있습니다: 체이닝은 노드 포인터에 메모리를 낭비하고, 개방 주소법은 클러스터링(clustering)(점령된 슬롯이 뭉쳐 탐사 시퀀스가 길어짐)의 문제가 있습니다. 실제로 테이블이 너무 가득 차지 않으면 둘 다 잘 작동합니다.

부하율 (Load Factor)과 재해싱 (Rehashing)

부하율(load factor)은 버킷 대비 항목의 비율입니다: n / table_size. 부하율이 증가함에 따라 충돌이 더 빈번해지고 성능이 O(n)으로 저하됩니다.

CPython은 부하율을 대략 2/3(0.67) 이하로 유지합니다. 새 항목을 삽입하면 이 임계값을 초과할 때, 테이블은 재해싱(rehashing)합니다: 새로운 더 큰 테이블(일반적으로 두 배 크기)을 할당하고 모든 기존 항목을 다시 삽입합니다. 이것은 O(n) 연산이지만, 드물게 발생합니다 — 동적 배열 크기 조정과 같은 패턴으로 삽입당 분할 상환 O(1)입니다.

# You can observe rehashing indirectly via memory usage
import sys

d = {}
prev = sys.getsizeof(d)
for i in range(100):
    d[i] = i
    curr = sys.getsizeof(d)
    if curr != prev:
        print(f"At {i} entries: dict size jumped from {prev} to {curr} bytes")
        prev = curr
# You'll see jumps at approximately 5, 10, 21, 42, 85 entries

Python 딕셔너리 내부

CPython 3.6부터 딕셔너리는 두 개의 별도 배열을 가진 컴팩트 레이아웃(compact layout)을 사용합니다:

  • 밀집 배열에 대한 인덱스를 저장하는 희소 해시 테이블(sparse hash table)(대부분 비어 있음)
  • 삽입 순서로 키와 값을 저장하는 밀집 항목 배열(dense entries array)

이것이 Python 딕셔너리가 3.7부터 삽입 순서를 보존하는 이유입니다 — 밀집 배열이 항목이 추가된 순서로 기록합니다. 또한 딕셔너리가 예상보다 더 많은 메모리를 사용하는 이유를 설명합니다: 희소 해시 테이블은 부하율을 낮게 유지하기 위해 빈 슬롯을 가집니다.

Python 딕셔너리에 대한 주요 사실:

  • 평균 조회, 삽입, 삭제: O(1)
  • 최악의 경우(모든 키가 충돌): O(n) — 하지만 Python의 해시 함수로는 사실상 발생하지 않음
  • 반복은 O(n)이고 삽입 순서를 따름
  • 메모리 사용량은 항목당 약 50–70바이트(리스트보다 훨씬 많음)

집합 (Set)

집합(set)은 값 없이 키만 저장하는 해시 테이블입니다. O(1) 멤버십 테스트를 제공하며, 이것이 리스트 대신 집합을 사용하는 가장 흔한 이유입니다.

# O(n) — membership test on a list
def has_overdue_list(task_ids, overdue_ids_list):
    for tid in task_ids:
        if tid in overdue_ids_list:  # O(n) each time
            return True
    return False
# Total: O(n * m) where n = task_ids, m = overdue list

# O(n) — membership test on a set
def has_overdue_set(task_ids, overdue_ids_set):
    for tid in task_ids:
        if tid in overdue_ids_set:  # O(1) each time
            return True
    return False
# Total: O(n) where n = task_ids

집합은 또한 효율적인 수학적 연산을 지원합니다:

team_a = {"Alice", "Bob", "Carol"}
team_b = {"Bob", "Dave", "Carol"}

print(team_a & team_b)   # Intersection: {"Bob", "Carol"}
print(team_a | team_b)   # Union: {"Alice", "Bob", "Carol", "Dave"}
print(team_a - team_b)   # Difference: {"Alice"}
print(team_a ^ team_b)   # Symmetric diff: {"Alice", "Dave"}

집합 대 리스트를 언제 사용할지: "X가 내 컬렉션에 있는지?" 가끔 이상으로 확인해야 한다면 집합으로 변환하세요. 100개의 원소에서는 차이가 무시할 만합니다. 백만 개의 원소에서는 x in my_list가 밀리초가 걸리는 반면 x in my_set은 마이크로초가 걸립니다 — 1000배 차이.

해싱이 실패할 때

해시 테이블에는 알아야 할 실패 모드가 있습니다:

  • 가변 키는 금지됩니다. 리스트를 딕셔너리 키로 사용한 후 수정할 수 있다면, 해시가 변경되어 딕셔너리가 더 이상 찾을 수 없습니다. Python은 이를 방지합니다: TypeError: unhashable type: 'list'. 먼저 튜플로 변환하세요.
  • 해시 충돌 공격(Hash collision attacks). 해시 함수를 아는 공격자는 모두 같은 버킷으로 매핑되는 입력을 만들어 O(1)을 O(n)으로 저하시킬 수 있습니다. Python은 해시 무작위화(프로세스당 다른 시드)로 이를 완화합니다.
  • 병리적 분포. 좋은 해시 함수에도 불구하고 많은 키가 같은 버킷으로 해시되면 성능이 저하됩니다. Python의 해시로는 극히 드물지만 디버깅을 위해 알아둘 가치가 있습니다.
# This fails: lists are mutable, so unhashable
try:
    d = {[1, 2, 3]: "value"}
except TypeError as e:
    print(e)  # unhashable type: 'list'

# This works: tuples are immutable, so hashable
d = {(1, 2, 3): "value"}
print(d[(1, 2, 3)])  # "value"
5개의 키-값 쌍을 보유한 8개의 버킷을 가진 해시 테이블. 버킷 3에 충돌이 있습니다: 두 키가 같은 인덱스로 해시되었습니다. 체이닝이 그것들을 체인으로 연결합니다. 조회는 버킷을 확인하고, 필요하면 체인을 순회합니다. 낮은 부하율에서 체인이 짧게 유지되고 조회가 O(1)을 유지합니다.
참고

조회에 dict, set, list 중 선택하는 것은 매일의 엔지니어링 결정입니다. if x in my_list는 O(n). if x in my_set는 O(1). 100개의 원소에서는 중요하지 않습니다. 백만 개에서는 1초와 1마이크로초의 차이입니다. 고유 ID를 리스트에 저장하고 루프에서 멤버십을 확인하는 AI 생성 코드를 보면, 집합으로 교체해야 함을 알 수 있습니다. 이것이 확장되는 최적화입니다.

챕터 22 트리 (Tree)

핵심 개념

트리는 보이지 않는 곳 어디에나 있습니다. 파일 시스템은 트리입니다. 데이터베이스 인덱스는 트리입니다. HTML DOM은 트리입니다. JSON은 트리입니다. 디렉토리를 순회하기 위해 os.walk()를 호출할 때, 트리를 순회하는 것입니다. 트리를 이해하면 이 모든 시스템이 어떻게 성능을 발휘하는지, 그리고 수백만 행이 있는 테이블에서도 데이터베이스 쿼리가 왜 빠른지 이해할 수 있습니다.

이진 트리 (Binary Tree) 기초

이진 트리(binary tree)는 각 노드가 최대 두 개의 자녀를 갖는 계층 구조로, 관습적으로 왼쪽(left)과 오른쪽(right)이라고 합니다. 최상위 노드는 루트(root)입니다. 자녀가 없는 노드는 잎(leaf)입니다. 노드의 깊이(depth)는 루트로부터의 거리입니다. 트리의 높이(height)는 가장 깊은 잎의 깊이입니다.

class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

# Build a small tree:
#        8
#       / \
#      3   10
#     / \    \
#    1   6    14
root = TreeNode(8,
    TreeNode(3, TreeNode(1), TreeNode(6)),
    TreeNode(10, None, TreeNode(14))
)

트리 순회 (Tree Traversals)

트리의 모든 노드를 방문하는 네 가지 표준 방법이 있습니다. 각 순회 순서는 다른 목적을 수행합니다:

def inorder(node):
    """Left -> Root -> Right. For BSTs, visits nodes in sorted order."""
    if node is None:
        return []
    return inorder(node.left) + [node.value] + inorder(node.right)

def preorder(node):
    """Root -> Left -> Right. Copies the tree structure."""
    if node is None:
        return []
    return [node.value] + preorder(node.left) + preorder(node.right)

def postorder(node):
    """Left -> Right -> Root. Safe for deletion (children before parent)."""
    if node is None:
        return []
    return postorder(node.left) + postorder(node.right) + [node.value]

def levelorder(node):
    """Layer by layer (BFS). Uses a queue."""
    from collections import deque
    if node is None:
        return []
    result, queue = [], deque([node])
    while queue:
        current = queue.popleft()
        result.append(current.value)
        if current.left:
            queue.append(current.left)
        if current.right:
            queue.append(current.right)
    return result

# Using our tree from above:
print("Inorder:   ", inorder(root))    # [1, 3, 6, 8, 10, 14] — sorted!
print("Preorder:  ", preorder(root))   # [8, 3, 1, 6, 10, 14]
print("Postorder: ", postorder(root))  # [1, 6, 3, 14, 10, 8]
print("Levelorder:", levelorder(root)) # [8, 3, 10, 1, 6, 14]

이진 탐색 트리 (Binary Search Tree, BST)

이진 탐색 트리(BST)는 순서 규칙을 강제합니다: 모든 노드에 대해, 왼쪽 서브트리의 모든 값이 더 작고 오른쪽 서브트리의 모든 값이 더 큽니다. 이 규칙으로 인해 탐색, 삽입, 삭제가 모두 O(h)입니다(h는 트리의 높이).

def bst_search(node, target):
    """Search a BST — O(h) where h = height."""
    if node is None:
        return False
    if target == node.value:
        return True
    elif target < node.value:
        return bst_search(node.left, target)   # go left
    else:
        return bst_search(node.right, target)  # go right

def bst_insert(node, value):
    """Insert into a BST — O(h)."""
    if node is None:
        return TreeNode(value)
    if value < node.value:
        node.left = bst_insert(node.left, value)
    elif value > node.value:
        node.right = bst_insert(node.right, value)
    # if value == node.value, duplicate — skip or handle as you wish
    return node

# Build a BST from scratch
tree = None
for val in [8, 3, 10, 1, 6, 14]:
    tree = bst_insert(tree, val)

print(bst_search(tree, 6))   # True
print(bst_search(tree, 7))   # False
print(inorder(tree))          # [1, 3, 6, 8, 10, 14] — always sorted

균형이 중요한 이유

BST의 성능은 완전히 높이에 달려 있습니다. n개의 노드를 가진 균형 잡힌 BST는 높이가 log n이어서 O(log n) 연산을 제공합니다. 하지만 정렬된 데이터를 BST에 삽입하면 모든 노드가 오른쪽으로 가며, 높이 n의 연결 리스트와 사실상 같은 퇴화된 트리를 만듭니다.

# Balanced insertion: [8, 3, 10, 1, 6, 14]
# Creates a nice tree with height 3:
#        8
#       / \
#      3   10
#     / \    \
#    1   6    14
# Search: O(log n) = O(log 6) ≈ 3 comparisons

# Sorted insertion: [1, 3, 6, 8, 10, 14]
# Creates a degenerate tree with height 6:
#    1
#     \
#      3
#       \
#        6
#         \
#          8
#           \
#            10
#              \
#               14
# Search: O(n) = 6 comparisons — it's a linked list!

자가 균형 트리(self-balancing trees)(AVL 트리, 레드-블랙 트리)는 삽입 및 삭제 후 자동으로 재구조화하여 O(log n) 높이를 보장합니다. 이것들을 직접 구현할 일은 거의 없습니다 — 라이브러리와 데이터베이스 내부에 존재합니다. 하지만 왜 그것들이 존재하는지 알아야 합니다: O(n) 퇴화 경우를 방지하기 위해. Python의 sorted containers 라이브러리와 C++의 std::map은 레드-블랙 트리를 사용합니다. 데이터베이스 인덱스는 B-트리를 사용합니다(아래 참조).

힙 (Heap)과 우선순위 큐 (Priority Queue)

최소 힙(min-heap)은 모든 부모가 자녀보다 작거나 같은 완전 이진 트리입니다. 최솟값은 항상 루트에 있습니다. 힙은 인덱스 산술을 사용하여 배열로 저장됩니다:

  • 노드 i의 자녀: 인덱스 2i+1과 2i+2
  • 노드 i의 부모: 인덱스 (i-1)//2

Python의 heapq 모듈은 일반 리스트를 기반으로 한 최소 힙을 제공합니다:

import heapq

# Build a heap from a list
tasks = [(3, "Low priority task"),
         (1, "URGENT task"),
         (2, "Medium task")]
heapq.heapify(tasks)  # O(n) — rearranges in place

# Pop the smallest (highest priority)
print(heapq.heappop(tasks))  # (1, "URGENT task")
print(heapq.heappop(tasks))  # (2, "Medium task")
print(heapq.heappop(tasks))  # (3, "Low priority task")

# Push a new item
heapq.heappush(tasks, (0, "Critical fix"))

# Get top-K without sorting the whole list
data = [42, 17, 93, 5, 68, 31, 84, 2]
print(heapq.nsmallest(3, data))  # [2, 5, 17] — O(n log k)
print(heapq.nlargest(3, data))   # [93, 84, 68]

핵심 통찰: 최솟값(또는 최댓값) 원소만 반복적으로 필요하다면, 힙이 정렬보다 낫습니다. 정렬은 모든 원소를 순서대로 줍니다(O(n log n)). 힙은 O(1)로 최솟값만 주고 O(log n)으로 제거할 수 있습니다. "10,000개 중 가장 중요한 10개 작업을 달라"에는 힙이 올바른 도구입니다.

트라이 (Trie)

트라이(trie)("트라이"로 발음)는 각 노드가 문자를 나타내는 트리입니다. 루트에서 노드까지의 경로가 접두사를 만듭니다. 트라이는 자동 완성 기능을 지원합니다: "pro"로 시작하는 모든 단어를 찾으려면 p-r-o 경로를 따라 내려가고 아래의 모든 것을 반환합니다.

핵심 속성: 조회 시간은 저장된 키의 수와 관계없이 키 길이 m에서 O(m)입니다. 백만 단어의 트라이는 "programming"을 정확히 11단계로 찾습니다.

class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_end = False

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for char in word:
            if char not in node.children:
                node.children[char] = TrieNode()
            node = node.children[char]
        node.is_end = True

    def search(self, word):
        node = self.root
        for char in word:
            if char not in node.children:
                return False
            node = node.children[char]
        return node.is_end

    def starts_with(self, prefix):
        node = self.root
        for char in prefix:
            if char not in node.children:
                return False
            node = node.children[char]
        return True

t = Trie()
for word in ["task", "tag", "test", "team"]:
    t.insert(word)
print(t.search("task"))       # True
print(t.search("tas"))        # False (prefix, not complete word)
print(t.starts_with("ta"))    # True
print(t.starts_with("te"))    # True
print(t.starts_with("tx"))    # False

B-트리 (B-Tree, 개념적)

데이터베이스는 BST를 사용하지 않습니다. 디스크 기반 저장을 위해 설계된 넓고 얕은 트리인 B-트리(B-tree)를 사용합니다.

디스크에서 BST의 문제: 각 노드는 하나의 비교이고, 디스크에서 노드를 읽는 것은 ~10밀리초(기계 디스크) 또는 ~0.1ms(SSD)가 걸립니다. 백만 개의 노드를 가진 균형 잡힌 BST는 높이가 20이어서 20번의 디스크 읽기가 필요합니다. 각 0.1ms에서, 그것은 쿼리당 2ms입니다 — 참을 수 있지만 좋지는 않습니다.

B-트리는 노드를 넓게 만들어 이를 해결합니다: 각 노드는 수백 개의 키를 보유하고 수백 개의 자녀를 갖습니다. 분기 계수 100과 3레벨의 B-트리는 탐색당 디스크 읽기 3번만으로 100³ = 백만 개의 키를 보유할 수 있습니다. 이것이 수백만 행이 있는 테이블에서 SELECT * FROM tasks WHERE id = 42가 즉시 반환되는 이유입니다.

B-트리는 직접 구현하지 않습니다. 데이터베이스(SQLite, PostgreSQL)를 통해 사용합니다. 하지만 그것들이 존재한다는 것을 알면 데이터베이스 인덱스가 왜 빠른지, 그리고 CREATE INDEX가 왜 쿼리를 빠르게 하는지 설명할 수 있습니다 — 인덱스된 열에 대한 B-트리를 구축하는 것입니다.

같은 7개의 숫자가 삽입 순서에 따라 매우 다른 트리를 만듭니다. 임의 순서는 O(log n) 탐색의 균형 잡힌 트리를 줍니다. 정렬된 순서는 O(n) 탐색의 퇴화된 트리 — 사실상 연결 리스트 — 를 줍니다. 자가 균형 트리는 퇴화 경우를 방지합니다.
참고

트리를 직접 구현하는 일은 거의 없습니다. 하지만 트리를 이해하는 것은 필수적입니다: 데이터베이스 인덱스는 B-트리이고, 파일 시스템은 트리 구조이며, heapq는 "가장 중요한 10개 작업을 달라"에 적합한 도구입니다. AI가 최댓값을 찾기 위해 리스트를 정렬하는 코드를 생성할 때, 단일 max() 호출이 O(n)임을 알고, 더 나아가 항목이 변경될 때 반복적으로 최댓값이 필요하다면 힙이 업데이트당 O(log n)을 주지만 재정렬은 O(n log n)임을 알 수 있습니다. 구조를 아는 것은 올바른 도구를 아는 것입니다.

챕터 23 그래프 (Graph)

핵심 개념

모든 관계가 계층적이지는 않습니다. 소셜 네트워크, 항공 노선, 작업 의존성, 웹 페이지 링크 — 이것들은 모두 그래프(graph)입니다. "A가 B에 의존한다" 또는 "앨리스가 밥을 안다" 또는 "X에서 Y까지 도로가 있다"를 모델링해야 할 때, 트리는 사이클, 복수 경로, 또는 피어-투-피어 관계를 나타낼 수 없기 때문에 충분하지 않습니다. 그래프는 연결을 모델링하는 가장 범용적인 자료 구조입니다.

그래프 용어

그래프(graph)는 간선(edge)으로 연결된 노드(node)(정점(vertex)이라고도 함)의 집합입니다. 이것이 전체 정의입니다. 나머지는 모두 특수화입니다:

  • 방향 그래프 대 무방향 그래프: 방향 그래프(directed graph)에서 간선 A→B는 단방향입니다(A가 B에 의존하지만, B는 A에 의존하지 않음). 무방향 그래프(undirected graph)에서 간선은 양방향입니다(앨리스가 밥을 안다는 것은 밥이 앨리스를 안다는 의미).
  • 가중 그래프 대 비가중 그래프: 가중 간선(weighted edge)은 값을 갖습니다(거리, 비용, 시간). 비가중 간선은 연결만 나타냅니다.
  • 경로(path): 두 노드를 연결하는 간선의 시퀀스. "A에서 D로 갈 수 있나요?"는 경로가 존재하는지에 대한 질문입니다.
  • 사이클(cycle): 같은 노드에서 시작하고 끝나는 경로. 사이클이 없는 그래프는 비순환(acyclic)입니다.
  • 연결 요소(connected component): 모든 노드가 다른 모든 노드에 도달할 수 있는 노드 그룹. 그래프는 여러 개의 분리된 요소를 가질 수 있습니다.
  • 차수(degree): 노드에 연결된 간선의 수. 방향 그래프에서 진입 차수(in-degree)는 들어오는 간선을 세고 진출 차수(out-degree)는 나가는 간선을 셉니다.

표현 방법

코드에서 그래프를 저장하는 두 가지 표준 방법이 있습니다. 둘 다 같은 정보를 나타내지만 성능 특성이 다릅니다.

인접 리스트 (Adjacency List)

각 노드를 이웃의 리스트(또는 집합)에 매핑하는 딕셔너리. 이것이 가장 흔한 표현이며 기본값으로 사용해야 합니다.

# Directed graph: A->B, A->C, B->C, C->D
adj_list = {
    "A": ["B", "C"],
    "B": ["C"],
    "C": ["D"],
    "D": []
}

# Checking neighbors of A: O(1) dict lookup
print(adj_list["A"])  # ["B", "C"]

# Checking if edge A->B exists: O(degree of A)
print("B" in adj_list["A"])  # True

인접 행렬 (Adjacency Matrix)

노드 i에서 노드 j로의 간선이 있으면 matrix[i][j] = 1(또는 가중치)이고, 그렇지 않으면 0인 2D 격자.

# Same graph: A=0, B=1, C=2, D=3
#       A  B  C  D
adj_matrix = [
    [0, 1, 1, 0],  # A -> B, A -> C
    [0, 0, 1, 0],  # B -> C
    [0, 0, 0, 1],  # C -> D
    [0, 0, 0, 0],  # D -> (none)
]

# Checking if edge A->B exists: O(1)
print(adj_matrix[0][1])  # 1 — edge exists

그래프 클래스 구축

class Graph:
    def __init__(self, directed=False):
        self.adj = {}
        self.directed = directed

    def add_node(self, node):
        if node not in self.adj:
            self.adj[node] = set()

    def add_edge(self, u, v):
        self.add_node(u)
        self.add_node(v)
        self.adj[u].add(v)
        if not self.directed:
            self.adj[v].add(u)

    def neighbors(self, node):
        return self.adj.get(node, set())

    def has_edge(self, u, v):
        return v in self.adj.get(u, set())

    def nodes(self):
        return list(self.adj.keys())

    def __repr__(self):
        lines = []
        for node, neighbors in self.adj.items():
            for n in neighbors:
                lines.append(f"  {node} -> {n}")
        return "Graph:\n" + "\n".join(lines)

# Example: social network (undirected)
social = Graph(directed=False)
social.add_edge("Alice", "Bob")
social.add_edge("Bob", "Carol")
social.add_edge("Alice", "Carol")
print(social.neighbors("Alice"))  # {"Bob", "Carol"}
print(social.has_edge("Bob", "Alice"))  # True (undirected)

# Example: task dependencies (directed)
deps = Graph(directed=True)
deps.add_edge("Write code", "Run tests")
deps.add_edge("Run tests", "Deploy")
deps.add_edge("Write code", "Code review")
print(deps.neighbors("Write code"))  # {"Run tests", "Code review"}
print(deps.has_edge("Deploy", "Write code"))  # False (directed)

방향 비순환 그래프 (DAG: Directed Acyclic Graph)

DAG는 사이클이 없는 방향 그래프입니다. DAG는 의존성 모델링에 중요합니다: 작업 A가 B에 의존하고 B가 C에 의존한다면 괜찮습니다. 하지만 C도 A에 의존한다면 사이클이 있습니다 — 유효한 실행 순서가 존재하지 않습니다.

# Valid DAG (no cycles):
#   Write code -> Run tests -> Deploy
#   Write code -> Code review -> Deploy

# Invalid (cycle):
#   A -> B -> C -> A
# No task can go first because each depends on another

따라서 사이클 감지는 중요한 검증 단계입니다: 의존성을 기반으로 작업을 스케줄링하기 전에 의존성 그래프가 DAG인지 검증하세요. 사이클이 포함되어 있으면 사용자의 요구 사항은 불가능합니다.

DFS를 이용한 사이클 감지

재귀 스택(recursion stack)을 사용한 깊이 우선 탐색(DFS)으로 방향 그래프의 사이클을 감지할 수 있습니다. 아이디어: DFS를 통해 더 깊이 탐색하면서 현재 탐색 경로에 있는 모든 노드를 추적합니다. 현재 경로에 이미 있는 노드에 도달하면 사이클을 찾은 것입니다.

def has_cycle(graph):
    """Detect cycles in a directed graph using DFS.

    Uses three states per node:
    - unvisited: not yet explored
    - in_stack: currently on the DFS path (ancestors of current node)
    - done: fully explored, no cycles through this node
    """
    visited = set()
    in_stack = set()

    def dfs(node):
        visited.add(node)
        in_stack.add(node)

        for neighbor in graph.neighbors(node):
            if neighbor not in visited:
                if dfs(neighbor):
                    return True
            elif neighbor in in_stack:
                return True  # found a cycle!

        in_stack.remove(node)
        return False

    for node in graph.nodes():
        if node not in visited:
            if dfs(node):
                return True
    return False

# Test: valid DAG
dag = Graph(directed=True)
dag.add_edge("A", "B")
dag.add_edge("B", "C")
dag.add_edge("A", "C")
print(has_cycle(dag))  # False

# Test: graph with cycle
cyclic = Graph(directed=True)
cyclic.add_edge("A", "B")
cyclic.add_edge("B", "C")
cyclic.add_edge("C", "A")  # creates cycle
print(has_cycle(cyclic))  # True
같은 방향 그래프를 인접 리스트(왼쪽)와 인접 행렬(오른쪽)로 나타냈습니다. 인접 리스트는 간선 수에 비례하는 공간을 사용합니다. 행렬은 간선이 몇 개이든 V² 공간을 사용합니다. 이 희소 그래프(6개 간선, 25개 행렬 셀)에서 리스트가 더 공간 효율적입니다.

그래프는 전문 엔지니어링 어디에나 나타납니다. pip는 DAG와 위상 정렬을 사용하여 패키지 의존성을 해결합니다. Google Maps는 가중 그래프와 최단 경로 알고리즘(다익스트라)을 사용하여 경로를 찾습니다. LinkedIn의 "알 수도 있는 사람"은 소셜 그래프에서 BFS를 사용합니다. Docker의 레이어 시스템은 DAG를 형성합니다. Git의 커밋 기록도 각 커밋이 부모를 가리키는 DAG입니다. 페이즈 6에서 이 알고리즘들 중 일부를 구현할 것입니다.

직접 해보기: 그래프 구축

g = Graph(directed=True)

g.add_edge("A", "C")   # A must finish before C

g.add_edge("C", "D")   # C must finish before D

직접 해보기: 유효하지 않은 의존성 감지

D에서 A로 다시 간선을 추가하면 어떻게 됩니까?

g.add_edge("D", "A")    # D depends on A, but A depends on B/C which depend on D!
print(has_cycle(g))      # True — impossible dependency cycle
