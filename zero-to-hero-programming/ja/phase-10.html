エージェントエンジニアリングのレベル6〜8

このフェーズは高度なオーケストレーションパターンのリファレンス資料です。フェーズ9直後にすべてを実践できるとは限りません。プロジェクトが成長するにつれて戻ってきてください。レベル5からレベル6への進歩は、通常、数日ではなく数ヶ月の日々の練習を要します。

第43〜47章フェーズゲート + TaskForge
補足

このフェーズは次のことができることを前提としています：AIが生成したコードを評価して一般的な失敗パターンを特定できる（第38章）、効果的なCLAUDE.mdファイルを書いてコンテキストを管理できる（第40章）、テストを含む単一タスク開発にClaude Codeを使用できる（第39〜42章）。フェーズ10はリファレンス資料です — プロジェクトが成長するにつれて戻ってきてください。

重要な概念

なぜ1つのエージェントでは不十分なのか — 単一のClaude Codeセッションは強力です。しかし、一部の作業は1つのセッションに収まりません：

  • コンテキストの制限：1つのセッションは大規模なコードベース全体を保持できません。専門エージェントは特定の領域に集中します。
  • 関心の分離：コードを書いたエージェントが自分の作業を採点すべきではありません。実装とレビューを分けることでより多くのバグを発見できます。
  • 並列性：3つの独立した機能を同時に構築すると、1つの場合の時間で完了します。
  • 専門化：セキュリティ監査用に設定されたエージェントは、後付けでセキュリティを確認するよう求められた汎用エージェントより良い結果を出します。

会社のチームのように考えてください：ある人がコードを書き、別の人がレビューし、別の人がテストを書き、別の人がドキュメントを更新します。人数が多いほどスループットが増しますが、調整のオーバーヘッドも増します。制約なしに多くのエージェントを使うと、生産性ではなく混乱が生じます。だからこそ、このフェーズでは作業の振り分けと同様に、ハーネス（harness）とガードレール（guardrail）に焦点を当てています。

第43章 レベル6 — ハーネスエンジニアリングと自動フィードバックループ

なぜ今重要なのか

レベル1〜5では、あなたが品質ゲートです — すべての変更をレビューします。これは単一の機能には機能しますが、スケールしません。レベル6では、エージェントが自分の間違いを発見できる自動品質ゲートを構築し、ラインごとのレビューではなく設計上の意思決定に集中できるようにします。

レベル5では、エージェントにケイパビリティを与えました。レベル6では、エージェントに自分の作業を検証できるガードレールを与えます。転換点：あなたの役割がコードをレビューすることからharnessを設計することに移ります。

バックプレッシャー

バックプレッシャーとは、人間の介入なしにエージェントが自己修正できる自動フィードバックです。コードレビューでエラーを発見するのではなく、ツールが即座に発見します：

自動エージェント修正のためのバックプレッシャーメカニズム
メカニズム発見するもの
TypeScript strict / mypy型エラー
リンター（ESLint、Ruff）スタイル違反
テストスイート動作の退行
プリコミットフックコミット前のフォーマット・リント・型チェック
CIパイプライン統合の失敗

エージェントがgit commitを実行してプリコミットフックが失敗すると、エージェントはエラーを確認し、コードを修正して、再試行します。これがバックプレッシャーの動作です。

警告：バックプレッシャーがない場合

エージェントに機能の追加を依頼します。エージェントは微妙な型エラーを含むコードを書きます。型チェッカーがなければ、コードは問題なく見えます。デプロイします。本番環境でクラッシュします。プリコミットフックが型チェッカーを実行していれば、エージェントはエラーがあなたに届く前に発見して修正します。修正は数時間ではなく数秒で済みます。

制約 > 指示

ステップバイステップの指示は、モデルにどのように作業するかを伝えます。制約は成功がどのように見えるかを伝え、方法はモデルに任せます。制約はモデルが問題へのアプローチを適応させるため、よりスケールします。

ステップバイステップ（脆弱）
1. ユーザーモデルを読む。
2. メールフィールドを追加する。
3. RFC 5322バリデーションを追加する。
4. マイグレーションを更新する。
5. テストを更新する。
制約ベース（堅牢）
ユーザーモデルにメールを追加する。

要件： RFC 5322バリデーション、ユニーク制約。
受け入れ基準： すべてのテストがパス、有効/無効/重複/nullのための新しいテスト。
実行： npm testがパスするまで作業する。

セキュリティの境界

エージェント、コード、シークレットは別々の信頼ドメインに存在します。エージェントに本番環境の認証情報への直接アクセスを与えないでください。環境変数、シークレットマネージャー、最小権限アクセスを使用してください。

ドキュメントをナビゲーションとして使う

CLAUDE.mdを約100行の目次として使用します。詳細なドキュメントは別の場所に（architecture.md、APIドキュメント、デプロイガイド）。モデルは第22章のプログレッシブディスクロージャーパターンを通じてオンデマンドで発見します。

harnessによってエージェントは自分の作業を検証できます。あなたの役割はコードのレビューからharnessの設計に移ります。

TaskForgeとの接続

TaskForgeのためにRuff（リンター）とpytestを実行するプリコミットフックを設定します。これによりClaude Codeは機能追加時に自己修正できます。

ケーススタディ：Dockerスキャフォールド

この章と第21章のスキャフォールドセクションのすべての原則を適用した実世界の指示スキャフォールドを示します。目標：マルチエージェント開発のためにあらゆるプロジェクトをコンテナ化する方法をClaude Codeに伝えること。このスキャフォールドは、Claudeが間違いを犯した数十回の反復を経て洗練されました — 各失敗が新しい制約になりました。

なぜ機能するのか

スキャフォールドはCLAUDE.mdファイルで、約300行です。各セクションが存在する理由：

1
一文のゴール

「1つのコンテナ。execで入ると、リポジトリを指すClaude Codeが使える。CMDはsleep infinity。」解釈の余地がありません。モデルは「完了」がどのように見えるかを知っています。

2
説明ではなく正確なファイル内容

Dockerfile、docker-composeサービス、シェルスクリプトは逐語的に提供されます — 説明されません。FROM debian:bookworm-slimであり「Debianベースのイメージを使用」ではありません。curl -fsSL https://claude.ai/install.sh | bashであり「Claude Codeをインストール」ではありません。モデルがゼロから生成するすべての行は、実行ごとに変わる可能性のある行です。

3
実際の失敗からの明示的な禁止事項

各「NEVER」行は、Claudeが実際に間違ったことをしたために存在します：

  • 「NEVER npm install を使う — 認証/PATHの破損を引き起こす」— Claudeのトレーニングデータには古いnpmベースのチュートリアルが含まれています
  • 「CLAUDE_MODELを環境ブロックに追加しない — 実際の環境変数ではない」— Claudeはこの環境変数を繰り返しハルシネーションしました
  • 「NEVER .envを上書きする — 追記のみ」— Claudeはファイルを書き直すことで既存の設定を破壊しました
  • 「claudeサービスにenvironment:ブロックを追加しない」— Claudeはenv_fileと競合する冗長な環境変数を追加しました

4
失敗モードテーブル

認証リファレンステーブルは症状から原因、修正方法をマッピングします。「オンボーディングウィザードが表示される」→「.claude.jsonが見つからない」→「DockerfileがあらかじめそれをCREATEしている。」これにより、モデルはビルドレシピだけでなくデバッグプレイブックを持てます。

5
バリデーションチェックリスト

正確性を検証する15のyes/noチェック：「ベースイメージはdebian:bookworm-slimか？」「.envは上書きされていないか？」「agent.shは--dangerously-skip-permissionsを使用しているか？」これらは決定論的なガードレールです。モデルはチェックリストを回避することで合理化できません。

重要な概念：非決定論の原則（実践）

このスキャフォールドが機能するのは、モデルが行う必要のある決定を最小化するからです。曖昧な「Dockerを設定して」はClaude に無数の可能な設定を与えます。スキャフォールドはそれを本質的に1つに絞ります。これが基本的なパターンです：制約が厳しいほど、出力はより決定論的になります。指定されていない詳細はすべてコイントスです。良いスキャフォールドはコイントスをなくします。

スキャフォールドは指示するだけでなく、保護します。.envのappend-onlyルールはデータ損失を防ぎます。別のDockerfile.claudeはプロジェクトの既存のDocker設定を上書きすることを防ぎます。認証のためのボリュームマウントは、再起動ごとの再認証を防ぎます。スキャフォールドは防御的なエンジニアリングです：モデルが破壊的なことをしようとすることを前提とし、構造的にそれを不可能にします。

デプロイ、テスト、コード生成、または反復可能なタスクのための独自のスキャフォールドを構築するときは、このパターンに従ってください：ゴール → 正確な内容 → 禁止事項 → 失敗モード → バリデーション。より具体的であればあるほど、あらゆる実行で、モデルの非決定論に関わらず、出力はより信頼できるものになります。

第44章 レベル7 — バックグラウンドエージェント

なぜ今重要なのか

レベル7は、AIがあなたが使うツールから、あなたが管理するチームへと変わる段階です。1度に1つのエージェントと作業する代わりに、プロジェクトマネージャーが異なる専門知識を持つチームメンバーにタスクを割り当てるように、専門家を派遣します。各エージェントは独自のスペースで作業し、結果はレビューのためにあなたに返ってきます。

レベル6では、エージェントは自己修正します。レベル7では、セッションをスリムに保ちながら、複数のエージェントを独立したタスクに派遣します。ここでレバレッジが倍増します。

バックグラウンドエージェントのオーケストレーション

メインセッションはコマンドセンターになります。ワーカーは分離されたコンテキスト（新しいコンテキストウィンドウ、別々のワークツリー）で実行されます。詰まったワーカーはあなたに質問を返します。

覚えておいてください：各「バックグラウンドエージェント」は、新しいコンテキスト、特定のタスク、プロジェクトのファイルとツールへのアクセスを持つClaude Codeセッションに過ぎません。魔法はありません — 同じモデル、同じCLAUDE.md、同じテストスイートを使用します。レバレッジは並列性と専門化から来ており、新しいケイパビリティからではありません。

現在のツール（2026年3月）

Dispatch。インストール：npx skills add bassimeledath/dispatch -g。ワーカーは新しいコンテキストウィンドウを取得します。詰まったワーカーは質問を返します。

/dispatch pre-launch sweep for TaskForge:
1) security audit the auth flow — use opus, worktree
2) write missing integration tests — use sonnet
3) update documentation — use haiku

タスクに適したモデルを選ぶ

すべてのタスクに最も強力なモデルが必要なわけではありません。モデルのケイパビリティをタスクの複雑さに合わせることで、品質を犠牲にすることなくコストを節約し、多くの場合速度が向上します。

タスクの複雑さ別の推奨モデル層
タスクタイプ推奨層理由例
アーキテクチャ上の決定、複雑なリファクタリング、曖昧な仕様 最高ケイパビリティ（例：Opus） 深い推論、曖昧さの処理、多くのファイルにわたる整合性の維持が必要 「TaskForgeのストレージ層をJSONファイルからSQLiteに再設計し、既存のテストをすべて維持する」
機能実装、バグ修正、テスト作成 中間層（例：Sonnet） 明確な受け入れ基準を持つ明確に定義されたタスク；より低いコストで強いケイパビリティ 「addコマンドに--priorityフラグを追加し、low/medium/highの値を持つ」
ドキュメント、フォーマット、ボイラープレート、シンプルな編集 高速/軽量（例：Haiku） 深い推論より速度が重要な機械的なタスク 「taskforge/api.pyのすべてのパブリック関数にdocstringを追加する」
注意：ここでも2層パターンが適用される

モデル名は変わります。「Opus」「Sonnet」「Haiku」は2026年3月現在のものです。原則は安定しています：モデルのケイパビリティをタスクの複雑さに合わせる。曖昧または重要な作業には最も強力なモデルを使用し、明確に定義された機械的なタスクには高速なモデルを使用してください。現在のラインアップについてはdocs.anthropic.com/modelsを確認してください。

迷った場合は、中間層モデルを使用してください。開発タスクの80%を適切に処理します。タスクが曖昧、多くのファイルに触れる、またはアーキテクチャ上の判断が必要な場合のみ最高層にエスカレートします。どちらにしても出力をレビューするような大量操作には軽量層に落としてください。

実装者/レビュワーの分離

同じモデルに自分の試験を採点させないでください。SonnetがコードをREVIEWするなら、OpusまたはClaude の別のモデルを使ってレビューしてください。偏った自己評価は既知の失敗モードです。

Ralphループ

自律的なエージェントループ：PRD（製品要求文書）が完成するまで実行し、各反復は新しいコンテキストを取得します。注意：仕様が不明確なPRDは後で問題を引き起こします。ループは曖昧さを増幅します — 曖昧な要件は、10回の反復にわたって繰り返される自信を持って間違った実装になります。

やってみよう：Ralphループを構築する

Ralphループは自律的なループエージェントパターンです：エージェントはタスクを実行し、成功基準に対して結果を確認し、基準が満たされていない場合は反復します — 問題を修正して再確認します — タスクが成功するか最大反復制限に達するまで。各反復は新しいコンテキストウィンドウを取得し、失敗した試みからのコンテキスト汚染を防ぎます。重要な洞察は、ループが自己修正であることです：エージェントは盲目的に再試行するのではなく、失敗の出力を読んでアプローチを適応させます。これにより、明確でテスト可能な成功基準を持つタスクに対して強力です。

TaskForgeのためのシンプルなRalphループを設定します：

claude --background "Fix all failing tests in TaskForge.
Run: python3 -m pytest
If any tests fail, read the failure output, fix the code, and re-run.
Repeat until all tests pass.
Maximum 5 iterations. If still failing after 5, stop and report what's left."

これをテストするには、まず意図的に何かを壊してください：models.pyにバグを導入します（例：戻り値を変更する、バリデーションチェックを壊す）。次にループを起動して、エージェントがそれを発見して修正するのを観察してください。

検証：エージェントの最終出力はすべてのテストがパスしていることを示します。git diffを確認して修正が合理的かを確認してください — 失敗したテストを削除するようなハックではないこと。

うまくいかない場合：（1）エージェントが永遠にループする → 反復制限は必須です、決して省略しないでください。（2）エージェントがコードを修正するのではなくテストを削除する → プロンプトに追加：「テストを削除またはスキップしないでください。ソースコードを修正してください。」（3）--backgroundが利用できない → 代わりにdispatchを使用：/dispatch fix all failing tests, max 5 iterations — use sonnet。

警告：常に反復制限を設ける

最大反復回数のないRalphループは無限に実行され、APIクレジットを消費し、各パスでコードベースをより悪化させる可能性があります。常にハードリミットを指定してください（ほとんどのタスクには3〜5回の反復）。エージェントが5回で修正できない場合、その問題は人間の判断が必要であり、より多くの反復ではありません。

CIトリガーエージェント

リポジトリイベントでアクティブになるエージェント：プルリクエストでのPRレビューボット、マージ時のドキュメント更新者、依存関係の変更時のセキュリティスキャナー。

あなたのセッションはコマンドセンターになります。ワーカーは新しいコンテキストで実行されます。質問はあなたに返ってきます。結果は完了時に返ってきます。

第45章 DockerでのClaude Code

なぜ今重要なのか

DockerでClaude Codeを実行することで、分離された再現可能なAIコーディング環境が得られます。これは、ローカルマシンを汚染することなく1つのエージェントから多数にスケールする方法です。

なぜDockerでClaude Codeを実行するのか？

ホストマシンでClaude Codeを直接実行することは単一セッションでは問題なく機能します。しかし、複数のエージェントにスケールするにつれて、Dockerは避けられない問題を解決します：

  • プロジェクト/エージェントごとの分離された環境：各エージェントは独自のファイルシステム、ツール、依存関係を取得します — クロス汚染はありません。
  • 再現可能なセットアップ：同じツール、同じ設定、毎回。マシン間またはセッション間でのドリフトがありません。
  • --dangerously-skip-permissionsを使った安全な実験：コンテナの分離によりこのフラグが安全になります — エージェントはコンテナの外に触れることができません。
  • 複数の並列エージェント：ボリュームマウントを使用して同じコードベース上で複数のエージェントを同時に実行します。
  • CI/CD統合：パイプラインからClaude Codeエージェントをトリガーします — 自動コードレビュー、テスト生成、ドキュメント更新。

前提条件

続ける前に、以下を確認してください：

  • Dockerがインストールされて実行中（第16章）
  • Claude MaxまたはProサブスクリプション（またはAnthropicのAPIキー）

Dockerfile.claudeパターン

コンテナでClaude Codeを実行するための本番グレードのDockerfileです。これは第43章のスキャフォールドパターンの結果であり、すべての行は省略することが失敗を引き起こしたために存在します。

FROM debian:bookworm-slim

# システムパッケージ — Claude Codeが動作するための最小セット
RUN apt-get update && apt-get install -y --no-install-recommends \
    git bash curl ca-certificates build-essential \
    jq tree ripgrep python3 python3-pip \
    && rm -rf /var/lib/apt/lists/*

# 非rootユーザー
RUN groupadd -r claude && useradd -r -g claude -m -s /bin/bash claude

# Claude Codeをインストール（ネイティブインストーラー — npmは使わない）
USER claude
WORKDIR /home/claude
RUN curl -fsSL https://claude.ai/install.sh | bash
ENV PATH="/home/claude/.local/bin:/home/claude/.claude/bin:${PATH}"

# オンボーディングウィザードが表示されないよう認証を事前設定する
RUN mkdir -p /home/claude/.claude && \
    echo '{"hasCompletedOnboarding":true,"installMethod":"native"}' \
      > /home/claude/.claude/.claude.json && \
    ln -sf /home/claude/.claude/.claude.json /home/claude/.claude.json && \
    echo '{"permissions":{"allow":["*"],"deny":[]},"env":{"CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC":"1"}}' \
      > /home/claude/.claude/settings.json

WORKDIR /workspace
CMD ["sleep", "infinity"]

各決定が重要な理由：

ヘッドレスClaude Code用のDockerfile設計上の決定
決定理由
debian:bookworm-slimglibcベースである必要があります。Alpine/muslはネイティブClaude Codeバイナリを壊します。
npmではなくネイティブインストーラーnpmパッケージは異なる設定パスを使用し、認証の破損を引き起こします。ネイティブインストーラーはAnthropicの公式方法です。
非rootのclaudeユーザーセキュリティのベストプラクティス。何か問題が起きた場合の影響範囲を制限します。
事前作成された.claude.jsonなければ、オンボーディングウィザードが毎回表示され、ヘッドレス操作がブロックされます。シンリンクは異なるバージョンがチェックする両方の設定パスをカバーします。
"allow":["*"]のsettings.json設定に焼き込まれた--dangerously-skip-permissionsと同等です。ヘッドレスモードでプロンプトなし。
CMD ["sleep", "infinity"]コンテナが生き続けます。各セッションにdocker execで入ります。複数のexec = 複数のClaude Codeセッション、すべてがワークスペースを共有します。
重要な概念：非決定論の教訓

このDockerfileはスキャフォールドパターンの実践です。曖昧な「Claude CodeのDockerfileを作成して」という指示は、まったく異なる結果を生み出します：npmvs ネイティブ、Alpine vs Debian、root vs 非root、ENTRYPOINT vs CMD。各間違った選択は特定の失敗を引き起こします — 認証ループ、バイナリクラッシュ、権限エラー。スキャフォールドはすべてのコイントスを排除します。これが非決定論的な出力を信頼できるものにする方法です。

認証方法

Claude Codeには認証が必要です。2つのアプローチがあり、同じ.envに共存させてはなりません：

OAuthトークン（Claude Max/Pro）

ホストマシンでトークンを生成し、環境変数として渡します：

ターミナル（ホスト上）claude setup-token

結果のトークンを.envファイルに追加します：

CLAUDE_CODE_OAUTH_TOKEN=sk-ant-oat01-your-token-here

トークンはdocker-compose.ymlのenv_file: .envを通じてコンテナに渡されます。Dockerイメージには決して焼き込まれません。

APIキー

トークンごとの課金（サブスクリプションなし）の場合：

ANTHROPIC_API_KEY=sk-ant-api03-your-key-here
重要な概念：両方設定しない

ANTHROPIC_API_KEYとCLAUDE_CODE_OAUTH_TOKENの両方が存在する場合、APIキーが優先され、サブスクリプションを使用する代わりにトークンごとに課金されます。どちらか一方を選んでください。

単一のClaude agentを実行する

パターン：コンテナを一度起動し、各セッションでdocker execで入ります。各execは同じワークスペースを共有する独立したClaude Codeセッションです。

ターミナル# コンテナのビルドと起動 docker compose up -d claude # インタラクティブなClaude Codeセッションを開く docker exec -it claude-myproject claude --dangerously-skip-permissions # 2番目のターミナルペインで別のセッションを開く docker exec -it claude-myproject claude --dangerously-skip-permissions

-v $(pwd):/workspaceボリュームマウントは現在のディレクトリをコンテナにマップします。エージェントがコンテナ内で行った変更はすぐにホストファイルシステムに表示されます。

--dangerously-skip-permissionsで実行する

--dangerously-skip-permissionsフラグは、Claude Codeにすべての権限プロンプトをスキップするよう伝えます — ファイルの書き込み、コマンドの実行、ネットワークリクエスト。エージェントは尋ねることなく動作します。

警告：このフラグを使うタイミング

ホストマシンでは、このフラグは本当に危険です — エージェントは任意のファイルを変更し、任意のコマンドを実行できます。Dockerコンテナ内では、リスクは封じ込められています：エージェントはコンテナのファイルシステムとマウントされたボリュームにのみ影響を与えることができます。これがコンテナと--dangerously-skip-permissionsが自然なペアである理由です：コンテナの分離が自律的な操作を安全にします。

完全な自律性で非インタラクティブタスクを実行する：

ターミナルdocker run -it -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY -v $(pwd):/workspace claude-agent --dangerously-skip-permissions -p "fix all failing tests"

-pフラグは、インタラクティブセッションを開く代わりにプロンプトを直接渡します。--dangerously-skip-permissionsと組み合わせることで、エージェントは完全に自律的に動作できます：ファイルを読み、コードを編集し、テストを実行し、完了するまで反復します。

Docker Composeのセットアップ

composeファイルは既存のプロジェクトサービスとともにclaudeサービスを定義します：

# docker-compose.yml
services:
  claude:
    build:
      context: .
      dockerfile: Dockerfile.claude
    container_name: claude-${COMPOSE_PROJECT_NAME:-agents}
    env_file: .env
    volumes:
      - .:/workspace
      - claude-auth:/home/claude/.claude
    working_dir: /workspace
    stdin_open: true
    tty: true

volumes:
  claude-auth:
    name: claude-auth-${COMPOSE_PROJECT_NAME:-default}

重要な詳細：claude-authボリュームはコンテナの再起動をまたいで認証を維持します。.:/workspaceはプロジェクトをマウントします。env_file: .envはOAuthトークンを渡します。コンテナ名にはプロジェクトディレクトリ名が含まれているため、各リポジトリは一意のコンテナを取得します。

ターミナル# コンテナを起動する docker compose up -d claude # 複数のエージェントセッションを実行する（各自のターミナルペインで） docker exec -it claude-myproject claude --dangerously-skip-permissions --model claude-opus-4-6 docker exec -it claude-myproject claude --dangerously-skip-permissions --model claude-opus-4-6
警告：調整の課題

複数のエージェントが同じボリュームマウントを共有する場合、同じファイルに同時に書き込むことができます。これにより競合が生じる可能性があります — 一方のエージェントがもう一方の変更を上書きします。戦略：（1）エージェントを重ならないディレクトリに割り当てる、（2）gitワークツリーを使用して各エージェントが独自の作業コピーを持つようにする、（3）並列ではなくパイプラインで順番にエージェントを実行する。

agent.shパターン

コンテナが実行されていることを確認してからexecで入る薄い便利スクリプトです。モデルは--modelフラグで強制されます（環境変数はモデル選択に機能しません）。

#!/usr/bin/env bash
set -euo pipefail

PROJECT_NAME="${COMPOSE_PROJECT_NAME:-$(basename "$(pwd)")}"
CONTAINER="claude-${PROJECT_NAME}"

# 実行中でなければコンテナを起動する
if ! docker ps --format '{{.Names}}' | grep -q "^${CONTAINER}$"; then
  docker compose up -d claude
  sleep 2
fi

# インタラクティブまたはワンショット
if [ $# -gt 0 ]; then
  docker exec -it "$CONTAINER" claude \
    --dangerously-skip-permissions --model claude-opus-4-6 -p "$*"
else
  docker exec -it "$CONTAINER" claude \
    --dangerously-skip-permissions --model claude-opus-4-6
fi

使用方法：

ターミナル./agent.sh # インタラクティブセッション ./agent.sh "refactor the auth module" # ワンショットプロンプト

複数のエージェントを実行するには、ターミナルペインをさらに開いてそれぞれで./agent.shを実行します。各execは独立したセッションで、すべてが同じワークスペースを共有します。

実践的なパターン

順次パイプライン

エージェント1がコードを書き、エージェント2がレビューし、エージェント3がテストを書きます。各エージェントの出力が次のエージェントの入力になります。これによりテストの前にレビューが確保され、問題が早期に発見されます。

並列ワーカー

複数のエージェントが独立した機能を同時に作業します。すべてが完了したら、結果をマージします。機能がファイルを共有しない場合に最適です。

ウォッチャーエージェント

テスト結果（またはCI出力）を監視し、テストが失敗したときに修正エージェントをdispatchするエージェント。これにより自己修復パイプラインが生まれます：何かを壊すと、ウォッチャーがエージェントを割り当てて修正します。

リソース管理

Dockerを使用すると、暴走したエージェントがマシン全体を消費しないよう、コンテナごとにリソースを制限できます：

# docker-compose.yml
services:

      context: .

      - .:/workspace
    environment:
      - ANTHROPIC_API_KEY
    deploy:
      resources:
        limits:
          memory: 2G
          cpus: "1.0"

ディスクスペースを回収するために定期的に停止したコンテナをクリーンアップします：

ターミナル# 停止したすべてのコンテナを削除 docker container prune # 未使用のイメージを削除 docker image prune
