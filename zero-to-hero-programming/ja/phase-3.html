第10章 ターミナルとシェルスクリプト

重要な概念

Claude Codeは完全にターミナル上で動作します。実行するコマンド、読み込むファイル、実行するテスト—すべてがここで学ぶシェル（Shell）コマンドを通じて行われます。ターミナルの習熟は、AI支援開発においてオプションではありません。ターミナルこそがインターフェースそのものです。

シェルスクリプト

シェルスクリプトとは、順番に実行されるコマンドのファイルです。すべてのスクリプトは、OSにどのインタープリタを使うかを伝えるシバン行から始めます：

#!/bin/bash
echo "Running tests..."
python3 -m pytest
echo "Done."

run_tests.shとして保存し、chmod +x run_tests.shで実行可能にしてから./run_tests.shで実行します。

パイプ：小さなツールを組み合わせる

|演算子は、あるコマンドの出力を別のコマンドの入力として送ります。>は出力をファイルに書き込みます。

パイプは小さなツールを強力なワークフローに組み合わせます。各ステージは次のステージのためにデータをフィルタリングまたは変換します。

環境変数とPATH

環境変数はシステム全体の設定です。PATHはシェルにコマンドの場所を伝えます。command not foundエラーが出たときは、そのプログラムがPATHにリストされているどのディレクトリにも存在しないことを意味します。

export API_KEY="sk-abc123"   # 変数を設定する
echo $PATH                    # PATHを確認する

SSHの基礎

SSH（Secure Shell）を使うと、ターミナル経由でリモートマシンに接続できます。コードをデプロイするときやクラウドサーバーにアクセスするときに使います。ssh-keygen -t ed25519でキーペアを生成し、公開鍵（~/.ssh/id_ed25519.pub）をリモートサービスに追加します（具体的なサービスへの設定は第12章で行います）。接続するには：ssh user@hostname。

tmux：ターミナルマルチプレクサ

概念：ターミナルマルチプレクサを使うと、1つのターミナルウィンドウ内で複数の永続的なシェルセッションを実行できます。ペインを分割したり、タブを切り替えたり、完全にデタッチしたりして—すべてをバックグラウンドで動かし続けることができます。これは複数のプロセスを同時に監視する必要があるワークフローで不可欠であり、複数のAIエージェントをオーケストレーションするときに特に重要になります。

補足：現在のツール（2026年3月）

tmux（全プラットフォーム、無料）。インストール：brew install tmux（macOS）またはsudo apt install tmux（Linux）。フェーズ5のマルチエージェントワークフロー向けの標準ターミナルマルチプレクサです。

重要な概念：tmuxがAI開発で欠かせない理由

フェーズ5では、複数のClaude Codeエージェントを同時に実行します。各エージェントには独自のターミナルセッションが必要です。tmuxなしでは、複数のターミナルウィンドウが必要になり、ノートPCを閉じるとすべて終了してしまいます。tmuxを使えば、セッションはバックグラウンドで維持されます—切断して再接続しても、エージェントは動き続けます。tmuxはバックグラウンドエージェントを実現するインフラです。

推奨tmux設定

~/.tmux.confを以下の設定で作成して、tmuxをより使いやすくします：

# ~/.tmux.conf

# Ctrl+BのかわりにCtrl+Aをプレフィックスにする（押しやすい）
unbind C-b
set -g prefix C-a
bind C-a send-prefix

# | と - で分割（覚えやすい）
bind | split-window -h -c "#{pane_current_path}"
bind - split-window -v -c "#{pane_current_path}"

# 新しいウィンドウは現在のディレクトリで開く
bind c new-window -c "#{pane_current_path}"

# Alt+矢印でペインをナビゲート（プレフィックス不要）
bind -n M-Left select-pane -L
bind -n M-Right select-pane -R
bind -n M-Up select-pane -U
bind -n M-Down select-pane -D

# マウスサポートを有効にする（クリックでペイン選択、スクロールでスクロール）
set -g mouse on

# ウィンドウ番号を1から始める（0ではなく）
set -g base-index 1
setw -g pane-base-index 1

# ウィンドウが閉じられたとき番号を振り直す
set -g renumber-windows on

# スクロールバックバッファを増やす
set -g history-limit 50000

# エスケープ遅延を減らす（vim/nanoユーザー向けに高速化）
set -sg escape-time 10

# ステータスバースタイリング
set -g status-style bg=black,fg=white
set -g status-left "[#S] "
set -g status-right "%H:%M"

保存後はtmux source-file ~/.tmux.confでリロード（または新しいセッションを開始）します。

注意：上記の設定を使った場合

この設定はプレフィックスをCtrl + BからCtrl + Aに変更し、分割を%と"のかわりに|と-で行います。上記のホットキー表はすべてデフォルトのバインディングを使っています。この設定を使う場合は、どこでもCtrl + BのかわりにCtrl + Aを使い、分割には|/-を使ってください。

第11章 Gitによるバージョン管理

重要な概念

Gitはオプションではありません。Claude Codeはブランチ（branch）を作成し、コミット（commit）を作り、プロジェクト履歴を読んでコードベースを理解します。Gitなしでは、Claude Codeを効果的に使うことができません—そしてGitを理解せずには、Claude Codeがあなたのコードに何をしているか監督できません。

注意：バージョン管理なしに何が起きるか

ファイルのリファクタリングに3時間費やします。何かを壊してしまいます。動いていたバージョンがどんなものだったか思い出せません。手動で元に戻そうとしてさらに悪化させます。Gitがあれば、1つのコマンド（git checkout -- file.py）で最後の動作バージョンを復元できます。なければ、すべての編集でギャンブルをしています。

補足：現在のツール（2026年3月）

Gitはユニバーサルなバージョン管理システムです。ほぼすべてのプロのプロジェクトで使われています。インストール：brew install git（macOS）、apt install git（Linux）、またはgit-scm.com（Windows）。

Gitはコードへのすべての変更を追跡します。プロジェクトのタイムマシンと考えてください—以前の任意の状態に戻ることができます。ブランチは平行したタイムラインのようなものです：メインのタイムラインを安定させたまま、新しい機能を別のタイムラインで作業できます。実験が失敗したら、そのタイムラインを破棄します。成功したら、メインに統合します。これが、AIエージェントが既存の動作を壊さずにコードを改善できる仕組みです。

注意：GitはClaude Codeの最重要前提条件

Claude CodeはGitの履歴を読み、コミットを作り、ブランチを作成し、コンフリクトを解消します。Claude CodeがGitで何をしているか理解できなければ、適切に監督することができません。

第12章 Gitリモートと GitHub

重要な概念

すべてのプロのプロジェクトはリモートリポジトリにあります。後のフェーズのClaude Codeワークフローには、プルリクエスト、リモートブランチ、CIパイプラインが含まれます—これらはすべてリモートGitへの理解が土台になります。

リモートとは何か？

これまでのGitの作業はすべてローカルでした—コミット、ブランチ、マージがあなたのマシン上にのみ存在していました。リモートとは、別のサーバー（通常はGitHub）にホストされているリポジトリのコピーです。ローカルリポジトリは作業場所です。リモートリポジトリは共有、バックアップ、コラボレーションの場所です。イメージとして言えば、ローカルリポジトリはあなただけのノート、リモートリポジトリはチーム全員が見られる共有ホワイトボードです。

プッシュすると、コミットをリモートにコピーします。プルすると、リモートのコミットをローカルリポジトリにコピーします。2つはこれらの明示的な操作によって同期を保ちます—何も自動的には起きません。

GitHubアカウントのセットアップ

GitHubアカウントをまだ持っていない場合：

  1. github.comにアクセスし、Sign upをクリックします。
  2. ユーザー名、メールアドレス、パスワードを設定します。
  3. ログインしたら、+アイコン（右上）→New repositoryをクリックします。
  4. 名前をtaskforgeとし、公開のままにして、READMEで初期化はしない（ローカルに既にあります）。
  5. GitHubがローカルリポジトリを接続するコマンドを表示します。次にそれを使います。

ローカルとリモートを接続する

GitHubに空のリポジトリを作成した後、ローカルリポジトリを接続します：

git remote add origin https://github.com/YOUR-USERNAME/taskforge.git

originはプライマリリモートの慣例的な名前です。何でも名前をつけられますが、originが普遍的です。接続を確認します：

git remote -v

2行表示されるはずです—fetch用とpush用で、両方ともGitHubのURLを指しています。

プッシュとプル

プッシュはローカルのコミットをリモートに送ります。プルはリモートのコミットをフェッチしてローカルブランチにマージします。

# コミットをGitHubに送る
git push origin main

# GitHubから新しいコミットを取得
git pull origin main

新しいブランチを初めてプッシュするとき、トラッキングを設定するために-uを使います：

git push -u origin main

その後は、git pushとgit pullでリモートとブランチを毎回指定する必要がなくなります。

注意：「リモートに作業が含まれているため拒否されました」エラー

! [rejected] main -> main (fetch first)が表示された場合、ローカルにないコミットがリモートにあることを意味します。修正方法：

git pull origin main    # リモートの変更をフェッチしてマージ
git push origin main    # これでプッシュが成功

プルがマージコンフリクトを作成した場合は、第11章で学んだ方法で解消します：ファイルを編集し、コンフリクトマーカーを削除し、git addしてgit commitします。

クローニング

git cloneはリモートからマシンに、すべてのファイル、全履歴、全ブランチを含むリポジトリ全体をダウンロードします：

git clone https://github.com/someone/their-project.git

これにより、内部に完全に機能するGitリポジトリを持つtheir-project/という新しいディレクトリが作成されます。originリモートは既にクローン元のURLを指すように設定されています。これが他の人のプロジェクトをあなたのマシンに取り込む方法です。

リモートのブランチ

ブランチはローカルマシンとリモートで独立して存在します。ローカルブランチをGitHubにプッシュするには：

git push origin feature-branch

存在しない場合はGitHubにブランチが作成されます。トラッキングを設定するには（ローカルブランチとリモートブランチが対応していることをGitに知らせる）：

git push -u origin feature-branch

トラッキングが設定されると、git pushとgit pullは使うリモートブランチを自動的に把握します。リモートのものを含むすべてのブランチを確認するには：

git branch -a

.gitignore

.gitignoreファイルはGitに追跡しないファイルとディレクトリを伝えます。生成されたファイル、シークレット、環境固有のアーティファクトを除外してリポジトリをクリーンに保ちます。

# バイトコンパイルされたPythonファイル
*.pyc
__pycache__/

# 仮想環境
.venv/
venv/

# 環境変数・シークレット
.env

# IDEの設定
.vscode/
.idea/

# OSファイル
.DS_Store

# Node（該当する場合）
node_modules/

パターン構文：

.gitignoreのパターン構文とマッチングルール
パターンマッチするもの
*.pyc任意のディレクトリの.pycで終わるすべてのファイル
__pycache__/__pycache__という名前のディレクトリとその中のすべて
.envプロジェクトルートの.envというファイル
.venv/.venvディレクトリとそのすべての内容
node_modules/node_modulesディレクトリ（JavaScriptの依存関係）
重要な概念：.gitignoreが必要な理由

.gitignoreなしでは、シークレット（APIキーを含む.envファイル）をコミットしてしまったり、依存関係（node_modules/、.venv/）で何百メガバイトもリポジトリが膨らんだり、OSのゴミ（.DS_Store）をプッシュしてすべての差分を汚したりするリスクがあります。最初のコミットの前に.gitignoreを追加してください。

プルリクエスト

プルリクエスト（pull request、PR）は、あるブランチを別のブランチにマージする提案です。コードレビューの核心です：変更を表示し、チームメートがレビューし、承認されてはじめてコードがmainにマージされます。ワークフロー：

  1. ブランチ：ローカルにフィーチャーブランチを作成。git switch -c add-delete-feature
  2. 作業：変更を加え、コミットする。
  3. プッシュ：git push -u origin add-delete-feature
  4. PRを開く：GitHubにアクセス。バナーが表示されます：「Compare & pull request」。クリックして変更内容と理由の説明を書く。
  5. レビュー：チームメートがコードを読み、コメントし、変更を要求する。
  6. マージ：承認されたら、GitHubの「Merge pull request」をクリックする。

個人開発者もPRを使います—何が変わったかとその理由の明確な記録を作り、後のフェーズでClaude Codeがあなたのレビュー用に作業を提出する方法です。

SSHキー

GitHubは2つの接続方法をサポートしています：HTTPSとSSH。HTTPSはプッシュのたびに認証情報を要求します。SSHはキーペアを使うので自動的に認証されます。

SSHキーを生成します：

ssh-keygen -t ed25519 -C "your-email@example.com"

Enterを押してデフォルトの場所を承認します。パスフレーズを設定するか空のままにします。次に公開鍵をGitHubに追加します：

  1. 公開鍵をコピー：cat ~/.ssh/id_ed25519.pub
  2. GitHub → Settings → SSH and GPG keys → New SSH key
  3. キーを貼り付けて保存。

接続をテストします：

ssh -T git@github.com

次のように表示されるはずです：Hi username! You've successfully authenticated。これでHTTPSのかわりにSSH URL（git@github.com:user/repo.git）を使えるようになります。

プッシュはローカルからGitHubにコミットを送ります。プルはGitHubからローカルにコミットをもたらします。クローンはリモートリポジトリから新しいローカルコピーを作成します。

よくある誤解

注意：「git pushでプロジェクト全体がアップロードされる」

いいえ。git pushはリモートがまだ持っていないコミットだけを送ります。100のコミットがあってリモートに98あれば、プッシュは新しい2つだけを送ります。Gitは効率的です—必要最小限のデータを転送します。

注意：「ローカルでファイルを削除するとGitHubからも削除される」

いいえ。マシン上のファイルを削除してもリモートには何も変わりません。削除がGitHubに到達するのは、git addで削除をステージし、git commitして、git pushした後だけです。それまでは、ファイルはGitHub上に以前の状態のままで存在し続けます。
