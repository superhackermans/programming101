Chapter 03Display Hardware & CAN Signal Architecture

Accuracy Warning for LLM Users

MFTBC does not publicly disclose display specifications (resolution, diagonal, panel type). If an LLM states "1920×720 12.3-inch TFT-LCD," these are unverified industry-typical values. Always verify against internal documentation.

CAN FD Migration (2025–Late 2020s)

Every cluster element is driven by CAN messages. ASIL B mandates 200 ms display latency and message integrity verification.

Key principles for CAN signal-to-UI mapping:

  • Signal latency budget: ASIL B requires telltale display within 200 ms of CAN message arrival.
  • Message integrity: Cluster must verify CRC and message counter. Corrupted/late messages trigger safe state.
  • DBC file as source of truth: Never trust LLM-generated CAN signal names. Verify against actual .dbc file.
Sources: CAN FD in J1939 • ELISA Telltale Safety

Chapter 04ADAS & Safety-Critical Telltale Design

The 2024+ Super Great carries one of the most comprehensive ADAS suites in the Japanese heavy-duty segment.

ABA6 Warning Escalation

ABA6 warning escalation: each stage increases visual urgency through color, animation rate, and screen real estate. Emergency brake commands full-screen takeover.

Active Sideguard Assist 2.0

ASA 2.0 monitors blind spots on both sides during turns and lane changes. Three-stage escalation at ≤20 km/h: visual alert → audible alarm + A-pillar lamp → damage mitigation brake.

Telltale Hierarchy Principle

When multiple ADAS systems trigger simultaneously, prioritize by criticality. Emergency interventions (ABA full brake, ASA emergency stop) take absolute priority. Lower-priority warnings queue and display once the critical event resolves.

Chapter 07Glance-Time Budgets, Ergonomics & Cognitive Load

At 85 km/h, a 1.5-second glance covers ~35 m of blind travel. Every display element must be readable within this budget.

SAE J941 Limitation

SAE J941's Class B (heavy truck) eye ellipses have not been updated since 2010 and don't account for height-adjustable seats. Treat J941-based cluster placement as approximate for trucks.

Long-Haul Cognitive Load

  • Monotony fatigue: After hours on expressways, alerts must break through attentional fatigue.
  • Exception-based info: Show only what has changed, not static dashboards.
  • Glove-compatible: Japanese truck drivers wear work gloves. Physical buttons are essential.
Sources: SAE J941 • Fleet Equipment: HMI Safety

Chapter 10Expert Prompting for FUSO HMI Work

Every prompt follows: Role → Context → Task → Format → Constraints.

Pattern 1: Cluster Layout Spec

Expert Prompt

Role: Senior automotive HMI designer, 15+ years heavy-duty commercial vehicle clusters, Japanese market. Familiar with ISO 26262 ASIL B, ISO 15005:2017, ISO 2575:2021, UNECE R121.

Context: Designing digital cluster for FUSO Super Great MY2025+. 6R30 12.8L diesel, SHIFTPILOT 12-speed AMT. ADAS: ABA6, ASA 2.0, FBIS, Active Attention Assist, LDW. Exhaust: BlueTec SCR + DPF. Telematics: Truckonnect. Right-hand drive (Japan market).

Task: Propose 3-tier visual hierarchy. Tier 1 always visible, Tier 2 contextual, Tier 3 on-demand. For each element: tier, screen zone (left/center/right/top/bottom), size allocation, display type (gauge/numeric/icon/bar/text), ASIL classification.

Format: Table grouped by tier. Constraints: ISO 15005 1.5s glance time. 10+ hour driver shifts. Night legibility. ASIL B telltales must support frame readback.

Always Specify RHD or LHD

The Super Great for Japan is right-hand drive. LLMs default to LHD assumptions. In RHD, "lower-right corner" is close to the driver — the opposite of LHD. Always state cab configuration explicitly.

Pattern 2: CAN Signal-to-UI State Matrix

Expert Prompt

Role: Automotive HMI systems engineer, CAN-to-display mapping, ISO 26262 ASIL B expertise.

Context: eCanter next-gen cluster. BMS signals (illustrative — verify against DBC):

BMS_StateOfCharge     (0-100%, 0.5% resolution)
BMS_ChargingStatus    (0=None, 1=AC L1, 2=AC L2, 3=DC)
BMS_CellTempWarning   (0=Normal, 1=High, 2=Critical)
BMS_IsolationFault    (0=OK, 1=Warning, 2=Fault)

Task: Define complete UI state matrix per signal: element type, color (ISO 2575), animation, audio, ASIL, priority. Format: One table per signal.

Chapter 12Claude Cowork for HMI Design Workflows

Claude Cowork, released in January 2026, extends Claude's agentic capabilities beyond coding into general knowledge work. Where a standard Claude conversation is reactive (you ask, it answers), Cowork is autonomous: you assign a task, grant folder or browser access, and Claude plans, executes, and delivers finished work — files created, data organized, research compiled.

For HMI designers at MFTBC, Cowork opens workflows that were previously impossible or prohibitively tedious with a chat interface.

Cowork operates autonomously: you provide a task and folder access, it plans and executes across files and browser, delivering finished work.

Use Case 1: Competitor Dashboard Scraping & Analysis

Services with slow-loading image galleries (dealer configurator sites, press image databases, internal asset management systems) can bottleneck your competitive analysis workflow. With Cowork + Claude in Chrome, you can instruct Claude to navigate to each competitor's press or product page, scroll through image galleries, capture screenshots of instrument cluster details, and compile them into an organized folder with filenames tagged by OEM, model, and feature. You review the compiled output in minutes instead of spending an afternoon clicking through slow-loading pages one by one.

When you need to cross-reference multiple ISO standards for a design decision (e.g., "Does my DPF icon sizing comply with ISO 15008 at 750mm viewing distance, and does its color comply with ISO 2575?"), Cowork can browse publicly available ISO previews, extract relevant clauses, and compile a focused compliance brief — with each claim linked to its source page.

Use Case 6: Design Review Preparation

Before presenting a cluster layout to stakeholders, have Cowork compile a review package: it reads your Figma export images, cross-references them against the telltale spec files in your folder, flags any missing states or ISO compliance gaps, and generates a presentation-ready summary document with screenshots annotated against requirements.

Use Case 7: CAN DBC File Processing

Given access to a folder containing .dbc files, Cowork can parse them, extract all signals relevant to the instrument cluster, generate a human-readable signal dictionary, and produce a draft UI state matrix for each signal — all saved as structured files ready for design review.

Cowork vs. Chat: When to Use Which

Use Cowork when: The task involves many files, requires browser access, is repetitive across items, or benefits from autonomous execution while you do other work.
Use regular Claude chat when: You need interactive back-and-forth reasoning, are exploring a single design decision, or want fine-grained control over each step.

Cowork Safety for ASIL-Rated Work

Cowork operates autonomously, which means it can create and modify files without step-by-step approval. For ASIL B-rated specifications, never grant Cowork write access to production spec directories. Use a separate working folder, and manually review + transfer files after verification. Anthropic explicitly warns about the risk of deleted files or prompt injection — keep production artifacts outside Cowork's sandbox.

Verification by Stakes Level

Low Stakes

Brainstorming, icon ideation, early concepts, comparative research.

Verification: Quick sanity check. If surprising, investigate. Creative value outweighs minor inaccuracy risk.

Medium Stakes

Internal design specs, layout proposals, style guides, competitor analysis.

Verification: Spot-check key claims. Verify standards references. Cross-reference CAN signals against DBC. Colleague review.

High Stakes

ASIL-rated elements, standards compliance, production specs, regulatory submissions.

Verification: Verify every claim. All ISO references checked. All ASIL confirmed by safety team. Never submit without qualified engineer review.

Non-Negotiable Rule

Never submit an ASIL B specification generated by an LLM without independent verification by a qualified functional safety engineer. The LLM is a drafting tool, not a safety authority.

Chapter 15Expert Workflow & Verification Checklist

Pre-Prompt Checklist

  1. What exactly do I need? (Layout spec? State matrix? Alert copy? Standards review?)
  2. What vehicle? (Super Great / eCanter / Canter / Fighter. Model year. RHD or LHD?)
  3. What context does the model need? (Display dimensions, CAN signals, ISO sections.)
  4. What output format? (Table, JSON, Figma component spec, prose.)
  5. Is this safety-critical? (If yes: plan full verification. No ASIL specs without engineer review.)

Post-Output Verification

  1. Does output match my request? (Format, scope, detail level.)
  2. Are ISO/SAE/UNECE references accurate? (Check every clause number.)
  3. Are CAN signal names real? (Verify against DBC. Assume fictitious until confirmed.)
  4. Are ASIL classifications correct? (Confirm with safety team.)
  5. Does placement analysis account for RHD? (Driver on right for Japan.)
  6. For safety-critical items: has a qualified engineer verified?
AI doesn't replace the judgment that makes a good HMI designer. It amplifies the throughput. You decide what the driver needs, when, and how. The LLM handles drafting, cross-referencing, and systematic enumeration. That division is where the productivity gain lives.
