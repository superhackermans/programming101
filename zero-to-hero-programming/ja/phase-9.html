重要な概念: LLMとは何か

AIコーディングツールを使い始める前に、それが何であるか—そして何でないか—を正確に理解しておきましょう。

大規模言語モデル（LLM）は、膨大なテキストから学習したパターンに基づいて、最も可能性の高い次の単語を予測します。人間のエンジニアのようにあなたのプロジェクトを「理解」しているわけではありません。セッション間のメモリはありません。自分の出力を検証することもできません。パターンに基づいて正しく見えるテキストを生成します—多くの場合、本当に役立ちますが—同時に、自信に満ちた、もっともらしいナンセンスを生成することもあります。

何百万ものコードベースを読んだことがある優秀なリサーチアシスタントのようなものですが、あなたのプロジェクトは見たことがありません。流暢なコードを書くことはできますが、アーキテクチャを誤解したり、存在しないAPIを発明したり、表面上はパスするが、エッジケースで失敗するロジックを生成したりすることがあります。

エージェント（agent）は魔法の存在ではありません。モデル呼び出しを中心に構築されたワークフローです: プロンプトが入力され、レスポンスが出力され、ツールがアクションを実行し、ルールが動作を制約します。エージェントが「インテリジェント」なことを行うすべては、あなたが提供したコンテキスト、設定したツール、設定した制約に遡ります。セットアップが優れているほど、エージェントのパフォーマンスも向上します。

これは2つのことを意味します: (1) あなたの判断がクオリティのゲートであり、(2) フェーズ1〜4で学んだすべてのこと—コードを読むこと、データフローを理解すること、テストを書くこと—がその判断を可能にするものです。

第38章 AIが生成したコードの評価と指示

なぜ今これが重要か

AIツールはコードを素早く生成します。しかし、判断力のないスピードは危険です。この章では、クオリティのゲートになる方法を学びます—AIの出力がコードベースに入る前に、それが正確で、完全で、安全かどうかを判断する人になるために。

これはフェーズ9で最も重要な章です。このスキルなしに続く各章を読んでも、出力を評価する判断力なしにAIツールを使うだけの読者になってしまいます。

核心的な問題: 非決定性

これまで学んできたすべてのプログラミングの概念は決定論的です。2 + 2を百万回実行しても、毎回4が得られます。sorted([3,1,2])を呼び出すと、結果は常に[1,2,3]です。関数には入力と出力があります。テストはパスするか失敗するかのどちらかです。この予測可能性がソフトウェアエンジニアリングの全基盤です。

大規模言語モデル（LLM）は根本的に非決定論的です。同じ質問を2回聞いても、異なる答えが返ってくる場合があります。関数を書くように頼むと、変数名、構造、エッジケースの処理、さらにはアルゴリズムの選択も実行間で異なる場合があります。これはバグではありません—テクノロジーの動作原理です。LLMはトークンの確率分布からサンプリングします; 小さな浮動小数点の差が異なる出力にカスケードします。

重要な概念

計算機を使うとき、出力を信頼します。LLMを使うとき、出力を検証します。これは哲学的な立場ではありません—エンジニアリングの要件です。AIを活用した開発という分野全体は、LLMの出力がデフォルトでは信頼できないために存在します。

これはパラドックスを生み出します: 信頼性の高いソフトウェアを信頼性の低いジェネレータから得たいのです。非決定論的なシステムを決定論的であるかのように動作させる唯一の方法は、極めて堅牢なガードレールで囲むことです:

  • テスト — コードはパスするかしないか、生成方法に関係なく
  • 仕様 — 正確な指示は可能な出力の空間を狭める
  • コンテキストドキュメント（CLAUDE.md、ルールファイル）— モデルの選択をプロジェクトのパターンに合わせるよう制約する
  • レビューパイプライン — diff、リンティング、型チェックがモデルの見逃しをキャッチする
  • 決定論的バリデーション — CI/CD、自動チェック、判断に依存しない受け入れ基準

フェーズ5〜6のすべてのテクニック—context engineering、複合ルール、ハーネスエンジニアリング、Docker分離—は非決定論的な出力を決定論的な境界に制約するために存在します。ガードレールはオプションの追加機能ではありません。それが核心的な目的です。ガードレールなしでは、宝くじを出荷していることになります。

AIが生成する。あなたが検証する。ガードレールが強制する。それが契約です。ここから先の各章は、より良いガードレールを構築することを教えます。

diffを読む

diffは行ごとの変更を示します: +=追加、-=削除。Claude Codeが変更を加えると、受け入れる前にdiffをレビューします。チェック: 変更は自分が求めたものと一致しているか? 無関係な変更はないか? 誤った削除はないか?

実例: 実際のdiffのレビュー

経験豊富な開発者がdiffをどのようにレビューするかを示します。Claudeに「load_tasks関数にエラー処理を追加して」と頼んだとします。diffは次のようになります:

 def load_tasks(filepath):
-    with open(filepath) as f:
-        data = json.load(f)
-    return data
+    if not isinstance(filepath, str) or not filepath.strip():
+        raise ValueError("filepath must be a non-empty string")
+    try:
+        with open(filepath) as f:
+            data = json.load(f)
+    except:
+        return []
+    if not isinstance(data, list):
+        raise TypeError(f"Expected list, got {type(data).__name__}")
-    import os
     return data

行ごとのアノテーション:

  • 1〜2行目（追加）: 良い—filepathの入力バリデーションを追加。Noneや空文字列がopen()に到達することによる不明なエラーを防ぐ。
  • 5行目（except:）: 問題—これはKeyboardInterruptやSystemExitを含むすべての例外をキャッチするベアのexceptです。予想される失敗のみをキャッチするためにexcept (FileNotFoundError, json.JSONDecodeError):にすべきです。
  • 6行目（return []）: 疑問—空のリストを静かに返すとエラーが隠れます。これは意図的ですか、それとも警告をログに記録すべきですか? プロジェクトのエラーポリシーによります。
  • 8〜9行目（dataの型チェック）: 良い—返す前にJSON構造を検証。破損したファイルを早期にキャッチ。
  • 10行目（import osの削除）: 疑問—なぜこのインポートが削除されたのか? 他にosを使うものがなければ、これは適切なクリーンアップです。しかし、まずファイルの残りの部分を確認してください。

判定: 修正が必要。ベアのexceptは本物のバグ—特定の例外をキャッチするようClaudeに求めてください。残りの部分はしっかりしています。

第40章 レベル3 — context engineering

なぜ今これが重要か

平凡なAI出力と優れた出力の違いは、ほとんどの場合モデルではありません—コンテキストです。モデルはどのファイルを見るか? どんな指示がガイドするか? どんな制約が悪い出力を防ぐか? context engineeringはAIを活用した開発で最も高い効果をもたらすスキルです。

context engineeringとは、モデルが見るものを制御して、すべてのトークンが有用な仕事をするようにすることです。「すべてのトークンはプロンプト（prompt）の中で自分の居場所を勝ち取る必要がある。」

コンテキストウィンドウ（context window）

コンテキストウィンドウ（context window）をモデルの作業メモリ予算と考えてください。モデルが一度に考慮できるすべてのもの—プロンプト（prompt）、システム指示、ファイルの内容、会話履歴—はこの予算に収まる必要があります。いっぱいになると、古い情報が押し出されます。この予算を管理することがcontext engineeringの核心スキルです。

4つのサーフェス

1
CLAUDE.md

毎セッション開始時に読まれる永続的な指示。200行以下、具体的、実行可能。

2
ツールの説明

モデルはこれを読んでどのツールを呼び出すかを決定します。明確な説明=より良いツール使用。

3
会話管理

コンテキストの約50%で/compact。探索にはサブエージェント。タスク切り替えには/clear。

4
選択的コンテキスト

コードベース全体ではなく、関連ファイルのみを含める。ノイズが少ない=より良い出力。

TaskForge用CLAUDE.mdの例

# TaskForge
Command-line task manager with JSON persistence and Flask API.

## Commands
- `python3 -m pytest` — run tests
- `python3 -m flask run` — start API server
- `python3 src/taskforge/main.py` — run CLI

## Architecture
- src/taskforge/ — core logic (main.py, models.py, api.py)
- tests/ — pytest test files
- data/ — JSON storage (gitignored)

## Code Style
- Type hints on all function signatures
- Docstrings on all public functions
- No bare try/except — always catch specific exceptions

## Gotchas
- Task IDs are auto-incrementing integers, NOT UUIDs
- JSON file is the single source of truth
- Flask API mirrors CLI functionality exactly

指示スキャフォールドを作る

小規模プロジェクトのCLAUDE.mdは簡単です。しかし、目標がより大きい場合—たとえばClaude Codeに環境全体をゼロから設定させる場合はどうでしょうか? ここで指示スキャフォールドを構築します: 非決定論的なモデルがほぼ毎回同じ正しい出力を生成するほど精密なドキュメント。

第38章の核心的な問題を覚えていますか: LLMは非決定論的です。「このプロジェクトにDockerをセットアップして」とClaudeに5回頼むと、5つの異なるDockerfile、異なるcomposeの設定、異なる認証アプローチが得られます。いくつかは動作します。いくつかは動作しません。このバリエーションが敵です。

解決策は、すべての決定ポイントで曖昧さを排除するスキャフォールドです:

1
目標を一文で述べる

「Dockerをセットアップして」ではなく「1つのコンテナ。execすると、リポジトリを指すClaude Codeが得られる。CMDはsleep infinity。以上。」

2
正確なバージョンとコマンドを指定する

「Claude Codeをインストールして」ではなくcurl -fsSL https://claude.ai/install.sh | bash。「Debianを使って」ではなくFROM debian:bookworm-slim。すべての未指定の選択はモデルが半分の確率で間違える硬貨投げです。

3
間違ったアプローチを明示的に禁止する

「npmインストールは絶対に使わないこと—認証/PATHの破損を引き起こす。」「CLAUDE_MODELを環境ブロックに追加しないこと—実際の環境変数ではない。」ネガティブな制約はポジティブなものと同様に重要です。モデルのトレーニングデータにはすべての間違ったアプローチも含まれているからです。

4
正確なファイル内容を提供する

Dockerfileに何が含まれるべきかを説明しないで—Dockerfileを書いてください。モデルはコピーすべきであり、即興すべきではありません。モデルがゼロから生成するすべての行は、実行間で変わる可能性のある行です。

5
バリデーションチェックリストを追加する

出力が正しいことを確認するyes/noチェックのリスト。「ベースイメージはdebian:bookworm-slimか?」「CMDはsleep infinityか?」「.envにはすべての既存行が含まれているか?」これらが非決定論的な出力を囲む決定論的なガードレールです。

6
既存ファイルを保護する

「.envを絶対に上書きしないこと—追記のみ。」「プロジェクトの既存のDockerfileに触れないこと。」破壊は不可逆的です。スキャフォールドは正しいことをすることより物事を壊すことを難しくする必要があります。

重要な概念: なぜこのレベルの精度が重要か

「このプロジェクト用にDockerをセットアップして」のような曖昧な指示は、モデルに数千の解釈の可能性を与えます。正確なファイル内容、禁止されたアプローチ、バリデーションチェックリストを持つ精密なスキャフォールドは、それを本質的に1つに絞ります。明確化の質問をできる人間の同僚への指示を書いているのではありません—解釈の余地を残すと自信を持って間違ったことをするシステムへの制約を書いています。ガードレールが厳密なほど、出力はより決定論的になります。

第43章（ハーネス（harness）エンジニアリング）では、これらの原則すべてをClaude Code用のDocker環境のセットアップに適用した完全なスキャフォールドを見ることになります。300行以上あります—そして各行は実際に欠落が失敗を引き起こしたから存在します。

複合ループ

仕様駆動ワークフロー

AIを活用した開発で最も効果的なワークフローです。コードに直接飛び込む代わりに: ブレインストーミング → 仕様 → 計画 → 実行という流れで進みます。

仕様駆動ワークフロー: コードから始めない。理解から始め、書き留め、ステップに分け、実行する。

ステップ1: ブレインストーミング

コードではなく、会話から始めます。AIにあなたに質問するよう求めます:

I want to build [feature]. Before we write any code, ask me clarifying questions
one at a time to understand: what problem this solves, who uses it, what the core
features are, what it should NOT do, and what edge cases matter.

AIが明確化の質問をします。あなたが答えます。出力: 問題の共通理解。

ステップ2: 仕様を書く（spec.md）

ブレインストーミングを書面仕様に変換します:

  • システムが何をするか（要件）
  • システムが何をしないか（明示的なスコープ境界）
  • データモデル（何を保存するか）
  • APIまたはインターフェース設計（入力、出力、エラー）
  • エッジケースとエラー処理
  • 受け入れ基準（完了の判断方法）

ステップ3: 計画を生成する（prompt_plan.md）

仕様を順序付きの実装ステップに分解します。各ステップはAIに与えるプロンプトであり、5分以内にレビューできるほど小さいもの。

ステップ4: ステップごとに実行する

計画から各プロンプトを一つずつ渡します。出力をレビューします。テストを実行します。次に進む前に問題を修正します。レビューを省略しない。複数のステップをまとめない。

なぜこれが機能するか: 仕様はAIのメモリではなく、真実のソースです。計画がチェックポイントを作ります。何か問題が発生した場合、正確にどこかがわかります。そして複合化します — 仕様の品質はプロジェクトごとに向上します。

補足: 仕様駆動アプローチの参考資料

このワークフローは、毎日AIを使って本番ソフトウェアを構築している実践者から適応しています:

  • Harper Reed: My LLM Codegen Workflow — 元の仕様駆動ワークフロー。
  • Harper Reed: Basic Claude Code — Claude Codeで作業するための実践的なパターン。

演習: 悪い仕様を分析する

悪く書かれた仕様を与えられた場合、問題を特定してください。欠けているセクションを持つdictを返します。

第42章 レベル5 — MCP・スキル・ケイパビリティ

なぜ今これが重要か

レベル1〜4はツールをそのまま使います。レベル5はツールを拡張し始めるところです。MCPはClaude Codeをデータベース、ブラウザ、外部サービスに接続します。スキルは再利用可能なワークフローを与えます。サブエージェントは特化した役割を与えます。あなたはユーザーからアーキテクトへと移行しています。

MCP（Model Context Protocol）

概念: Claude Codeが外部ツールやサービスと対話できるようにする標準化されたコネクタ。

現在のセットアップ（2026年3月）
// .mcp.json (project root)
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": { "DATABASE_URL": "postgresql://..." }
    }
  }
}

カスタムスキル（skill）

概念: 要求に応じて読み込まれる再利用可能な指示セット。スキル（skill）はワークフロー、ツールの制限、モデルの選択を定義します。

# .claude/skills/pr-review/SKILL.md
---
name: pr-review
description: Reviews PRs for bugs, security, performance, style.
tools: Read, Glob, Grep, Bash
model: opus
---
You are a senior code reviewer. Check:
## Security — hardcoded secrets, injection, XSS
## Performance — N+1 queries, missing indexes
## Output — BLOCKING, SUGGESTION, or APPROVED

カスタムサブエージェント（sub-agent）

概念: ツールが制限された特化した役割。各サブエージェントは必要なツールにのみアクセスできます。

# .claude/agents/test-writer.md
---
name: test-writer
description: Writes tests. Can read source and write test files only.
tools: Read, Glob, Write
model: sonnet
---
Write comprehensive tests. Cover: happy path, edge cases, error cases.
1つのスキルが特化したサブエージェントを生成でき、各エージェントが異なる観点をチェックします。結果は統合されたレビューに収束します。
