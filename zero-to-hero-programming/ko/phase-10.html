43~47장페이즈 게이트 + TaskForge
페이즈 10을 시작하기 전에

이 페이즈는 다음을 할 수 있다고 가정합니다: AI가 생성한 코드를 평가하고 흔한 실패 패턴을 식별할 수 있어야 하며(38장), 효과적인 CLAUDE.md 파일을 작성하고 컨텍스트(context, 문맥 정보)를 관리할 수 있어야 하고(40장), 테스트와 함께 단일 작업 개발에 Claude Code를 사용할 수 있어야 합니다(39~42장). 페이즈 10은 참고 자료이므로 — 프로젝트가 성장함에 따라 다시 돌아오세요.

에이전트 하나로는 항상 충분하지 않은 이유

단일 Claude Code 세션은 강력합니다. 그러나 일부 작업은 하나의 세션에 맞지 않습니다:

  • 컨텍스트 한계: 하나의 세션은 전체 대규모 코드베이스(codebase)를 담을 수 없습니다. 특화된 에이전트는 특정 영역에 집중합니다.
  • 관심사 분리(Separation of concerns): 코드를 작성한 에이전트가 자신의 작업을 검토해서는 안 됩니다. 구현과 리뷰를 분리하면 더 많은 버그를 잡을 수 있습니다.
  • 병렬 처리(Parallelism): 독립적인 기능 세 개를 동시에 구축하면 하나를 구축하는 시간에 완성됩니다.
  • 전문화(Specialization): 보안 감사에 맞게 구성된 에이전트는 사후에 보안을 확인하도록 요청받은 범용 에이전트보다 더 나은 결과를 생성합니다.

회사의 팀처럼 생각해보세요: 한 사람은 코드를 작성하고, 다른 사람은 검토하고, 다른 사람은 테스트를 작성하고, 다른 사람은 문서를 업데이트합니다. 더 많은 사람은 더 많은 처리량을 의미하지만 — 동시에 더 많은 조정 오버헤드도 의미합니다. 제약 없는 더 많은 에이전트는 생산성이 아닌 혼돈을 만들어냅니다. 그래서 이 페이즈는 작업을 디스패치(dispatch, 발송)하는 것만큼이나 하네스(harness, 검증 도구 모음)와 가드레일(guardrail, 안전 경계)에 집중합니다.

43장 레벨 6 — 하네스 엔지니어링 & 자동화된 피드백 루프

지금 이것이 중요한 이유

레벨 1~5에서는 여러분이 품질 게이트(quality gate)입니다 — 모든 변경 사항을 직접 검토합니다. 단일 기능에는 효과적이지만 확장되지 않습니다. 레벨 6은 자동화된 품질 게이트를 구축하여 에이전트가 자신의 실수를 스스로 잡을 수 있게 하고, 여러분은 줄 단위 검토 대신 설계 결정에 집중할 수 있게 합니다.

레벨 5에서는 에이전트에게 능력을 부여했습니다. 레벨 6에서는 에이전트에게 자신의 작업을 검증할 수 있는 가드레일을 부여합니다. 전환점: 여러분의 역할이 코드 검토에서 하네스 설계로 이동합니다.

백프레셔(Backpressure)

백프레셔는 인간 개입 없이 에이전트가 스스로 수정할 수 있게 하는 자동화된 피드백입니다. 코드 리뷰에서 여러분이 오류를 잡는 대신, 도구들이 즉시 잡아냅니다:

자동화된 에이전트 수정을 위한 백프레셔 메커니즘
메커니즘감지 대상
TypeScript strict / mypy타입 오류
린터(Linter) (ESLint, Ruff)스타일 위반
테스트 스위트(Test suite)동작 회귀(Behavioral regressions)
프리-커밋 훅(Pre-commit hooks)모든 커밋 전 포맷, 린트, 타입 검사
CI 파이프라인(CI pipeline)통합 실패

에이전트가 git commit을 실행하다가 프리-커밋 훅이 실패하면, 에이전트는 오류를 확인하고 코드를 수정한 후 다시 시도합니다. 이것이 백프레셔의 작동 방식입니다.

주의: 백프레셔 없이는 어떻게 되는가

에이전트에게 기능을 추가하도록 요청합니다. 에이전트는 미묘한 타입 오류가 있는 코드를 작성합니다. 타입 검사기 없이는 코드가 정상적으로 보입니다. 배포하면 프로덕션에서 충돌이 발생합니다. 타입 검사기를 실행하는 프리-커밋 훅이 있으면, 에이전트는 오류가 여러분에게 도달하기 전에 잡고 수정합니다. 수정에 몇 시간이 아닌 몇 초가 걸립니다.

제약 조건 > 지시 사항

단계별 지시 사항은 모델에게 어떻게 작업할지 알려줍니다. 제약 조건은 성공이 어떤 모습인지를 알려주고 방법은 모델이 스스로 파악하게 합니다. 제약 조건은 모델이 문제에 맞게 접근 방식을 조정하기 때문에 더 잘 확장됩니다.

단계별 방식 (취약)
1. 사용자 모델을 읽는다.
2. 이메일 필드를 추가한다.
3. RFC 5322 유효성 검사를 추가한다.
4. 마이그레이션을 업데이트한다.
5. 테스트를 업데이트한다.
제약 기반 방식 (견고)
사용자 모델에 이메일을 추가한다.

요구 사항: RFC 5322 유효성 검사, 고유 제약 조건.
승인 기준: 모든 테스트 통과, 유효/무효/중복/null에 대한 새 테스트.
실행: npm test가 통과될 때까지 작업한다.

보안 경계

에이전트, 코드, 시크릿(secrets, 비밀 정보)은 별도의 신뢰 도메인에 있습니다. 에이전트에게 프로덕션 자격 증명에 대한 직접 접근을 절대로 주지 마세요. 환경 변수, 시크릿 매니저, 최소 권한 접근(least-privilege access)을 사용하세요.

탐색 수단으로서의 문서

CLAUDE.md는 약 100줄의 목차 역할을 합니다. 자세한 문서는 다른 곳에 있습니다(architecture.md, API 문서, 배포 가이드). 모델은 22장의 점진적 공개(progressive disclosure) 패턴을 통해 필요할 때 발견합니다.

하네스는 에이전트가 자신의 작업을 검증할 수 있게 합니다. 여러분의 역할이 코드 검토에서 하네스 설계로 이동합니다.

TaskForge 연결

Ruff(린터)와 pytest를 실행하는 TaskForge의 프리-커밋 훅을 설정합니다. 이를 통해 Claude Code가 기능을 추가할 때 스스로 수정할 수 있습니다.

사례 연구: Docker 스캐폴드

이 장과 21장의 스캐폴드(scaffold, 구조 틀) 섹션의 모든 원칙을 적용하는 실제 지시 스캐폴드입니다. 목표: Claude Code에게 멀티 에이전트 개발을 위해 모든 프로젝트를 컨테이너화하는 방법을 알려주는 것입니다. 이 스캐폴드는 Claude가 잘못된 작업을 한 수십 번의 반복을 거쳐 정제되었습니다 — 각 실패는 새로운 제약 조건이 되었습니다.

효과적인 이유

스캐폴드는 약 300줄의 CLAUDE.md 파일입니다. 각 섹션이 존재하는 이유:

1
한 문장 목표

"컨테이너 하나. exec로 들어가면 저장소를 가리키는 Claude Code를 얻는다. CMD는 sleep infinity다." 해석의 여지가 전혀 없습니다. 모델은 "완료"가 어떤 모습인지 알고 있습니다.

2
설명이 아닌 정확한 파일 내용

Dockerfile, docker-compose 서비스, 쉘 스크립트가 그대로 제공됩니다 — 설명이 아닙니다. "Debian 기반 이미지 사용"이 아닌 FROM debian:bookworm-slim. "Claude Code 설치"가 아닌 curl -fsSL https://claude.ai/install.sh | bash. 모델이 처음부터 생성하는 모든 줄은 실행 간에 달라질 수 있는 줄입니다.

3
실제 실패에서 나온 명시적 금지 사항

각 "NEVER" 줄은 Claude가 실제로 잘못된 작업을 했기 때문에 존재합니다:

  • "NEVER npm install을 사용하지 않는다 — 인증/PATH 손상을 유발한다" — Claude의 훈련 데이터에는 오래된 npm 기반 튜토리얼이 포함됨
  • "환경 블록에 CLAUDE_MODEL을 추가하지 않는다 — 실제 환경 변수가 아니다" — Claude가 이 환경 변수를 반복적으로 환각(hallucination)함
  • "NEVER .env를 덮어쓰지 않는다 — 추가만 한다" — Claude가 파일을 다시 작성하여 기존 구성을 파괴함
  • "claude 서비스에 environment: 블록을 추가하지 않는다" — Claude가 env_file과 충돌하는 불필요한 환경 변수를 추가함

4
실패 모드 표

인증 참조 표가 증상을 원인에서 수정으로 매핑합니다. "온보딩 마법사 나타남" → ".claude.json 없음" → "Dockerfile이 미리 생성함." 이것은 모델에게 빌드 레시피만이 아닌 디버깅 플레이북을 제공합니다.

5
검증 체크리스트

정확성을 확인하는 15개의 예/아니오 체크: "기본 이미지가 debian:bookworm-slim인가?" ".env가 덮어쓰이지 않았는가?" "agent.sh가 --dangerously-skip-permissions를 사용하는가?" 이것들은 결정론적 가드레일입니다. 모델은 체크리스트를 합리화로 통과할 수 없습니다.

핵심 개념: 실제 비결정성 원칙

이 스캐폴드가 효과적인 이유는 모델이 내려야 할 결정을 최소화하기 때문입니다. 모호한 "Docker 설정"은 Claude에게 수천 가지 가능한 구성을 제공합니다. 스캐폴드는 그것을 본질적으로 하나로 좁힙니다. 이것이 기본 패턴입니다: 제약 조건이 강할수록 출력이 더 결정론적(deterministic)이 됩니다. 명시되지 않은 모든 세부 사항은 동전 던지기입니다. 좋은 스캐폴드는 동전 던지기를 제거합니다.

스캐폴드가 단순히 지시만 하는 것이 아님을 주목하세요 — 보호합니다. .env에 대한 추가 전용 규칙은 데이터 손실을 방지합니다. 별도의 Dockerfile.claude는 프로젝트의 기존 Docker 설정을 덮어쓰는 것을 방지합니다. 인증을 위한 볼륨 마운트는 매번 재시작 시 재인증을 방지합니다. 스캐폴드는 방어적 엔지니어링입니다: 모델이 무언가 파괴적인 작업을 시도할 것이라고 가정하고 구조적으로 불가능하게 만듭니다.

배포, 테스트, 코드 생성 또는 반복 가능한 작업을 위한 자신만의 스캐폴드를 구축할 때 이 패턴을 따르세요: 목표 → 정확한 내용 → 금지 사항 → 실패 모드 → 검증. 구체적일수록 모델의 비결정성과 관계없이 모든 실행에서 출력이 더 안정적이 됩니다.

직접 해보기: Ralph 루프 구축

Ralph 루프는 자율 반복 에이전트 패턴입니다: 에이전트가 작업을 실행하고, 성공 기준에 대해 결과를 확인하며, 기준이 충족되지 않으면 작업이 성공하거나 최대 반복 한도에 도달할 때까지 — 문제를 수정하고 다시 확인하면서 — 반복합니다. 각 반복은 신선한 컨텍스트 윈도우를 얻어 실패한 시도의 컨텍스트 오염을 방지합니다. 핵심 통찰은 루프가 자기 수정적이라는 것입니다: 에이전트는 맹목적으로 재시도하는 것이 아니라 실패 출력을 읽고 접근 방식을 조정합니다. 이것이 명확하고 테스트 가능한 성공 기준이 있는 작업에 강력합니다.

TaskForge를 위한 간단한 Ralph 루프를 설정하세요:

claude --background "Fix all failing tests in TaskForge.
Run: python3 -m pytest
If any tests fail, read the failure output, fix the code, and re-run.
Repeat until all tests pass.
Maximum 5 iterations. If still failing after 5, stop and report what's left."

테스트하려면, 먼저 의도적으로 무언가를 망가뜨리세요: models.py에 버그를 도입합니다(예: 반환 값 변경, 유효성 검사 깨뜨리기). 그런 다음 루프를 시작하고 에이전트가 버그를 찾아 수정하는 것을 지켜보세요.

검증: 에이전트의 최종 출력이 모든 테스트 통과를 보여줍니다. 수정이 합리적인지 확인하기 위해 git diff를 확인하세요 — 실패한 테스트 삭제 같은 꼼수가 아닌지 확인합니다.

작동하지 않을 경우: (1) 에이전트가 무한 루프 → 반복 한도는 필수이며 절대 생략하지 마세요. (2) 에이전트가 코드 수정 대신 테스트 삭제 → 프롬프트에 추가: "테스트를 삭제하거나 건너뛰지 마세요. 소스 코드를 수정하세요." (3) --background 사용 불가 → 대신 dispatch 사용: /dispatch fix all failing tests, max 5 iterations — use sonnet.

주의: 항상 반복 한도를 설정하세요

최대 반복 횟수 없는 Ralph 루프는 무한정 실행되어 API 크레딧을 소진하고 코드베이스를 반복할 때마다 악화시킬 수 있습니다. 항상 하드 한도(대부분의 작업에서 3~5회)를 지정하세요. 에이전트가 5번의 시도에서 수정할 수 없다면 문제는 더 많은 반복이 아닌 인간의 판단이 필요합니다.

Claude Code를 Docker에서 실행하는 이유

호스트 머신에서 직접 Claude Code를 실행하는 것은 단일 세션에는 잘 작동합니다. 그러나 여러 에이전트로 확장할 때 Docker는 피할 수 없는 문제들을 해결합니다:

  • 프로젝트/에이전트별 격리된 환경: 각 에이전트는 자체 파일 시스템, 도구, 의존성을 얻습니다 — 교차 오염 없이.
  • 재현 가능한 설정: 항상 같은 도구, 같은 설정. 머신이나 세션 간 드리프트(drift, 설정 불일치) 없이.
  • --dangerously-skip-permissions로 안전한 실험: 컨테이너 격리로 이 플래그가 안전해집니다 — 에이전트가 컨테이너 외부의 어떤 것도 건드릴 수 없습니다.
  • 여러 병렬 에이전트: 볼륨 마운트를 사용하여 동일한 코드베이스에서 여러 에이전트를 동시에 실행합니다.
  • CI/CD 통합: 파이프라인에서 Claude Code 에이전트를 트리거합니다 — 자동화된 코드 리뷰, 테스트 생성, 문서 업데이트.

Dockerfile.claude 패턴

컨테이너에서 Claude Code를 실행하기 위한 프로덕션급 Dockerfile입니다. 이것은 43장의 스캐폴드 패턴의 결과입니다 — 모든 줄은 빠뜨렸을 때 실패를 초래했기 때문에 존재합니다.

FROM debian:bookworm-slim

# System packages — minimal set for Claude Code to operate
RUN apt-get update && apt-get install -y --no-install-recommends \
    git bash curl ca-certificates build-essential \
    jq tree ripgrep python3 python3-pip \
    && rm -rf /var/lib/apt/lists/*

# Non-root user
RUN groupadd -r claude && useradd -r -g claude -m -s /bin/bash claude

# Install Claude Code (NATIVE installer — never npm)
USER claude
WORKDIR /home/claude
RUN curl -fsSL https://claude.ai/install.sh | bash
ENV PATH="/home/claude/.local/bin:/home/claude/.claude/bin:${PATH}"

# Pre-configure auth so the onboarding wizard never appears
RUN mkdir -p /home/claude/.claude && \
    echo '{"hasCompletedOnboarding":true,"installMethod":"native"}' \
      > /home/claude/.claude/.claude.json && \
    ln -sf /home/claude/.claude/.claude.json /home/claude/.claude.json && \
    echo '{"permissions":{"allow":["*"],"deny":[]},"env":{"CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC":"1"}}' \
      > /home/claude/.claude/settings.json

WORKDIR /workspace
CMD ["sleep", "infinity"]

핵심 개념: 비결정성 교훈

이 Dockerfile은 스캐폴드 패턴의 실제 적용입니다. 모호한 "Claude Code용 Dockerfile 만들기" 지시는 매우 다른 결과를 만들어냅니다: npm vs 네이티브, Alpine vs Debian, 루트 vs 비루트, ENTRYPOINT vs CMD. 각 잘못된 선택은 특정 실패를 유발합니다 — 인증 루프, 바이너리 충돌, 권한 오류. 스캐폴드는 모든 동전 던지기를 제거합니다. 그것이 비결정적 출력을 신뢰할 수 있게 만드는 방법입니다.

인증 방법

Claude Code는 인증이 필요합니다. 두 가지 접근 방식이 있으며, 같은 .env에 공존해서는 안 됩니다:

OAuth 토큰 (Claude Max/Pro)

호스트 머신에서 토큰을 생성한 다음 환경 변수로 전달합니다:

터미널 (호스트에서)claude setup-token

결과 토큰을 .env 파일에 추가하세요:

CLAUDE_CODE_OAUTH_TOKEN=sk-ant-oat01-your-token-here

토큰은 docker-compose.yml의 env_file: .env를 통해 컨테이너에 전달됩니다. Docker 이미지에는 절대로 내장되지 않습니다.

API 키

토큰별 청구를 위해 (구독 없이):

ANTHROPIC_API_KEY=sk-ant-api03-your-key-here
주의: 둘 다 설정하지 마세요

ANTHROPIC_API_KEY와 CLAUDE_CODE_OAUTH_TOKEN이 모두 존재하면 API 키가 우선하여 구독을 사용하는 대신 토큰당 청구됩니다. 하나만 선택하세요.

단일 Claude 에이전트 실행

패턴: 컨테이너를 한 번 시작한 다음 각 세션에 docker exec으로 접속합니다. 각 exec는 같은 워크스페이스를 공유하는 독립적인 Claude Code 세션입니다.

Terminal# Build and start the container docker compose up -d claude # Open an interactive Claude Code session docker exec -it claude-myproject claude --dangerously-skip-permissions # Open ANOTHER session in a second terminal pane docker exec -it claude-myproject claude --dangerously-skip-permissions

-v $(pwd):/workspace 볼륨 마운트는 현재 디렉토리를 컨테이너에 매핑합니다. 에이전트가 컨테이너 내부에서 만드는 변경 사항은 호스트 파일 시스템에 즉시 나타납니다.

--dangerously-skip-permissions로 실행

--dangerously-skip-permissions 플래그는 Claude Code에게 모든 권한 프롬프트를 건너뛰도록 지시합니다 — 파일 쓰기, 명령 실행, 네트워크 요청. 에이전트가 묻지 않고 행동합니다.

주의: 이 플래그를 사용해야 할 때

호스트 머신에서 이 플래그는 진정으로 위험합니다 — 에이전트가 어떤 파일도 수정하고 어떤 명령도 실행할 수 있습니다. Docker 컨테이너 내부에서는 위험이 격리됩니다: 에이전트는 컨테이너의 파일 시스템과 마운트된 볼륨에만 영향을 미칠 수 있습니다. 이것이 컨테이너와 --dangerously-skip-permissions가 자연스러운 쌍인 이유입니다: 컨테이너 격리로 자율 작동이 안전해집니다.

완전한 자율성으로 비대화형 작업 실행:

Terminaldocker run -it -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY -v $(pwd):/workspace claude-agent --dangerously-skip-permissions -p "fix all failing tests"

-p 플래그는 대화형 세션을 여는 대신 프롬프트를 직접 전달합니다. --dangerously-skip-permissions와 결합하면 에이전트가 완전히 자율적으로 작동할 수 있습니다: 파일 읽기, 코드 편집, 테스트 실행, 완료될 때까지 반복.

Docker Compose 설정

Compose 파일은 기존 프로젝트 서비스와 함께 claude 서비스를 정의합니다:

# docker-compose.yml
services:
  claude:
    build:
      context: .
      dockerfile: Dockerfile.claude
    container_name: claude-${COMPOSE_PROJECT_NAME:-agents}
    env_file: .env
    volumes:
      - .:/workspace
      - claude-auth:/home/claude/.claude
    working_dir: /workspace
    stdin_open: true
    tty: true

volumes:
  claude-auth:
    name: claude-auth-${COMPOSE_PROJECT_NAME:-default}

주요 세부 사항: claude-auth 볼륨은 컨테이너 재시작 시 인증을 유지합니다. .:/workspace는 프로젝트를 마운트합니다. env_file: .env는 OAuth 토큰을 전달합니다. 컨테이너 이름에 프로젝트 디렉토리 이름이 포함되어 각 저장소가 고유한 컨테이너를 갖습니다.

Terminal# Start the container docker compose up -d claude # Run multiple agent sessions (each in its own terminal pane) docker exec -it claude-myproject claude --dangerously-skip-permissions --model claude-opus-4-6 docker exec -it claude-myproject claude --dangerously-skip-permissions --model claude-opus-4-6
주의: 조정 과제

여러 에이전트가 같은 볼륨 마운트를 공유할 때 동시에 같은 파일에 쓸 수 있습니다. 이는 충돌을 유발할 수 있습니다 — 하나의 에이전트가 다른 에이전트의 변경 사항을 덮어씁니다. 전략: (1) 에이전트를 겹치지 않는 디렉토리에 할당, (2) 각 에이전트가 자체 작업 복사본을 갖도록 git 워크트리(worktree) 사용, (3) 병렬 대신 파이프라인에서 에이전트를 순차적으로 실행.

agent.sh 패턴

컨테이너가 실행 중인지 확인하고 접속하는 얇은 편의 스크립트입니다. 모델은 --model 플래그를 통해 강제됩니다(환경 변수는 모델 선택에 작동하지 않음).

#!/usr/bin/env bash
set -euo pipefail

PROJECT_NAME="${COMPOSE_PROJECT_NAME:-$(basename "$(pwd)")}"
CONTAINER="claude-${PROJECT_NAME}"

# Start container if not running
if ! docker ps --format '{{.Names}}' | grep -q "^${CONTAINER}$"; then
  docker compose up -d claude
  sleep 2
fi

# Interactive or one-shot
if [ $# -gt 0 ]; then
  docker exec -it "$CONTAINER" claude \
    --dangerously-skip-permissions --model claude-opus-4-6 -p "$*"
else
  docker exec -it "$CONTAINER" claude \
    --dangerously-skip-permissions --model claude-opus-4-6
fi

사용법:

터미널./agent.sh # 대화형 세션 ./agent.sh "refactor the auth module" # 원샷 프롬프트

여러 에이전트를 실행하려면 터미널 창을 더 열고 각각 ./agent.sh를 실행합니다. 각 exec는 독립적인 세션이며 모두 같은 워크스페이스를 공유합니다.
