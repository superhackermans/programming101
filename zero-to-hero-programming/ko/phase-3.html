챕터 10 터미널(Terminal)과 셸(Shell) 스크립팅

핵심 개념

Claude Code는 완전히 터미널에서 실행됩니다. 실행하는 모든 명령, 읽는 모든 파일, 실행하는 모든 테스트—이 모두가 여기서 배울 동일한 셸 명령을 통해 이루어집니다. 터미널 능숙함은 AI 보조 개발에서 선택 사항이 아닙니다. 터미널 자체가 인터페이스이기 때문입니다.

Ch 03에서 Python을 실행하기 위해 터미널을 사용했습니다. 이제 강력한 도구로서 배울 것입니다. 아래의 ~15개 명령은 전문가들이 매일 사용하는 것들입니다. 나머지는 필요할 때 찾아볼 수 있습니다.

셸 스크립트

셸 스크립트는 순서대로 실행되는 명령어 파일입니다. 모든 스크립트는 OS에게 어떤 인터프리터를 사용할지 알려주는 shebang 줄로 시작합니다:

#!/bin/bash
echo "Running tests..."
python3 -m pytest
echo "Done."

run_tests.sh로 저장하고, chmod +x run_tests.sh으로 실행 가능하게 만든 후, ./run_tests.sh로 실행합니다.

파이프: 소형 도구 조합하기

| 연산자는 한 명령의 출력을 다른 명령의 입력으로 보냅니다. >는 출력을 파일에 씁니다.

파이프는 소형 도구들을 강력한 워크플로로 조합합니다. 각 단계가 다음을 위해 데이터를 필터링하거나 변환합니다.

환경 변수와 PATH

환경 변수는 시스템 전체 설정입니다. PATH는 shell에게 명령어를 찾을 위치를 알려줍니다. command not found 오류가 발생하면, 해당 프로그램이 PATH에 나열된 디렉토리에 없다는 의미입니다.

export API_KEY="sk-abc123"   # 변수 설정
echo $PATH                    # PATH 확인

SSH 기초

SSH (Secure Shell)를 사용하면 터미널을 통해 원격 머신에 연결할 수 있습니다. 코드를 배포하거나 클라우드 서버에 액세스할 때 접하게 됩니다. ssh-keygen -t ed25519로 키 쌍을 생성하고, 공개 키(~/.ssh/id_ed25519.pub)를 원격 서비스에 추가합니다(Ch 12에서 특정 서비스와 함께 설정합니다). 연결: ssh user@hostname.

tmux: 터미널 멀티플렉서

개념: 터미널 멀티플렉서를 사용하면 단일 터미널 창 안에서 여러 영구적인 셸 세션을 실행할 수 있습니다. 창을 분할하고, 탭 사이를 전환하고, 완전히 분리(detach)할 수 있습니다—모든 것이 백그라운드에서 계속 실행됩니다. 이것은 여러 프로세스를 동시에 모니터링해야 하는 워크플로에 필수적이며, 여러 AI 에이전트를 조율할 때 핵심적입니다.

참고

tmux (모든 플랫폼, 무료). 설치: brew install tmux (macOS) 또는 sudo apt install tmux (Linux). Phase 5의 멀티 에이전트 워크플로를 위한 표준 터미널 멀티플렉서입니다.

핵심 개념

Phase 5에서는 여러 Claude Code 에이전트를 동시에 실행합니다. 각각은 자체 터미널 세션이 필요합니다. tmux 없이는 여러 터미널 창이 필요하고, 노트북을 닫으면 모두 종료됩니다. tmux를 사용하면 세션이 백그라운드에서 지속됩니다—연결을 끊고 다시 연결해도 에이전트는 계속 실행됩니다. tmux는 백그라운드 에이전트를 가능하게 하는 인프라입니다.

시작하기

세션 내의 모든 tmux 명령은 prefix 키로 시작합니다: Ctrl + B. prefix를 누르고 뗀 다음 다음 키를 누릅니다. 이것이 가장 흔한 걸림돌입니다—모든 것을 한 번에 누르는 것이 아니라 두 단계 조합을 누르는 것입니다.

Terminal# 새로운 이름 있는 세션 시작 tmux new -s taskforge # 이제 tmux 내부에 있습니다. 하단의 초록 표시줄이 확인해 줍니다. # 분리 (세션을 백그라운드에서 실행 유지) # 누름: Ctrl+B, 그 다음 d # 실행 중인 세션 목록 보기 tmux ls # 세션에 다시 연결 tmux attach -t taskforge # 완료 후 세션 종료 tmux kill-session -t taskforge

권장 tmux 설정

이 설정으로 ~/.tmux.conf를 만들어 tmux를 더 사용하기 편하게 만드세요:

# ~/.tmux.conf

# Ctrl+B 대신 Ctrl+A를 prefix로 사용 (누르기 쉬움)
unbind C-b
set -g prefix C-a
bind C-a send-prefix

# | 과 - 로 pane 분할 (기억하기 쉬움)
bind | split-window -h -c "#{pane_current_path}"
bind - split-window -v -c "#{pane_current_path}"

# 새 window는 현재 디렉토리에서 열림
bind c new-window -c "#{pane_current_path}"

# Alt+화살표로 pane 탐색 (prefix 없이)
bind -n M-Left select-pane -L
bind -n M-Right select-pane -R
bind -n M-Up select-pane -U
bind -n M-Down select-pane -D

# 마우스 지원 활성화 (클릭으로 pane 선택, 스크롤)
set -g mouse on

# window 번호를 1부터 시작 (0이 아닌)
set -g base-index 1
setw -g pane-base-index 1

# window가 닫힐 때 번호 재정렬
set -g renumber-windows on

# 스크롤백 버퍼 증가
set -g history-limit 50000

# 이스케이프 지연 줄이기 (vim/nano 사용자를 위해 더 빠르게)
set -sg escape-time 10

# 상태 표시줄 스타일링
set -g status-style bg=black,fg=white
set -g status-left "[#S] "
set -g status-right "%H:%M"

저장 후, tmux source-file ~/.tmux.conf로 다시 로드합니다(또는 새 세션 시작).

주의

이 설정은 prefix를 Ctrl + B에서 Ctrl + A로 변경하고, %와 " 대신 |와 -로 분할합니다. 위의 모든 단축키 표는 기본 바인딩을 사용합니다. 이 설정을 사용한다면, 어디서나 Ctrl + B 대신 Ctrl + A를 사용하고, 분할에는 |/-를 사용하세요.

실제 tmux 워크플로

일반적인 멀티 에이전트 개발 세션은 이렇게 보입니다:

Terminal# 프로젝트를 위한 세션 생성 tmux new -s taskforge # Window 1 (기본): Claude Code 메인 세션 claude # 테스트를 위한 Window 2 생성 (prefix + c) # 이름을 "tests"로 지정 (prefix + ,) python3 -m pytest --watch # 서버를 위한 Window 3 생성 (prefix + c) # 이름을 "server"로 지정 (prefix + ,) python3 -m flask run # git log를 보기 위해 Window 1 분할 (prefix + %) # 오른쪽 pane에서: watch -n 2 git log --oneline -10 # 분리하고 점심 (prefix + d) # 모든 것이 계속 실행됩니다. # 돌아와서 다시 연결: tmux attach -t taskforge
직접 해보기

목표: 3개의 pane이 있는 세션 만들기. 단계: (1) tmux new -s practice (2) Prefix + %로 수직 분할 (3) Prefix + "로 오른쪽 pane을 수평으로 분할 (4) Prefix + 화살표 키로 3개 pane 사이를 탐색 (5) Prefix + z로 한 pane을 확대하고, z를 다시 눌러 원래대로 (6) Prefix + d로 분리 (7) tmux attach -t practice로 다시 연결. 확인: 다시 연결한 후에도 3개의 pane이 내용과 함께 그대로 있습니다.

챕터 11 Git으로 버전 관리

핵심 개념

Git은 선택 사항이 아닙니다. Claude Code는 브랜치를 만들고, 커밋을 하고, 프로젝트 기록을 읽어 코드베이스를 이해합니다. Git 없이는 Claude Code를 효과적으로 사용할 수 없습니다—그리고 Git을 이해하지 못하면 Claude Code가 코드로 하는 일을 감독할 수 없습니다.

주의

파일을 리팩토링하는 데 3시간을 씁니다. 무언가가 망가집니다. 작동하던 버전이 어땠는지 기억할 수 없습니다. 수동으로 되돌리려 하다가 더 망가뜨립니다. Git이 있으면, 명령 하나(git checkout -- file.py)로 마지막 작동 버전을 복원합니다. 없으면 모든 편집에 도박을 하는 셈입니다.

참고

Git은 범용 버전 관리 시스템입니다. 거의 모든 전문 프로젝트가 이것을 사용합니다. 설치: brew install git (macOS), apt install git (Linux), 또는 git-scm.com (Windows).

Git은 코드의 모든 변경을 추적합니다. 프로젝트를 위한 타임머신이라고 생각하세요—이전의 어떤 상태로도 돌아갈 수 있습니다. 커밋(commit)은 특정 시점의 코드 스냅샷입니다. 브랜치(branch)는 평행 타임라인과 같습니다. 메인 타임라인을 안정적으로 유지하면서 별도 타임라인에서 새로운 기능을 작업할 수 있습니다. 실험이 실패하면 그 타임라인을 버립니다. 성공하면 다시 병합(merge)합니다. 이것이 AI 에이전트가 이미 작동하는 것을 망가뜨리지 않고 코드에 작업하는 방법입니다.

주의

Claude Code는 git 기록을 읽고, 커밋을 만들고, 브랜치를 생성하고, 충돌을 해결합니다. git으로 무엇을 하는지 이해하지 못하면 감독할 수 없습니다.

핵심 명령어

챕터 12 Git 원격과 GitHub

핵심 개념

모든 전문 프로젝트는 원격 저장소에 있습니다. 이후 phase의 Claude Code 워크플로는 pull request, 원격 브랜치, CI 파이프라인을 포함합니다—이 모두는 원격 Git을 이해해야 합니다.

원격이란 무엇인가?

지금까지 모든 Git 작업은 로컬이었습니다—커밋, 브랜치, 병합이 오직 여러분의 컴퓨터에만 존재했습니다. 원격은 다른 서버, 일반적으로 GitHub에 호스팅된 저장소의 복사본입니다. 로컬 저장소는 여러분이 작업하는 책상이고, 원격 저장소는 팀 전체가 공유하는 화이트보드입니다. 변경 내용은 명시적으로 push하거나 pull해야만 동기화됩니다.

push는 로컬 커밋을 원격으로 올리고, pull은 원격의 새 커밋을 로컬로 내려받습니다. 이 두 작업을 명시적으로 실행해야 하며, 자동으로 동기화되지는 않습니다.

GitHub 계정 설정

아직 GitHub 계정이 없다면:

  1. github.com으로 가서 Sign up을 클릭합니다.
  2. 사용자 이름을 선택하고, 이메일을 입력하고, 비밀번호를 설정합니다.
  3. 로그인 후 + 아이콘(오른쪽 상단) → New repository를 클릭합니다.
  4. taskforge로 이름 짓고, 공개로 두고, README로 초기화하지 않습니다(이미 로컬에 있습니다).
  5. GitHub이 로컬 저장소를 연결하는 명령을 보여줍니다. 바로 다음 단계에서 그 명령을 사용합니다.

로컬을 원격에 연결

GitHub에 빈 저장소를 만든 후, 로컬 저장소를 연결합니다:

git remote add origin https://github.com/YOUR-USERNAME/taskforge.git

origin은 주 원격의 관례적인 이름입니다. 뭐든 이름을 붙일 수 있지만 origin이 보편적입니다. 연결 확인:

git remote -v

두 줄이 표시되어야 합니다—fetch와 push 각각 하나씩—모두 GitHub URL을 가리킵니다.

Push와 Pull

Push는 로컬 커밋을 원격으로 보냅니다. Pull은 원격의 커밋을 가져와 로컬 브랜치에 병합합니다.

# 커밋을 GitHub로 보내기
git push origin main

# GitHub에서 새 커밋 가져오기
git pull origin main

새 브랜치를 처음 push할 때 -u를 사용하여 추적을 설정합니다:

git push -u origin main

그 후, git push와 git pull은 매번 원격과 브랜치를 지정하지 않아도 작동합니다.

주의

! [rejected] main -> main (fetch first)가 표시되면, 원격에 로컬에 없는 커밋이 있다는 뜻입니다. 수정 방법:

git pull origin main    # 원격 변경사항 가져와 병합
git push origin main    # 이제 push 성공

pull이 병합 충돌을 발생시키면, Ch 11에서 배운 것과 같은 방식으로 해결합니다: 파일을 편집하고, 충돌 마커를 제거하고, git add하고, git commit합니다.

클로닝

git clone은 원격에서 여러분의 컴퓨터로 전체 저장소를—모든 파일, 모든 기록, 모든 브랜치를—다운로드합니다:

git clone https://github.com/someone/their-project.git

이것은 완전한 Git 저장소가 내부에 있는 their-project/라는 새 디렉토리를 만듭니다. 원격 origin이 이미 설정되어, 클론한 URL을 가리킵니다. 이것이 다른 사람의 프로젝트를 여러분의 컴퓨터로 가져오는 방법입니다.

원격 브랜치

브랜치는 로컬 컴퓨터와 원격에 독립적으로 존재합니다. 로컬 브랜치를 GitHub에 push하려면:

git push origin feature-branch

존재하지 않으면 GitHub에 브랜치를 만듭니다. 추적을 설정하려면 (Git이 로컬 브랜치가 원격 브랜치에 해당한다는 것을 알도록):

git push -u origin feature-branch

추적이 설정되면, git push와 git pull이 자동으로 어떤 원격 브랜치를 사용할지 알게 됩니다. 원격 브랜치를 포함한 모든 브랜치를 보려면:

git branch -a

.gitignore

.gitignore 파일은 Git에게 어떤 파일과 디렉토리를 절대 추적하지 않을지 알려줍니다. 생성된 파일, 비밀, 환경별 아티팩트를 제외하여 저장소를 깨끗하게 유지합니다.

# 바이트 컴파일된 Python 파일
*.pyc
__pycache__/

# 가상 환경
.venv/
venv/

# 환경 변수 / 비밀
.env

# IDE 설정
.vscode/
.idea/

# OS 파일
.DS_Store

# Node (해당하는 경우)
node_modules/

패턴 문법:

.gitignore 패턴 문법 및 매칭 규칙
패턴매칭 대상
*.pyc어느 디렉토리에서나 .pyc로 끝나는 모든 파일
__pycache__/__pycache__라는 이름의 모든 디렉토리와 그 안의 모든 것

.gitignore 없이는 비밀(.env API 키가 있는 파일)을 커밋하거나, 수백 메가바이트의 의존성(node_modules/, .venv/)으로 저장소를 부풀리거나, 모든 diff를 어지럽히는 OS 쓰레기(.DS_Store)를 push할 위험이 있습니다. 첫 번째 커밋 전에 .gitignore를 추가하세요.

풀 리퀘스트(Pull Request)

풀 리퀘스트(pull request, PR)는 하나의 브랜치를 다른 브랜치로 병합하는 제안입니다. 코드 리뷰의 핵심입니다: 변경 사항을 보여주면, 팀원들이 검토하고, 승인 후에만 코드가 main에 병합됩니다. 워크플로:

  1. 브랜치: 로컬에서 피처 브랜치 생성. git switch -c add-delete-feature
  2. 작업: 변경하고 커밋합니다.
  3. Push: git push -u origin add-delete-feature
  4. PR 열기: GitHub으로 이동합니다. 배너가 표시됩니다: "Compare & pull request." 클릭합니다. 무엇을 변경했고 왜 변경했는지 설명을 작성합니다.
  5. 검토: 팀원들이 코드를 읽고, 댓글을 달고, 변경을 요청합니다.
  6. 병합: 승인되면 GitHub에서 "Merge pull request"를 클릭합니다.

솔로 개발자도 PR을 사용합니다—무엇이 변경되었고 왜 변경되었는지 깔끔한 기록을 만들고, 이후 phase에서 Claude Code가 검토를 위해 작업을 제출하는 방식입니다.

SSH 키

GitHub은 두 가지 연결 방법을 지원합니다: HTTPS와 SSH. HTTPS는 매번 push할 때 자격증명을 요청합니다. SSH는 키 쌍을 사용하여 자동으로 인증합니다.

SSH 키 생성:

ssh-keygen -t ed25519 -C "your-email@example.com"

Enter를 눌러 기본 위치를 수락합니다. 암호를 설정하거나 비워둡니다. 그런 다음 공개 키를 GitHub에 추가합니다:

  1. 공개 키 복사: cat ~/.ssh/id_ed25519.pub
  2. GitHub → Settings → SSH and GPG keys → New SSH key
  3. 키를 붙여넣고 저장합니다.

연결 테스트:

ssh -T git@github.com

다음이 표시되어야 합니다: Hi username! You've successfully authenticated. 이제 HTTPS 대신 SSH URL(git@github.com:user/repo.git)을 사용할 수 있습니다.

Push는 커밋을 로컬에서 GitHub로 보냅니다. Pull은 커밋을 GitHub에서 로컬로 가져옵니다. Clone은 원격 저장소에서 새로운 로컬 복사본을 만듭니다.

흔한 오해

주의

아닙니다. git push는 원격에 없는 커밋만 보냅니다. 100개의 커밋을 만들었고 원격에 98개가 있다면, push는 새로운 2개만 보냅니다. Git은 효율적입니다—최소한의 필요한 데이터만 전송합니다.

주의

아닙니다. 컴퓨터에서 파일을 삭제해도 원격에는 아무런 변화가 없습니다. 삭제는 git add로 스테이징하고, git commit하고, git push한 후에야 GitHub에 반영됩니다. 그 전까지 파일은 그대로 GitHub에 존재합니다.

TaskForge 연결

TaskForge를 GitHub 저장소에 push합니다. __pycache__/, .venv/, *.pyc, .env를 제외하는 .gitignore를 만듭니다. 피처 브랜치를 만들고 GitHub에 push한 후 pull request를 엽니다—혼자서 검토하더라도. 이것은 Phase 4에서 Claude Code가 브랜치와 PR을 만들어 검토를 요청할 때 사용하게 될 동일한 워크플로입니다.

작동하지 않는 경우: (1) push가 거부되면 먼저 git pull --rebase origin main을 시도합니다. (2) SSH가 실패하면 ssh -T git@github.com으로 키 설정을 확인합니다. (3) Fork 버튼을 찾을 수 없으면 GitHub에 로그인되어 있고 다른 사람의 공개 저장소를 보고 있는지 확인합니다.

AI 에이전트의 Diff 검토

AI 에이전트에게 "delete_task 함수에 오류 처리를 추가하세요"라고 요청했습니다. 아래는 에이전트가 생성한 diff입니다. 주의 깊게 읽은 다음, 세 가지 키를 가진 딕셔너리를 반환하는 evaluate_diff() 함수를 작성하세요: "correct_changes" (유효한 오류 처리 추가가 있는 줄 번호 목록), "unrelated_change" (요청하지 않은 변경이 있는 줄 번호), "missing" (에이전트가 처리하는 것을 잊은 것을 설명하는 문자열).

+가 있는 줄은 추가입니다. isinstance 확인은 관련 있는 오류 처리입니다. 하지만 list_tasks 변경은 어떨까요?

에이전트는 delete_task에 오류 처리를 추가하도록 요청받았습니다. list_tasks를 변경하는 것은 범위 초과입니다—요청하지 않은 변경입니다.

tasks가 None이거나 리스트가 아니면 어떻게 됩니까? 에이전트는 task_id에 대한 타입 체크를 추가했지만 tasks 자체를 검증하는 것을 잊었습니다.
